# 지원 문의에 답변하는 방법

이 페이지는 Langfuse의 인간 지원 엔지니어와 AI 지원 에이전트가 함께 사용하는 플레이북입니다. 아래 의사결정 트리의 각 분기는 반복적으로 발생하는 질문 패턴을 다룹니다: 어떻게 트리아지할지, 무엇을 확인할지, 참고해 조정할 수 있는 답변 템플릿, 그리고 언제 에스컬레이션할지를 포함합니다.

이 트리는 이메일, Slack, MS Teams, 인앱 채팅 위젯, GitHub Discussions 전반에 걸친 약 1,500건의 종결된 Pylon 티켓을 분석하여 도출되었습니다. 패턴과 답변 문구는 팀이 실제로 해결한 사례에서 가져온 것입니다.

<Callout type="info">

대부분의 티켓에 답변할 때 사용할 도구:

- **Pylon**: 기본 인박스, 티켓 메타데이터, 고객 등급, 내부 노트
- **Metabase**: 사용량 통계, 수집 볼륨, ClickHouse 쿼리
- **PostHog**: 제품 분석, 사용자 활동, 세션 리플레이
- **Stripe**: 구독, 인보이스, 청구, 환불 이력
- **Impersonation View**: 고객이 보는 것과 동일한 Langfuse UI 확인
- **Google Forms**: 스타트업 할인 신청(폼이 신뢰할 수 있는 원본 데이터)
- **DataDog**: 수집 큐 깊이, 워커 상태, ClickHouse 지연 시간
- **status.langfuse.com**: 공개 장애 타임라인

</Callout>

## 답변 전 확인 사항 (프리플라이트)

답변 창을 열기 전에 아래 네 가지를 순서대로 수행하세요. 이 페이지의 나머지 대부분은 이 작업을 이미 완료했다고 가정합니다.

1. **고객과 등급을 확인합니다.** Pylon 사이드바에 조직명, 플랜 등급(Hobby / Core / Pro / Pro + Teams Add-on / Enterprise / 셀프 호스팅 EE), 데이터 리전, 계약 관련 메모가 표시됩니다. 등급에 따라 SLA와 에스컬레이션의 적극성이 달라집니다.
2. **고객의 환경을 파악합니다.** Langfuse Cloud(어느 리전인지: EU `cloud.langfuse.com`, US `us.cloud.langfuse.com`, HIPAA `hipaa.cloud.langfuse.com`, Japan `jp.cloud.langfuse.com`)인지 셀프 호스팅인지 확인합니다. 셀프 호스팅이라면 어떤 버전인지도 확인합니다. 같은 증상이라도 Cloud와 셀프 호스팅에서는 원인이 다른 경우가 많습니다.
3. **`status.langfuse.com`과 DataDog을 확인합니다.** 고객이 오류나 지연을 보고한다면, 디버깅에 앞서 알려진 진행 중인 장애가 아닌지 먼저 배제하세요.
4. **지난 7일간 같은 증상이 없었는지 Pylon에서 검색합니다.** 지금 다른 세 명의 고객이 같은 문제를 보고하고 있다면 이는 장애(incident) 상황일 가능성이 높으므로, 개별적으로 답변하기보다 엔지니어링 팀으로 에스컬레이션하세요.

네 가지를 모두 확인했다면 아래 트리를 따라가세요.

## 의사결정 트리

아래 제목들은 최상위 질문 카테고리입니다. 각 항목을 클릭하면 구체적인 하위 질문으로 들어갈 수 있으며, 각 하위 질문에는 트리아지 단계, 답변 템플릿, 에스컬레이션 규칙이 포함되어 있습니다. 고객 메시지의 키워드로 바로 이동하려면 Cmd/Ctrl-F를 사용하세요.

### 1. 계정, 로그인, 접근 권한

<details>

<summary>"로그인이 안 돼요" / "잘못된 자격 증명" / "계정을 찾을 수 없음"</summary>

**가장 흔한 원인은 잘못된 데이터 리전입니다.** 사용자가 한 리전에서 가입한 뒤 다른 리전으로 로그인을 시도하는 경우입니다. 비밀번호 재설정 흐름에서 "계정이 없습니다"라는 메시지가 뜨는 것은 계정이 실제로 존재하지 않아서가 아니라, _해당 사용자가 보고 있는 리전에는_ 존재하지 않기 때문입니다.

**트리아지 단계:**

1. 고객이 어느 리전에서 가입했는지 물어보거나(또는 이메일 서명/도메인에서 확인), 확인합니다. 모른다면 다음 리전들을 모두 시도해보게 하세요: EU `cloud.langfuse.com`, US `us.cloud.langfuse.com`, HIPAA `hipaa.cloud.langfuse.com`, Japan `jp.cloud.langfuse.com`.
2. 원래 SSO(Google, GitHub, Azure AD)로 로그인했다면 이메일+비밀번호 로그인은 "Please sign in with the identity provider that is linked to your account."라는 오류와 함께 실패합니다. SSO 제공자로 시도하게 하세요.
3. 여전히 계정을 찾을 수 없다면 Impersonation View에서 이메일로 조회하여 어느 리전에 계정이 있는지 확인하세요.
4. 리전이 맞고 SSO도 확인되었다면, 해당 이메일이 이메일 억제 목록(suppression list)에 있는지 확인하세요(아래 "비밀번호 재설정 이메일이 오지 않음" 참조).

**답변 템플릿:**

```text
안녕하세요 {name}님,

불편을 드려 죄송합니다. 가장 흔한 원인은 잘못된 데이터 리전에서 로그인을 시도하는 경우입니다. 저희는 네 개의 독립된 리전을 운영하며, 한 리전의 계정은 다른 리전에서 보이지 않습니다:

- EU:    https://cloud.langfuse.com
- US:    https://us.cloud.langfuse.com
- HIPAA: https://hipaa.cloud.langfuse.com
- Japan: https://jp.cloud.langfuse.com

참고: https://langfuse.com/security/data-regions

또 다른 가능성은, 원래 Google / GitHub / Azure AD SSO로 가입하셨다면 이메일+비밀번호 로그인은 실패합니다. 대신 SSO 제공자 버튼을 클릭해보세요.

어느 리전과 로그인 방식을 사용하셨는지 확인해 주시면 거기서부터 살펴보겠습니다.

감사합니다,
{you}
```

**에스컬레이션 시점:** 고객이 리전과 제공자를 모두 확인했는데도 로그인이 안 될 경우 → 이메일과 조직 ID와 함께 엔지니어링 팀에 문의하세요. 계정 상태를 직접 조회해야 할 수 있습니다.

관련 FAQ: [/faq/all/cannot-see-organization](/faq/all/cannot-see-organization), [/faq/all/forgot-password](/faq/all/forgot-password), [/faq/all/where-is-my-project](/faq/all/where-is-my-project).

</details>

<details>

<summary>"비밀번호 재설정 이메일이 오지 않아요"</summary>

**일반적인 원인은 이메일 억제 목록(suppression list)입니다.** 이전에 해당 주소로 보낸 이메일이 반송되었거나 수신자가 스팸으로 표시하면, 이메일 제공업체가 해당 주소로의 발송을 중단합니다. 이는 도메인 전체가 아니라 해당 사용자 한 명에게만 영향을 줍니다.

**트리아지 단계:**

1. 먼저 고객이 올바른 데이터 리전을 사용 중인지 확인합니다(위의 "로그인이 안 돼요" 참조). 잘못된 리전이라면 그곳에는 계정이 존재하지 않으므로 이메일도 오지 않습니다.
2. 스팸함을 확인하도록 안내한 뒤, 엔지니어링 팀에 요청하여 해당 이메일을 억제 목록에서 제거합니다.
3. 차단이 해제되면 비밀번호 재설정을 다시 시도하도록 안내합니다.

**답변 템플릿:**

```text
안녕하세요 {name}님,

확인 감사합니다. 먼저 간단히 확인할게요. 원래 가입하신 데이터 리전이 어디인가요(cloud.langfuse.com EU vs. us.cloud.langfuse.com US vs. hipaa.cloud.langfuse.com HIPAA vs. jp.cloud.langfuse.com Japan)?

올바른 리전인데도 이메일이 계속 오지 않는다면, 저희 이메일 제공업체가 해당 주소를 억제 목록에 올렸을 수 있습니다(이전 반송이나 스팸 표시 이후 발생합니다). 저희 쪽에서 차단을 해제해드릴 테니, 약 10분 후 비밀번호 재설정을 다시 시도해 보시고 결과를 알려주세요.

감사합니다,
{you}
```

**에스컬레이션 시점:** 억제 목록이 원인이 아니고 고객이 어떤 Langfuse 이메일도 정말로 받을 수 없는 경우 → 엔지니어링 팀. 보안상의 이유로 저희가 비밀번호를 직접 재설정해드릴 수는 없다는 점에 유의하세요. 엔지니어링 팀은 계정 상태를 확인해줄 수 있지만, 재설정 자체는 사용자가 직접 완료해야 합니다.

</details>

<details>

<summary>SSO 설정 (Okta / Azure AD / Entra / Google Workspace)</summary>

Cloud에서 SSO는 Pro + Teams Add-on 이상 플랜에 포함되며, 설정은 화이트글러브 방식으로 진행됩니다(지원팀이 자격 증명을 수집하고 엔지니어링 팀이 적용). 셀프 호스팅(OSS 및 EE) 고객은 환경 변수를 통해 직접 SSO를 설정합니다.

**트리아지 단계:**

1. 고객이 SSO가 포함된 플랜 등급인지 확인합니다(Cloud에서는 Pro + Teams Add-on 이상, 셀프 호스팅 OSS에는 기본 포함이며 SCIM / Org Management API만 EE 전용입니다). 해당하지 않으면 영업팀으로 안내하고, 할인을 약속하지 마세요.
2. 다음 네 가지 정보를 수집합니다: 인스턴스 URL, 발급자(issuer) URL, 클라이언트 ID, 클라이언트 시크릿.
3. 고객에게 시크릿을 비밀번호 관리자 공유 링크(1Password 공유 링크, Bitwarden Send 등)를 통해 전달하도록 권장합니다. 평문 이메일로는 시크릿을 받지 마세요.
4. 수집한 정보를 엔지니어링 팀에 전달하여 적용합니다.

**답변 템플릿:**

```text
안녕하세요 {name}님,

SSO 설정을 도와드리겠습니다. 다음 정보가 필요합니다:

- 인스턴스 URL (어느 Langfuse 리전인지: cloud.langfuse.com, us.cloud.langfuse.com, hipaa.cloud.langfuse.com, jp.cloud.langfuse.com, 또는 셀프 호스팅 URL)
- 발급자(Issuer) URL (예: https://example.okta.com)
- 클라이언트 ID
- 클라이언트 시크릿

클라이언트 시크릿은 평문 이메일이 아닌 비밀번호 관리자 공유 링크(1Password / Bitwarden 등)로 전달해 주세요. 네 가지 정보를 모두 받으면 영업일 기준 1일 이내에 팀에서 적용해드립니다.

IdP 쪽에서 궁금한 점이 있으면 언제든 알려주세요.

감사합니다,
{you}
```

**에스컬레이션 시점:** 고객이 SCIM, 커스텀 클레임 매핑, 또는 비표준 IdP를 요청하는 경우 엔지니어링 팀의 검토가 필요합니다.

<details>

<summary>2FA 복구 / 인증 앱 분실 / 백업 코드</summary>

2FA 복구는 신뢰도가 높은 작업으로 취급합니다. 고객은 소유권을 증명해야 합니다.

**트리아지 단계:**

1. 보조 신호를 통해 고객의 신원을 확인합니다: 이메일이 청구 기록과 일치하는지, 업무용 이메일 도메인이 조직 도메인과 일치하는지, 또는 이미 신뢰하는 Slack Connect 채널에 있는지 확인합니다.
2. 신뢰가 확립되면 엔지니어링 팀이 계정의 2FA를 비활성화하여 사용자가 다시 등록할 수 있게 합니다. 직접 처리하지 마세요.
3. 고객이 복구용 이메일에도 접근할 수 없다면 조직 소유자(org owner)가 나서야 합니다. 조직 소유자도 잠겨 있다면 전체 맥락과 함께 엔지니어링 팀에 에스컬레이션하세요. 이는 드문 케이스로 사안별로 처리합니다.

**답변 템플릿:**

```text
안녕하세요 {name}님,

2FA 복구를 위해서는 MFA를 비활성화하기 전에 소유권 확인이 필요합니다. 가장 빠른 방법은 다음과 같습니다:

1. 이 문제가 발생한 조직/프로젝트를 확인해 주세요.
2. 계정에 연결된 이메일이 현재도 사용 중인 이메일인지 확인해 주세요.

확인이 완료되면 다음 로그인 시 다시 등록할 수 있도록 2FA를 비활성화해 드리겠습니다. 복구용 이메일에도 접근할 수 없으시다면, 같은 조직의 다른 인증된 주소로 회신하시거나 조직 소유자에게 직접 연락을 요청해 주세요.

감사합니다,
{you}
```

관련 FAQ: [/faq/all/enforcing-2fa](/faq/all/enforcing-2fa).

</details>

<details>

<summary>"조직/프로젝트가 보이지 않아요" (RBAC, 뷰어 권한, 초대)</summary>

보통 다음 중 하나입니다: 잘못된 리전(위 참조), 사용자가 같은 이메일로 다른 조직에 초대됨, 초대한 관리자가 사용자를 `VIEWER`로 설정하여 관리 화면이 숨겨짐, 또는 SCIM/SSO 그룹 매핑이 적용되지 않음.

**트리아지 단계:**

1. 리전 확인(이 섹션 상단 참조).
2. Impersonation View에서 사용자를 조회하여 어느 조직에 속해 있는지 확인합니다.
3. 역할을 확인합니다: VIEWER, MEMBER, ADMIN, OWNER. 더 높은 권한이 필요하다면 조직의 OWNER가 변경해야 하며, 조직 소유자의 승인 없이는 고객을 대신해 역할을 변경하지 않습니다.
4. 셀프 호스팅 EE의 경우: 내장된 "인스턴스 관리자/슈퍼유저" 역할은 없습니다. 프로젝트 전반에 대한 관리 권한을 부여하려면 [Instance Management API](https://langfuse.com/self-hosting/administration/instance-management-api)를 사용하여 관리자 사용자를 모든 조직에 일회성으로 초대하는 스크립트를 작성하세요.

**답변 템플릿 (cloud):**

```text
안녕하세요 {name}님,

먼저 간단히 확인해 주세요. 초대받으신 데이터 리전과 동일한 곳에 로그인되어 있나요(EU cloud.langfuse.com vs. US us.cloud.langfuse.com vs. HIPAA hipaa.cloud.langfuse.com vs. Japan jp.cloud.langfuse.com)?

맞다면, 조직 관리자에게 (a) 사용자님의 이메일이 올바른 조직에 초대되어 있는지, (b) 최소 MEMBER 역할을 가지고 있는지 확인을 요청해 주실 수 있을까요? 소유자는 Organization Settings → Members에서 확인할 수 있습니다.

감사합니다,
{you}
```

이는 가장 민감한 청구 관련 질문입니다. 공감을 표하고 *사실*에 근거해 답변하며, 원인을 추측하지 마세요.

<summary>ClickHouse: 대체 백엔드, 사이징, 마이그레이션</summary>

**절대 원칙:** ClickHouse가 유일하게 지원되는 OLAP 백엔드입니다. Elasticsearch, BigQuery 등을 대체재로 지원하지 않습니다. 이에 대해 문의하는 고객은 기능 요청 채널로 안내하고, 지원을 약속하지 마세요.

**흔한 ClickHouse 관련 질문의 트리아지 단계:**

- **"\<대체재\>를 사용할 수 있나요?"** 불가능합니다. 기능 요청 아이디어나 기존 GitHub Discussion(있는 경우)으로 안내하세요.
- **"마이그레이션 실패 / 마이그레이션 데드락"** → [/faq/all/self-hosting-clickhouse-handling-failed-migrations](/faq/all/self-hosting-clickhouse-handling-failed-migrations) 참조. 큰 버전 점프의 경우, 마이그레이션 중간에 컨테이너가 종료되지 않도록 readiness/liveness 프로브 대기 시간을 일시적으로 늘리고 마이그레이션 중 web 레플리카를 1개로 줄이도록 안내하세요.
- **"웹/API를 우회하는 직접 DB 수집이 가능한가요?"** 지원되지 않습니다. web/worker 계층이 유일하게 보장되는 인터페이스입니다. 지금은 동작하더라도 스키마는 어떤 마이너 릴리스에서든 변경될 수 있습니다.
- **디스크 사용량이 너무 높음** → [/faq/all/reduce-clickhouse-disk-size](/faq/all/reduce-clickhouse-disk-size).

<summary>Redis / BullMQ / 큐 / Valkey / Elasticache</summary>

**트리아지 단계:**

1. Redis에 접근 가능한지 확인합니다: `redis-cli -h $REDIS_HOST ping`. Redis 7 이상 또는 호환 버전(Valkey, ElastiCache)이 필요합니다.

<summary>S3 / 블롭 스토리지 / 미디어 업로드 / 이벤트 내보내기</summary>

Langfuse는 원시 이벤트 업로드와 미디어에 S3 호환 스토리지를 사용합니다. 여기서 발생하는 문제는 보통 수집 실패(이벤트는 수락되었지만 처리되지 않음) 또는 "블롭 스토리지 내보내기 실패" 이메일로 나타납니다.

**트리아지 단계:**

1. `LANGFUSE_S3_EVENT_UPLOAD_*` 환경 변수가 설정되어 있고 버킷이 존재하는지 확인합니다.

3. "블롭 스토리지 내보내기 실패" 알림의 경우, 버킷 정책과 라이프사이클 규칙이 최근에 변경되지 않았는지 확인하세요.
4. 미디어 업로드의 경우 `LANGFUSE_S3_MEDIA_UPLOAD_*`도 설정하세요.

관련 FAQ: [/faq/all/self-hosting-missing-events-after-ingestion](/faq/all/self-hosting-missing-events-after-ingestion).

<details>

<summary>Langfuse 버전 간 업그레이드 (셀프 호스팅)</summary>

**트리아지 단계:**

1. 현재 버전(`docker images | grep langfuse`, 또는 Helm의 `appVersion`)과 목표 버전을 확인합니다.
2. 각 중간 메이저 버전의 업그레이드 노트를 따라갑니다. 대부분의 v3.x → v3.x 업그레이드는 매끄럽게 진행됩니다. v2 → v3 및 v3 → v4는 마이그레이션 가이드를 따라야 합니다.
3. 아주 큰 버전 점프(예: v3.132 → v3.175)의 경우 마이그레이션에 몇 분 걸릴 수 있습니다. K8s readiness/liveness 프로브 대기 시간을 일시적으로 늘리고, **마이그레이션 중에는 web 레플리카를 1개로 스케일 다운**하여 Prisma/Postgres 마이그레이션 데드락을 방지하세요.
4. 고객에게 스테이징 환경이 있다면 먼저 테스트하도록 안내하세요.

**답변 템플릿:**

```text
안녕하세요 {name}님,

그 정도로 큰 버전 점프에서는 마이그레이션 소요 시간이 가장 큰 리스크입니다. 업그레이드 전 다음 두 가지를 해주세요:

1. web 컨테이너의 readiness/liveness 프로브 initial-delay와 failure-threshold를 일시적으로 늘려서 마이그레이션 도중 종료되지 않도록 하세요.
2. 마이그레이션 중에는 `web`을 1개 레플리카로 스케일 다운하세요. 여러 레플리카가 동시에 있으면 Prisma/Postgres 마이그레이션에서 데드락이 발생할 수 있습니다. 마이그레이션 완료 후 다시 스케일 업하세요.

메이저 버전 내에서는 완전한 호환성을 목표로 하고 있으며, v3.132와 최신 v3.x 사이에 알려진 breaking change는 없습니다.

문서: https://langfuse.com/self-hosting/upgrade

감사합니다,
{you}
```

관련 FAQ: [/faq/all/upgrade-langfuse](/faq/all/upgrade-langfuse).

<summary>EE 라이선스 사용 / "프로덕션에 EE 라이선스가 필요한가요?"</summary>

이는 기술적인 질문이 아니라 거버넌스/컴플라이언스 질문입니다. 고객은 보통 내부 OSS 검토를 준비 중입니다.

- EE 기능에는 `LANGFUSE_EE_LICENSE_KEY`가 필요합니다. 전체 목록은 [/self-hosting/license-key](/self-hosting/license-key)를 참조하세요.

<summary>Docker 이미지의 CVE / 취약점 보고</summary>

컨테이너 스캐너(Wiz, Snyk, Trivy, Black Duck)는 종종 전이적(transitive) Node.js 의존성에 대한 긴 CVE 목록을 만들어냅니다. 대부분은 저희 사용 방식에서는 악용 가능하지 않습니다. 올바른 대응은 다음과 같습니다:

**트리아지 단계:**

1. 고객이 스캔한 버전을 확인합니다. 최신 버전이 아니라면, 먼저 현재 이미지를 다시 스캔해달라고 요청하세요. 많은 CVE가 이미 다음 릴리스에서 패치되었을 수 있습니다.
2. 실제로 우려되는 사안이라면 고객을 [ClickHouse Bugcrowd 프로그램](https://bugcrowd.com/engagements/clickhouse)으로 안내하여 범위 검토와 제출을 진행하도록 하세요. 민감한 개념 증명(PoC) 세부 정보를 지원 채널로 보내달라고 요청하지 마세요.
3. 보고서가 실제 악용이나 고객 데이터 노출을 시사한다면, 동시에 Slack `#security`에서 엔지니어링 팀을 호출하세요.
4. 수정 일정을 약속하지 마세요. 저희는 매 릴리스마다 순차적으로 패치합니다.

**답변 템플릿:**

```text
안녕하세요 {name}님,

스캔 결과 감사합니다. 최신 이미지({current_version}, {date} 릴리스)로 다시 스캔해 주실 수 있을까요? 보고서에 있는 심각도 높은 CVE 중 다수가 최근 릴리스에서 이미 해결되었습니다.

그럼에도 남아 있는 항목이 있다면 저희 보안팀이 검토하고 우선순위를 정하겠습니다. 전이적 Node.js 의존성의 CVE 대부분은 Langfuse가 실행하지 않는 코드 경로에 있어, 모든 CVE에 대해 수정본을 배포하지는 않지만 도달 가능한 것에 대해서는 수정합니다.

남은 발견 사항이 실제로 악용 가능하다고 판단되시면 ClickHouse Bugcrowd 프로그램을 통해 범위를 검토하고 제출해 주세요: https://bugcrowd.com/engagements/clickhouse

감사합니다,
```

<summary>"트레이스가 안 보여요 / 느려요 / 나타나지 않아요"</summary>

**순서대로 진행하는 트리아지 단계:**

1. **status.langfuse.com**: 먼저 현재 발생 중인 장애가 아닌지 배제합니다.
2. **DataDog**: 수집 큐 깊이, ClickHouse 지연 시간을 확인합니다. 큐가 깊다면 이는 플랫폼 이슈이므로 개별 고객별로 디버깅하지 말고 에스컬레이션해야 합니다.
3. **고객의 SDK 버전**: 물어보세요. 오래된 SDK(Python pre-v3, JS pre-v4)는 알려진 성능 문제가 있는 레거시 엔드포인트를 사용합니다. 최신 스코프 패키지(`@langfuse/client`, `@langfuse/tracing`, `@langfuse/otel` 또는 `langfuse` Python v3+)로 업그레이드를 권장하세요.
4. **고객의 flush 동작**: 단기 실행 프로세스(Lambda, CLI, 엣지 런타임)는 종료 전에 `langfuse.flush()`를 호출해야 합니다. 그렇지 않으면 처리 중이던 이벤트가 유실됩니다.
5. **고객의 필터/시간 범위**: 올바른 프로젝트, 올바른 환경 태그, 그리고 "지금-5분"을 포함하는 시간 범위를 보고 있는지 확인하세요(정상적인 운영에서도 수집은 약 1~2분 지연될 수 있습니다).

**답변 템플릿 (cloud, 상태 확인 후):**

```text

상태 페이지는 정상이며 저희 쪽 큐도 정상으로 보입니다. 몇 가지 확인해 주세요:

1. 최신 SDK를 사용 중이신가요? Python은 `langfuse` v3+, JS는 v4+ 스코프 패키지(`@langfuse/client` / `@langfuse/tracing` / `@langfuse/otel`)입니다. 레거시 `langfuse` JS v3 패키지와 Python v2 SDK는 모두 알려진 지연이 있는 구버전 엔드포인트를 사용했습니다.
```

<details>

<summary>OTEL / OpenTelemetry: 원치 않는 스팬, 이중 집계, 시맨틱 컨벤션</summary>

흔한 OTEL 설정 문제입니다: 고객의 기존 OTEL 설정이 LLM 스팬과 함께 모든 HTTP 요청, DB 쿼리, 프레임워크 스팬을 Langfuse로 내보냅니다. 내보내진 모든 스팬은 관찰(observation)로 수집되어 과금되므로, 내보내기 필터를 조정하기 전까지는 비용과 UI 혼잡이 함께 증가합니다.

**트리아지 단계:**

1. 고객이 Langfuse를 OTEL 제공자에 어떻게 연결했는지 물어보세요(TracerProvider를 공유하는지, exporter만 사용하는지, 자동 계측인지).
2. HTTP / DB / 프레임워크 자동 계측과 함께 전역 TracerProvider를 공유하고 있다면, LLM이 아닌 스팬을 제외하도록 `should_export_span` 필터(Python SDK, 지원 중단된 `blocked_instrumentation_scopes`를 대체) 또는 `shouldExportSpan`(JS SDK)을 권장하세요.
3. 에이전트 프레임워크(특히 pydantic-ai, 이슈 #1819 참조)의 비용 이중 집계의 경우, 저희가 추적 중인 알려진 버그입니다. 인지하고 이슈를 파일링/링크해 드리되, 수정 일정을 약속하지 마세요.
4. `langfuse.experiment.*` 속성의 경우, non-Python SDK를 사용하는 고객이 실험 속성을 수동으로 전파하려다 평가기가 실행되지 않는 경우가 있습니다. LLM-as-a-Judge는 현재 OTEL로 수집된 트레이스에서만 실행되므로, 레거시 SDK 경로가 사용되고 있지 않은지 확인하세요.

**답변 템플릿 (원치 않는 스팬):**

```text
안녕하세요 {name}님,

기존 OTEL 설정에서 흔히 발생하는 문제입니다. 전역 TracerProvider가 LLM 스팬과 함께 HTTP/DB/프레임워크 스팬을 내보내고 있어 볼륨이 높은 것입니다.

해결 방법 (Python):
  from langfuse import Langfuse
  from langfuse.span_filter import is_default_export_span

  blocked = {
      "opentelemetry.instrumentation.fastapi",
      "opentelemetry.instrumentation.asgi",
      "opentelemetry.instrumentation.httpx",
      # ... 필요한 항목 추가
  }
  langfuse = Langfuse(
      should_export_span=lambda span: (
          is_default_export_span(span)
          and (
              span.instrumentation_scope is None
              or span.instrumentation_scope.name not in blocked
          )
      ),
  )

이렇게 하면 보통 수집 볼륨이 50~90% 줄어들며 LLM/에이전트 스팬만 Langfuse에 남게 됩니다.

전체 문서: https://langfuse.com/faq/all/existing-otel-setup#unwanted-spans-in-langfuse

감사합니다,
{you}
```

관련 FAQ: [/faq/all/existing-otel-setup](/faq/all/existing-otel-setup), [/faq/all/unwanted-http-database-spans](/faq/all/unwanted-http-database-spans).

<details>

<summary>비용 / 토큰 추적 불일치 ("비용이 이상해요")</summary>

**트리아지 단계:**

1. 저희가 지원하는 가격 목록에 있는 모델인가요? UI의 "Model" 정의에서 모델을 확인하세요. 커스텀 모델은 Langfuse가 비용을 계산할 수 있도록 입력/출력 토큰 가격이 포함된 `Model` 항목이 필요합니다.
2. SDK/프레임워크가 토큰 수를 전송하나요? 그렇다면 Langfuse가 이를 사용하고, 그렇지 않다면 저희가 모델의 토크나이저로 입력/출력을 자체적으로(best-effort) 토큰화합니다.
3. 에이전트 프레임워크(특히 pydantic-ai)의 경우, 부모 에이전트 스팬과 자식 LLM 스팬이 모두 사용량을 보고할 때 토큰 이중 집계가 발생할 수 있습니다. 알려진 이슈이므로 트레이스 링크와 함께 에스컬레이션하세요.
4. 프레임워크도 비용을 보고함에도 Langfuse가 비용을 계산하는 경우, 프레임워크의 `otel operation.cost` 속성이 오버라이드됩니다. 저희의 가격표가 신뢰할 수 있는 소스입니다.

**답변 템플릿:**


### 5. SDK 및 통합

<details>

<summary>Python SDK</summary>

**흔한 이슈:**

- **레거시 `langfuse` Python v2 패키지 사용.** `@observe` 데코레이터와 OTEL 기반 수집은 v3+부터 제공됩니다. 업그레이드를 권장하세요.

- **`get_prompt()` 오류**: 보통 잘못된 리전, 누락된 API 키, 또는 잘못된 `label`을 참조하는 경우입니다.

<details>

<summary>JS / TypeScript SDK</summary>

**흔한 이슈:**

- **레거시 `langfuse` npm 패키지는 v3.x에 머물러 있습니다.** v4+는 `@langfuse/*` 스코프 패키지(`@langfuse/client`, `@langfuse/tracing`, `@langfuse/otel`) 아래에 있습니다. 인앱 평가기 경고 "JS SDK v4+ required"는 이 스코프 패키지들로 전환하라는 의미입니다.

### 7. 평가

<details>

<summary>"평가기가 제 트레이스에서 실행되지 않아요"</summary>

**가장 흔한 원인: 트레이스가 OTEL 이전의 레거시 SDK 경로를 통해 수집되었습니다.** LLM-as-a-Judge는 현재 OTEL 기반 관찰에 대해서만 실행됩니다.

**진단 방법:** UI에서 트레이스를 열어보세요. `metadata.scope.*`와 `metadata.resourceAttributes.*` 필드가 존재하면 OTEL을 통해 수집된 것이며 평가기가 이를 처리해야 합니다. 이 필드들이 없다면 트레이스는 레거시 `/api/public/ingestion` 엔드포인트를 통해 들어온 것이며 점수가 매겨지지 않습니다.

**트리아지 단계:**

1. 고객의 최근 트레이스 중 하나를 확인하여 OTEL 메타데이터가 있는지 확인합니다.
2. 레거시라면: SDK 업그레이드를 요청하세요(Python `langfuse` v3+, JS `@langfuse/*` v4+).
3. OTEL이라면: 평가기 설정이 트레이스와 일치하는지 확인하세요(변수 매핑, 필터 조건, 대상 관찰 유형). 일부 평가기는 `traces`가 아니라 `observations`를 대상으로 합니다.
4. UI → Evaluators → 해당 설정 클릭 → 최근 실행 이력에서 평가기 로그를 확인하세요.

**답변 템플릿:**

```text
안녕하세요 {name}님,

LLM-as-a-Judge는 현재 OTEL로 수집된 관찰만 평가합니다. 점수가 매겨지지 않은 트레이스 중 하나를 열어서 metadata.scope.* / metadata.resourceAttributes.* 필드가 있는지 확인해 주세요:

- 있음 → OTEL 기반이며 점수가 매겨져야 함
- 없음 → 레거시 SDK 경로로 수집되어 점수가 매겨지지 않음

없음에 해당한다면, SDK를 업그레이드하는 것이 해결책입니다(Python langfuse v3+, JS @langfuse/* v4+). traceId 몇 개를 공유해 주시면 어떤 트레이스가 어느 경우에 해당하는지 함께 확인해 드리겠습니다.

감사합니다,
{you}
```

관련 FAQ: [/faq/all/observation-eval-not-executing](/faq/all/observation-eval-not-executing).

</details>

<summary>데이터셋과 실험</summary>

### 10. API 오류

<details>

<summary>5xx 오류 (502 / 503 / 504 / 524 Bad Gateway / Gateway Timeout)</summary>

**트리아지 단계:**

1. **status.langfuse.com**을 먼저 확인합니다. 진행 중인 장애가 있다면 상태 페이지로 안내하고 인지했음을 알립니다.
2. 상태가 정상이라면 지난 30분간 DataDog에서 오류율이 상승했는지 확인합니다. 짧은 장애가 발생했지만 상태 페이지에 아직 게시되지 않았을 수 있습니다. 짧은 순간의 문제는 정상적인 범위이지만, 반복된다면 내부적으로 기록해 두세요.
3. 오직 한 고객만 겪고 있고 저희 쪽이 정상으로 보인다면, 타임스탬프, 리전, `cloud.langfuse.com` / `us.cloud.langfuse.com` 등 중 어디에 접속 중인지 또는 프록시를 거치는지 물어보세요.

```text
{time} 무렵에 아주 짧은 장애가 있었습니다. 지금은 정상으로 복구되었을 것입니다. 여전히 오류가 발생하는지 확인해 주시겠어요? 만약 그렇다면 가장 최근 타임스탬프를 공유해 주시면 확인해 드리겠습니다.

감사합니다,
{you}
```

관련 FAQ: [/faq/all/api-524-http-errors](/faq/all/api-524-http-errors), [/faq/all/self-hosting-502-504-network-errors](/faq/all/self-hosting-502-504-network-errors).

</details>

<details>

<summary>429 / 요청 제한(rate limit) 오류</summary>

**트리아지 단계:**

1. 어떤 엔드포인트에 요청 중인지 확인합니다. 트레이스 수집은 프롬프트/API 읽기보다 훨씬 관대한 제한을 가집니다.

3. 실제로 높은 처리량이 필요한 경우 엔터프라이즈 팀으로 라우팅하세요. 계약에 따라 제한을 상향할 수 있습니다.

관련 FAQ: [/faq/all/api-limits](/faq/all/api-limits).

### 13. 기능 요청

<details>

<summary>"X 기능을 추가해 주실 수 있나요?" / "Langfuse가 …할 수 있으면 좋겠어요"</summary>

기능 요청은 스레드에서 해결할 사안이 아니라 제품 팀이 수요를 파악할 수 있도록 라우팅할 사안입니다. 처리 방식은 요청이 어디서 왔는지에 따라 달라집니다.

**유료 고객 (Pylon):**

1. 요청을 인지하고 +1을 기록합니다. 예: "말씀하신 부분이 타당하다고 생각합니다! 이 기능 요청에 고객님의 +1을 추가하겠습니다."
2. 새 Linear 티켓을 생성하거나, 기존 Linear 이슈를 Pylon 이슈에 연결하여 수요가 구체적인 작업 항목에 기록되도록 합니다.

**커뮤니티 / GitHub 아이디어 보드:**

1. 요청을 인지합니다. 예: "여기에 등록해 주셔서 감사합니다!"
2. 해당 제품 영역을 담당하는 Langfuse 제품 엔지니어를 @-멘션하여([담당자 배정](/handbook/how-we-work/ownership) 참조) 올바른 사람에게 전달되도록 합니다. 예: "`@nimar` FYI - 더 빠른 평가에 대한 기능 요청입니다."

**에스컬레이션 시점:** 요청이 거래나 갱신과 연관된 경우(엔터프라이즈 팀으로 라우팅), 또는 짧은 기간 안에 여러 고객이 동일한 것을 요청하는 경우, +1만 기록하지 말고 담당 제품 엔지니어에게 직접 알리세요.

</details>

### 14. 실제로는 지원 문의가 아닌 인박스 (빠르게 걸러내기)

<summary>스팸 / 파트너십 / 스폰서십 / 게스트 포스트 / 링크 삽입</summary>

인박스의 약 1~2%는 다음과 같은 홍보성 메일입니다: "게스트 포스트를 작성하고 싶습니다", "파트너십을 판매합니다", "저희 행사를 후원해 주세요", "백링크를 구매하세요". 정중히 거절하거나 응답하지 않고 종료하세요.

**답변 템플릿:**

```text
안녕하세요 {name}님,

연락 주셔서 감사합니다. 현재로서는 이러한 유형의 파트너십을 검토하고 있지 않습니다. 하시는 일에 좋은 결과가 있기를 바랍니다.

감사합니다,
{you}
```

## 확신이 서지 않을 때

고객의 질문이 위의 어느 분기에도 맞지 않는다면:

1. **이 페이지를 검색합니다**: Cmd/Ctrl-F로 고객 메시지의 키워드를 검색하세요.
2. **Pylon을 검색합니다**: 지난 30일간 같은 증상이 있었는지 확인하세요. 누군가 이미 답변했을 가능성이 높습니다.
3. **`#support` Slack 채널에 문의합니다**: 티켓 링크와 본인의 가설을 함께 남기세요. Pylon 티켓의 내부 노트도 유효합니다.
4. **담당자에게 전달합니다**: [담당자 배정](/handbook/how-we-work/ownership)을 참조하세요. 담당자를 알 수 없다면 기술적인 사안은 엔지니어링 팀으로, 상업적/법적 사안은 엔터프라이즈 팀으로 에스컬레이션하세요.

동일한 질문에 세 번째로 답변하고 있다는 것을 깨달으면, 이 페이지에 추가하거나 `content/faq/all/` 아래에 FAQ 항목을 추가하고 여기서 링크하세요. 저희가 문서화하는 반복적인 질문 하나하나가 Inkeep, Dosu, 그리고 앞으로의 지원 엔지니어가 사람의 개입 없이 답변할 수 있는 질문이 됩니다.

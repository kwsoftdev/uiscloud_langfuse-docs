# 문서화

_"문서화는 제품이다"_ 라는 생각을 가지고 있으며, 이는 우리가 크고 계속 성장하는 커뮤니티로 효율적으로 확장하는 데 도움이 되기 때문에 매우 중요하게 여깁니다.
문서는 Product Engineering 팀과 GTM 팀이 함께 관리하며, 제품 기능/영역의 [오너](/handbook/how-we-work/ownership)가 해당 문서를 정리된 상태로 유지할 책임을 집니다.

## Langfuse 문서를 수정하는 방법

문서 페이지(그리고 웹사이트)는 [langfuse/langfuse-docs](https://github.com/langfuse/langfuse-docs) 저장소에 있습니다.
로컬 머신에 프로젝트를 설정하려면 [README.md](https://github.com/langfuse/langfuse-docs/blob/master/README.md)를 따르세요.
변경 사항은 Pull Request를 통해 반영합니다. 누구나 변경할 수 있으며, 기여를 환영합니다.

## 좋은 기술 문서를 작성하는 방법

- 목적과 대상 독자를 명확히 하세요
- 관련된 다른 문서 페이지로의 링크를 포함하세요
- 초보 사용자를 위해 높은 수준(high-level)에서 시작한 뒤 필요에 따라 점진적으로 커스터마이징/복잡성을 쌓아가세요
- [diataxis](https://diataxis.fr)와 다음 항목들의 차이를 숙지하세요:
  - 튜토리얼(Tutorials)
  - 하우투 가이드(How-to Guides)
  - 레퍼런스(References)
  - 설명(Explanations)
- SDK 설정 옵션이나 API 라우트의 전체 목록이 필요한 경우 레퍼런스를 참고하세요

추천 자료:

- [leerob.com/docs](https://leerob.com/docs)
- [Posthog _How we write docs?_](https://posthog.com/handbook/content/docs)
- [Mintlify Docs Guide](https://www.mintlify.com/guides/introduction)

## 페이지 이동/이름 변경 시 리다이렉트

문서 페이지를 이동하거나 이름을 변경할 때는 항상 이전 URL에서 새 URL로의 리다이렉트를 추가하세요. 이는 사용자 경험(북마크, 공유된 링크)과 SEO(검색 엔진이 새 URL로 랭킹을 이전함)를 보존합니다.

**위치:** 모든 리다이렉트는 [lib/redirects.js](https://github.com/langfuse/langfuse-docs/blob/master/lib/redirects.js)에 있습니다. Next.js와 Cloudflare Pages 모두 이 파일을 사용합니다.

**방법:**

1. 페이지 이동/이름 변경의 경우 **영구 리다이렉트(permanent redirects)**를 사용하세요(301로 적용되어 검색 엔진이 색인을 업데이트합니다).
2. `lib/redirects.js` 파일 하단에 변경 사항을 짧게 설명하는 주석과 함께 새로운 날짜 기반 상수(예: `myPageMoveRedirects202601`)를 추가하세요.
3. 하나 이상의 `[oldPath, newPath]` 튜플을 추가하세요. 경로는 사이트 루트를 기준으로 한 상대 경로입니다(예: `/docs/evaluation/data-model` → `/docs/evaluation/experiments/data-model`). 특정 섹션을 위해 해시 프래그먼트를 포함할 수 있습니다(예: `"/docs/foo#section"`).
4. 새로 만든 상수를 `permanentRedirects` 배열에 스프레드(spread)하세요.

**중요:** 영구 리다이렉트는 추가만 가능하며 제거할 수 없습니다(브라우저와 검색 엔진이 이를 캐시할 수 있기 때문입니다). 나중에 읽는 사람이 리다이렉트가 존재하는 이유를 이해할 수 있도록 설명적인 상수 이름과 주석을 사용하세요.

**Cursor / Claude Code용 예시 프롬프트:** 현재 브랜치의 모든 페이지 변경 사항에 대한 리다이렉트를 추가해 달라고 AI에게 요청할 수 있습니다. 아래와 같은 내용을 복사-붙여넣기 하세요.

> `content/` 디렉토리에 대한 현재 브랜치의 git diff를 확인하세요. 경로가 이동되었거나 이름이 변경된(또는 경로가 바뀐) 모든 파일에 대해 `lib/redirects.js`에 영구 리다이렉트(이전 URL → 새 URL)를 추가하세요. 기존 리다이렉트와 동일한 패턴을 사용하세요: 짧은 주석과 함께 새로운 날짜 기반 상수(예: `myChangeRedirects202601`)를 만들고, `[oldPath, newPath]` 튜플(사이트 루트 기준 상대 경로, 예: `content/docs/foo.mdx`가 아니라 `/docs/foo`)을 추가한 뒤 `permanentRedirects` 배열에 스프레드하세요. 실제로 경로가 변경된 페이지에 대해서만 리다이렉트를 추가하고, 기존 파일의 편집에는 리다이렉트를 추가하지 마세요.

## 체인지로그

새로운 기능과 업데이트에 대해서는 [체인지로그](/changelog) 게시물을 작성합니다.

동기:

- 이미 Langfuse를 사용하고 있는 사람들이 새로운 기능을 사용하기 시작하면서 새로운 가치를 얻습니다
- 한동안 Langfuse 도입을 고려해온 사람들이 마침내 원하던 기능을 발견하고 이를 계기로 시작하게 됩니다
- Langfuse를 사용하지 않는 사람들이 "이 팀은 계속 출시하는구나(this team ships)"라고 생각하게 됩니다

## 문서/체인지로그 리뷰

기능/영역의 [오너](/handbook/how-we-work/ownership)가 해당 문서를 유지 관리할 책임을 집니다.

다만 세부 사항에 너무 깊이 빠져 있으면 좋은 문서를 작성하기 어려울 때도 있습니다. Felix와 Jannik이 PR 리뷰를 통해 문서를 올바르게 다듬는 데 기꺼이 도움을 줄 것입니다.

여기에는 엄격한 규칙이 없으며, 판단은 여러분에게 달려 있습니다.

- 변경 사항이 크거나 다른 사람의 검토가 필요하다고 생각되는 경우(어떤 경우든 저희는 기꺼이 살펴봅니다!)
- "on this page"에 표시되는 헤드라인(h2, h3)을 추가한 경우, 리뷰를 받아보는 것이 좋습니다
- 체인지로그 게시물을 작성하는 경우, 대개 리뷰를 받는 것이 좋습니다

절차:

1. 문서 리뷰가 필요한지, 그리고 기능 출시 전과 후 중 언제 진행할지 결정합니다
2. 문서 변경 PR에 Felix 또는 Jannik을 리뷰어로 지정합니다. 두 사람은 당일 리뷰를 약속합니다. 시간이 너무 늦었다면 다음 날 아침에 진행합니다.
3. 대부분의 경우 저희가 대신 문서를 게시하고 알려드립니다. 때로는 다시 논의를 되돌릴 수도 있습니다

# 제품 엔지니어링 원칙

## 우리가 하는 일

- 각 제품 영역에 대해 더 긴밀하게 반복 작업을 할 수 있는 리드 사용자 그룹을 정의합니다
- 제품 영역은 엔지니어 간에 명확하게 분담됩니다([소유권](/handbook/how-we-work/ownership) 참조)
- 문서를 작성하고 지원을 제공합니다
  - 문서는 우리가 확장하는 방법이자 제품의 핵심 요소입니다
  - 지원은 우리가 배우는 방법이며, 제품이 기대만큼 훌륭하지 않을 때 고객이 언제든 우리에게 알려주도록 보장합니다
- 품질/속도/유지보수성을 확보하기 위해 범위를 질문하거나 축소합니다
- [Product Ops 워크플로우](/handbook/product-engineering/how-we-work/product-ops)를 따릅니다
  - 로드맵/전략의 핵심에 해당하는 고객 피드백(버그, 개선 사항)에는 신속하게 대응합니다. 규모가 작은 경우 같은 주에 출시하는 것을 목표로 합니다. 이는 신뢰를 쌓고 복리 효과를 냅니다
  - 출시하기로 한 경우 Slack/Pylon/… 에 있는 피드백을 최대한 빨리 Linear로 옮깁니다. 이를 통해 공개 채널의 노이즈를 줄이고 개선 사항의 실행 상황을 추적합니다

이를 위해서는 다음이 필요합니다

- Linear Inbox와 Pylon의 Inbox 0 유지(하루 한 번이면 충분합니다)
- 무언가를 출시할 때 문서/체인지로그 게시물 작성

## 우리가 하지 않는 일

- 로드맵과 맞지 않는 단일 사용자의 요구사항에 맞춰 최적화하는 것
- 외부에 일정을 공유하는 것
- 높은 스트레스 상태에 빠지는 것; 우리는 차분하게 높은 속도와 품질로 출시합니다

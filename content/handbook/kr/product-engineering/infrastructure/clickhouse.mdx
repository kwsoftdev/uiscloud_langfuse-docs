# ClickHouse Cloud

우리는 트레이스, observation, 점수와 같은 관측성 데이터를 저장하고 쿼리하기 위한 주요 데이터 웨어하우스로 [ClickHouse Cloud](https://clickhouse.com/cloud)를 사용하고 있습니다.

우리는 `Langfuse GmbH`와 `Langfuse HIPAA`라는 두 개의 조직을 운영하고 있습니다.
전자는 스테이징, EU, US 계정을 위한 별도의 웨어하우스를 포함합니다.
후자는 HIPAA 클라우드 리전을 위한 전용 HIPAA 준수 조직입니다.

## 용어

- **Organization(조직)**: ClickHouse Cloud에서 여러 웨어하우스를 포함하는 최상위 엔터티입니다. 지원 티켓과 청구는 조직 수준에서 관리됩니다.
- **Warehouse(웨어하우스)**: 공통 데이터 저장소와 ClickHouse-Keeper 구성을 공유하는 서비스들의 논리적 그룹입니다. 즉, 웨어하우스는 Compute-Compute 분리를 가능하게 합니다.
- **Service(서비스)**: 공통 엔드포인트를 공유하는 레플리카 집합입니다. 서비스는 수평 및 수직으로 확장될 수 있습니다.
- **Node/Replica(노드/레플리카)**: 서비스 내의 개별 인스턴스입니다. Kubernetes Pod에 대응됩니다.
- **Part(파트)**: 디스크상의 논리적 "파일"입니다. ClickHouse는 시간이 지남에 따라 파트들을 더 큰 파트로 병합합니다.
  초기에는 모든 데이터가 동일한 파일 내에 있는 "Compact" 파트로 기록되며, 이는 파트 전체를 쓰고 읽는 것을 더 저렴하게 만듭니다.
  시간이 지나 파트가 커지면, 각 컬럼이 자체 파일로 분리되는 "Wide" 파트로 이동됩니다.
- **Granule(그래뉼)**: ClickHouse에서 가장 낮은 수준의 인덱싱 엔터티입니다. 파트 내 행 집합에 대한 포인터 역할을 합니다.

## 공통 개념

이 섹션에서는 Langfuse에서 사용하는 몇 가지 공통 개념과 그 의미를 다룹니다.

### ReplacingMergeTree

모든 트레이싱 테이블은 [ReplacingMergeTree](https://clickhouse.com/docs/engines/table-engines/mergetree-family/replacingmergetree) 엔진을 사용합니다.
이를 통해 더 높은 버전 번호와 동일한 정렬 키를 가진 새 행을 작성하여 업데이트를 수행할 수 있으며, 백그라운드에서 자동으로 중복이 제거됩니다.
트레이스, observation, 점수에 대한 업데이트를 허용하므로, 이는 핵심적인 기능입니다.

ReplacingMergeTree의 중복 제거는 언젠가는 일어나지만, 반드시 일어난다는 보장은 없습니다.
따라서 대부분의 쿼리에서 일관된 결과를 반환하기 위해 추가적인 읽기 측 중복 제거를 수행합니다.

우리가 사용하는 읽기 시점 중복 제거의 세 가지 주요 방법은 다음과 같습니다:

- **FINAL 수정자**: ClickHouse가 병합 시 수행하는 전체 중복 제거를 자동으로 수행합니다.
  특히 대규모 테이블의 경우, 이는 매우 비용이 많이 들고 느리며 메모리 소비가 높을 수 있습니다.
  또한, 이전 버전의 ClickHouse에서는 데이터 스키핑 인덱스의 사용을 차단합니다.
- **ORDER BY event_ts LIMIT 1 BY ...**: 정렬 순서가 업데이트 시 중복을 피하도록 보장하면(즉, 시간과 event_ts로만 정렬), LIMIT 1 BY 절을 사용하여 식별자당 하나의 행만 반환할 수 있습니다. 이는 계산 비용이 가장 저렴하고 데이터 스키핑 인덱스를 활용하는 경우가 많지만,
  테이블의 정렬 순서를 신중하게 설계해야 합니다. 또한 업데이트 간에 타임스탬프가 변경되어 겉보기에 임의적인 결과가 발생하는 함정도 관찰했습니다.
- **argMax/argMin 함수**: 이 함수들은 각 정렬 키에 대해 가장 높은/낮은 버전을 가진 행을 선택할 수 있게 합니다.
  이는 공통 식별자에 대한 group-by 호출에 의존합니다. 예: `GROUP BY project_id, trace_id`.
  전체 결과 집합을 메모리에 유지한 후 반환해야 하므로 메모리 소비가 높아, 우리는 이 방법을 거의 사용하지 않습니다.

### Insert

ClickHouse는 대규모 insert의 이점을 크게 누리는데, 각 insert가 나중에 백그라운드에서 병합되어야 하는 초기 데이터 파트를 생성하기 때문입니다.
따라서 우리는 insert를 시작하기 전에 데이터를 누적하는 여러 배치 메커니즘을 갖추고 있습니다.

워커 컨테이너는 모든 ClickHouse 쓰기를 `ClickHouseWriter` 내의 인메모리 버퍼에 저장합니다.
거기서부터 구성 가능한 배치 크기와 간격으로 데이터를 ClickHouse에 정기적으로 플러시합니다.
또한, ClickHouse의 `async_inserts`를 사용하여 서버 측에서 데이터를 수집한 후 insert를 확인하고 디스크에 기록합니다.

## 성능 고려 사항

### 여러 컬럼 요청

디스크상의 컬럼형 레이아웃, 특히 Wide 파트에서는 쿼리에 추가 컬럼을 더하면 비용이 더 커집니다.
이는 특히 `input`, `output`, `metadata`와 같은 대용량 바이너리 컬럼에 해당됩니다.
따라서 테이블 데이터를 요청할 때 이러한 컬럼을 선택적으로 제외할 수 있는 여러 최적화가 코드에 포함되어 있습니다.
사용 사례에 필요한 컬럼이 무엇인지 적극적으로 고려하고 쿼리를 해당 컬럼으로 제한하도록 하십시오.

### 삭제

삭제는 ClickHouse에서 영향을 받는 파트의 뮤테이션을 발생시킵니다.
처음에는 새로운 `_row_exists` 컬럼이 기록되는데, 이는 Wide 파트의 경우 전체 컬럼 재작성을, Compact 파트의 경우 전체 파트 재작성을 필요로 합니다.
따라서 우리는 삭제를 백그라운드에서 비동기적으로 실행하려고 시도하며, 시스템 상태에 따라 삭제를 배치 처리하고 속도를 제한하는 여러 최적화를 갖추고 있습니다.

### 단일 레코드 조회

ClickHouse에서 주소 지정 가능한 가장 작은 단위는 일반적으로 8192개의 행을 보유하는 그래뉼입니다.
따라서 단일 행 조회는 각각 최소한 그만큼의 행을 읽습니다.
가능하다면 여러 레코드에 대한 쿼리를 한 번에 실행하기 위해 배치 처리를 사용해야 합니다.

### Join

ClickHouse의 join은 쿼리를 최대한으로 최적화하는 경우가 드뭅니다.
대규모 프로젝트에서 런타임 문제가 관찰된다면, 다음 사항들이 해당되는지 조사하는 것이 일반적으로 유용합니다:

- 테이블의 왼쪽과 오른쪽으로 필터를 푸시다운할 수 있는가?
- 더 적합한 join 알고리즘이 있는가([ClickHouse join 문서](https://clickhouse.com/blog/clickhouse-fully-supports-joins-part1))?
- 더 작은 relation이 join의 오른쪽에 있는가?

## 운영

ClickHouse는 클러스터 모드와 단일 인스턴스 모드를 지원합니다. 이는 DDL 쿼리에 영향을 미치는데, 다중 노드 설정에서 복제를 달성하려면 ON CLUSTER 키워드가 필요하지만, 클러스터 구성이 없는 단일 인스턴스에서는 오류가 발생합니다.

애플리케이션의 단순성을 위해, 우리는 마이그레이션 스크립트를 복제하여 두 경우를 모두 지원합니다. 실행되는 마이그레이션은 CLICKHOUSE_MIGRATION_CLUSTER_ENABLED 환경 변수에 따라 달라집니다. 기본값은 true이며, 클러스터 모드를 비활성화하려면 명시적으로 false로 설정해야 합니다.

새 마이그레이션을 생성하는 경우, 다음 네 개의 파일을 추가해야 합니다:

```
packages/shared/clickhouse/migrations/clustered/<name>.up.sql
packages/shared/clickhouse/migrations/clustered/<name>.down.sql
packages/shared/clickhouse/migrations/unclustered/<name>.up.sql
packages/shared/clickhouse/migrations/unclustered/<name>.down.sql
```

비클러스터 모드는 우리 CI에서 테스트되고, 클러스터 모드는 스테이징에서 테스트됩니다. 즉, 추가한 마이그레이션 중 잘못된 것이 있는지 관찰해야 합니다.

또한, 저장소 내 [ClickHouse 데이터베이스 마이그레이션](https://github.com/langfuse/langfuse/blob/main/REVIEW.md#clickhouse)에 대한 가이드라인도 검토하십시오.

## 제한 사항

Langfuse v3 메이저 릴리스 주기 내에서, 우리는 ClickHouse 24.3과의 하위 호환성을 보장합니다.
따라서 일부 CI 실행은 이 버전에 대해 실행되며, 이후 버전에서 도입된 기능이나 설정 플래그를 사용하면 오류가 발생할 수 있습니다.
주요 차이점은 다음과 같습니다:

- `JSON` 컬럼 지원이 없습니다. 즉, JSON 타입의 새 컬럼 도입은 Langfuse의 메이저 릴리스를 필요로 합니다.

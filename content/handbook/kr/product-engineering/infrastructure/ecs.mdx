# AWS ECS Fargate

저희는 웹 및 워커 애플리케이션 컨테이너의 런타임으로 [AWS ECS Fargate](https://docs.aws.amazon.com/ecs/)를 사용하고 있습니다.

트래픽과 부하 패턴을 분리하기 위해 각 서비스를 서로 다른 구성으로 여러 배포로 나누어 운영하며, 특정 라우팅이나 큐 워커 구성을 적용합니다.
또한 모든 컨테이너는 애플리케이션 부하와 메트릭을 기반으로 동적으로 오토스케일링됩니다.

## 용어

- **Cluster(클러스터)**: 서비스와 태스크의 논리적 그룹입니다.
- **Service(서비스)**: 동일한 구성을 공유하는 태스크의 모음입니다. 서비스는 장기간 지속되며 태스크의 집합입니다.
- **Task(태스크)**: 특정한, 일시적인 컨테이너 그룹입니다. Kubernetes의 `Pod`와 비교할 수 있습니다. 태스크는 단명하며 수평적으로 스케일링될 수 있습니다.
- **Task Definition(태스크 정의)**: 태스크를 위한 특정 구성입니다. 환경 변수, 도커 이미지, 기타 설정을 포함합니다.
- **Sidecar(사이드카)**: 태스크 내에서 메인 컨테이너와 함께 실행되는 추가 컨테이너입니다.

## 공통 개념

### 컴포넌트 분리

`web`과 `worker` 서비스를 여러 다른 구성으로 운영합니다.

- **web**: 대부분의 사용자 대면 트래픽을 처리하는 표준 웹 애플리케이션입니다.
- **web-iso**: 특정 라우트가 전송되는 추가 웹 컨테이너입니다. 현재 이는 `/api/public/traces`, `/api/public/observations` 등을 포함한 모든 트레이스 관련 퍼블릭 API 라우트에 영향을 미칩니다.
  이러한 워크로드는 CPU 부하가 크기 때문에, 메인 컨테이너 그룹에 함께 두면 지연 시간에 민감한 다른 라우트에 노이지 네이버(noisy-neighbor) 영향을 줄 수 있습니다.
  트래픽 분리는 LoadBalancer 레벨에서 경로 기반 규칙을 사용하여 이루어집니다.
- **web-ingestion**: 미디어 API, `/api/public/ingestion`, `/api/public/otel/v1/traces` 요청을 처리하는 컨테이너 그룹입니다. 이를 통해 더 세밀한 오토스케일링이 가능해지고,
  인제스천 라우트의 안정성이 나머지 구현부로부터 보호됩니다.
- **worker**: 대부분의 큐 처리를 수행하는 표준 워커 컨테이너입니다. ingestion-queue와 otel-ingestion-queue 처리는 비활성화되어 있습니다.
- **worker-cpu**: CPU 부하가 큰 처리, 주로 ingestion-queue와 otel-ingestion-queue를 처리하는 워커 배포입니다. 이는 처리량을 높이고 다른 워커 작업의 CPU 스톨로 인한 오류율을 줄여줍니다.

### 사이드카

각 태스크는 세 개의 컨테이너로 구성됩니다. Langfuse-OSS용으로 배포하는 핵심 애플리케이션 컨테이너와, 모니터링을 위한 두 개의 사이드카 컨테이너입니다.
하나의 사이드카 컨테이너는 Fluent Bit을 사용하여 로깅을 처리하고 모든 애플리케이션 로그를 DataDog으로 직접 전달합니다.
두 번째 사이드카 컨테이너는 Datadog Agent로, 메트릭을 추적하며 트레이스를 위한 OpenTelemetry Collector 역할도 합니다.

## 배포

### 구성 변경

컨테이너 구성(예: 환경 변수)은 Terraform을 통해 변경할 수 있습니다.
이는 기반이 되는 태스크 정의를 업데이트하고 서비스에 대한 새로운 배포를 트리거합니다.

수동 스케일링 오버라이드는 다음 Terraform apply 시 되돌려집니다.

### 버전 변경

GitHub의 `Deploy to ECS` 액션은 애플리케이션 컨테이너의 새 버전을 빌드하여 클러스터에 배포합니다.
이는 새로운 태스크 정의 버전을 생성하고 서비스에 대한 새로운 배포도 트리거합니다.

### 일반적인 문제

- 컨테이너 배포가 간혹 멈추어 "Deploying" 상태로 오랫동안 남아 있는 경우가 있습니다.
  이는 배포 실행 시 GitHub에서 배포 타임아웃으로 나타날 수 있습니다. 이는 여러 배포가 동시에 발생하여 AWS ECS가 어떤 컨테이너를 중지할지 결정하지 못하고
  추가 컨테이너를 시작할 용량도 없을 때 자주 발생합니다. 저희의 스케일링 설정은 스케일링 용량이 100%를 절대 초과하지 않고 원하는 용량 아래로도 떨어지지 않도록 강제합니다.
  이는 원하는 용량을 일시적으로 늘려서 해결할 수 있습니다.
- 오래된 계획(plan)으로 배포된 버전을 덮어쓰는 경우입니다. Terraform은 `plan` 시점의 현재 컨테이너 버전을 이후 쓰기 작업의 기준선으로 사용합니다.
  plan과 apply 단계 사이에 배포가 발생했다면, apply가 해당 배포를 되돌려버릴 수 있습니다.
  따라서 plan과 apply 사이의 시간 간격을 최대한 짧게 유지하는 것을 권장합니다.

## 성능 관련 참고 사항

### 대용량 본문 로깅

로깅 호출 내에서 대규모 연산을 수행하면, 해당 로그가 비활성화되어 있더라도 애플리케이션 성능에 큰 영향을 줍니다.
예를 들어 `logger.debug(JSON.stringify(bigObject))`는 로깅 여부를 결정하기 전에 먼저 객체를 문자열화(stringify)합니다.
Datadog의 프로파일링 뷰와 플레임그래프를 사용하여 이러한 부분을 식별할 수 있습니다.

## 스케일링

- `web` 컨테이너의 경우 요청 기반 오토스케일링을 사용합니다. 목표 요청 수를 설정하고, 목표에서 벗어나면 용량을 추가하거나 제거합니다.
- `worker` 컨테이너의 경우 큐 길이 기반 오토스케일링을 사용합니다. 인스턴스당 목표 백로그를 정의하고, 백로그가 목표를 초과하면 용량을 추가합니다.

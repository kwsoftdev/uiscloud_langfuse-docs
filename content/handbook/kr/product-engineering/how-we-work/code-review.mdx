# 코드 리뷰

## PR 작성자가 리뷰를 요청해야 하는 경우

- 낮은 리스크(양방향 문): 바로 병합해도 됩니다. 엔지니어는 문제가 발생할 경우를 대비해 Datadog을 모니터링할 책임이 있습니다. 롤백이 필요한 경우 [온콜 엔지니어](/handbook/product-engineering/how-we-work/on-call)에게 알리세요.
  - 제품 전문가 리뷰: 때로는 다른 사람의 담당 영역에서 더 큰 변경을 진행하기도 합니다. [DRI](https://docs.google.com/spreadsheets/d/1gOvWf_uSAtcXxWkR_gMuWX8-OYmSJNIPXTmPGfZ2DGY/edit?gid=0#gid=0)에게 피드백을 요청해도 좋습니다.
- 리스크(단방향 문): Max를 리뷰어로 지정하세요(Max는 Linear에서 열려 있는 PR 리뷰를 확인하며, SLA는 24시간이고 더 급한 경우 핑을 보내세요). **리뷰가 필요한 PR의 예시(전체 목록 아님)**: 데이터베이스 마이그레이션, 공개 API 변경, SDK 시그니처 변경, 인증 변경, 수집 파이프라인의 주요 변경, 대규모 인프라 변경. 확신이 서지 않으면 Max에게 리뷰를 요청하세요.
- 신규 입사자는 최소 입사 첫 1개월 동안 모든 PR에 대해 리뷰를 받아야 합니다. 이를 통해 신규 입사자는 코드베이스와 시스템 동작 방식을 익힐 수 있습니다.

## PR 작성자의 책임

- 작성자는 Pull Request의 품질에 대한 책임이 있습니다. 문제가 발생하면 본인이 수정해야 합니다.
- 작성자는 리뷰어의 코멘트를 모두 확인해야 하지만, 제안을 반드시 수용할 필요는 없습니다.

## 리뷰어의 책임

- 서로의 작업을 막지 않는 것이 중요하므로 PR 리뷰는 24시간 이내에 처리해야 합니다. 오전에 리뷰할 PR을 받았다면 같은 날 안에 피드백을 제공해 주세요.
- 리뷰 시 확인해야 할 사항(전체 목록 아님):
  - 잠재적인 보안 문제(예: 데이터 유출)
  - 잠재적인 성능 문제(예: 느린 쿼리)
  - 잠재적인 버그, 고려되지 않은 엣지 케이스 등
  - 구현의 코드 품질과 우리의 [원칙](/handbook/product-engineering/principles) 준수 여부
  - 코드가 의도한 대로 동작하는지 확인
- 리뷰어가 하지 않아도 되는 것:
  - PR을 아주 세세한 부분까지 이해하는 것.
  - 코드를 로컬에서 실행하여 엔드투엔드로 테스트하는 것.

## UI/UX 리뷰

- UI/UX 변경 사항의 경우 Max 또는 Marc를 리뷰에 참여시켜 주세요. 변경 사항을 담은 짧은 동영상을 만들어 리뷰어와 공유하세요.
- 빠르게 출시하기 위해 V0를 먼저 배포할 수는 있지만, 이후에는 반드시 시간을 들여 UI/UX를 개선해야 합니다.

## Clickhouse 리뷰

- 더 복잡한 Clickhouse 쿼리를 작업하는 경우, ClickHouse DRI에게 PR 리뷰를 요청하세요. 가능하면 구현 전에 DRI에게 미리 조언을 구하는 것이 좋습니다.
- 리뷰를 위해 프로덕션 ClickHouse에서 실제와 유사한 데이터로 쿼리 성능을 검증할 수 있는 쿼리를 준비해 두세요.

# 우리는 이렇게 출시합니다

## 우선순위 지정

매주 티켓을 분류하고 우선순위를 정하는 데 시간을 쏟지 않기 위해, 각자가 자신의 우선순위를 관리하고 작업이 쌓이면 리드에게 에스컬레이션합니다. 이는 SLA와 분기별 [로드맵 프로세스](/handbook/product-engineering/how-we-work/roadmapping)에 대한 공통된 이해가 있을 때만 가능합니다.

애매할 때는 리드에게 태그하세요(Linear 인박스 기준 24시간 SLA). 현재 진행 중인 주요 작업은 하나의 Linear 프로젝트나 이슈로 관리하고, 더 작은 작업은 [버그 뷰](https://linear.app/langfuse/view/bugs-confirmedopen-97856f9f745c)나 "My issues"를 이용하세요. 모든 티켓에는 우선순위가 있습니다.

### 우선순위 레벨 [#priority-levels]

| 우선순위      | 타임라인             | 설명                                                                                                                                                                                                 | 예시                                                  |
| ------------- | -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------- |
| **P0 (긴급)** | 즉시, 모든 것을 중단 | - 보안 사고(예: 데이터 유출)<br/>- 성능 문제(수집 지연, ClickHouse CPU/메모리 문제)<br/>- 애플리케이션을 망가뜨리는 대규모 영향 이슈                                                                 | 트레이스 테이블이 로드되지 않음, 로그인 불가          |
| **P1 (높음)** | 같은 주 내           | - 영향 범위가 작은 이슈<br/>- 1시간 미만의 작업으로 많은 고객에게 큰 영향을 주는 개선 사항; 사용자들이 빠른 출시에 기뻐하므로 빠르게 처리하기 좋음<br/>- 사용자를 돕는 작은 변경으로 당일 만족시키기 | 대시보드에서 특정 엣지 케이스가 동작하지 않음         |
| **P2 (중간)** | 같은 달 내           | - 광범위한 영향은 없는 사소한 수정                                                                                                                                                                   | 사용자가 점수가 많을 때 트레이스 트리 UI가 깨짐       |
| **P3 (낮음)** | 백로그               | - 시급하지 않은 Langfuse의 있으면 좋은 추가 기능                                                                                                                                                     | 최신이 아닌 프롬프트 버전을 기반으로 새 프롬프트 생성 |

## 일상적인 책임

우리는 바쁜 작업과 우리가 추진 중인 가장 중요한 프로젝트의 진행 사이에서 균형을 맞춥니다. 이 균형을 유지하기 위해 몇 가지 응답 시간을 스스로 지킵니다:

- **Linear 인박스**: 하루에 한 번 비우기.
- **Pylon 인박스**: 하루에 두 번 비우기. 사용자의 요청을 최대한 빨리 인지한 후, 조사하거나 수정 작업을 진행합니다.
- **PR 리뷰**: 리뷰어라면 24시간 이내에 처리하세요. 아침에 도착했다면 당일에 리뷰하세요. 자세한 내용은 [코드 리뷰](/handbook/product-engineering/how-we-work/code-review)를 참고하세요.
- **버그 수정 및 지원**: 버그 수정, 지원 티켓 등에는 하루 최대 약 2시간을 사용하세요. 버그 수정이 회사에 중요한 경우도 있지만, 하루 2시간을 지속적으로 초과한다면 리드와 상의해 동의를 얻거나 팀 내에 작업을 분산하세요.

## 스펙 작성

목표: 변경 사항당 반복 횟수를 최소화하고 개별 기여자가 최대한 많은 결정을 내릴 수 있도록 합니다. 변경 사항이 사전에 얼마나 많은 조율이 필요한지는 그 규모와 리스크에 따라 달라집니다.

### 작고 중간 규모의 변경 (예: 테이블에 새 필터 추가, 폼에 새 필드 추가 등)

- 엔지니어가 주도합니다: Linear에 아주 간단한 메모를 남기거나, 설명적인 제목만 작성하세요.
- 속도를 높이고 리스크/불확실성을 줄일 수 있다면 누구와든 짧게 논의해도 좋습니다.
- 리뷰어를 둘 계획이라면, 계획 단계부터 그들을 참여시키세요.
- 코딩 에이전트에게 "내가 놓치고 있는 것은 무엇인가요?" 또는 "당신이라면 이것을 어떻게 만들겠어요?"라고 묻는 것은 피드백을 얻고 불확실성을 줄이는 빠르고 좋은 방법입니다.

### 더 큰 프로젝트 (예: Langfuse에서 에이전트 지원, SDK 재구축 등)

1. 엔지니어가 초기 조사를 수행하고, Linear에 짧은 RFC를 작성한 뒤 미팅을 잡습니다.
   - 프로젝트 시작 전에 리드와 함께 역할을 정의하세요:
     - **드라이버(Driver)**: 담당 엔지니어
     - **승인자(Approver)**: 파드 리드, Max, 또는 Marc(의사결정 미팅 참석자)
     - **정보 공유 대상(Informed)**: 추가적인 맥락을 가지고 있을 수 있는 엔지니어(자세한 내용은 [도움 받기](#getting-help) 참고)
   - 때로는 RFC 없이 15분짜리 미팅으로도 시간을 절약할 수 있습니다. 이는 엔지니어가 판단해 일정을 잡습니다.

2. RFC를 논의하는 미팅에서:
   - 복잡한 엔지니어링 프로젝트마다 몇 가지 중요한 항목이 있습니다. 미팅 중에는 이 항목들에 집중하고, 확신을 가지고 결정을 내릴 수 있도록 최대한 많은 데이터를 준비하세요.
   - 미팅은 녹화되므로, 스펙이나 이슈 설명을 작성하는 데 필요한 세부 정보가 기록됩니다.
   - 타임라인과 작업을 더 빠르게 진행할 수 있는 방법을 논의하세요.
   - 이후 구현 계획에 합의하고, 필요하다면 리드와 동기화하세요.

### 도움 받기 [#getting-help]

언제든 팀 누구든 끌어들일 수 있습니다. 15분간의 공유 논의가 종종 결과물을 크게 개선합니다.

- **UI/UX**: 규모가 큰 UI/UX 변경의 경우, 디자이너나 Nikita의 의견을 구하세요. 초기 단계부터 훌륭한 UI/UX를 고안하는 데 기꺼이 도와줍니다.
- **ClickHouse**: 더 복잡한 ClickHouse 쿼리의 경우, ClickHouse [DRI](https://docs.google.com/spreadsheets/d/1gOvWf_uSAtcXxWkR_gMuWX8-OYmSJNIPXTmPGfZ2DGY/edit?gid=0#gid=0)의 의견을 구하세요. 그렇지 않으면 성능 및 유지보수성에 관한 안티패턴의 위험이 있습니다.
- **프로덕트 오너**: 다른 엔지니어의 제품 영역에서 작업하는 경우, 그들의 의견을 구하는 것이 도움이 될 때가 있습니다.
- **제품**: 엔지니어는 RFC 프로세스의 일환으로 제품 관련 의사결정을 주도합니다.
- **출시 및 마케팅**: 기능이 언제 출시할 준비가 되었는지는 여러분이 가장 잘 알고 있으므로, 출시 계획을 세우기 위해 사전에 마케팅 팀을 참여시키세요.

## 구현

- **불확실성 없음**: 엔지니어가 이슈/기능의 과제와 구현 방법을 이해하고 있는 경우. 바로 빌드를 진행하세요.
- **불확실성 또는 질문이 있는 경우**: Linear 티켓에서 리드에게 명확하고 구체적인 질문을 태그해 답변이나 변경에 대한 동의를 얻으세요.

## 릴리스

- **복잡한 변경/프로젝트**: 어떤 PR을 어떤 순서로 병합하고 릴리스할지에 대한 릴리스 계획에 합의하세요.
- 무언가를 출시할 때는 항상 [공개 문서](/docs)나 [체인지로그 포스트](/changelog)를 작성하세요.
- **기능 가용성**: 해당되는 각 기능에 대해:
  - 관련 헤딩 바로 아래에 `<AvailabilityBanner />` 컴포넌트를 사용해, 해당 기능이 어디에서 사용 가능한지와 어떤 제한 사항(플랜, 사용량 제한, 버전 요구사항)이 있는지를 문서에 명확히 명시하세요.
  - 해당된다면 [프라이싱 페이지](/pricing)에 기능을 추가하세요.
  - 셀프 호스팅과 Langfuse Cloud 간의 차이를 보여주세요(문서의 가용성 배너, 그리고 해당되는 경우 [셀프 호스팅 프라이싱 페이지](/pricing-self-host)).

## 도구

### Linear

우리는 Linear를 내부 프로젝트 계획 및 티켓 관리 도구로 사용합니다. Linear는 다음을 도와줍니다:

- 사용자 피드백을 한곳에 모으기
- 제품 요구사항과 구현 세부사항 논의
- 프로젝트 완료까지 남은 작업 파악
- 버그 분류 및 우선순위 지정
- 가능한 한 많은 내용을 Linear에 유지함으로써 Linear/Slack 분산을 줄이고 지식을 한곳에 유지
- 다른 도구(Pylon, GitHub, Cursor 등)와 통합해 워크플로우를 원활하게 만들기

#### 이슈 상태

| 상태                   | 설명                                                                                                                                                                                                                                                                                                            |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triage**             | - 담당자 미지정, 라벨이나 우선순위 없음<br/>- GitHub Issue 통합을 통해 생성되었거나, 엔지니어링팀이 아닌 팀원이 Linear에서 생성<br/>- Marc/Max가 구독하고, 분류하고, 담당자를 지정하고, 라벨을 추가하고, 제목을 다듬음<br/>- 담당자 지정 후, 엔지니어가 기존 티켓과 중복 여부를 확인하고 사용자와의 소통을 처리 |
| **Backlog**            | - 각자가 Linear에서 자신의 백로그를 관리<br/>- 앞으로 1~2분기 내에 계획하거나 하고 싶은 작업만 백로그에 넣고, 나머지는 [GitHub Discussions](https://langfuse.com/ideas)로 보냄<br/>- 제목, 라벨, Pylon 링크에 대해서는 [아래 컨벤션](#linear-conventions) 참고                                                  |
| **Todo / In progress** | 처리 지침은 [우선순위 레벨](#priority-levels) 참고                                                                                                                                                                                                                                                              |

#### 컨벤션 [#linear-conventions]

- **제목**: 정확하고 간결하게 유지해, 티켓 목록을 훑어보는 어떤 엔지니어라도 대략적으로 각 항목을 이해할 수 있도록 하세요. 설명은 선택 사항이며, 작성할 경우 짧고 정확하게 유지하세요.
- **라벨**: 모든 이슈는 라벨을 가지며, Linear 뷰를 사용해 특정 라벨의 모든 티켓을 볼 수 있습니다. 제품 영역에는 라벨을 사용하고, 대부분의 이슈에는 "feat-..." 라벨을 사용하세요. 새 라벨을 만들기 전에 기존 라벨 중 적합한 것이 없는지 확인하세요([라벨](https://linear.app/langfuse/settings/issue-labels)). 새 라벨은 기본적으로 프로젝트 레벨에서 생성되므로 워크스페이스/조직 레벨로 이동하세요.
- **담당자 지정**: "생성 시 자신에게 할당" 옵션을 활성화하고, 서드파티 도구에서 생성할 때도 자신에게 할당하세요.
- **프로젝트**: 명확한 종료 시점이 있는 작업에만 프로젝트를 생성하세요(끝없이 티켓이 쌓이는 버킷이 되지 않도록).
- **Pylon**: "Thread links"로 Pylon 스레드를 Linear 이슈에 연결하세요. Linear 티켓을 닫을 때는 Pylon 스레드도 해결 처리해 사용자에게 수정 사항이나 요청 사항이 출시되었음을 알리세요. 단 2분의 노력으로 우리가 빠르게 출시한다는 것을 보여줄 수 있습니다. 나중에 다시 다룰 스레드(예: 다음 주)는 스누즈하세요.
- **내부 Slack 메시지**: 내부(사용자에게 노출되지 않는) 메시지의 경우, Slack의 Linear 앱을 사용해 이슈를 생성하세요. 이렇게 하면 Slack 스레드가 이슈에 연결되고, 티켓 상태가 변경될 때 관련자에게 알림이 갑니다.

### Slack

Slack은 바쁘고 시끄러운 채널로 가득 차 있으며, 알림이 집중 작업을 방해하는 것을 원하지 않습니다. [Slack 채널을 소비하는 방법](/handbook/tools-and-processes/using-slack)에 대한 짧은 가이드를 참고하세요.

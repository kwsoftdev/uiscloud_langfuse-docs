# 평가

## 평가가 루프에서 차지하는 역할

오프라인 평가는 실험 실행과 변경 사항 배포 사이에 위치하는 단계입니다. 데이터셋이 있고, 이를 대상으로 애플리케이션을 실행했다면, 이제 출력이 좋은지 판단해야 합니다.

<LoopDiagram highlight="eval" />

## 평가는 일반적으로 어떻게 발전하는가 [#how-evaluation-typically-evolves]

대부분의 경우 **출력을 수동으로 검토**하는 것부터 시작해 애플리케이션에서 좋은 결과와 나쁜 결과가 어떤 모습인지에 대한 직관을 쌓습니다. 그다음 **점검할 가치가 있는 구체적인 실패 모드를 식별**합니다. 이를 정확히 정의할 수 있게 되면 **전용 평가기로 자동화**합니다.

<EvaluationEvolutionDiagram />

이 페이지의 나머지 부분에서는 다양한 종류의 평가를 자세히 다룹니다. 실제로는 이들을 모두 조합해서 사용하게 될 가능성이 높습니다. 하지만 잘 작동하는 자동화된 평가 체계로 가는 경로는 거의 항상 수동 검토에서 시작합니다.

<Callout type="info">
수동 평가는 한 번 하고 끝나는 단계가 아닙니다. 잘 갖춰진 프로덕션 환경에서는 사람 전문가의 지속적인 검토를 통해 새로운 실패 모드를 포착하고 자동화된 평가기의 보정 상태를 유지합니다.
</Callout>

## 평가 방법 [#three-kinds-of-evaluation]

평가하는 방법은 크게 세 가지입니다: 수동, 코드, LLM. 각각은 서로 다른 종류의 품질 검사에 적합합니다.

### 수동 평가

수동 평가란 출력을 직접 살펴보고 점수를 매기거나 품질에 대한 생각을 기록하는 과정입니다.

이는 중요한 과정입니다. 출력을 읽는 것은 애플리케이션이 실제로 무엇을 하는지, 어디서 어려움을 겪는지, 그리고 여러분의 특정 사용 사례에서 "좋음"이 무엇을 의미하는지에 대한 이해를 만들어 줍니다. 이러한 이해가 이후에 어떤 자동화된 평가기를 만들지, 그 기준을 어떻게 정의할지를 알려주는 근거가 됩니다. 이 단계를 건너뛰고 곧바로 자동화된 평가로 넘어가는 팀은 중요하지 않은 것을 측정하게 되는 경우가 많습니다.

수동 평가는 또한 이후 자동화된 평가기를 검증하는 데 기준(ground truth)으로 사용할 수 있는 사람 라벨을 만들어냅니다.

### 코드 평가기

[코드 평가기](/docs/evaluation/evaluation-methods/code-evaluators)는 결정론적 로직으로 검증할 수 있는 속성을 확인합니다. Langfuse에서는 UI에서 작성하는 Python 또는 TypeScript 평가기로, 트레이스 수준의 대상이 아니라 관측치(observation)나 실험에 대해 실행됩니다. 빠르고 저렴하며 매번 동일한 결과를 냅니다.

코드 평가기가 자연스럽게 잘 맞는 체크의 예시는 다음과 같습니다:

- 출력이 유효한 JSON이거나 필수 스키마를 따르는지
- 출력에 특정 키워드나 패턴이 포함되어 있는지(또는 포함되어 있지 않은지)
- 출력이 길이 제한 내에 있는지
- 생성된 SQL이 문법적으로 유효한지

코드 평가기의 한계는 의미를 평가할 수 없다는 점입니다. 코드 평가기는 출력에 "환불"이라는 단어가 포함되어 있는지는 확인할 수 있지만, 출력이 환불 정책을 올바르게 설명하고 있는지는 확인할 수 없습니다.

### LLM-as-a-judge

LLM-as-a-judge 평가기는 언어 모델을 사용해 출력에 점수를 매깁니다. 이는 AI 애플리케이션/에이전트의 품질이 텍스트 출력의 품질을 채점하는 데 달려 있다는 근본적인 문제를 극복하기 위해 필요합니다.

이는 언어에 대한 이해가 필요한 품질을 평가하는 데 적합한 방법입니다. 예를 들어 응답이 질문과 관련이 있는지, 어조가 의도한 대상 독자와 맞는지, 요약이 원문의 핵심 내용을 담고 있는지 등입니다.

LLM 심사자는 완벽하지 않으며 잘못 사용하기 쉽습니다. 이는 다음을 의미합니다:

- 모델은 전문가가 가진 맥락을 갖고 있지 않기 때문에, 사람 전문가처럼 자동으로 채점하지 못합니다
- 실제로 여러분이 측정하고자 하는 것을 측정하고 있는지 검증하려면 사람의 선호도에 대해 [보정(calibration)](/guides/llm-as-a-judge-calibration-skill)이 필요합니다
- 특히 애플리케이션의 LLM과 같은 모델 계열을 사용하는 경우, LLM 심사자는 애플리케이션과 동일한 맹점을 공유할 수 있습니다

이러한 한계가 LLM 심사자를 피해야 할 이유가 되지는 않습니다. 사람 라벨에 대해 보정되고 코드 평가기 체크가 뒷받침하는 LLM 심사자는 신뢰할 수 있는 평가기입니다.

## 참조 기반 평가기 vs 참조 없는 평가기 [#reference-based-vs-reference-free]

코드 평가기와 LLM-as-a-judge 평가기는 모두 참조 기반(reference-based) 또는 참조 없는(reference-free) 방식이 될 수 있습니다. 참조 기반 평가기는 정답이나 골든 응답과 같이 미리 정의된 예상 출력과 실제 출력을 비교합니다. 참조 없는 평가기는 비교할 기준(ground truth) 없이 출력 자체만을 평가합니다.

|           | 코드 평가기           | LLM-as-a-judge / 수동                             |
| --------- | --------------------- | ------------------------------------------------- |
| 참조 기반 | 정확 문자열 일치 체크 | "이 응답이 특정 정책을 올바르게 설명하고 있는가?" |
| 참조 없음 | JSON 구조 검증        | 어조(tone-of-voice) 평가; 언어 감지               |

참조 없는 평가기의 장점은 아직 본 적 없는 프로덕션 데이터에도 적용할 수 있다는 점입니다. 반면 참조 기반 평가기는 항상 미리 정의된 참조 응답이 필요합니다.

## 실전에서

### 언제 평가기를 구축해야 하는가

[앞서 언급했듯이](#how-evaluation-typically-evolves) 항상 수동 검토부터 시작합니다. 이를 마치고 나면, 발견한 문제에 대해 자동화된 평가기를 구축해야 하는지가 다음 질문이 됩니다.

문제가 **일회성 수정**인지 **일반화가 필요한 문제**인지 스스로에게 물어보세요. 간단한 프롬프트 변경으로 해결된다면 그냥 변경하면 됩니다. 평가기가 필요하지 않습니다. 하지만 서로 다른 입력들에 대해 반복적으로 테스트하고 싶은 실패 모드를 명확히 식별할 수 있다면, 그때가 평가기를 구축할 시점입니다.

### 무엇을 평가해야 하는가

"유용성"이나 "품질" 같은 일반적인 속성은 시작점으로 매력적으로 보이지만, 유용한 신호를 만들어내는 경우는 드뭅니다. 모호한 기준을 체크하는 평가기는 모호한 결과를 낼 뿐입니다. 여러분의 애플리케이션에서 "좋음"과 "나쁨"이 무엇을 의미하는지 더 정밀하게 정의할수록 평가기가 더 유용해집니다.

<Callout type="info">
실전 조언 하나: 평가기를 설계할 때는 등급 척도(1~5점) 대신 이진 점수(합격/불합격)를 선호하세요. 이진 점수는 무엇이 허용 가능하고 무엇이 허용 불가능한지에 대한 명확한 정의를 강제합니다. 등급 척도는 3점과 4점의 차이가 무엇인지에 대한 모호함을 만들어내고, 이는 평가기 간, 그리고 시간에 따른 점수 해석과 일관성을 더 어렵게 만듭니다.
</Callout>

### 평가 방법 조합하기

관심 있는 각 품질 속성은 각자의 평가기를 갖습니다.

**가장 성숙한 평가 체계는 [세 가지 평가 방법](#three-kinds-of-evaluation)을 모두 사용합니다.** 이들을 함께 사용하면 애플리케이션의 전반적인 품질에 대한 관점을 얻을 수 있습니다.

이를 구체적으로 보여주기 위해, 세 가지 예시 애플리케이션과 각각의 평가기 스택을 소개합니다:

<Tabs items={["SQL agent", "Tone-of-voice rewriter", "Document extraction tool"]}>
<Tab>

_자연어 질문을 회사 데이터 웨어하우스에 대한 SQL로 변환하는 SQL 에이전트입니다. 사용자는 "지난 분기 지역별 매출이 어땠나요?"와 같은 질문을 하고, 에이전트는 쿼리를 생성하고 실행합니다._

| 품질                            | 방법           |
| ------------------------------- | -------------- |
| 쿼리가 문법적으로 유효한가      | 코드 평가기    |
| 쿼리가 사용자의 의도에 답하는가 | LLM-as-a-judge |
| 연속 샘플에 대한 보정           | 수동 검토      |

</Tab>
<Tab>

_초안 마케팅 카피를 받아 회사의 브랜드 어조에 맞게 다시 쓰는 어조 재작성기입니다. 출력은 순수한 산문 형태로, 확인할 구조도 없고 정확한 참조 답변도 없습니다._

| 품질                              | 방법           |
| --------------------------------- | -------------- |
| 어조가 브랜드 보이스와 일치하는가 | LLM-as-a-judge |
| 원문 메시지가 충실히 반영되었는가 | LLM-as-a-judge |
| 연속 샘플에 대한 보정             | 수동 검토      |

</Tab>
<Tab>

_업로드된 PDF에서 구조화된 필드(공급업체, 총액, 만기일)를 추출하는 인보이스 처리 도구입니다. 데이터셋에는 각 필드에 대한 정확 일치 라벨이 있습니다._

| 품질                                | 방법                                      |
| ----------------------------------- | ----------------------------------------- |
| 추출된 JSON이 스키마와 일치하는가   | 코드 평가기                               |
| 각 필드가 예상 값과 일치하는가      | 코드 평가기 (데이터셋 라벨과의 정확 일치) |
| 엣지 케이스 및 새로운 공급업체 형식 | 수동 검토 (샘플)                          |

</Tab>
</Tabs>

## 시작하는 방법

수동 검토부터 시작한 다음, 반복적으로 실행해야 하는 체크만 자동화하세요.

1. [출력을 수동으로 검토](#how-evaluation-typically-evolves)하여 애플리케이션에서 좋은 결과와 나쁜 결과가 어떤 모습인지에 대한 직관을 쌓습니다.
2. 포착하고 싶은 구체적인 실패 모드를 기록하고 최대한 명확하게 정의합니다.
3. 그 실패 모드를 많은 입력에 걸쳐, 또는 시간이 지나도 반복적으로 테스트해야 할 때만 자동화된 평가기를 구축합니다.

## 다음 단계

결과가 충분히 좋다면 변경 사항을 배포할 수 있습니다. 배포가 완료되면 루프는 다시 시작됩니다. 업데이트된 시스템은 새로운 [트레이스](/academy/tracing), 새로운 [모니터링](/academy/monitoring) 신호, 그리고 새로운 개선 기회를 만들어냅니다.

일부 평가기는 오프라인 실험을 넘어서도 사용해야 합니다. 참조 없는 평가기, 사용자 피드백 신호, 그리고 프로덕션에 안전한 다른 체크들은 실제 트래픽에 적용해 배포 전에 확인했던 품질이 프로덕션에서도 그대로 유지되는지 확인하는 데 사용할 수 있습니다.

프로덕션 동작이 기대에 부합한다면 더 큰 확신을 갖고 계속 확장할 수 있습니다. 그렇지 않다면, 해당 케이스를 트레이스에서 포착해 [데이터셋](/academy/datasets) 항목으로 만들고 다음 라운드의 [실험](/academy/experiments)을 실행하세요. 이렇게 루프를 닫습니다.

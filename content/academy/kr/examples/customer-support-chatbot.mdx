# 고객 지원 챗봇

<Callout type="info">
  이 예시는 [Langfuse Academy](/academy)의 개념을 설명하기 위한 것입니다.
</Callout>

## 배경

한 SaaS 기업이 고객 지원 채팅을 자동화하고자 합니다. 현재는 지원팀이 모든 대화에 직접 응답하고 있으며, 이미 해결된 티켓들이 수년치 티켓팅 시스템에 쌓여 있습니다. 이 봇은 회사를 대신해 고객과 대화하게 되므로, 잘못된 답변 하나가 해당 대화를 넘어서는 피해를 줄 수 있습니다. 고객이 지원팀 전체에 대한 신뢰를 잃을 수 있기 때문입니다.

이런 위험 때문에 곧바로 고객에게 배포하는 방식은 선택지가 될 수 없습니다. 대신 출시는 세 단계로 진행되며, 각 단계는 다음 단계로 넘어가기 위한 신뢰를 쌓는 과정입니다.

| 단계         | 현재 운영 중인 기능                                    | 다음 단계로 넘어가기 위한 조건                                                                    |
| ------------ | ------------------------------------------------------ | ------------------------------------------------------------------------------------------------- |
| 1. 오프라인  | 없음, 에이전트는 개발 중                               | 지원팀이 실제로 사용하고 싶어할 만큼 좋은 초안                                                    |
| 2. 내부 활용 | 에이전트가 답변 초안을 작성하고, 지원팀이 수정 후 발송 | 티켓 카테고리 전반에서 초안과 발송된 답변의 차이가 작게 유지되고, 지원팀의 정성적 피드백이 긍정적 |
| 3. 고객 대응 | 봇이 고객에게 직접 응답                                | 없음, 이것이 최종 목표이며 이 단계에서 에이전트는 지속적으로 개선됨                               |

이러한 단계적 접근 방식을 통해 위험을 크게 줄이면서도 프로덕션 신호를 조기에 확보할 수 있습니다.

## 1단계: 초기 에이전트 버전 만들기

지원팀이 실제로 내부 에이전트를 사용하게 하려면, 초기 버전부터 그들에게 유용해야 합니다. 이를 위해 이미 보유하고 있는 해결된 티켓의 과거 데이터를 활용할 수 있으며, 별도의 합성 데이터 생성은 필요하지 않습니다.

AI를 활용해 지원 티켓팅 시스템의 과거 데이터에 어떤 패턴이 존재하는지 파악하고, 이를 바탕으로 반복되는 행동 유형별로 몇 개의 데이터셋으로 그룹화할 수 있습니다.

<Callout type="info">
  여러 데이터셋으로 나눌지 하나로 유지할지는
  [트레이드오프](/academy/datasets#what-makes-a-good-dataset)입니다. 그룹화하면 사용 사례별로 성능을 측정할 수 있고 필요할 때 일부만 실행할 수 있습니다. 특히 초기에는 최적의 분할 기준이 아직 명확하지 않을 수 있으므로, 하나의 큰 데이터셋으로 시작하는 것도 합리적일 수 있습니다.
</Callout>

이제 프롬프트를 반복적으로 개선하면서 [각 버전을 이 데이터셋들에 대해 실행](/academy/experiments#how-experiments-are-used)하고, 초안이 지원팀이 실제로 유용하다고 느낄 만한 수준에 도달할 때까지 반복합니다. 각 실행은 다음 기준으로 평가됩니다.

<EvaluatorBlock
  evaluators={[
    {
      name: "resolution_match",
      type: "LLM-as-a-judge",

        "Does the draft refer to the same sources and recommend the same thing as the human reply did?",
      returns: "binary, per dataset item",
      details: (
        <>
          A{" "}
          <a href="/academy/evaluate#reference-based-vs-reference-free">
            reference-based
          </a>{" "}
          evaluator: the expected output gives it the sources and
          recommendation from the human reply to compare against. It has to be an LLM-as-a-judge
          rather than a code check because two replies can recommend the same
          thing in completely different words.
        </>
      ),
    },
    {
      name: "manual review",
      type: "Human",
      check: "Vibe check on a sample of drafts",
    },
  ]}
/>

여기서는 어조를 자동으로 평가하지 않는다는 점에 주목하세요. 이는 의도적인 선택입니다. 몇 가지 [수동 점검](/academy/evaluate#how-evaluation-typically-evolves)만으로도 초안의 어조가 받아들일 만한 수준인지 확인하기에 충분하며, 어조를 완벽하게 맞추는 작업은 2단계에서 진행합니다.

평가기 점수가 충분히 좋고 수동 검토 결과도 양호하면, 에이전트는 지원 티켓팅 시스템으로 이동합니다.

다음 섹션으로 넘어가기 전에, 트레이스가 어떤 모습일 수 있는지 살펴보겠습니다.

## 2단계: 내부 롤아웃

이 단계에서는 들어오는 모든 티켓에 대해 에이전트가 답변 초안을 작성하고, 지원팀 구성원이 이를 사용하거나 수정한 후 발송할 수 있습니다. 에이전트의 초안과 실제로 팀 구성원이 발송한 내용 간의 차이를 매우 신뢰할 수 있는 [사용자 신호](/academy/monitoring#user-feedback)로 활용할 수 있습니다.

또 다른, 더 명시적인 사용자 신호는 초안에 대한 좋아요/싫어요 버튼이며, 팀이 여기에 텍스트 피드백을 남길 수도 있습니다.

<AnnotatedLoop
  ariaLabel="2단계의 루프: 완전히 활성화되며, 초안과 발송된 답변의 차이로 구동됨"
  stations={{
    trace: { note: "모든 초안이 트레이싱됨" },
    monitor: { note: "모든 트레이스에 대한 edit_type" },
    dataset: { note: "많이 수정된 초안이 새 항목이 됨" },
    change: { note: "반복되는 수정 패턴에 대한 수정" },
    eval: { note: "배포 전 데이터셋 재실행" },
  }}
/>

두 신호 모두 모든 초안 [트레이스](/academy/tracing#anatomy-of-a-trace)에 점수로 기록됩니다.

<EvaluatorBlock
  evaluators={[
    {
      name: "edit_type",
      type: "LLM-as-a-judge",
      check: (
        <>
          사람의 수정을 <code>none</code>, <code>tone</code>,{" "}
          <code>corrected</code>, <code>added</code> 중 하나로 분류
        </>
      ),
      returns: "categorical, on the trace",
      details: (
        <>
          초안과 발송된 답변을 비교하는 판정 모델입니다. <code>tone</code>{" "}
          은 팀 구성원이 내용은 유지한 채 표현만 바꾼 경우, <code>corrected</code>{" "}
          는 잘못된 내용이 제거되거나 수정된 경우, <code>added</code>는 초안을 유지한 채
          누락된 정보를 추가한 경우를 의미합니다. 표현 변경과 내용 변경을 구분하려면
          두 텍스트를 모두 읽어야 하므로 판정 모델이 필요합니다.
        </>
      ),
    },
    {
      name: "human_feedback",
      type: "Human",
      check: "Thumbs up or down on the draft, with optional written feedback",
      returns: "binary plus comment, on the trace",
      details: (
        <>
          티켓팅 시스템의 버튼에서 직접 수집되며, 별도의 평가기가
          필요하지 않습니다. 클릭이 곧 점수를 기록합니다. 텍스트 코멘트는
          단순 비교만으로는 알 수 없는 맥락을 제공합니다.
        </>
      ),
    },
  ]}
/>

점수가 낮은 트레이스는 이후 검토하고 개선하여, 아직 다루지 않은 경우라면 데이터셋에 추가할 수 있습니다. 이를 위한 한 가지 방법은 [오류 분석](/academy/monitoring/error-analysis) 프로세스를 활용하는 것입니다.

루프의 한 번의 반복은 다음과 같은 모습입니다. [모니터링](/academy/monitoring#metrics-and-signals) 결과 `troubleshooting` 카테고리에서 `corrected` 초안이 급증한 것이 확인됩니다. 트레이스를 살펴보니 도움말 문서 인덱스가 오래되어 에이전트가 계속 구버전 내보내기 대화상자를 인용하고 있었습니다. 인덱스를 갱신한 후 `troubleshooting` 데이터셋을 재실행해 수정 사항을 확인하면, `corrected` 초안의 비율이 감소합니다.

수정된 초안의 비율은 시간이 지남에 따라 감소해야 합니다. 모든 티켓 카테고리에서 초안이 대체로 수정 없이 발송되면, 에이전트는 고객을 직접 상대할 준비가 된 것입니다.

## 3단계: 고객 대응 롤아웃

이제 봇이 고객에게 직접 응답하며, 2단계를 이끌던 신호는 사라집니다. 고객이 답변을 보기 전에 아무도 수정하지 않기 때문입니다. 대신 시간이 지남에 따라 학습하고 개선하는 데 도움이 되는 몇 가지 새로운 암묵적 사용자 신호를 도입합니다.

<Callout type="info">
  바로 완전 자동화로 전환하는 대신, 2단계 조건을 통과한 요청 카테고리만 자동화하고 나머지는 좀 더 오랫동안 지원팀을 거치도록 할 수도 있습니다.
</Callout>

<AnnotatedLoop
  ariaLabel="3단계의 루프: 사람의 수정 대신 암묵적 사용자 신호로 구동됨"
  stations={{
    trace: { note: "모든 고객 대화" },
    monitor: { note: "사용자 신호 및 리스크 모니터링" },
    dataset: { note: "문제가 있는 대화가 새 항목이 됨" },
    change: { note: "모니터링에서 드러난 문제에 대한 수정" },
    eval: { note: "배포 전 데이터셋 재실행" },
  }}
/>

고객이 지원 채팅을 직접 평가하는 경우는 드물기 때문에, 주된 초점은 암묵적 사용자 신호를 수집하는 데 있습니다.

<EvaluatorBlock
  evaluators={[
    {
      name: "handoff_requested",
      type: "LLM-as-a-judge",
      check: "Detects the customer asking for a human",
      returns: "binary, on the trace",
      details: (
        <>
          고객은 자유 텍스트로 사람과의 연결을 요청하기 때문에, 코드 기반
          평가기가 매칭할 대상이 없습니다. 트레이스에 점수를 매기면 고객이
          봇을 포기한 정확한 시점에 플래그를 지정할 수 있습니다.
        </>
      ),
    },
    {
      name: "question_repeated",
      type: "LLM-as-a-judge",
      check: "Detects the customer rephrasing the same question",
      returns: "binary, on the trace",
      details: (
        <>
          <a href="/academy/monitoring#user-feedback">암묵적 피드백</a>의
          한 형태입니다. 고객이 같은 말을 반복한다는 것은 이전 답변이
          제대로 전달되지 않았다는 의미입니다. 대화를 읽는 판정 모델만이
          표현을 바꾼 반복 질문과 후속 질문을 구분할 수 있습니다.
        </>
      ),
    },
    {
      name: "session_outcome",
      type: "LLM-as-a-judge",
      check: "Classifies how the conversation ended",
      returns: "categorical, on the session",
      details: (
        <>
          대화가 종료될 때 실행되며, 결과를{" "}
          <code>resolved</code>, <code>abandoned</code>,{" "}
          <code>handed_off</code> 중 하나로 분류합니다. 세 가지 결과가 각각
          봇의 성능에 대해 서로 다른 의미를 가지므로 범주형입니다. 결과는
          대화 전체에 속하는 값이므로 트레이스가 아닌 세션에 점수로
          기록됩니다.
        </>
      ),
    },
  ]}
/>

더 이상 사람이 루프에 개입하지 않으므로, 평판에 손상을 줄 수 있는 행동이 있는지 모든 답변을 모니터링합니다. 이 평가기들은 어떤 것도 차단하지 않습니다. 단지 발송된 답변에 플래그를 지정해 팀이 신속히 고객에게 후속 조치를 취할 수 있도록 할 뿐입니다.

<EvaluatorBlock
  evaluators={[
    {
      name: "data_leak",
      type: "LLM-as-a-judge",
      check:
        "Does the reply reveal data about anyone other than this customer?",
      returns: "binary, on the trace",
      details: (
        <>
          <a href="/academy/evaluate#reference-based-vs-reference-free">
            참조 없는(Reference-free)
          </a>
          방식이므로, 기대 출력값 없이도 모든 프로덕션 답변에 대해 실행할
          수 있습니다. 어떤 표현으로든 개인 정보를 인식하려면 언어 이해가
          필요하므로 LLM-as-a-judge를 사용합니다.
        </>
      ),
    },
    {

# 트레이싱

## 트레이싱이 루프에서 하는 역할

전통적인 소프트웨어는 대체로 결정론적이며, 실행이 미리 정의된 형식을 따릅니다. 하지만 LLM 애플리케이션은 그렇지 않습니다. 에이전트 실행은 지저분할 수 있으며, 우리는 예상치 못한 풍부한 입력과 출력, 그리고 실행 순서를 가진 창발적 동작을 다루고 있습니다. 에이전트의 동작을 따라가려면 다른 무언가가 필요합니다. 바로 [트레이스](/docs/observability/overview)입니다.

트레이스는 애플리케이션이 단일 요청에 대해 수행한 작업을 구조화된 형태로 기록한 것입니다. 어떤 단계를 거쳤는지, 어떤 데이터를 보았는지, 무엇을 생성했는지를 담고 있습니다.

<LoopDiagram highlight="trace" />

트레이싱은 전체 개선 루프의 중심입니다. 검토, 데이터셋 구축, 실험 실행, 평가 등 다른 모든 단계는 트레이스를 기반으로 동작합니다.

<Callout type="info">
이미 전통적인 관측성 개념에 익숙하다면, 이어지는 내용 중 일부는 반복처럼 느껴질 수 있습니다. 편하게 훑어보거나 건너뛰어도 좋습니다.
</Callout>

## 트레이스의 구조 [#anatomy-of-a-trace]

트레이스는 애플리케이션이 요구하는 만큼 복잡하거나 단순할 수 있지만, 모든 트레이스는 동일한 기본 구조를 공유합니다. 트레이스는 에이전트가 거쳐 간 경로를 나타내는 일련의 [관찰(observation)](/docs/observability/data-model) 집합으로 구성됩니다.

<Callout type="info">
  *관찰(observation)은 프로세스의 단일 단계입니다. 입력, 출력, 시작/종료 시간, 그리고
  해당 단계에서 발생한 일에 대한 메타데이터를 가지고 있습니다.*
</Callout>

### 계층 구조

트레이스는 계층적인 트리 구조를 가집니다. 그 안에는 다른 관찰을 포함할 수 있는 관찰들이 중첩되어 있으며, 이는 실제 AI 애플리케이션 실행을 그대로 반영하는 부모-자식 구조를 형성합니다.

<RagTraceViewDiagram />

어떤 순서로 무슨 일이 일어났는지, 그리고 어떤 단계가 더 큰 단계의 일부였는지를 확인할 수 있습니다.

### 관찰 데이터

**입력과 출력.** 모든 관찰은 입력과 출력을 가질 수 있습니다. 대부분의 경우 둘 다 가지고 있지만, 특정 경우에는 둘 중 하나만 가질 수도 있습니다. 해당 관찰에서 일어나는 액션의 유형에 맞는 입력 및/또는 출력을 설정하는 것이 해석 가능성 측면에서 중요합니다.

**관찰 유형.** 작업을 쉽게 구분할 수 있도록 다양한 [관찰 유형](/docs/observability/features/observation-types)을 확인할 수 있습니다. 각 관찰 유형은 에이전트의 서로 다른 종류의 상호작용을 포착하는 데 사용됩니다.

| 에이전트의 액션                     | 관찰 유형    | 일반적인 관찰 입력/출력                                                                                          |
| ----------------------------------- | ------------ | ---------------------------------------------------------------------------------------------------------------- |
| 언어 모델 호출                      | `generation` | 입력으로는 전체 프롬프트 또는 메시지 히스토리, 출력으로는 완성 결과, 그리고 모델명이나 토큰 수와 같은 메타데이터 |
| 외부 소스에서 정보를 가져오는 단계  | `retriever`  | 쿼리와 반환된 문서                                                                                               |
| 에이전트에 의한 도구 또는 함수 호출 | `tool`       | 호출된 도구, 인자, 반환 값                                                                                       |
| 일반적인 프로세스                   | `span`       | 사용 사례에 따라 크게 달라짐                                                                                     |

관찰 유형은 트레이스를 더 쉽게 읽고 필터링할 수 있게 해줍니다. 관찰이 20개 있는 트레이스에서 LLM 호출을 빠르게 찾아낼 수 있다면 시간을 절약할 수 있습니다.

### 비용, 지연 시간, 토큰 사용량

입력과 출력 외에도, LLM 애플리케이션이라면 당연히 갖춰야 할 관찰 속성이 몇 가지 있습니다. 바로 비용, 지연 시간, 그리고 [토큰 사용량](/docs/observability/features/token-and-cost-tracking)입니다. 이들은 관찰 단위로 기록되며 트레이스 수준에서 집계됩니다.

## 트레이스와 세션의 차이 [#traces-vs-sessions]

대부분의 경우 에이전트의 전체 생명주기 실행이 하나의 트레이스로 보이지는 않을 것입니다. 트레이스는 [세션](/docs/observability/features/sessions)으로 그룹화될 수 있습니다. 그렇다면 트레이스와 세션의 경계는 어디에 두어야 할까요?

<TracingHierarchyDiagram />

일반적인 경험 법칙은 다음과 같습니다. **하나의 트레이스는 시스템에 대한 한 번의 호출에 대응합니다.** 일반적으로 하나의 API 호출 또는 하나의 에이전트 실행입니다. 그리고 세션은 여러 트레이스를 하나로 묶는데, 예를 들어 멀티턴 대화의 모든 턴이 여기에 해당합니다.

이를 구체적으로 보여주기 위해, 두 가지 애플리케이션이 트레이스-세션 분할을 어떻게 설계했는지 예시를 들어보겠습니다.

<Tabs items={["고객 지원 챗봇", "PR 리뷰 에이전트"]}>
<Tab>

_SaaS 기업의 도움말 페이지에 내장된 고객 지원 챗봇입니다. 사용자는 계정, 청구, 제품 사용에 관한 질문을 하기 위해 이 챗봇을 엽니다. 일반적인 세션은 문제가 해결되거나 대화가 상담원에게 넘겨질 때까지 몇 차례의 주고받는 대화 턴으로 이루어집니다._

트레이스와 세션 분할은 다음과 같습니다.

```
Session: conversation_8f2a
├── Trace: "Why was I charged twice this month?"
├── Trace: "Can you refund the duplicate?"
└── Trace: "Thanks, when will I see it?"
```

이렇게 하면 특정 입력에 대한 챗봇 실행 평가를 하나의 독립된 트레이스에서 수행할 수 있습니다. 전체 대화를 살펴보고 싶다면 세션 전체를 확인하면 됩니다.

</Tab>
<Tab>

_열려 있는 풀 리퀘스트에 새로운 커밋이 푸시될 때마다 실행되는 자동화된 코드 리뷰 에이전트입니다. 매 실행마다 에이전트는 diff와 주변 파일을 읽고, 정적 검사를 수행하며, 인라인 리뷰 댓글을 게시합니다. PR은 작성자가 반복 작업을 하는 동안 그 생명주기에서 여러 차례의 리뷰 실행을 거치게 됩니다._

트레이스와 세션 분할은 다음과 같습니다.

```
Session: PR #1234
├── Trace: review run on commit a3f9b
├── Trace: review run on commit c7d2e
└── Trace: review run on commit 8b1f0
```

여기서는 하나의 리뷰 반복이 하나의 트레이스입니다. 댓글의 품질은 무엇을 읽었는지에 좌우됩니다. 리뷰 반복의 각 단계를 별도의 트레이스로 나누면 컨텍스트가 흩어져 리뷰를 따라가기 어려워질 것입니다.

</Tab>
</Tabs>

공통적인 트레이드오프는 다음과 같습니다. 너무 작게 나누면 트레이스가 하나의 이야기를 전달할 응집력을 잃고, 너무 크게 나누면 개별 실패가 읽기 어려운 트레이스 안에 묻혀버립니다.

## 어디서부터 시작할까

이제 막 시작하는 단계라면, 가능한 모든 경로를 다루려 하기보다 하나의 실제 워크플로를 처음부터 끝까지 계측하는 데 집중하세요.

1. 애플리케이션에서 중요한 요청 경로 하나에 대해 [트레이싱을 설정](/docs/observability/get-started)합니다.
2. 각 관찰이 해당 단계를 나타내는 데 유용한 입력, 출력, 메타데이터를 포착하도록 합니다.
3. 실제 트레이스를 몇 개 직접 검토하여 구조가 이해하기 쉽고 디버깅에 유용한지 확인합니다.

## 다음 단계

# 실험

## 실험이 루프에서 차지하는 위치

시스템이 어떻게 동작하는지 체계적으로 이해하고 개선하려면 원인과 결과를 분리할 수 있는 방법이 필요합니다. 실험이 제공하는 것이 바로 그것입니다. 변수 하나 또는 여러 변수를 선택하고, 데이터셋을 시스템의 두 가지 버전에 통과시킨 뒤, 나온 결과를 비교합니다. 그 결과는 변경 사항이 실제로 도움이 되었는지, 그리고 얼마나 도움이 되었는지를 알려줍니다. 얼마나 도움이 되었는지 파악하려면 실험 출력을 평가해야 합니다([평가하기 참고](/academy/evaluate)). 이 섹션에서는 평가에 앞서 체계적인 실험 진행 방법을 다룹니다.

## 실험의 구성 요소

모든 실험은 네 가지 구성 요소로 이루어집니다.

| 구성 요소       | 설명                                                                                                                             |
| --------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| **베이스라인**  | 현재 프로덕션 시스템 — 다른 모든 것을 비교하는 기준이 되는 대조군입니다. 한 가지를 변경하는 동안 이 값은 고정해 둡니다.          |
| **데이터셋**    | 두 조건 모두에 대해 실행하는 입력입니다. 시간이 지나도 결과를 비교할 수 있도록 실험 간에 동일한 데이터셋을 유지합니다.           |
| **변수**        | 변경하는 설정 — 모델, 프롬프트, 컨텍스트, 도구 접근 권한, 또는 에이전트 아키텍처입니다. 아래 [변수](#on-variables)를 참고하세요. |
| **비교할 출력** | 각 조건에서 시스템이 생성하는 결과입니다. 이를 비교하는 것이 실험을 실행하는 실질적인 작업입니다.                                |

한 번에 하나의 변수만 변경하는 것이 도움이 되는 경우가 많습니다. 다만 변수들은 서로 상호작용하며, 일부 설정은 여러 변수를 동시에 변경해야 할 수도 있습니다.

### 변수 [#on-variables]

- **모델.** 사용 중인 AI 모델입니다. 추론에 특화된 모델, 저렴한 모델, 빠른 모델이 있으며, 각각 결과 품질, 속도, 비용 면에서 서로 다른 트레이드오프를 가집니다.
- **프롬프트.** 가장 흔히 사용하는 레버입니다. 프롬프트 실험을 실행하기 전에 다음을 자문해 보세요. 실패의 원인이 명세 문제(모호하거나 불완전한 프롬프트)인지, 아니면 일반화 문제(모델이 명확한 지시를 일관성 없이 적용)인지 말이죠. 후자는 측정해 볼 가치가 있습니다.
- **컨텍스트.** 프롬프트에 포함하는 정보 — 검색된 문서, 대화 기록, 사용자 메타데이터입니다.
- **도구 접근 권한.** 도구를 추가하거나 제거하면 시스템이 취할 수 있는 경로가 바뀝니다.
- **에이전트 아키텍처.** 단일 에이전트 대 다중 에이전트, 어떤 프레임워크를 사용하는지, 작업을 어떻게 분해하는지입니다. 가장 큰 판돈이 걸려 있으면서도 분리하기 가장 어려운 변수입니다.

## 실험 활용 방법 [#how-experiments-are-used]

핵심 흐름은 다음과 같습니다. 변수를 선택하고 가설을 세운 뒤, 데이터셋에 대해 두 조건을 모두 실행하고, 출력을 비교하고, 무언가를 배운 다음, 이를 반복합니다.

![실험 루프](/images/academy/experimentation.png)

일반적인 시도에는 다음이 포함될 수 있습니다.

- 새 모델이 나왔다: 우리 시스템의 성능을 개선할까?
- 프롬프트를 변경하면 시스템의 출력 품질이 개선될까?
- 우리의 새 에이전트 하네스 아키텍처가 기존 다중 에이전트 시스템보다 더 나은 결과를 내고 있을까?

정성적인 방식으로 시작하세요. 동일한 입력에 대해 두 조건의 트레이스를 나란히 놓고 비교합니다. 이 방법으로 자신의 애플리케이션에서 "더 낫다"는 것이 무엇을 의미하는지 배우게 됩니다. 실제 출력을 정기적으로 읽지 않으면 메트릭을 잘못 해석하기 쉽습니다.

그런 다음 점수를 통해 비교를 구체화할 수 있습니다 — 승률, 승리가 입력 전반에 고르게 퍼져 있는지 아니면 특정 부분에 집중되어 있는지, 그리고 비용이나 지연 시간 트레이드오프입니다. 품질, 가격, 속도는 함께 움직이는 경우가 거의 없습니다. 실험은 이런 상충 관계를 추상적으로가 아니라 실제 데이터에서 보여줍니다.

## 시작하는 방법

더 많은 인프라를 구축하기 전에 작고 수동적인 비교부터 시작하세요. 트레이스를 나란히 놓고 비교하는 몇 가지 예시만으로도 첫 한 시간 동안 일주일간의 설정 작업보다 더 많은 것을 배울 수 있습니다.

1. 실제 예시 20~30개를 확보하세요. 프로덕션 트레이스에서 가져오거나 현실적인 예시를 직접 만드세요. 모든 경우를 다룰 필요는 없으며, 애플리케이션이 처리하는 상황의 실제 단면이면 충분합니다.
2. 설정을 변경하고 두 버전을 모두 실행하세요. 나머지는 모두 동일하게 유지합니다.
3. 트레이스를 나란히 놓고 읽으세요. 아직 평가기는 필요 없습니다. 그냥 읽으세요. 무엇이 다른가요? 어느 쪽이 실제로 더 낫고, 왜 그런가요? 실패의 유형에 주목하세요 — 프롬프트가 불명확한 것인지, 아니면 모델이 명확한 지시를 일관성 없이 적용하는 것인지. 이 구분에 따라 다음에 시도해야 할 수정 방식이 달라집니다.
4. 직관이 생기면 평가기를 추가하세요. 몇 차례 수동으로 진행하고 나면 무엇을 찾고 있는지 알게 될 것입니다. 그것을 코드화하세요. 이제 규모를 확장할 수 있습니다.

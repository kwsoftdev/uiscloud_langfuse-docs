# 모니터링

## 모니터링이 루프에서 하는 역할

[트레이싱](/academy/tracing)은 LLM 애플리케이션이 수행하는 모든 작업—모든 요청, 모든 모델 호출, 모든 도구 사용—에 대한 완전한 기록을 제공합니다. 모니터링은 그 데이터를 이해하는 방법입니다. 모니터링은 두 가지를 제공합니다: 시스템이 시간에 따라 어떻게 동작하는지에 대한 지속적인 관점, 그리고 조사할 가치가 있는 특정 트레이스—오류, 사용자 행동 패턴, 예상치 못하게 잘못된 경우—를 표면화하는 방법입니다. 이 두 가지가 함께 작용하면 단순히 데이터를 보유한 상태에서 벗어나 시스템을 실제로 개선할 수 있을 만큼 충분히 이해하는 단계로 나아가게 됩니다.

## 메트릭과 신호 [#metrics-and-signals]

모니터링을 두 가지 별개의 활동으로 구분하는 것이 도움이 됩니다. 각각이 답하는 질문이 다르기 때문입니다.

**집계 메트릭 추적(aggregate metrics tracking)**은 시간이 지남에 따라 상황이 좋아지고 있는지 나빠지고 있는지를 알려줍니다. 비용, 지연 시간, 평가 점수—이러한 값들은 지켜보고 추론할 수 있는 추세가 됩니다. 지난 화요일의 프롬프트 변경이 무언가를 개선했나요? 사용량이 늘어남에 따라 품질이 저하되고 있나요?

**신호 감지(signal detection)**는 지금 당장 어디를 봐야 하는지 알려줍니다. 조사할 가치가 있는 개별 트레이스—오류, 재시도 클러스터, 대화 중간에 이탈하는 사용자—를 표면화합니다. 신호는 그것을 유발한 특정 트레이스에 연결되어 있기 때문에 유용합니다. 그 트레이스가 바로 무엇이 잘못되었는지 이해하기 위한 시작점입니다.

## 메트릭과 신호는 어디에서 오는가

집계 메트릭과 신호 감지는 모두 관측(observation)에 첨부된 필드에 의존합니다. 제대로 계측만 하면 필요한 것의 상당 부분은 이미 확보됩니다: 지연 시간, 토큰 기반 비용, 모델 및 라우트 메타데이터, 도구 실행 결과, 오류는 일반적으로 별도의 배선 없이 클라이언트와 프로바이더 API로부터 흘러 들어옵니다.

내장 필드 외에도, 사용자 피드백(명시적 평점이나 세션 이탈과 같은 암묵적 신호), 사람에 의한 주석(human annotation), 자동화된 평가기 점수와 같은 평가(evaluation)를 추가할 수 있습니다. 트레이스에 수동으로 주석을 달거나 해당되는 곳에 평가기를 실행하면 됩니다. 이 데이터는 시간에 따른 추세를 추적하기 위한 집계 차트에 반영되고, 여러분이 신경 쓰는 임계값을 넘었을 때 개별 트레이스를 표면화하는 신호 규칙에도 반영됩니다.

### 명시적 및 암묵적 사용자 피드백 [#user-feedback]

사용자 피드백은 가장 풍부한 신호 소스 중 하나지만, 서로 다른 트레이드오프를 가진 두 가지 형태로 나뉩니다.

**명시적 피드백(explicit feedback)**은 직접적입니다: 좋아요/싫어요, 별점, 사용자가 남긴 댓글. 신호는 명확하지만, 응답률이 낮고 편향되어 있습니다—불만족한 사용자가 만족한 사용자보다 더 자주 응답합니다.

**암묵적 피드백(implicit feedback)**은 행동에서 도출됩니다: 사용자가 쿼리를 재시도했는지, 시스템에 동의하지 않았는지, 응답을 복사했는지, 제안을 수락했는지, 대화를 중간에 이탈했는지 등입니다. 사용자의 노력이 필요 없고 대량의 데이터를 생성하지만, 신호가 간접적이어서 해석이 필요합니다. 이러한 신호는 자동화된 평가기를 사용해 표면화할 수 있습니다.

이를 구체적으로 보여주기 위해, 명시적 피드백과 암묵적 피드백의 균형을 어떻게 맞추는지 보여주는 두 가지 예시 애플리케이션을 소개합니다:

_SaaS 회사의 고객센터에 내장된 고객 지원 챗봇. 사용자는 대화가 끝난 후 평가할 수 있으며, 채팅 중 언제든 사람에게 인계를 요청할 수 있습니다._

팀이 수집하는 피드백:

```
Explicit:  thumbs up / thumbs down at the end of a conversation
Implicit:  user requests human handoff mid-conversation
```

</Tab>
<Tab>

_업로드된 PDF에서 구조화된 필드(공급업체, 총액, 만기일)를 추출하는 인보이스 처리 도구. 각 추출 결과는 회계 시스템에 기록되기 전, 후속 승인 단계에서 사람이 검토합니다._

팀이 수집하는 피드백:

```
Explicit:  none — there's no point in the workflow where a rating fits
Implicit:  extracted field manually edited before approval
```

</Tab>

두 피드백 모두 점수(score)로 등록되므로, 다른 평가 데이터와 동일한 대시보드, 추세 차트, 신호 규칙에 반영됩니다. 어떤 피드백 신호를 자동화된 평가기로 전환할 가치가 있는지 파악하려면 [오류 분석](/academy/monitoring/error-analysis)에 대한 심층 자료를 참고하세요.

## 시작하는 방법

무엇이 중요할지에 대한 추상적인 아이디어가 아니라 실제 트레이스로부터 모니터링 설정을 구축하세요. 작게 시작하는 것이 좋습니다.

1. **먼저 데이터를 수동으로 살펴보세요.** 트레이스를 읽고 어떤 종류의 것들이 반복적으로 나타나는지 확인하세요. 무엇을 찾고 있는지 알기 전에는 유용한 모니터링을 설정할 수 없습니다.

2. **[오류 분석](/academy/monitoring/error-analysis)을 사용해 추적할 가치가 있는 것을 표면화하세요.** 오류 분석은 트레이스 전반에서 패턴을 찾는 구조화된 방법을 제공합니다. 지속적으로 실행할 수 있는 자동화된 평가기로 전환할 가치가 있는, 반복적으로 발생하는 종류의 문제들을 찾아낼 수 있습니다.

3. **여러분의 특정 애플리케이션에서 실패가 어떻게 나타나는지 생각해보세요.** 지원 채팅에서 사용자가 동의하지 않거나, 프로세스 자동화 흐름에서 수정이 발생하는 것과 같은 애플리케이션별 암묵적 신호는 일반적인 점수보다 더 실행 가능한 경우가 많으며, 수동 라벨링 없이도 문제를 표면화합니다.

4. **이를 반복적인 프로세스로 취급하세요.** 제대로 동작하는 모니터링 설정은 한 번 구성해두고 방치하는 것이 아닙니다. 사용 패턴이 변화하고, 모델이 업데이트되고, 새로운 실패 모드가 나타납니다. 노이즈를 걸러내고 실제로 중요한 것에 계속 집중할 수 있도록 설정을 지속적으로 다듬어야 합니다.

## 다음 단계

모니터링을 통해 조사할 가치가 있는 무언가가 표면화되면, 몇 가지 선택지가 있습니다: 원인이 명확하다면 직접 수정하거나, 패턴처럼 보인다면 데이터셋에 캡처하거나, 시스템적인 문제가 의심된다면 구조화된 평가를 실행하는 것입니다. 어느 경로를 택할지는 원인에 대한 확신 정도에 따라 달라집니다.

- [데이터셋](/academy/datasets): 평가를 위해 프로덕션 트레이스를 캡처하기

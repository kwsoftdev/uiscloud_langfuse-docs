# Langfuse API 호출에서 524 오류가 발생하는 이유는 무엇인가요?

Langfuse Cloud에서 524 오류 클래스는 ClickHouse가 쿼리를 처리하는 동안 리소스나 시간이 부족하여 요청이 타임아웃되었음을 나타냅니다.
이는 일반적으로 필터 조건이 지나치게 넓어 쿼리가 너무 많은 데이터를 스캔할 때 발생합니다.
이전 버전의 데이터 조회 엔드포인트의 경우, 기존 쿼리를 튜닝하는 대신 v2 데이터 API로 이전하는 것이 권장되는 해결책입니다. 기존 API는 대규모 환경에서 성능이 떨어지기 때문입니다.

## 일반적인 접근 방법

API 엔드포인트 전반에서 524 오류를 방지하려면:

1. **추출에는 v2 데이터 API를 사용하세요**: 행 단위 데이터에는 [Observations API v2 업그레이드 가이드](/docs/api-and-data-platform/features/observations-api#upgrade-from-older-reads)를 따르고, 집계 데이터에는 [Metrics API v2](/docs/metrics/features/metrics-api#v2)를 사용하세요.
2. **항상 엔드포인트의 타임스탬프 필터를 사용하세요**: v2 Observations는 `fromStartTime`과 `toStartTime`을 사용하며, Metrics API v2는 JSON `query` 내부에서 `fromTimestamp`와 `toTimestamp`를 사용합니다. 이전 버전의 트레이스 조회는 URL 파라미터로 `fromTimestamp`와 `toTimestamp`를 사용합니다.
3. **구체적인 필터를 추가하세요**: 쿼리 범위를 좁히기 위해 `userId`, `sessionId`, `name`, `tags` 등의 필터를 사용하세요.

## 이전 버전의 트레이스 조회

이전 버전의 트레이스 조회 엔드포인트는 계속 사용할 수 있지만, 새로운 데이터 추출 워크플로우의 기본값으로는 권장되지 않습니다. 특히 대규모 프로젝트에서는 524 오류가 발생하기 쉽습니다.

트레이스 및 observation 추출 워크플로우를 v2 API로 이전하려면 [Observations API v2 업그레이드 가이드](/docs/api-and-data-platform/features/observations-api#upgrade-from-older-reads)를 따르세요. 집계 리포팅에는 [Metrics API v2](/docs/metrics/features/metrics-api#v2)를 사용하세요.

### 추가 셀프 호스팅 옵션

이전 버전의 트레이스 조회를 여전히 사용하는 셀프 호스팅 배포 환경에서는 환경 변수를 통해 `GET /api/public/traces` 엔드포인트에 대한 서버 측 기본값과 제한을 설정할 수 있습니다.
적용 순서: 거부 → 기본 날짜 범위 → 기본 필드.

| 변수                                          | 유형                                     | 설명                                                                                                                                                                                                                                                         |
| --------------------------------------------- | ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `LANGFUSE_API_TRACES_REJECT_NO_DATE_RANGE`    | `"true"` / `"false"` (기본값: `"false"`) | `fromTimestamp` 파라미터를 포함하지 않은 요청을 HTTP 400으로 거부합니다.                                                                                                                                                                                     |
| `LANGFUSE_API_TRACES_DEFAULT_DATE_RANGE_DAYS` | 양의 정수 (선택 사항)                    | `fromTimestamp`가 제공되지 않은 경우, `toTimestamp`(또는 현재 시각)로부터 N일의 조회 기간을 자동으로 적용합니다. 거부가 활성화된 경우에는 무시됩니다.                                                                                                        |
| `LANGFUSE_API_TRACES_DEFAULT_FIELDS`          | 쉼표로 구분된 문자열 (선택 사항)         | 호출자가 `fields` 쿼리 파라미터를 지정하지 않은 경우 반환되는 기본 필드 그룹입니다. 유효한 그룹: `core`, `io`, `scores`, `observations`, `metrics`. 예: `"core"` 또는 `"core,io"`. 명시적으로 지정된 `fields` 쿼리 파라미터는 항상 이 기본값보다 우선합니다. |

**대규모 프로젝트를 위한 권장 시작 설정:** `LANGFUSE_API_TRACES_DEFAULT_FIELDS=core`와 `LANGFUSE_API_TRACES_DEFAULT_DATE_RANGE_DAYS=3`을 설정하세요.
이렇게 하면 기본 응답에서 `input`/`output`이 제외되며(사용자는 여전히 `fields=core,io`를 통해 명시적으로 요청할 수 있습니다), 스캔 범위가 제한됩니다.

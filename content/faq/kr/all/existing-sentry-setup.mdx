# Sentry와 함께 Langfuse 사용하기

이 가이드에서는 Sentry와 함께 Langfuse를 구성하는 방법을 다룹니다. 아직 읽지 않았다면 [기존 OpenTelemetry 설정에서 Langfuse 사용하기](/faq/all/existing-otel-setup)를 먼저 읽고 일반적인 개념을 이해하시기 바랍니다.

## Sentry가 Langfuse와 충돌할 수 있는 이유

Sentry의 JavaScript/Node SDK(v8 이상)와 Python SDK(v3 이상)는 `Sentry.init()`을 호출하면 OpenTelemetry를 자동으로 초기화합니다. 여기에는 다음이 포함됩니다.

- `SentrySpanProcessor` 생성
- 분산 트레이싱을 위한 `SentryPropagator` 설정
- `SentryContextManager` 설치
- 자신을 전역 TracerProvider로 등록

Sentry가 전역 TracerProvider를 "선점"하기 때문에, 이후에 Langfuse를 단순히 초기화하는 것만으로는 동작하지 않습니다. Langfuse의 스팬 프로세서가 Sentry가 제어하는 provider에 연결되지 않기 때문입니다.

> **참고:** Python `sentry_sdk` v2.x는 기본적으로 OpenTelemetry를 사용하지 **않습니다**. 대신 Sentry 자체의 계측 엔진을 사용하므로, Python v2.x 사용자는 일반적으로 별도의 설정이 필요 없으며 [옵션 A](#sentry-without-otel)를 바로 사용할 수 있습니다.

## 설정 옵션

### 옵션 A: OpenTelemetry 없이 Sentry 사용하기(가장 간단함) [#sentry-without-otel]

Sentry의 성능 트레이싱이 필요하지 않다면(즉, 주로 오류 모니터링 용도로만 Sentry를 사용한다면) Sentry의 OTEL 통합을 완전히 비활성화할 수 있습니다. Sentry와 Langfuse가 OpenTelemetry 레벨에서 전혀 상호작용하지 않으므로 충돌을 피하는 가장 간단한 방법입니다.

<LangTabs items={["JS/TS", "Python"]}>
<Tab>

```typescript
import * as Sentry from "@sentry/node";
import { NodeSDK } from "@opentelemetry/sdk-node";
import { LangfuseSpanProcessor } from "@langfuse/otel";

// 오류 모니터링 전용으로 Sentry 초기화, OTEL 및 트레이싱 없음
Sentry.init({
  dsn: process.env.SENTRY_DSN,
  skipOpenTelemetrySetup: true,
  // tracesSampleRate를 설정하지 마세요 - Sentry의 성능 트레이싱을 비활성화합니다
});

// Langfuse가 전역 TracerProvider를 소유하며 충돌이 없습니다
const sdk = new NodeSDK({
  spanProcessors: [new LangfuseSpanProcessor()],
});
sdk.start();
```

</Tab>
<Tab>

```python
import sentry_sdk
from langfuse import Langfuse

# Python sentry_sdk v2.x는 기본적으로 자체 계측을 사용하므로 OTEL 충돌이 없습니다
sentry_sdk.init(
    dsn=os.environ["SENTRY_DSN"],
    traces_sample_rate=1.0,  # Sentry 트레이싱은 OTEL과 독립적으로 동작합니다
)

# Langfuse는 정상적으로 초기화됩니다
langfuse = Langfuse()
```

내부적으로 OTEL을 사용하는 Python `sentry_sdk` v3 이상에서는 [옵션 B](#shared-tracer-provider) 또는 [옵션 C](#isolated-tracer-provider)를 대신 사용하세요.

</Tab>
</LangTabs>

### 옵션 B: 공유 TracerProvider [#shared-tracer-provider]

Sentry의 성능 트레이싱**과** Langfuse를 동일한 분산 트레이스 안에서 함께 사용해야 한다면, Sentry의 자동 OTEL 설정을 비활성화하고 두 프로세서를 모두 포함하는 공유 TracerProvider를 구성하세요.

<Callout type="warning" title="공유 분산 트레이스가 필요할 때만 옵션 B를 사용하세요">
  옵션 B를 사용하면 Langfuse는 Sentry의 활성 OpenTelemetry 컨텍스트를
  상속받습니다. 따라서 수신되는 `sentry-trace` 헤더가 Langfuse 트레이스
  ID와 샘플링 결정을 변경할 수 있습니다. 각 AI 작업에 대해 신뢰할 수
  있는 독립적인 Langfuse 트레이스를 원한다면
  [옵션 C](#isolated-tracer-provider)를 사용하는 것이 좋습니다.
</Callout>

```bash
npm install @sentry/opentelemetry
```

```typescript
import * as Sentry from "@sentry/node";
import { LangfuseSpanProcessor } from "@langfuse/otel";
import {
  SentryPropagator,
  SentrySampler,
  SentrySpanProcessor,
} from "@sentry/opentelemetry";
import { NodeTracerProvider } from "@opentelemetry/sdk-trace-node";

// 1단계: 자동 OTEL 설정 없이 Sentry 초기화
const sentryClient = Sentry.init({
  dsn: process.env.SENTRY_DSN,
  skipOpenTelemetrySetup: true,  // 중요: Sentry가 전역 provider를 선점하지 못하도록 방지합니다
  tracesSampleRate: 1.0,
});

// 2단계: 두 프로세서를 모두 포함하는 공유 TracerProvider 생성
const provider = new NodeTracerProvider({
  sampler: sentryClient ? new SentrySampler(sentryClient) : undefined,
  spanProcessors: [
    // Langfuse 프로세서 - 기본 스마트 필터(Langfuse + GenAI/LLM 스팬)
    new LangfuseSpanProcessor(),
    // Sentry 프로세서 - 모든 스팬을 수신합니다
    new SentrySpanProcessor(),
  ],
});

// 3단계: Sentry의 propagator 및 context manager로 등록
provider.register({
  propagator: new SentryPropagator(),
  contextManager: new Sentry.SentryContextManager(),
});
```

#### Sentry의 샘플링 비율이 Langfuse에 영향을 미침

공유 TracerProvider를 사용할 때, Sentry의 `tracesSampleRate`는 Langfuse로 전송되는 트레이스를 포함한 **모든** 트레이스에 적용됩니다.

```typescript
Sentry.init({
  tracesSampleRate: 0.1,  // 트레이스의 10%만 생성됩니다
  // ...
});
```

이 값을 `0.1`로 설정하면 LLM 호출의 10%만 Langfuse에 표시됩니다. Sentry는 샘플링하면서 모든 트레이스를 Langfuse로 보내려면 대신 [독립 TracerProvider](#isolated-tracer-provider) 방식(옵션 C)을 사용하세요.

#### 수신되는 `sentry-trace` 헤더도 Langfuse에 영향을 미침

`SentryPropagator`가 활성화되어 있으면 백엔드 요청은 수신된
Sentry 분산 트레이스를 계속 이어갑니다. 해당 컨텍스트에서 생성된 Langfuse 스팬은
상위 트레이스 ID와 상위 샘플링 결정을 모두 상속받습니다.

- 수신된 트레이스가 샘플링되지 않은 경우, 백엔드 스팬이 기록되지 않는 상태가 되어 Langfuse에 아무것도 표시되지 않을 수 있습니다.
- 여러 백엔드 작업이 동일한 상위 트레이스를 이어받는 경우, 하나의 Langfuse 트레이스로 병합되어 나타날 수 있습니다.

AI 워크플로에 대해 Langfuse가 항상 별도의 트레이스를 생성하도록 하려면
[옵션 C](#isolated-tracer-provider)를 사용하는 것이 좋습니다. 앱의 나머지 부분에는 옵션 B가 필요하다면,
새로운 루트 컨텍스트에서 시작하여 특정 Langfuse 트레이스 블록만 분리할 수 있습니다.

```typescript
import { context, ROOT_CONTEXT } from "@opentelemetry/api";
import { startActiveObservation } from "@langfuse/tracing";

await context.with(ROOT_CONTEXT, async () => {
  await startActiveObservation("generateText", async () => {
    // Langfuse로 트레이싱되는 AI 호출
  });
});
```

이렇게 하면 해당 블록에 대해 수신된 Sentry 트레이스와의 부모 관계가 의도적으로 끊어집니다.

#### Langfuse 스팬 필터링

Langfuse는 이미 기본적으로 LLM 중심의 필터를 적용합니다. 대부분의 Sentry 설정에서는 추가 필터링 코드가 필요하지 않습니다.

더 엄격한 라우팅이 필요하다면 `shouldExportSpan`을 지정할 수 있습니다.

```typescript
new LangfuseSpanProcessor({
  shouldExportSpan: ({ otelSpan }) =>
    otelSpan.instrumentationScope.name === "langfuse-sdk",
}),
```

이렇게 하면 Sentry는 여전히 모든 것을 받는 동안, Langfuse에는 Langfuse SDK 스팬만 유지됩니다.

디버깅 중에 모든 것을 내보내려면 일시적으로 `shouldExportSpan: () => true`로 설정하세요.

Langfuse에 포함하고 싶은 내용에 따라 허용할 [스코프](/faq/all/existing-otel-setup#instrumentation-scopes)를 조정하세요. Langfuse UI에서 임의의 스팬을 클릭하고 `metadata.scope.name`을 확인하면 해당 스팬의 스코프 이름을 찾을 수 있습니다.

#### 필요한 Sentry 구성 요소

`skipOpenTelemetrySetup: true`를 사용할 때는 Sentry의 모든 OTEL 구성 요소를 수동으로 설정해야 합니다.

| 구성 요소              | 목적                                       |
| ---------------------- | ------------------------------------------ |
| `SentrySampler`        | Sentry의 샘플링 결정을 적용합니다          |
| `SentrySpanProcessor`  | 스팬을 Sentry로 전송합니다                 |
| `SentryPropagator`     | 분산 트레이싱 헤더를 처리합니다            |
| `SentryContextManager` | Sentry를 위한 비동기 컨텍스트를 관리합니다 |

이 중 하나라도 누락하면 Sentry의 트레이싱이 올바르게 작동하지 않을 수 있습니다.

### 옵션 C: 독립 TracerProvider [#isolated-tracer-provider]

Sentry와 Langfuse 스팬 간의 분산 트레이싱이 필요하지 않다면, Langfuse에 완전히 [독립된 TracerProvider](/docs/observability/sdk/advanced-features#isolated-tracer-provider)를 사용할 수 있습니다. Langfuse 트레이스를 Sentry의 샘플링 및 전파 동작과 무관하게 유지할 수 있으므로 대부분의 AI 애플리케이션에 권장되는 설정입니다.

<LangTabs items={["JS/TS", "Python"]}>
<Tab>

```typescript
import { NodeTracerProvider } from "@opentelemetry/sdk-trace-node";
import { LangfuseSpanProcessor } from "@langfuse/otel";
import { setLangfuseTracerProvider } from "@langfuse/tracing";
import * as Sentry from "@sentry/node";

// Sentry는 자체 자동 OTEL 설정을 사용합니다
Sentry.init({
  dsn: process.env.SENTRY_DSN,
  tracesSampleRate: 1.0,
  // skipOpenTelemetrySetup을 설정하지 않아 Sentry가 전역 provider를 관리하도록 둡니다
});

// Langfuse는 완전히 별도의 provider를 사용합니다
const langfuseProvider = new NodeTracerProvider({
  spanProcessors: [new LangfuseSpanProcessor()],
});
setLangfuseTracerProvider(langfuseProvider);
```

</Tab>
<Tab>

```python
import sentry_sdk
from opentelemetry.sdk.trace import TracerProvider
from langfuse import Langfuse

# Sentry를 정상적으로 초기화합니다
sentry_sdk.init(
    dsn=os.environ["SENTRY_DSN"],
    traces_sample_rate=1.0,
)

# Langfuse는 완전히 별도의 provider를 사용합니다
langfuse = Langfuse(tracer_provider=TracerProvider())
```

</Tab>
</LangTabs>

#### 장단점

- 더 간단한 구성
- Sentry의 샘플링이 Langfuse 트레이스에 영향을 주지 않습니다
- Langfuse와 Sentry 트레이스가 컨텍스트를 공유하지 않습니다
- 부모 스팬이 Sentry의 provider에 있는 경우 일부 스팬이 Langfuse에서 고아 상태로 나타날 수 있습니다

## 일반적인 문제

#### Sentry 추가 후 Langfuse에 트레이스가 표시되지 않음

**원인:** Langfuse가 프로세서를 연결하기 전에 Sentry가 OTEL을 초기화했습니다.

**해결 방법:** 옵션 A(OTEL 없이 Sentry 사용), `skipOpenTelemetrySetup: true`를 사용하는 옵션 B(공유 설정), 또는 옵션 C(독립 provider)를 사용하세요.

#### skipOpenTelemetrySetup을 설정하면 Sentry 트레이싱이 중단됨

**원인:** 필요한 모든 Sentry OTEL 구성 요소를 수동으로 설정하지 않았습니다.

**해결 방법:** 옵션 B에 나온 것처럼 `SentryPropagator`와 `SentryContextManager`로 provider를 등록했는지 확인하세요.

#### 인프라 스팬이 Langfuse에 표시됨

**원인:** 커스텀 `shouldExportSpan`이 지나치게 허용적입니다.

**해결 방법:** `shouldExportSpan` 규칙을 더 엄격하게 조정하세요.

#### 일부 트레이스만 Langfuse에 표시됨

**원인:** 옵션 B에서는 Langfuse가 Sentry의 샘플링 결정을 상속받습니다. 이는 Sentry의 로컬 `tracesSampleRate`에서 비롯되거나, 이미 업스트림에서 샘플링되지 않은 상태로 수신된 `sentry-trace` 헤더에서 비롯될 수 있습니다.

**해결 방법:** 모든 트레이스를 원한다면 `tracesSampleRate: 1.0`으로 설정하고 업스트림 요청이 샘플링되도록 하거나, 공유 샘플링 문제를 완전히 피하려면 옵션 C(독립 provider)를 사용하세요.

#### Sentry로 계측된 프런트엔드에서 오는 요청에 대해 Langfuse에 트레이스가 표시되지 않음

**원인:** 옵션 B에서는 `SentryPropagator`가 수신된 `sentry-trace` 헤더를 이어받습니다. 업스트림 트레이스가 샘플링되지 않았다면 백엔드 스팬이 기록되지 않는 상태가 되어 Langfuse가 아무것도 받지 못합니다.

**해결 방법:** Langfuse가 자체 트레이스를 생성하도록 하려면 옵션 C를 사용하는 것이 좋습니다. 다른 곳에서 옵션 B가 필요하다면, Langfuse 트레이스 블록을 `context.with(ROOT_CONTEXT, ...)`로 감싸서 수신된 Sentry 트레이스로부터 분리하세요.

#### 별개의 Langfuse 트레이스가 하나로 병합되어 표시됨

**원인:** 옵션 B에서는 여러 백엔드 작업이 동일한 수신 `sentry-trace` 헤더를 이어받아 동일한 트레이스 ID를 공유할 수 있습니다.

**해결 방법:** 독립적인 Langfuse 트레이스를 원한다면 옵션 C를 사용하거나, 수신된 Sentry 트레이스로부터 분리해야 하는 경우 해당 Langfuse 트레이스 블록을 새로운 `ROOT_CONTEXT`에서 시작하세요.

## AWS Lambda 관련 고려 사항

AWS Lambda와 같은 서버리스 환경에서는 추가 설정이 필요할 수 있습니다.

```typescript
new LangfuseSpanProcessor({
  exportMode: "immediate",  // 배치 처리하지 않음 - Lambda가 정지되기 전에 내보냅니다
}),
```

`exportMode: "immediate"` 설정은 스팬을 배치로 처리하지 않고 즉시 내보내도록 보장합니다. Lambda가 배치된 스팬이 플러시되기 전에 실행 컨텍스트를 정지시킬 수 있기 때문에 이 설정이 중요합니다. 자세한 내용은 [Langfuse가 스팬을 캡처하고 전송하는 방법](/docs/observability/data-model#background-processing)을 참고하세요.

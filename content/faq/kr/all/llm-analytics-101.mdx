# LLM 분석 101 — LLM 애플리케이션을 측정하고 개선하는 방법

<Frame fullWidth>
  ![LLM Analytics 101](/images/blog/faq/llm-analytics.png)
</Frame>

_이 가이드는 LLM 애플리케이션 계층에서 빌드하는 개발자들에게 트레이싱과 분석이 LLM 애플리케이션을 개선하는 데 있어 **왜**, **무엇을**, **어떻게** 필요한지에 대한 이해를 제공합니다_

## LLM은 소프트웨어 제공 방식을 바꿨습니다

생성형 AI의 출력은 결정론적이지 않습니다. 즉, 신뢰할 수 있게 예측할 수 없습니다. 이는 더 '전통적인' 소프트웨어 엔지니어링과 비교했을 때 소프트웨어가 제공되는 방식을 바꿉니다. 출력이 어떤 모습일지, 그리고 무엇이 '_좋은_' 출력인지 명확하지 않다면, 코드를 배포하기 전에 품질을 보장하고 견고한 테스트를 구축하기가 더 어렵습니다.

LLM 애플리케이션 계층에서는 광범위한 소프트웨어 설계와 테스트 대신 프로덕션 데이터로부터 학습하는 방식이 자리 잡았습니다. 하지만 프로덕션으로부터 학습하려면 LLM을 트레이싱하고 무엇이 효과가 있고 무엇이 없는지 분석해야 합니다.

## LLM 앱 트레이싱 - 무엇이 다른가?

LLM 기반 앱을 구축한다는 것은 여러 복잡한 요소와 상호작용을 코드에 통합하는 것을 의미합니다. 여기에는 체인, 에이전트, 다양한 기반 모델, 도구, 임베딩 검색과 라우팅이 포함될 수 있습니다. 전통적인 로깅 및 분석 도구는 LLM과 상호작용하는 이러한 새로운 방식을 수집, 표시, 분석하기에 적합하지 않습니다.
새로운 로깅 스택은 처음부터 LLM 네이티브 방식으로 설계되어야 합니다. 즉, 호출을 그룹화하고 팀이 이해하고 디버깅할 수 있는 방식으로 시각화해야 한다는 의미입니다.

## 본격적으로: 무엇을 측정해야 할까요?

```ts
// Example generation creation
import { startObservation } from "@langfuse/tracing";

const generation = startObservation(
  "chat-completion",
  {
    model: "gpt-3.5-turbo",
    modelParameters: {
      temperature: 0.9,
      max_tokens: 2000,
    },
    input: messages,
  },
  { asType: "generation" },
);
```

LLM 기반 앱을 개선하기 위한 기본 요구 사항은 활동을 트레이싱하는 것입니다. 하지만 이는 무엇을 의미하며, 무엇을 기록해야 할까요? 최전선에서 LLM을 다루는 사용자들과 함께 작업하면서, 우리는 추적해야 할 다섯 가지 메트릭을 확인했습니다.

- **볼륨:** 다른 모든 메트릭의 기반입니다 - 모든 LLM 호출과 그 내용을 추적하고, 프롬프트와 완료 결과 모두에 관련 메타데이터를 첨부합니다.
- **비용:** 각 호출의 비용을 계산하기 위해 토큰 수와 가격을 기록합니다. 셀프 호스팅 모델의 경우 GPU 사용 시간과 가격을 추적합니다.
- **지연 시간:** 모든 호출의 지연 시간을 측정합니다. 이 데이터를 사용해 어떤 단계가 지연 시간을 늘리는지 분석하고 사용자 경험 개선을 시작합니다.
- **품질:** 사용자 피드백을 적극적으로 수집하고, 수동 평가를 수행하며, 모델 기반 평가를 사용해 출력에 점수를 매깁니다.
- **오류/예외:** 시스템적 문제를 나타내는 타임아웃과 속도 제한(rate limit) 같은 HTTP 오류를 모니터링합니다.

## KPI를 통한 효과적인 분석 구현

성공적인 팀들은 위의 다섯 가지 메트릭(볼륨, 비용, 지연 시간, 품질, 오류)을 다음 기준으로 세분화하여 모범 사례 KPI를 구현하는 것을 확인했습니다.

- **유스 케이스:** 사용자가 LLM과 어떻게 상호작용하는지 이해하기 위해 프롬프트와 완료 결과를 유스 케이스별로 클러스터링합니다
- **모델과 구성:** 서로 다른 모델과 모델 구성이 품질, 지연 시간, 오류에 어떤 영향을 미치는가?
- **체인과 단계:** 체인을 더 깊이 파고들어 무엇이 성능을 좌우하는지 이해합니다
- **사용자 데이터:** 특정 특성에 따라 사용자를 그룹화하여 제품 내 페르소나와 특정 사용자층에 대한 인사이트를 얻습니다
- **체인과 단계:** 체인을 더 깊이 파고들어 무엇이 성능을 좌우하는지 이해합니다
- **모델과 구성:** 서로 다른 모델과 모델 구성이 품질, 지연 시간, 오류에 어떤 영향을 미치는지 추적합니다
- **유스 케이스:** 사용자가 LLM과 어떻게 상호작용하는지 이해하기 위해 프롬프트와 완료 결과를 유스 케이스별로 클러스터링합니다
- **시간:** 시간 경과에 따라 KPI를 검사하고 추세를 파악합니다
- **버전:** 프롬프트, 체인, 소프트웨어 릴리스를 버전별로 추적하고 성능 변화를 이해합니다
- **지역:** 특히 지연 시간에 중요합니다
- **언어:** 사용자 언어별로 앱이 얼마나 잘 작동하는지 이해합니다

## 단계별 가이드: LLM 애플리케이션에 트레이싱 및 분석 구현하기

1. **목표 정의:** 무엇을 달성하고 싶은지, 그리고 목표가 사용자의 요구 사항과 어떻게 부합하는지 파악합니다. 위의 메트릭을 애플리케이션에 맞는 고유한 KPI를 정의하는 출발점으로 삼습니다.
2. **트래킹 통합:** 이는 백엔드 실행과 점수(예: 프론트엔드에서의 사용자 피드백 수집)를 포함합니다.
3. **검사 및 디버깅:** 시각적 UI를 통해 런타임 트레이스를 검사하여 사용자를 이해합니다
4. **분석:** 모델/사용자 및 시간별 비용, 제품 기능별 비용, 체인 단계별 지연 시간을 측정하는 것부터 시작하고, 실험 또는 프로덕션 버전별로 그룹화된 품질/지연 시간/비용을 산점도로 분석합니다.

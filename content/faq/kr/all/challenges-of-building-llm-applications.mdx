# LLM 애플리케이션을 구축할 때 흔히 겪는 어려움은 무엇인가요?

검색 증강 생성(retrieval augmented generation), 내부 도구 및 API를 사용하는 에이전트, 백그라운드 추출/분류 작업 등 널리 사용되는 LLM 사용 사례를 구현할 때, 개발자는 전통적인 소프트웨어 엔지니어링과는 다른 독특한 과제에 직면합니다.

**트레이싱 & 제어 흐름(Control Flow)**: 가치 있는 LLM 애플리케이션 상당수는 파운데이션 모델에 대한 복잡하고 반복적이며 연쇄적이거나 에이전틱한 호출에 의존합니다. 이로 인해 확장된 제어 흐름에서 문제의 근본 원인을 정확히 찾아내기 어려워, 이러한 애플리케이션을 디버깅하는 일이 힘들어집니다.

_Langfuse를 사용하면_ LLM 애플리케이션의 전체 컨텍스트를 손쉽게 캡처할 수 있습니다. 저희 클라이언트 SDK와 통합은 모델 및 프레임워크에 구애받지 않으며, 실행의 전체 컨텍스트를 캡처할 수 있습니다. 사용자는 일반적으로 LLM 추론, 임베딩 검색, API 사용량, 그리고 문제를 정확히 찾아내는 데 도움이 되는 내부 시스템과의 기타 상호작용을 추적합니다. LangChain과 같은 프레임워크 사용자는 자동 계측(instrumentation)의 혜택을 받을 수 있으며, 그렇지 않은 경우에도 SDK를 통해 Langfuse가 추적할 단계를 인체공학적으로 정의할 수 있습니다.

**출력 품질**: 전통적인 소프트웨어 엔지니어링에서 개발자는 예외의 부재와 테스트 케이스 준수 여부를 테스트하는 데 익숙합니다. LLM 기반 애플리케이션은 비결정적(non-deterministic)이며, 품질을 평가할 확고한 표준이 거의 존재하지 않습니다. 애플리케이션의 품질, 특히 대규모 환경에서의 품질을 이해하고 '좋은' 평가가 무엇인지 파악하는 것이 주요 과제입니다. 이 문제는 사용자가 통제할 수 없는 호스팅 모델의 변경으로 인해 더욱 가속화됩니다.

_Langfuse를 사용하면_ 사용자는 프로덕션 트레이스(또는 그 하위 단계에도)에 점수를 부여하여 품질 측정에 한 걸음 더 다가갈 수 있습니다. 사용 사례에 따라 이러한 점수는 모델 기반 평가, 사용자 피드백, 수동 라벨링, 또는 기타 암묵적 데이터 신호 등을 기반으로 할 수 있습니다. 이렇게 얻은 메트릭은 이후 프로덕션에 배포된 변경 사항의 영향을 파악하고자 할 때, 시간 경과에 따른 품질, 특정 사용자별 품질, 애플리케이션 버전/릴리스별 품질을 모니터링하는 데 사용할 수 있습니다.

**혼재된 의도(Mixed intent)**: 많은 LLM 애플리케이션은 사용자 입력을 엄격하게 제한하지 않습니다. 대화형 및 에이전틱 애플리케이션은 흔히 매우 다양한 입력과 사용자 의도를 다뤄야 합니다.
이는 하나의 과제를 제기합니다. 팀은 자신들만의 사고 모델을 바탕으로 애플리케이션을 구축하고 테스트하지만, 실제 사용자는 종종 다른 목표를 가지고 있어 놀랍고 예상치 못한 결과로 이어지곤 합니다.

_Langfuse를 사용하면_ 사용자는 애플리케이션의 일부로 입력을 분류하고 이 추가 컨텍스트를 수집하여, 이후 사용자 행동을 심층적으로 분석할 수 있습니다.

# Langfuse Cloud Fast Preview에서 관측 탐색하기

## 새로운 모델을 이해하는 방법

Langfuse의 다음 메이저 버전(v4)에서는 애플리케이션이 수행하는 모든 작업 — LLM 호출, 도구 실행, 검색, 에이전트 단계 — 이 하나의 관측 테이블에서 1급 행(row)으로 취급됩니다. 각 행이 하나의 작업을 나타내고 어떤 열로도 필터링할 수 있는 스프레드시트를 떠올려 보세요. 트레이스는 관련된 작업들을 묶는 하나의 열(`trace_id`)에 불과하며, 이는 `session_id`나 `user_id`가 항목을 묶는 방식과 동일합니다.

에이전틱 애플리케이션이 점점 복잡해짐에 따라 하나의 트레이스에 수백, 수천 개의 작업이 포함될 수 있습니다. 기존 트레이스 테이블은 요청당 한 행만 보여줬기 때문에 비용이 많이 든 LLM 호출이나 실패한 도구 호출을 찾으려면 트레이스를 하나씩 열어 하위 항목을 확인해야 했습니다. 관측 테이블에서는 어떤 트레이스에 속하는지 알 필요 없이 모든 작업에 걸쳐 직접 쿼리할 수 있습니다. 예를 들어 $0.10를 초과하는 모든 GPT-4 호출, 2초 이상 걸린 모든 검색, 오류가 발생한 모든 도구 호출을 찾을 수 있습니다.

이전보다 테이블에 더 많은 행이 보이는 것은 정상입니다. 이제 각 행은 하나의 트레이스가 아니라 하나의 작업을 나타내기 때문입니다. 핵심은 필터와 [저장된 뷰(saved view)](/changelog/2025-05-20-save-table-views)를 사용해 중요한 부분에 집중하는 것입니다. 관측 이름, 유형, 모델을 기준으로 필터링해 관심 있는 작업을 찾아낸 다음 뷰로 저장하세요. 필터 사이드바에는 사용 가능한 값이 개수와 함께 표시되므로, 필터를 적용하기 전에 어떤 모델, 관측 이름, 오류 레벨이 가장 흔한지 미리 파악할 수 있습니다.

개념적인 배경 지식은 [관측 중심 데이터 모델](/docs/v4#under-the-hood)을 참고하세요. 더 폭넓은 릴리스 맥락은 [Fast Preview 문서](/docs/v4)를 참고하세요.

## 자주 사용하는 뷰

다음은 유용한 출발점들입니다. 각각을 [저장된 뷰](/changelog/2025-05-20-save-table-views)로 저장해두면 한 번의 클릭으로 접근할 수 있습니다.

| 보고 싶은 것…                    | 필터링 방법                                                                                                       |
| -------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| 핵심 작업                        | 가장 중요한 작업과 일치하는 관측 `name` 또는 `type`으로 필터링                                                    |
| 모든 LLM 생성(generation)        | `type = generation`으로 필터링                                                                                    |
| 비용이 많이 드는 LLM 호출        | `type = generation`으로 필터링한 뒤 `total_cost` 기준 내림차순 정렬                                               |
| 특정 사용자의 오류               | `user_id` 및 `level = ERROR`로 필터링                                                                             |
| 특정 세션에 한정된 작업          | `session_id`로 필터링한 뒤 `latency` 기준 내림차순 정렬                                                           |
| 하나의 트레이스에 속한 모든 작업 | `trace_id`로 필터링                                                                                               |
| 애플리케이션 진입점              | `Is Root Observation = true`로 필터링                                                                             |
| 모델별 지연 시간(latency) 비교   | `type = generation`으로 필터링한 뒤 [대시보드](/docs/metrics)에서 `model` 기준으로 그룹화하거나 `model` 열로 정렬 |
| 차원별 비용, 지연 시간, 오류     | [셀프서브 대시보드](/docs/metrics)                                                                                |

가장 자주 사용하는 디버깅 워크플로우에 대해 2~3개의 저장된 뷰를 미리 설정해두면 팀이 관측 테이블을 곧바로 유용하게 사용할 수 있습니다.

## 루트 관측부터 시작하기

Python SDK v4.7.0 이상 또는 JS/TS SDK v5.4.0 이상으로 수집(ingest)하는 프로젝트의 경우, 저장된 뷰나 공유된 필터 링크, 기타 테이블 상태 없이 관측 테이블을 열면 Langfuse가 `Is Root Observation = true`를 자동으로 적용합니다. 이를 통해 하위 작업을 살펴보기 전에 애플리케이션 진입점만 모은 목록을 먼저 볼 수 있습니다.

이 필터에는 두 종류의 루트가 모두 포함됩니다.

- **외부 루트(Outer roots)**: 부모가 없는 최상위 관측입니다.
- **앱 루트(App roots)**: SDK 스팬 필터링으로 제외된 부모 아래에서 가장 상위에 위치한, 내보내진(exported) 애플리케이션 관측입니다.

앱 루트는 OpenTelemetry 계측(instrumentation)이나 인프라 스팬이 애플리케이션 스팬의 외부 부모일 때 중요해집니다. Langfuse SDK가 해당 부모를 내보내기(export)에서 제외하더라도, 내보내진 자식 스팬은 여전히 부모 스팬 ID를 가지고 있습니다. 부모가 없는 관측만을 기준으로 필터링하면 애플리케이션 진입점을 놓치게 됩니다. 앱 루트를 인식하는 SDK는 해당 진입점을 표시해 `Is Root Observation`이 외부 루트와 함께 이를 포함할 수 있도록 합니다.

이 필터는 사이드바에 표시되며 제거하면 모든 작업을 확인할 수 있습니다. Langfuse는 이 선택을 기억합니다. 기존에 저장된/기본 뷰와 테이블 필터가 포함된 링크는 각자의 상태를 유지합니다. 자동 필터를 적용했을 때 일치하는 루트가 없고 다른 관측은 존재하는 경우, Langfuse는 필터를 자동으로 해제합니다.

## 업그레이드 체크리스트

### 호환되는 설정으로 수집하기

데이터가 실시간으로 통합 테이블에 나타나도록 다음 수집 경로 중 하나를 사용하세요.

- Python SDK v4.7.0 이상: [마이그레이션 가이드](/docs/observability/sdk/upgrade-path/python-v3-to-v4)
- JS/TS SDK v5.4.0 이상: [마이그레이션 가이드](/docs/observability/sdk/upgrade-path/js-v4-to-v5)
- 직접 OpenTelemetry 수집: OTEL 스팬 익스포터에 `x-langfuse-ingestion-version: 4`를 설정하세요. OTLP HTTP 환경 변수를 사용하는 경우, `OTEL_EXPORTER_OTLP_HEADERS`에 `x-langfuse-ingestion-version=4`를 추가하세요. 자세한 내용은 [OpenTelemetry 설정](/integrations/native/opentelemetry)을 참고하세요.

<Callout type="info">
아직 **이전 버전의 SDK**를 통해 데이터를 전송 중이거나 **`x-langfuse-ingestion-version: 4` 없이 직접 OpenTelemetry로 수집**하고 있다면, 트레이스가 통합 트레이싱 테이블에 나타나기까지 **최대 10분의 지연**이 발생할 수 있습니다.
</Callout>

### 관측 간 상관 속성(correlating attributes) 전파하기

SDK의 전파(propagation) 헬퍼를 사용해 자식 관측 전반에 걸쳐 상관 속성을 전파하세요.

- Python: `propagate_attributes()`
- JS/TS: `propagateAttributes()`

이는 필터가 정확하게 동작하는 데 매우 중요합니다.
`userId`, `sessionId`, `tags`, `metadata`, `version`, `traceName`과 같은 속성은 관측에서 사용 가능해야 하며, 이를 통해 요청 수준의 컨텍스트로 하나의 테이블을 필터링하면서도 관측 단위의 세밀함(granularity)을 유지할 수 있습니다.

SDK 구현 세부 사항은 [관측에 속성 추가하기](/docs/observability/sdk/instrumentation#add-attributes)를 참고하세요.

## 피드백 공유하기

저희는 통합 트레이싱 경험을 계속해서 개선해 나가고 있습니다.
헷갈리거나 부족하거나 예상보다 느리다고 느껴지는 부분이 있다면 [지원팀](/support)을 통해 피드백을 공유해 주세요.

## 자주 묻는 질문

### 통합 테이블에서 관측이 지연되거나 표시되지 않는 이유는 무엇인가요?

가장 흔한 원인은 데이터가 여전히 이전 버전의 SDK를 통해 전송되고 있거나, `x-langfuse-ingestion-version: 4` 없이 직접 OpenTelemetry 익스포터를 통해 전송되고 있기 때문입니다.
새로운 Fast Preview 환경에서는 이러한 설정이 통합 테이블에 트레이스가 표시되기까지 몇 분 단위의 지연으로 이어질 수 있습니다.
이 문제가 발생하면 Python SDK v4.7.0 이상 또는 JS/TS SDK v5.4.0 이상으로 업그레이드하거나, OTEL 익스포터에 수집 버전 헤더를 추가하세요.

### 가장 중요한 작업만 집중해서 볼 수 있는 방법이 있나요?

워크플로우와 가장 관련 있는 관측 이름, 유형, 모델을 기준으로 필터링하는 저장된 뷰를 만드세요. 예를 들어 `type = generation`으로 필터링하면 모든 LLM 호출을 볼 수 있고, 특정 관측 `name`으로 필터링하면 파이프라인의 핵심 단계를 추적할 수 있습니다. 이러한 뷰를 2~3개 저장해두면 테이블 헤더에서 원클릭으로 접근할 수 있는 바로가기가 됩니다.

### 테이블에 행이 너무 많이 표시됩니다. 어떻게 파악해야 하나요?

이는 관측 테이블로 처음 전환할 때 가장 흔히 나타나는 반응입니다. 에이전틱 애플리케이션이 점점 복잡해짐에 따라 하나의 트레이스에는 수백, 수천 개의 작업이 포함될 수 있는데, 트레이스당 한 행만 보여주는 방식은 사실 필요한 신호를 가리고 있었던 것입니다. 해결책은 스크롤이 아니라 필터와 저장된 뷰입니다. 워크플로우와 가장 관련 있는 관측 유형이나 이름(예: "모든 생성", "오류만", 또는 특정 에이전트 단계)을 필터링하는 것부터 시작하세요. 디버깅 워크플로우에 맞는 뷰를 2~3개 저장하고 나면, 테이블이 기존 트레이스 테이블보다 오히려 더 집중된 느낌을 줄 것입니다.

### 이전 경험으로 되돌아갈 수 있나요?

네. 지금은 UI에서 Fast Preview 환경을 끄고 이전 트레이싱 경험으로 돌아갈 수 있습니다.
다만 이는 일시적인 조치입니다. 통합 테이블이 표준 경험이 될 예정이므로, 새로운 모델에 맞춰 저장된 뷰와 계측을 미리 설정해두는 것이 좋습니다.

# Fast Preview에서의 대시보드 변경 사항

이 페이지에서는 레거시 Langfuse와 비교했을 때 [Fast Preview](/docs/v4)에서 대시보드가 어떻게 다르게 동작하는지 설명합니다. Fast Preview 토글을 활성화했다면, 관찰되는 차이를 이해하기 위한 참고 자료로 이 페이지를 활용하세요.

Langfuse에는 두 가지 유형의 대시보드가 있습니다:

- **홈 대시보드** — 프로젝트 홈 페이지에 표시되는 내장 대시보드로, 고정된 타일(트레이스 수, 지연 시간 백분위수, 점수 히스토그램, 모델 사용량 등)을 제공합니다.
- **[커스텀 대시보드](/docs/metrics/features/custom-dashboards)** — 사용자가 만든, 위젯을 구성할 수 있는 대시보드입니다.

아래 변경 사항 중 일부는 한 유형에만 영향을 미치고, 일부는 두 유형 모두에 영향을 미칩니다. 각 섹션에는 해당 내용이 어떤 대시보드에 적용되는지 명시되어 있습니다.

## 배경: 트레이스에서 관측치로

Langfuse에서 **관측치(observations)**(OpenTelemetry에서는 "스팬"이라고도 함)는 트레이스 내의 개별 작업 — LLM 호출, 도구 실행, 검색 단계 등 — 을 의미합니다.

레거시 Langfuse에서는 `trace`가 자체 테이블을 가진 일급(first-class) 레코드이고, 관측치는 외래 키를 통해 이를 참조하는 별도의 레코드입니다. 쿼리는 읽기 시점에 이 테이블들을 JOIN하여 트레이스 메타데이터와 관측치 메트릭을 결합합니다.

Fast Preview는 [OpenTelemetry 네이티브 데이터 모델](/docs/observability/data-model)을 채택합니다. OTEL에는 "트레이스"라는 객체가 존재하지 않으며, 오직 부모-자식 계층 구조로 배열된 관측치만 존재합니다. 루트 관측치(부모가 없는 관측치)가 곧 트레이스입니다. 트레이스 ID는 관련된 관측치들을 그룹화하는 공유 식별자입니다.

Fast Preview에서는 모든 데이터가 단일한 비정규화된 ClickHouse 테이블, 즉 **wide observations table**에 저장됩니다. 각 관측치 행은 전체 트레이스 컨텍스트(사용자 ID, 세션 ID, 태그, 환경 등)를 이미 구체화된 형태로 담고 있어 JOIN이 필요 없습니다. (Langfuse와 OTEL 모두 포함하여) 이전 버전의 SDK를 통해 수집된 데이터의 경우, Langfuse는 통합 모델에 맞추기 위해 각 트레이스에 대해 **합성 루트 관측치(synthetic root observation)**를 생성합니다([섹션 6](#6-synthetic-root-observations-appear-in-the-wide-observations-table) 참조). 직접 이벤트 쓰기(direct event writes) 방식을 사용하는 최신 OTEL SDK는 합성 루트 관측치가 필요하지 않습니다. 실제 OTEL 루트 관측치가 이미 자연스러운 루트 항목 역할을 하기 때문입니다. 마이그레이션 방법은 [SDK 업그레이드 가이드](/docs/observability/sdk/upgrade-path)를 참조하세요.

이는 대시보드에 두 가지 결과를 가져옵니다.

1. **관측치가 주된 쿼리 대상**이 되며, 트레이스가 아닙니다. 트레이스 수준의 메트릭은 관측치로부터 파생됩니다 — 예를 들어 트레이스 테이블의 행 수를 세는 대신 고유한 트레이스 ID 수를 세는 방식입니다.
2. **레거시 "트레이스 뷰"**는 하위 호환성을 위해 여전히 존재하지만, 트레이스에 속한 모든 관측치를 집계하여 트레이스 수준 필드를 재구성해야 하므로 대규모 환경에서 쿼리 비용이 큽니다. 이 뷰에 의존하는 새로운 [커스텀 위젯](/docs/metrics/features/custom-dashboards)은 더 이상 생성할 수 없으며, 기존 위젯도 이 뷰에서 벗어나도록 마이그레이션해야 합니다.

---

## 동일하게 유지되는 것

- 모든 대시보드 필터(name, userId, tags, environment, type, model 등)가 계속 지원됩니다. 일부 필터 동작은 Fast Preview에서 더 정교해졌습니다(아래 섹션 9와 11 참조).
- 쿼리 API의 형태(dimensions, metrics, filters, time series granularity)는 변경되지 않았습니다.
- 점수 집계 테이블, 히스토그램, 시계열은 동일한 출력 형식을 생성합니다.
- 모든 메트릭에 걸쳐 추세, 분포, 상대 비교는 일관되게 유지됩니다. 개별 절대값(예: 관측치 수)은 고정된 오프셋만큼 다를 수 있는데, 이는 레거시 Langfuse에서는 별도 레코드로 존재하지 않던 합성 루트 관측치가 이제 관측치 수에 포함되기 때문입니다.

## 달라지는 것

### 1. 트레이스 수는 wide observations table에서 계산됩니다

_적용 대상: 홈 대시보드 및 커스텀 대시보드_

레거시 Langfuse에서 트레이스 수는 전용 트레이스 테이블(`SELECT count(*) FROM traces`)에서 가져옵니다. Fast Preview에서는 wide observations table에서 `uniq(trace_id)`로 계산됩니다 — 모든 관측치에 걸쳐 고유한 트레이스 ID 수를 세는 방식입니다.

이는 의도적인 성능 선택입니다. Fast Preview의 트레이스 뷰는 트레이스 수준 필드를 재구성하기 위해 트레이스당 여러 행을 집계해야 하며, 이는 대규모 프로젝트에서 비용이 큽니다. wide observations table은 이를 완전히 피할 수 있습니다 — 각 행이 이미 트레이스 컨텍스트를 담고 있으므로, 고유한 트레이스 ID를 세는 것은 가벼운 스캔 작업입니다.

<Callout type="info">
`uniq()`는 근사 카운팅 함수이므로, 트레이스 수는 레거시 Langfuse의 정확한 수치와 약간 다를 수 있습니다. 실질적인 용도에서는 이 차이가 무시할 만한 수준입니다.
</Callout>

### 2. "Traces by time"이 "Observations by time"으로 바뀝니다

_적용 대상: 홈 대시보드_

레거시 Langfuse에서 이 차트는 "Traces"(시간에 따른 트레이스 수)와 "Observations by Level" 두 개의 탭으로 구성됩니다. Fast Preview에서는 "Traces" 탭이 제거됩니다. 차트는 "Observations by Level"만 표시하며, 제목도 "Observations by time"으로 변경됩니다.

### 3. 트레이스 지연 시간 백분위수가 제거될 수 있습니다

_적용 대상: 홈 대시보드_

트레이스 지연 시간 테이블은 현재 트레이스 뷰를 쿼리하는데, 이는 대규모 환경에서 가장 느린 대시보드 쿼리 중 하나입니다. 출시 전까지 성능을 허용 가능한 수준으로 개선하지 못하면, 이 타일은 Fast Preview 대시보드에서 제거됩니다. (wide observations table을 직접 쿼리하는) 관측치 수준의 지연 시간 메트릭은 그대로 유지됩니다.

### 4. 트레이스 시간 버킷팅은 루트 관측치의 시작 시간을 사용합니다

_적용 대상: 홈 대시보드 및 커스텀 대시보드_

레거시 Langfuse는 트레이스 레코드의 `timestamp` 필드를 사용하여 트레이스를 시간 버킷에 할당합니다. Fast Preview는 루트 관측치의 `start_time`, 즉 트레이싱된 작업이 실제로 시작된 시점을 사용합니다. 이전 버전 SDK로 수집된 데이터의 경우 합성 루트 관측치가 원본 트레이스 타임스탬프를 그대로 반영하므로, 두 값은 거의 일치해야 합니다. 트레이스 레코드의 생성 시간과 루트 관측치의 시작 시간이 다른 예외적인 경우, 경계에서 트레이스가 한 시간 버킷만큼 이동할 수 있습니다.

트레이스에 루트 관측치가 전혀 없는 경우, Fast Preview는 버킷팅을 위해 가장 이른 관측치의 시작 시간으로 대체(fallback)합니다. 이를 통해 트레이스가 시계열 차트에서 계속 보이도록 유지합니다. 다만 여러 수집 경로가 혼합된 프로젝트 — 예를 들어 (항상 루트 관측치를 생성하는) 이전 버전 SDK로 수집된 일부 트레이스와, (분산 트레이스처럼 루트 관측치가 다른 서비스에 존재하여 루트 관측치를 생성하지 않을 수 있는) 직접 쓰기 OTEL 경로로 수집된 일부 트레이스가 섞여 있는 경우 — 루트 관측치가 없는 트레이스는 시계열 차트에서 누락될 수 있습니다.

<Callout type="warn">
직접 쓰기 OTEL 경로를 사용하고 있고 시계열 차트에서 트레이스가 누락된 것을 발견했다면, 해당 트레이스에 루트 관측치가 있는지 확인하세요. 이는 여러 수집 경로를 함께 사용하는 프로젝트에서 가장 흔하게 발생합니다.
</Callout>

### 5. 시간 범위 내에 관측치가 없는 트레이스가 더 이상 누락되지 않습니다

_적용 대상: 홈 대시보드 및 커스텀 대시보드_

레거시 Langfuse에서는 트레이스 수와 (지연 시간이나 토큰 사용량 같은) 관측치 수준 메트릭을 결합한 대시보드 타일이, 자식 관측치가 선택된 시간 범위 밖에 있는 트레이스를 조용히 결과에서 제외하는 경우가 있었습니다. 예를 들어 선택한 범위 이전에 시작되어 그 범위 안에는 관측치가 없는 장시간 실행 트레이스는, 트레이스 자체는 범위 안에 있음에도 불구하고 결과에서 사라졌습니다.

Fast Preview에서는 자식 관측치가 언제 발생했는지와 무관하게, 트레이스가 시간 범위 안에 속하기만 하면 항상 결과에 포함됩니다.

### 6. wide observations table에 합성 루트 관측치가 나타납니다

_적용 대상: 홈 대시보드 및 커스텀 대시보드_

(Langfuse와 OTEL 모두 포함하여) 이전 버전 SDK를 통해 수집된 데이터의 경우, Fast Preview는 각 트레이스에 대해 `t-{traceId}` 형식의 ID를 가진 합성 루트 관측치를 생성합니다. 이 항목들은 트레이스의 메타데이터(이름, 태그, 사용자 등)를 담고 있지만 지속 시간, 비용, 토큰 사용량 정보는 없습니다. 관측치 ID로 쿼리하거나 필터링할 경우 이러한 항목을 마주칠 수 있습니다.

<Callout type="info">
직접 이벤트 쓰기 방식을 사용하는 최신 OTEL SDK는 합성 루트 관측치를 생성하지 않습니다 — (실제 지속 시간과 메트릭을 가진) 진짜 OTEL 루트 관측치가 직접 루트 항목 역할을 합니다.
</Callout>

### 7. 점수 히스토그램이 서버 사이드에서 계산됩니다

_적용 대상: 홈 대시보드_

레거시 Langfuse는 ClickHouse에서 최대 10,000개의 원시 점수 값을 가져와 브라우저에서 히스토그램 구간(bin)을 생성합니다. Fast Preview는 히스토그램을 서버 사이드에서 계산하며, 데이터셋 크기와 무관하게 모든 점수가 포함됩니다. 점수가 10,000개를 초과하는 경우, Fast Preview 히스토그램이 실제 분포를 더 잘 반영합니다. Fast Preview가 다른 구간(binning) 알고리즘을 사용하기 때문에, 히스토그램 구간 경계와 막대 높이도 약간 다르게 보일 수 있습니다.

### 8. 트레이스 이름은 루트 관측치로부터 결정됩니다

_적용 대상: 홈 대시보드 및 커스텀 대시보드_

레거시 Langfuse에서 트레이스 이름은 SDK가 설정한 명시적인 `trace.name` 필드에서 가져옵니다. Fast Preview에서는 전용 `trace_name` 필드가 비어 있는 경우(명시적인 트레이스 객체가 전송되지 않는 OTEL 네이티브 수집에서 흔함), 시스템은 루트 관측치 자체의 `name` 필드로 대체합니다. 즉, 이전에는 대시보드에 이름 없이 표시되던 OTEL 트레이스가 이제는 루트 관측치의 이름으로 표시됩니다.

이 대체 로직은 관측치 수준 및 점수 차트를 포함하여 트레이스 이름이 사용되는 모든 곳에 적용됩니다.

### 9. 점수 환경(environment) 필터링이 더 정확해집니다

_적용 대상: 홈 대시보드 및 커스텀 대시보드_

레거시 Langfuse에서는 환경별로 점수를 필터링할 때 *부모 트레이스*의 환경을 사용합니다. Fast Preview에서는 이 필터가 점수 자체의 `environment` 필드에 직접 적용됩니다. 점수의 환경이 부모 트레이스의 환경과 다른 경우, Fast Preview는 이를 반환하지만 레거시 Langfuse는 반환하지 않을 수 있습니다. 즉, Fast Preview는 레거시 Langfuse가 조용히 제외하던 추가적인 점수 행을 반환할 수 있습니다.

### 10. 카디널리티가 높은 차원(dimension)은 상위 N개 쿼리가 필요합니다

_적용 대상: 홈 대시보드 및 커스텀 대시보드_

`userId`, `traceId`, `sessionId` 같은 차원은 수백만 개의 고유 값을 가질 수 있습니다. 레거시 Langfuse에서는 이러한 차원으로 그룹화하는 쿼리가 무제한 결과 집합을 생성하여 속도가 느려지거나 타임아웃이 발생할 수 있었습니다.

Fast Preview는 가드레일을 적용합니다. 쿼리가 카디널리티가 높은 차원으로 그룹화될 경우, 행 제한과 정렬 순서를 반드시 지정해야 합니다(예: "비용 기준 상위 20명 사용자, 내림차순"). 시계열 차트는 카디널리티가 높은 차원을 아예 사용할 수 없습니다 — 수많은 고유 값과 수많은 시간 버킷의 조합은 지나치게 많은 행을 만들어 내기 때문입니다. "User consumption" 차트 같은 내장 대시보드 타일은 이미 이 패턴을 따르고 있습니다. 이러한 제약을 위반하는 [커스텀 위젯](/docs/metrics/features/custom-dashboards)은 무엇을 수정해야 하는지 설명하는 유효성 검사 오류를 표시합니다.

또한 카디널리티가 매우 높은 일부 필드(`id`, `traceId`, `parentObservationId`)는 더 이상 커스텀 위젯 빌더에서 차원으로 사용할 수 없습니다. 이러한 필드는 필터링에는 유용하지만 그룹화에는 적합하지 않습니다 — 개별 트레이스 ID로 그룹화된 차트는 의미가 없기 때문입니다.

### 11. 필터에서 빈 문자열과 NULL이 이제 동일하게 취급됩니다

_적용 대상: 홈 대시보드 및 커스텀 대시보드_

레거시 Langfuse에서는 빈 문자열(`""`)과 NULL이 서로 다른 값입니다 — "is null" 필터는 빈 문자열과 일치하지 않으며, 그 반대도 마찬가지입니다. Fast Preview에서는 이 둘을 동일하게 취급합니다. "is null"은 NULL과 빈 문자열 값 모두와 일치하며, "does not contain"은 두 값을 모두 제외합니다. 예를 들어 "parentObservationId is null"로 필터링하면, 기본 값이 NULL로 저장되었는지 빈 문자열로 저장되었는지와 무관하게 부모가 없는 모든 관측치가 반환됩니다.

### 12. 커스텀 위젯이 최소 필요 버전을 추적합니다

_적용 대상: 커스텀 대시보드_

[커스텀 대시보드 위젯](/docs/metrics/features/custom-dashboards)은 이제 위젯이 저장될 때 자동으로 감지되는 `min_version` 필드를 가집니다. 위젯이 Fast Preview에만 존재하는 차원이나 measure(예: `costByType` 또는 `usageByType`)를 사용하는 경우, 해당 위젯은 자동으로 Fast Preview가 필요한 것으로 태그됩니다. 두 버전 모두에서 사용 가능한 필드만 사용하는 위젯은 레거시 Langfuse와도 계속 호환됩니다.

Fast Preview가 활성화되면 위젯 빌더는 Fast Preview 뷰 정의를 사용하여 Fast Preview 전용 measure와 차원을 노출합니다. 위젯 빌더에서는 트레이스 뷰가 Fast Preview에서 제외되며, 위젯이 여전히 이 뷰를 대상으로 하는 경우 경고가 표시되고 해당 뷰에 대해서는 레거시 Langfuse 정의로 대체됩니다.

## 예상되는 수치상의 차이

대부분의 대시보드에서 레거시 Langfuse와 Fast Preview의 수치는 서로 밀접하게 일치합니다. 알려진 차이는 다음과 같습니다.

| 메트릭               | 예상되는 차이                                                             |
| -------------------- | ------------------------------------------------------------------------- |
| 관측치 수            | 다를 수 있음 (레거시 Langfuse에는 없던 합성 루트 관측치가 이제 포함됨)    |
| 트레이스 총 개수     | 약간 다를 수 있음 (`uniq()`를 통한 근사 카운팅)                           |
| 이름별 트레이스 수   | 약간 다를 수 있음 (`uniq()`를 통한 근사 카운팅)                           |
| 트레이스 시계열 버킷 | 윈도우 경계에서 한 버킷만큼 이동할 수 있음                                |
| 점수 수 및 평균      | 쿼리 및 필터 경로 변경으로 인해 약간 다를 수 있음                         |
| 점수 히스토그램 총합 | ±1 이내 (ClickHouse 히스토그램 반올림)                                    |
| 비용 및 토큰 합계    | 추세는 일치하나, 루트 관측치로 인해 총합이 고정된 오프셋만큼 다를 수 있음 |
| 사용자 소비량        | 약간 다를 수 있음 (근사 카운팅, 시간 경계)                                |

<Callout type="info">
데이터는 Fast Preview 내부적으로 일관성을 유지합니다. 위에서 설명한 변경 사항으로 인해 절대 수치는 레거시 Langfuse와 약간 다를 수 있지만, 추세, 분포, 상대 비교는 일치합니다.
</Callout>

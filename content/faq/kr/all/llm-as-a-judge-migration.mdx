# 평가기 업그레이드 가이드

Langfuse는 실시간 데이터 LLM-as-a-Judge 평가를 위한 권장 방식으로 옵저베이션에서 평가기를 실행하는 기능을 도입했습니다. 이 가이드는 기존의 실시간 데이터 평가기를 새로운 시스템으로 업그레이드하는 데 도움을 줍니다.

<Callout type="info">
**새 평가기를 설정하시나요?** 대신 [LLM-as-a-Judge 설정 가이드](/docs/evaluation/evaluation-methods/llm-as-a-judge#set-up-step-by-step)를 사용하세요. 최상의 경험을 위해 처음부터 옵저베이션 레벨 평가기로 시작하는 것이 좋습니다. 이 가이드는 기존 트레이스 레벨 평가기를 업그레이드하는 경우에만 해당합니다.
</Callout>

## 왜 옵저베이션 레벨 평가기로 업그레이드해야 하나요?

옵저베이션 레벨 평가기는 다음과 같은 이점을 제공합니다:

- **훨씬 빠른 실행 속도**: 평가가 몇 분이 아닌 몇 초 만에 완료됩니다. 평가 지연 및 백로그 문제가 사라집니다. 최적화된 아키텍처가 분당 수천 건의 평가를 처리합니다.
- **오퍼레이션 레벨 정밀도**: 전체 워크플로우가 아닌 특정 오퍼레이션(최종 LLM 응답, 검색 단계 등)을 평가합니다. 평가 볼륨과 비용을 줄일 수 있습니다.
- **조합형 평가**: 서로 다른 오퍼레이션에 대해 서로 다른 평가기를 동시에 실행할 수 있습니다
- **향상된 안정성**: 더 예측 가능한 동작, 개선된 오류 처리 및 재시도 로직
- **미래 지향적 아키텍처**: 향후 추가될 기능과 역량을 위해 설계되었습니다

트레이스 레벨 평가기는 계속 작동합니다. 이는 폐기가 아니라 업그레이드 경로입니다.

## 언제 업그레이드해야 하나요

### ✅ 지금 업그레이드해야 하는 경우:

- **평가 지연, 백로그, 또는 느린 평가 처리를 겪고 있는 경우** (옵저베이션 레벨 평가는 몇 초 만에 완료됩니다)
- OTel 기반 SDK(Python v3 이상 또는 JS/TS v4 이상)를 사용 중이거나 사용할 계획인 경우
- 새 평가기를 설정하는 경우

### ⏸️ 당분간 트레이스 레벨을 유지해야 하는 경우:

- 레거시 SDK(Python v2 또는 JS/TS v3)를 사용 중이며 아직 업그레이드할 수 없는 경우
- 트레이스 평가기가 사용 사례에 완벽하게 맞는 경우
- 여러 오퍼레이션에 걸친 집계 워크플로우 데이터를 평가해야 하는 경우

## 사전 요구 사항

### 옵저베이션 레벨 평가기를 위한 SDK 요구 사항

| 요구 사항             | Python                                                                 | JS/TS                                                             |
| --------------------- | ---------------------------------------------------------------------- | ----------------------------------------------------------------- |
| **최소 SDK 버전**     | v3 이상 (OTel 기반)                                                    | v4 이상 (OTel 기반)                                               |
| **업그레이드 가이드** | [Python v2 → v3](/docs/observability/sdk/upgrade-path/python-v2-to-v3) | [JS/TS v3 → v4](/docs/observability/sdk/upgrade-path/js-v3-to-v4) |

### 무중단 마이그레이션 전략

SDK 버전을 업그레이드하기 전에 옵저베이션 레벨 평가기를 설정할 수 있습니다. 이를 통해 다음이 가능합니다:

1. 기존 트레이스 레벨 평가기와 병행하여 새 평가기를 배포
2. 평가 다운타임 없이 준비가 되었을 때 SDK 업그레이드

이 방법은 필수는 아니지만 프로덕션 시스템에 가장 매끄러운 업그레이드 경로를 제공합니다.

## 업그레이드 프로세스

Langfuse는 평가기를 업그레이드할 수 있는 내장 마법사를 제공합니다. 이 마법사는 트레이스 레벨 구성을 옵저베이션 레벨로 변환하는 데 도움을 줍니다.

### 1단계: 업그레이드 마법사 접근

1. 프로젝트 → Evaluation → LLM-as-a-Judge로 이동합니다
2. "Legacy"로 표시된 평가기를 찾습니다
3. 업그레이드할 평가기 행을 클릭합니다
4. 오른쪽 상단의 "Upgrade"를 클릭합니다

### 2단계: 검토 및 구성

마법사는 구성을 나란히 보여줍니다:

- **왼쪽 (읽기 전용)**: 현재 트레이스 레벨 구성
- **오른쪽 (편집 가능)**: 옵저베이션 레벨 구성

**주요 구성 단계**:

1. **필터**: 이전과 동일한 트레이스 레벨 속성을 대상으로 하도록 필터를 추가합니다. 트레이스 내에서 특정 옵저베이션을 대상으로 하는 새로운 옵저베이션 레벨 필터를 추가하는 것을 권장합니다.
   - 트레이스 레벨 필터 (예: `traceName`, `userId`, `tags`)
   - 옵저베이션 유형 (예: `generation`, `span`, `event`)
   - 옵저베이션 이름 (예: `chat-completion`)

2. **변수 매핑**: 옵저베이션 필드에서 변수를 매핑합니다
   - `trace.input` → `observation.input`으로 변경
   - `trace.output` → `observation.output`으로 변경
   - 커스텀 필드에는 `observation.metadata`를 사용

3. **업그레이드 방식 선택**:
   - **둘 다 활성 상태로 유지**: 새 평가기를 기존 평가기와 함께 테스트
   - **기존 평가기를 비활성으로 표시**: 기존 평가기는 중지되고 새 평가기가 대신 실행됨 (초기에 권장)
   - **기존 평가기 삭제**: 레거시 평가기를 영구적으로 제거

### 3단계: 실행 확인

업그레이드 후:

1. **실행 로그 확인**
   - Evaluator Table로 이동 → 새 평가기를 찾음 → "Logs" 클릭
   - 평가가 실행되고 있는지 확인

2. **결과 비교** (둘 다 실행 중인 경우)
   - [점수 분석](/docs/evaluation/scores/score-analytics)을 사용하여 점수를 비교합니다
   - 평가 로직의 일관성을 확인합니다
   - 참고: 옵저베이션 레벨 평가기의 점수는 트레이스 자체가 아니라 트레이스 트리 내 특정 옵저베이션에 표시됩니다. 필터 조건에 따라 여러 옵저베이션이 조건에 일치하여 트레이스당 여러 점수가 발생할 수 있습니다.

## 업그레이드 예시: 간단한 트레이스 평가기

대부분의 트레이스 평가기는 특정 옵저베이션의 입력/출력에 직접 매핑되는 입력/출력을 가지고 있습니다. 업그레이드는 이 옵저베이션을 직접 대상으로 합니다.

**시나리오**: 챗봇의 응답을 평가하는 트레이스 레벨 평가기가 있습니다. 트레이스에는 트레이스와 동일한 입력/출력을 담고 있는 "user-workflow"라는 이름의 스팬 옵저베이션이 있습니다.

<Frame fullWidth>
  ![업그레이드 예시](/images/docs/evaluator-upgrade-example.png)
</Frame>

**변경 전 (트레이스 레벨)**:

```
Target: Traces
Filter: name = "user-workflow"
        AND environment not in ["sdk-experiment"]
Variables:
  - query: trace.input.key
  - generation: trace.output
```

**변경 후 (옵저베이션 레벨)**:

```
Target: Observations
Filter: traceName = "user-workflow"
        AND environment not in ["sdk-experiment"]
        AND type = "span"
Variables:
  - query: observation.input.key
  - generation: observation.output
```

**주요 변경 사항**:

1. **옵저베이션 레벨 필터 추가**: 트레이스 트리 내에서 특정 옵저베이션을 식별합니다
   - `observation.type = "span"`이 특정 스팬 옵저베이션을 대상으로 합니다

2. **변수 소스 변경**: 변수가 이제 트레이스가 아닌 옵저베이션에서 옵니다
   - `trace.input.key` → `observation.input.key`
   - `trace.output` → `observation.output`

3. **결과**: 평가기가 이제 일치하는 각 옵저베이션에 대해 개별적으로 실행되며, 다음을 제공합니다:
   - 훨씬 빠른 실행 속도(몇 분이 아닌 몇 초 만에 완료—전체 트레이스를 로드할 필요 없음)
   - 더 정확한 타겟팅(특정 오퍼레이션만 평가)
   - 더 나은 비용 관리(특정 오퍼레이션 유형으로 필터링 가능)

**중요**: 결과 점수는 이제 트레이스가 아니라 **옵저베이션에 연결**됩니다. 이는 다음을 의미합니다:

- 점수가 트레이스 트리 내 특정 옵저베이션에 표시됩니다
- 트레이스당 여러 점수를 가질 수 있습니다(평가된 옵저베이션마다 하나씩)

옵저베이션 레벨 평가기는 일치한 옵저베이션에서만 변수를 매핑할 수 있습니다. 동일한 트레이스 내의 형제 또는 자식 옵저베이션을 자동으로 읽지 않습니다. 저지(judge)가 종단 간(end-to-end) 요청과 응답이 필요한 경우, 해당 값을 기록하는 루트 옵저베이션을 대상으로 하고 추가로 필요한 컨텍스트를 해당 루트 옵저베이션에 기록하세요.

## SDK 버전별 가이드

### 레거시 SDK(Python v2, JS/TS v3) 사용자용

**옵션 1: OTel 기반 SDK로 업그레이드** (권장)

1. Python SDK v3 이상 또는 JS/TS SDK v4 이상으로 업그레이드합니다 ([사전 요구 사항](#prerequisites) 참고)
2. 마이그레이션 가이드를 사용해 계측 코드를 업데이트합니다
3. 마법사를 사용해 평가기를 업그레이드합니다

**옵션 2: 트레이스 레벨 평가기 계속 사용**

- 변경이 필요하지 않습니다—트레이스 레벨 평가기는 계속 작동합니다
- 준비가 되면 업그레이드하세요

### OTel 기반 SDK(Python v3 이상, JS/TS v4 이상) 사용자용

기존 트레이스 레벨 평가기가 있는 경우, 새 아키텍처의 모든 이점을 얻기 위해 마법사를 사용하여 옵저베이션 레벨 평가기로 업그레이드하는 것을 권장합니다.

## 롤백 계획

업그레이드 후 되돌려야 하는 경우:

1. **두 평가기를 모두 유지한 경우**: 새 평가기를 비활성으로 표시하고, 기존 평가기를 다시 활성화합니다
2. **기존 평가기를 삭제한 경우**: 이전 구성으로 새로운 트레이스 레벨 평가기를 생성합니다
3. **데이터는 보존됩니다**: 모든 과거 평가 결과는 계속 접근 가능합니다

# 애플리케이션에 사용자 피드백 루프를 구축하는 방법

트레이스가 수집되기 시작했다면, 다음으로 좋은 단계는 사용자 피드백을 수집하는 것입니다. 사용자는 클릭하고, 수정하고, 다시 시도하거나, 직접 말하는 모든 행동을 통해 응답을 평가합니다. 이를 트레이스에 기록하면 실제 사용 맥락에 기반한 품질 신호를 얻을 수 있고, 각 점수는 열어서 학습할 수 있는 구체적인 트레이스를 가리키게 됩니다.

<Callout type="info">
  많은 팀이 "트레이스는 수집되고 있는데, 이제 평가는 어떻게 시작하나요?"라고 묻습니다. 이런 팀들은 대개 사용자 피드백부터 시작하는 편이 낫습니다.
</Callout>

이 피드백 수집 단계는 AI 엔지니어링 프로세스에서 일반적으로 매우 높은 ROI를 가집니다. 이 과정은 [Langfuse Academy](/academy)에서 자세히 다루며, [AI 엔지니어링 루프](/academy/ai-engineering-loop)를 단계별로 살펴봅니다. 사용자 피드백 수집은 아래에 강조된 모니터링 단계에서 이루어집니다.

<LoopDiagram highlight="monitor" />

## 사전 준비 사항

Langfuse로 트레이스가 이미 수집되고 있어야 합니다. 아직 설정하지 않았다면 [트레이싱](/docs/observability/get-started)을 참고하세요.

## 진행 방법

<Steps>

### 피드백 신호 선택하기 [#choose-signals]

피드백은 두 가지 형태로 나뉩니다. **명시적 피드백**은 좋아요/싫어요나 별점처럼 사용자가 의도적으로 남기는 평가로, 의미는 명확하지만 드물고 불만족한 사용자 쪽으로 편향되는 경향이 있습니다. **암묵적 피드백**은 질문을 다시 시도하거나 초안을 수정하는 등 사용자의 행동에서 유추되는 신호로, 모든 트레이스에 대한 데이터를 얻을 수 있지만 해석이 필요합니다.

가장 좋은 신호는 사용 사례에 따라 크게 달라집니다. 참고할 수 있는 몇 가지 예시입니다:

| 신호                                       | 알려주는 것                               | 일반적인 수집 방법 |
| ------------------------------------------ | ----------------------------------------- | ------------------ |
| 응답에 대한 좋아요/싫어요                  | 해당 응답에 대한 직접적인 평가            | Browser SDK        |
| 같은 질문을 다른 표현으로 다시 묻는 경우   | 이전 답변이 만족스럽지 않았음             | LLM-as-a-Judge     |
| 사람과 상담을 요청하는 경우                | 사용자가 어시스턴트에 대한 신뢰를 잃음    | LLM-as-a-Judge     |
| 사용자가 응답을 복사한 경우                | 그대로 재사용할 만큼 결과가 충분히 좋았음 | Browser SDK        |
| 전송 전에 수정된 초안 답변                 | 초안에서 무엇이 틀렸거나 부족했는지       | SDK / API          |
| 코파일럿 제안을 수락하거나 무시한 경우     | 제안이 맥락에 맞았는지 여부               | Browser SDK        |
| 검토 단계에서 수정된 추출 필드             | 어떤 필드가 잘못 추출되었는지             | SDK / API          |
| 결과를 클릭하지 않고 재구성된 검색어       | 결과가 의도를 충족하지 못했음             | SDK / API          |
| 추천 아이템이 시작되자마자 건너뛰어진 경우 | 추천이 사용자 취향과 맞지 않았음          | SDK / API          |

팁:

- 여러 신호를 함께 사용해도 좋습니다. 하나만 구현할 필요는 없습니다
- 자유 형식 텍스트는 가능한 곳마다 기록하세요([점수에 대한 코멘트](/docs/evaluation/scores/data-model)로 남길 수 있습니다). 이런 분석의 상당 부분을 이제는 에이전트가 수행하며, 맥락이 많을수록 더 잘 작동합니다
- [academy 예제](/academy/examples)는 특정 애플리케이션에서 신호를 어떻게 선택하고 수명 주기에 걸쳐 어떻게 바꾸는지 보여줍니다

### 신호를 점수로 기록하기 [#capture]

모든 신호는 결과를 만들어낸 트레이스의 [점수](/docs/evaluation/scores/data-model)로 남게 됩니다. 점수는 이름, 데이터 타입(불리언, 숫자, 카테고리 또는 자유 텍스트)을 가진 값, 그리고 선택적인 `comment`로 구성됩니다. 평가기(evaluator) 역시 결과를 점수로 저장하므로, 사용자 피드백도 동일한 필터, 대시보드, 분석 화면에 함께 나타납니다.

<FeedbackCaptureDiagram />

점수로 이어지는 경로는 신호가 어디에서 발생했는지에 따라 달라집니다:

<Tabs items={["UI에서 발생한 신호", "백엔드의 앱 이벤트", "LLM-as-a-Judge를 통한 방법"]}>
<Tab>

UI에서 발생한 평가는 퍼블릭 키를 사용해 브라우저에서 바로 전송됩니다:

```typescript
import { LangfuseBrowserClient } from "@langfuse/browser";

const langfuse = new LangfuseBrowserClient({
  publicKey: process.env.NEXT_PUBLIC_LANGFUSE_PUBLIC_KEY!,
});

// 좋아요/싫어요 클릭 시
await langfuse.score({
  traceId, // 백엔드가 응답과 함께 반환한 값
  id: `response_rating-${traceId}`, // 고정 id로, 재평가 시 중복 대신 업데이트되도록 함
  name: "response_rating", // 신호마다 하나의 설명적인 이름
  value: 1, // 좋아요는 1, 싫어요는 0
  dataType: "BOOLEAN",
});
```

Next.js 챗봇 기준의 전체 설정(프론트엔드가 트레이스 ID를 얻는 방법 포함)은 [사용자 피드백](/docs/observability/features/user-feedback) 페이지를 참고하세요.

</Tab>
<Tab>

수락된 제안, 수정된 초안, 닫힌 티켓처럼 앱이 관찰하는 이벤트는 발생하는 즉시 점수로 기록됩니다:

<Tab>

```python
from langfuse import get_client

langfuse = get_client()

# 사용자가 전송 전에 초안 답변을 수정한 경우
langfuse.create_score(
    trace_id=trace_id,  # 초안을 생성한 트레이스
    name="draft_edited",
    value=1,
    data_type="BOOLEAN",
    comment=edit_diff,  # 이후 분석을 위한 맥락으로서의 수정 내용 자체
)
```

</Tab>
<Tab>

```typescript
import { LangfuseClient } from "@langfuse/client";

const langfuse = new LangfuseClient();

// 사용자가 전송 전에 초안 답변을 수정한 경우
await langfuse.score.create({
  traceId, // 초안을 생성한 트레이스
  name: "draft_edited",
  value: 1,
  dataType: "BOOLEAN",
  comment: editDiff, // 이후 분석을 위한 맥락으로서의 수정 내용 자체
});

// 짧게 실행되는 환경에서는 점수를 flush 하세요
await langfuse.flush();
```

사람에게 연결해달라는 요청이나 질문을 다시 표현하는 것처럼 자유 텍스트로만 존재하는 신호는 코드가 감지할 수 있는 이벤트를 남기지 않습니다. 프로덕션 트레이스에 적용된 [LLM-as-a-Judge 평가기](/docs/evaluation/evaluation-methods/llm-as-a-judge)가 대화를 읽고 애플리케이션 변경 없이 점수를 기록합니다.

</Tab>

### 피드백을 기반으로 조치하기 [#act]

#### 품질 추세 모니터링하기 [#monitor-trends]

[점수 분석](/docs/evaluation/scores/score-analytics)과 [커스텀 대시보드](/docs/metrics/features/custom-dashboards)로 각 신호가 시간에 따라 어떻게 변화하는지 확인할 수 있습니다. 한 가지 주의할 점은, 모든 신호가 깔끔한 품질 지표는 아니라는 것입니다. 예를 들어 좋아요/싫어요 피드백은 불만족한 사용자 쪽으로 편향되는 경향이 있습니다.

#### 흥미로운 트레이스 찾아내기 [#surface-traces]

`response_rating = 0`과 같이 점수로 트레이스를 필터링하면 조사할 구체적인 트레이스 목록을 얻을 수 있습니다. [오류 분석](/academy/monitoring/error-analysis)은 이를 구조적으로 읽고 클러스터링하는 방법이며, [주석 큐(annotation queue)](/docs/evaluation/evaluation-methods/annotation-queues)를 통해 더 많은 검토자를 참여시킬 수 있습니다.

#### 구조화된 개선의 입력으로 활용하기 [#structured-improvement]

실패한 사례를 [데이터셋](/docs/evaluation/experiments/datasets)에 추가하고, 배포 전에 [실험](/docs/evaluation/experiments/experiments-via-ui)으로 수정 사항을 테스트한 뒤, 반복되는 실패 패턴을 이후 모든 프로덕션 트레이스에 점수를 매기는 [자동화된 평가기](/docs/evaluation/core-concepts#evaluation-methods)로 발전시키세요.

**이 과정의 상당 부분은 에이전트에게 맡길 수도 있습니다.** 아래 [피드백을 기반으로 에이전트가 조치하도록 하기](#agents)를 참고하세요.

</Steps>

## 피드백을 기반으로 에이전트가 조치하도록 하기 [#agents]

Langfuse는 [Langfuse CLI](/docs/api-and-data-platform/features/cli), [Agent Skill](/docs/api-and-data-platform/features/agent-skill), [MCP 서버](/docs/api-and-data-platform/features/mcp-server)를 통해 에이전트가 접근할 수 있도록 설계되었습니다. 이 중 하나를 프로젝트에 연결하면 사용자 피드백을 통해 드러난 동작을 해석하고 그에 따라 조치할 수 있습니다. 예시 작업:

```text
지난 7일간 response_rating 점수가 0인 트레이스를 가져오세요.
코멘트와 출력을 읽고 실패 유형을 클러스터링한 뒤,
가장 큰 두 가지 유형에 대한 프롬프트 수정안을 제안하세요.
```

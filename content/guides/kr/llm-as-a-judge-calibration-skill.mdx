# Langfuse 스킬로 LLM-as-a-judge 캘리브레이션하는 방법

[LLM-as-a-judge](/docs/evaluation/evaluation-methods/llm-as-a-judge)는 편리합니다. 프롬프트를 작성하고 트레이스에 적용하면 점수를 얻을 수 있습니다. 하지만 이 judge도 결국 여러분이 작성한 다른 프롬프트와 다르지 않습니다. 여러분이 직접 라벨링하는 방식대로 케이스를 라벨링하는지 어떻게 알 수 있을까요?

그 답이 바로 **judge 캘리브레이션**입니다. 수동으로 할 수도 있지만, [Langfuse 스킬](https://github.com/langfuse/skills/tree/main/skills/langfuse)에는 이를 위한 전용 워크플로가 있습니다. 이 가이드에서는 이 워크플로를 처음부터 끝까지 살펴봅니다.

## 사전 준비 사항

**Langfuse 스킬이 설치된 코딩 에이전트.** [Langfuse 스킬](https://github.com/langfuse/skills/tree/main/skills/langfuse)을 사용할 수만 있다면 어떤 에이전트든(Claude Code, Cursor, Codex 등) 사용할 수 있습니다. 사용 중인 스킬 버전에 `references/judge-calibration.md`가 포함되어 있는지 확인하세요.

**환경에 설정된 Langfuse 자격 증명.** 이 스킬은 내부적으로 [Langfuse CLI](https://github.com/langfuse/langfuse-cli)를 사용하며, 셸에서 `LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY`, `LANGFUSE_BASE_URL`을 읽어옵니다.

## 전체적인 구성

여러분은 LLM-as-a-judge가 평가할 시나리오로 구성된 데이터셋, 즉 정답 라벨(ground truth labels)을 만듭니다. 테스트하려는 judge 프롬프트는 [Langfuse 프롬프트 관리](/docs/prompt-management/get-started)에 있습니다.

캘리브레이션 실행은 [Langfuse 실험](/docs/evaluation/experiments)으로, 데이터셋의 모든 항목에 대해 judge 프롬프트를 실행합니다. 각 항목마다 judge가 산출한 점수와 여러분이 정의한 예상 출력값이라는 두 가지 라벨을 얻게 됩니다. 에이전트는 이 둘을 비교하여 얼마나 자주 일치하는지 보고합니다. 일치율이 높을수록 judge가 여러분의 판단과 더 가깝게 정렬되어 있다는 뜻입니다.

## 워크플로

고객 지원 대화에서 실행되는 LLM-as-a-judge를 캘리브레이션하는 예시를 통해 살펴보겠습니다. 이 judge는 해당 대화가 사람에게 에스컬레이션되었는지 여부를 판단해야 합니다.

<Steps>

### 정답 데이터셋 만들기

[Langfuse 데이터셋](/docs/evaluation/experiments/datasets)을 만듭니다. 이 가이드에서는 예시로 `escalated-not-escalated-scenarios`라는 데이터셋을 만듭니다. 각 항목에는 고객 지원 대화와, LLM-as-a-judge가 이 대화에 부여하기를 기대하는 라벨이 담겨 있습니다.

데이터셋의 두 예시 항목:

| 입력값                                                                                                                                                                                                    | 예상 출력값     |
| --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------- |
| User: My subscription was charged twice this month.<br/>Agent: I can see the duplicate charge. I've issued a refund for the duplicate — it should appear in 3-5 business days.<br/>User: Perfect, thanks! | `not_escalated` |
| User: I can't log in.<br/>Agent: Can you try resetting your password at the link I just sent?<br/>User: This is the third time I'm trying. Just get me a human, please.                                   | `escalated`     |

### 프롬프트 관리에 judge 프롬프트 추가하기

LLM-as-a-judge에서 사용하는 프롬프트를 복사하여 [Langfuse 프롬프트 관리](/docs/prompt-management/get-started)에 새 프롬프트로 추가합니다. 캘리브레이션 실행 시 각 데이터셋 항목에 대해 실행될 프롬프트가 바로 이것입니다.

<Callout type="info">
  프로덕션에서 사용하는 프롬프트와 evaluator 프롬프트를 분리해서 관리하고
  싶다면, `evaluator-prompts/` 폴더 아래에 정리하거나 전용 프로젝트에
  넣어두세요.
</Callout>

### Langfuse 스킬로 캘리브레이션 실행하기

데이터셋과 프롬프트를 준비했다면, AI 어시스턴트를 열어 캘리브레이션을 실행하도록 요청합니다.

<AgentPromptCallout
  prompt={`I have a Langfuse dataset called <escalated-not-escalated-scenarios> and a judge prompt called <evaluator-prompts/escalation-judge> in Langfuse Prompt Management. Use the Langfuse skill's judge calibration workflow to test whether this judge agrees with the ground-truth labels in the dataset.`}
/>

이후로는 스킬이 작업을 이어받아, 실제로 실행하기 전에 몇 가지 간단한 질의응답을 거칩니다. 다음과 같은 흐름을 예상하면 됩니다.

#### 에이전트가 simple 모드와 advanced 모드 중 선택을 요청합니다

무언가를 실행하기 전에, 에이전트는 **simple** 캘리브레이션과 **advanced** 캘리브레이션 중 어느 쪽을 원하는지 묻습니다.

- **Simple 모드**는 단일 정확도 점수, 즉 judge가 여러분과 동일하게 라벨링한 데이터셋 항목의 비율을 보고합니다. "이 judge가 나와 충분히 자주 일치하는가?"에 대한 빠른 답을 얻고 싶을 때 사용하세요.
- **Advanced 모드**는 정밀도(precision), 재현율(recall), F1, TPR, TNR을 포함한 전체 혼동 행렬(confusion matrix)을 보고합니다. 한쪽 클래스가 다른 쪽보다 훨씬 드물거나, 높은 위험도의 자동화 작업에 judge를 신뢰하기 전에 사용하세요.

확신이 서지 않는다면 simple 모드부터 시작하세요.

#### 에이전트가 실험을 실행합니다

에이전트는 [Langfuse Experiments SDK](/docs/evaluation/experiments/experiments-via-sdk)를 사용해 스크립트를 작성하고 대신 실험을 실행해줍니다. 실행 결과는 [실험 UI](/docs/evaluation/experiments)에 데이터셋 실행으로 표시되며, 데이터셋 항목마다 judge의 출력값을 담은 트레이스가 하나씩 생성됩니다.

#### 에이전트가 캘리브레이션 리포트를 반환합니다

에이전트는 실험 출력값을 데이터셋의 예상 출력값과 비교하여 리포트를 반환합니다. 상세 수준은 선택한 모드에 따라 달라집니다.

<Details>
<Summary>캘리브레이션 리포트 예시 (advanced 모드)</Summary>

**설정**

- 데이터셋: `escalated-not-escalated-scenarios` — 35개 항목, 유효하지 않은 라벨 0개
- Judge 프롬프트: `evaluator-prompts/escalation-judge` v1 · 모델 `gpt-4o-mini` · temp 0
- Positive = `escalated`, Negative = `not_escalated`

**혼동 행렬**

|                       | Pred: escalated | Pred: not_escalated |
| --------------------- | --------------- | ------------------- |
| Actual: escalated     | TP = 13         | FN = 1              |
| Actual: not_escalated | FP = 0          | TN = 21             |

**메트릭**

| 메트릭       | 값    |
| ------------ | ----- |
| Accuracy     | 0.971 |
| Precision    | 1.000 |
| Recall / TPR | 0.929 |
| TNR          | 1.000 |
| F1           | 0.963 |

**오류 방향:** false negative가 1건 있었습니다. judge가 에스컬레이션 케이스를 놓친 경우이며, 잘못 플래그를 단 경우는 없었습니다. TPR(0.929)과 TNR(1.000) 모두 스킬의 advanced 품질 기준에서 정한 0.90 임계값을 통과합니다.

유일한 불일치 항목 (item `73cb3a52`):

> User: "Forget it. I'll just email your CEO directly."<br/>
> Agent: "I understand you're frustrated. Before you do that, would you like me to escalate this internally so we can resolve it today?"<br/>
> User: "No, I'll handle it."

- 정답: `escalated` (사용자가 사람이 응대하는 채널—CEO 이메일—로 직접 연락하겠다고 밝힘)
- Judge: `not_escalated` (사용자가 에이전트의 전달 제안을 거절함)

이 케이스는 실제로 경계선상에 있는 사례입니다. 사용자는 앱 내 에스컬레이션은 거절했지만 외부에서 사람에게 연락하겠다고 명시했습니다. 여러분의 프롬프트에는 "특정 채널로 사람에게 직접 연락하겠다고 말함(특정 팀에 이메일 등)" → escalated 규칙과 "에이전트의 전달 제안을 거절함" → not_escalated 규칙이 모두 있는데, 이 항목이 두 규칙 모두를 발동시킵니다. 어느 신호를 우선시할지 정하고, judge가 이런 케이스를 일관되게 처리하도록 하려면 프롬프트를 다듬을 필요가 있습니다.

**권장 사항:** 배포해도 좋습니다. 메트릭이 견고하며 (과잉 플래깅 없이) 바람직한 방향으로 편향되어 있습니다. 다만 모든 에스컬레이션을 놓치지 않고 잡아내야 하는 의사 결정에 이 judge를 사용하기 전에는 위의 규칙 충돌을 먼저 해결하세요. 프롬프트를 수정한 후 FN이 줄어드는지 확인하기 위해 다시 실행하세요.

</Details>

### 프롬프트 반복 개선하기

리포트에서 judge가 충분히 자주 여러분의 라벨과 일치한다고 나오면 바로 배포해도 됩니다. 결과를 더 개선하고 싶다면, 불일치 사항을 해결하도록 judge 프롬프트를 수정하고 캘리브레이션을 다시 실행합니다. 두 가지 방법이 있습니다.

- **수동으로** Langfuse 프롬프트 관리에서 프롬프트를 열어 수정한 뒤, 위의 캘리브레이션 단계를 다시 실행합니다.
- **AI 어시스턴트에게 요청**하여 결과를 바탕으로 프롬프트를 변경합니다. 이 스킬은 이미 내부적으로 [Langfuse CLI](https://github.com/langfuse/langfuse-cli)를 사용하고 있으므로, 에이전트가 프롬프트를 직접 업데이트할 수 있습니다.

</Steps>

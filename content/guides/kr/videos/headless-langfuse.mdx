# 코딩 에이전트로 사용하는 헤드리스 Langfuse

LLM 관측성과 평가를 설정하려면 보통 SDK 코드를 직접 추가하고, UI를 클릭하며 테스트 세트를 만들고, 평가기를 하나씩 연결해야 합니다. 이 가이드에서는 Langfuse UI를 한 번도 열지 않고 코딩 에이전트와 대화하는 것만으로 이 모든 작업을 수행합니다.

이러한 방식을 **헤드리스(Headless)**라고 부릅니다. [Langfuse Agent Skill](/docs/api-and-data-platform/features/agent-skill)은 Langfuse를 자연어로 제어할 수 있는 플랫폼으로 바꿔줍니다. 작업 내용을 설명하기만 하면 에이전트가 리서치를 기반으로 정립된 플레이북을 에디터 안에서 그대로 실행합니다.

## 핵심 재료: Langfuse Agent Skill [#agent-skill]

[Langfuse Agent Skill](/docs/api-and-data-platform/features/agent-skill)은 오픈소스이며 공개된 [Agent Skills](https://github.com/anthropics/skills) 표준을 따르므로 Claude Code, Cursor, Codex, Windsurf 등 사용 중인 어떤 도구에서도 동작합니다.

내부적으로 에이전트에게 다음 두 가지를 제공합니다.

- [Langfuse CLI](/docs/api-and-data-platform/features/cli)를 통한 전체 Langfuse REST API에 대한 **프로그래밍 방식 접근**.
- 에이전트가 추측 대신 최신 모범 사례를 조회할 수 있도록 하는 **문서 검색** 기능.

이 스킬은 **점진적 공개(progressive disclosure)** 방식을 사용합니다. 에이전트의 컨텍스트에는 아주 짧은 설명만 유지되고, 실제 작업과 관련이 있을 때만 전체 지침이 로드됩니다. 덕분에 컨텍스트 사용량을 낮게 유지하면서도 필요한 순간에 전문 지식을 제공할 수 있습니다.

## 1단계: 앱에 트레이싱 추가하기 [#add-tracing]

LLM을 호출하지만 관측성이 전혀 없는 앱에서 시작해 보겠습니다. 프로덕션에서 어떤 일이 일어나는지 알 수 없는 상태입니다. 에이전트에게 다음과 같이 요청합니다.

> Langfuse 스킬을 사용해서 모범 사례에 따라 이 애플리케이션에 트레이싱을 추가해줘.

에이전트는 문서를 읽고 여러분의 스택을 감지한 뒤 알맞은 통합을 연결합니다.

- **JavaScript 또는 TypeScript** 앱이라면 OpenTelemetry 스팬 프로세서와 Langfuse 래퍼를 사용합니다.
- **Python**이라면 `pip`으로 `langfuse`를 설치합니다. 이 SDK는 내부적으로 OpenTelemetry 기반입니다.

또한 필요한 세 가지 환경 변수(public key, secret key, region)를 스캐폴딩하고, 프로젝트 설정에서 값을 붙여넣도록 안내합니다. 키를 직접 가져오지는 않지만 그 외의 모든 작업을 대신 처리해줍니다.

## 2단계: 실제 프로덕션 데이터 생성하기 [#generate-data]

이제 앱을 잠시 사용해 보세요. 잘 처리될 만한 질의 몇 개와, 실패할 것으로 예상되는 질의 몇 개를 섞습니다. Langfuse에서는 [트레이스](/docs/observability/get-started)가 쌓이기 시작합니다. 모든 호출, 입력, 출력, 지연 시간, 비용이 기록됩니다. 이것이 바로 원재료입니다.

## 3단계: 헤드리스 방식으로 트레이스 분석하기 [#analyze]

직접 트레이스를 클릭하며 살펴보는 대신, 에이전트에게 분석을 요청합니다.

> 내 Langfuse 트레이스 중 최근 20개를 가져와서 분석해줘. 앱이 어디서 성능이 떨어지고 있고, 어떤 트레이스가 실패했어?

내부적으로 에이전트는 CLI를 호출해 트레이스를 JSON 형태로 가져온 뒤 그 내용을 분석합니다. 그리고 어떤 응답이 환각(hallucination)을 일으켰는지, 긴 입력에서 지연 시간이 어디서 급증하는지, 실패한 구체적인 트레이스 ID까지 담긴 실제 분석 결과를 제시합니다.

이렇게 실패한 트레이스는 매우 값진 자산입니다. 앞으로 반드시 방어해야 할 정확한 케이스이기 때문입니다.

## 4단계: 실패 사례를 데이터셋으로 전환하기 [#build-dataset]

Langfuse의 [데이터셋](/docs/evaluation/experiments/datasets)은 테스트에 사용할 입력과 기대 출력의 모음입니다. 에이전트에게 다음과 같이 요청합니다.

> `edge-cases`라는 데이터셋을 만들고, 앞서 확인한 실패한 트레이스 3개를 항목으로 추가해줘.

에이전트는 데이터셋을 생성하고 각 실패한 트레이스를 데이터셋 항목으로 바로 연결합니다. 이는 문서에서 권장하는 것과 동일한 워크플로입니다. 프로덕션에서 실패한 트레이스를 선별한 뒤 전문가가 기대 출력을 추가하는 방식이지만, 번거로운 작업은 에이전트가 대신 처리해줍니다. 이제 실제 실패 사례로 만들어진 재사용 가능한 회귀 테스트 세트가 생겼습니다.

## 5단계: LLM-as-a-judge 평가기 설정하기 [#evaluate]

출력 결과를 계속 수작업으로 채점할 수는 없으므로, [LLM-as-a-judge](/docs/evaluation/evaluation-methods/llm-as-a-judge) 평가기를 설정합니다. 이는 여러분이 정의한 기준에 따라 출력을 채점하는 성능 좋은 모델입니다. 에이전트에게 다음과 같이 요청합니다.

> 환각(hallucination)과 유용성(helpfulness)을 평가하는 LLM-as-a-judge 평가기를 설정하고, `edge-cases` 데이터셋에서 실행해줘.

Langfuse는 환각, 유해성(toxicity), 유용성 등에 대한 관리형 평가기를 프롬프트 작성 없이 바로 제공하며, 원한다면 커스텀 루브릭을 정의할 수도 있습니다. 점수는 숫자형, 범주형, 불리언형이 될 수 있으며 각 실행에 바로 연결됩니다.

권장하는 패턴은 반복 개선을 진행하는 동안 데이터셋에 대해 [실험](/docs/evaluation/experiments/data-model)을 사용하고, 이후 샘플링을 적용해 실제 프로덕션 트레이스에 평가기를 배포하는 것입니다. 이렇게 하면 앱의 향후 모든 버전이 여러분이 확보한 최악의 사례들을 기준으로 자동으로 채점됩니다.

## 전체 루프 [#the-full-loop]

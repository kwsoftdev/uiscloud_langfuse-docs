# 트레이싱으로 팬아웃 RAG 에이전트 디버깅하기

이것은 RAG를 기반으로 제품 관련 질문에 답하는 지원 에이전트입니다. 하나의 고객 질문이 단일 LLM 호출을 트리거하는 단순한 구조와 달리, 이 에이전트는 팬아웃 방식으로 동작합니다. 하나의 질문에 대해 다음과 같은 작업을 수행합니다.

- 주제 분류
- 계획 수립
- 쿼리를 세 가지 방식으로 재작성
- 각 재작성 결과를 임베딩
- 벡터 스토어 검색
- 결과 재순위화
- 백엔드 도구 호출
- 답변 작성
- 두 명의 별도 심사자로 자체 결과물 채점

에이전트가 이렇게 많은 단계로 분기될 때, 출력만 읽어서는 디버깅할 수 없습니다 — [트레이스](/docs/observability/get-started)를 읽어야 합니다.

## 확신에 차 있고 유창하지만 틀린 답변 [#the-symptom]

에이전트에게 한 가지를 물어봅니다.

> Pro 플랜에서 API 요청 한도를 늘리려면 어떻게 해야 하나요?

돌아온 답변은 확신에 차 있고 유창하지만 완전히 틀렸습니다. 답변이 가리킨 엔드포인트는 요청 한도가 아니라 결제에 관한 것으로, 실제 작업과는 아무 관련이 없었습니다.

최종 답변이 틀렸을 때 "에이전트가 고장났다"는 알림만으로는 아무 도움이 되지 않습니다. 실제로 실패한 단계는 어디일까요? 그것을 알려주는 것이 바로 트레이스입니다.

## 트레이스를 따라 잘못된 점수 거슬러 올라가기 [#tracing-it-back]

이 특정 실행에 대한 [트레이스](/docs/observability/get-started)를 열어봅니다. 상단의 [점수](/docs/evaluation/evaluation-methods/scores-via-ui)를 보면 **검색 관련성이 매우 낮게** 나타나 있습니다 — 상류 어딘가에서 문제가 발생한 것이 분명합니다. 단계별로 거슬러 올라가 봅니다.

- **분류 단계** — 가장 먼저 의심되는 부분입니다. 질문을 잘못 이해했을까요? 아닙니다: 출력을 보면 이 질문을 요청 한도(rate limits) 질문으로 정확히 분류했습니다. 여기가 문제는 아닙니다.
- **검색 단계** — 메타데이터에는 `rate_limits`로 분류되어 있지만, 실제로 적용된 필터는 `billing`입니다. 가져온 모든 문서는 결제 관련 문서이며, 유사도 점수도 0에 가깝습니다. 돌아온 결과는 애초에 질문과 아무 관련이 없었습니다.
- **도구 호출** — 다른 각도에서 같은 문제가 반복됩니다. 도구 선택 역시 동일한 카테고리 필드를 기준으로 이루어지므로, `get_billing_summary`를 호출해 요청 한도 도구 대신 고객의 청구서 데이터를 가져왔습니다.
- **답변 단계** — 전달받은 모든 내용이 결제 관련 콘텐츠입니다: 문서와 계정 데이터뿐이며, 요청 한도에 관한 내용은 어디에도 없습니다.

결국 에이전트는 주어진 자료로 가장 자연스러운 일을 했습니다: 가장 그럴듯해 보이는 조각들을 이어 붙여 확신에 찬, 근거 없는 답변을 만들어낸 것입니다. 이것이 바로 **환각(hallucination)**입니다 — 모델이 거짓말을 한 것이 아니라, 에이전트에게 잘못된 자료가 주어진 것입니다.

## 하나의 깨진 라인 [#root-cause]

모든 증상 — 잘못된 검색, 잘못된 도구, 환각 — 은 단 한 줄로 귀결됩니다. 라우팅 테이블의 잘못된 항목 하나가 `rate_limits`를 `billing` 필터에 매핑하고 있었습니다. 예외도, 오류 로그도 없었고, 아무것도 크래시되지 않았습니다. 그저 설정 파일의 한 행에 유창하고 확신에 차 있지만 틀린 답변이 자리 잡고 있었을 뿐입니다.

## 수정하고 몇 초 만에 확인하기 [#fix-and-confirm]

정확한 문제 상황을 코딩 에이전트에게 전달합니다.

> 이 라우팅 테이블은 `rate_limits` 카테고리를 잘못된 필터에 매핑하고 있습니다 — `rate_limits`가 아니라 `billing`을 가리키고 있습니다. 매핑을 수정해 주세요.

에이전트가 해당 항목을 찾아 수정합니다. 같은 질문을 다시 던져봅니다.

> Pro 플랜에서 API 요청 한도를 늘리려면 어떻게 해야 하나요?

이번에는 `rate_limits`로 올바르게 라우팅되어 실제 요청 한도 문서를 검색하고, 요청 한도 도구를 호출한 뒤, 설정에서 한도 증가를 요청하거나 요청 한도 증가 엔드포인트를 이용하도록 사용자에게 올바르게 안내합니다. **관련성과 충실도(faithfulness) 모두 높은 수준으로 회복됩니다.** 겉으로는 동일한 단계 트리이지만, 이제는 모든 분기가 제 역할을 하고 있습니다.

## 문제는 애초에 LLM이 아니었습니다 [#takeaway]

분류는 정확했고, 모델은 충분한 역량을 갖추고 있었으며, 어떤 오류도 발생하지 않았습니다. 실패는 모델이 답변을 작성하기 세 단계 전, 그보다 앞선 체인 어딘가에 있었습니다.

트레이싱이 없다면 그저 추측만 하게 됩니다: 모델에 다시 프롬프트를 넣어보거나, 검색 품질을 탓하거나, 도움이 되지 않는 재시도를 추가하는 식입니다. 하나의 트레이스만 있으면 잘못된 점수를 곧장 따라가 정확히 어느 단계가 깨졌는지 찾아내고, 실제 원인을 수정한 뒤, 몇 초 만에 확인할 수 있습니다.

에이전트가 여러 단계로 팬아웃될 때는 트레이스를 읽으십시오.

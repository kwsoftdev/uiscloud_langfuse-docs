# OpenTelemetry를 통해 모든 언어에서 Langfuse 사용하기

Langfuse SDK는 Python과 JavaScript/TypeScript를 지원하지만, Langfuse 자체는 [OpenTelemetry 백엔드](/integrations/native/opentelemetry)이기 때문에 OpenTelemetry SDK가 있는 어떤 언어에서든 Langfuse OTLP 엔드포인트로 트레이스를 전송할 수 있습니다. 스택이 Go, Java, C#, Ruby이거나 OTel을 지원하는 다른 언어라면, 네이티브 SDK가 나올 때까지 기다릴 필요가 없습니다.

**요약:** OpenTelemetry 익스포터가 OTLP를 HTTP(protobuf 또는 JSON, gRPC는 지원되지 않음)로 프로젝트 키로 생성한 Basic Auth 헤더와 함께 `https://cloud.langfuse.com/api/public/otel`로 전송하도록 구성하세요. [GenAI semantic convention](https://opentelemetry.io/docs/specs/semconv/gen-ai/) 속성(`gen_ai.*`)을 가진 스팬은 Langfuse에서 모델, 토큰, 비용 데이터를 포함한 생성(generation)으로 표시되며, 나머지 스팬은 그 주위에 일반 관측값(observation)으로 중첩됩니다.

## 모든 언어가 공유하는 설정

OpenTelemetry SDK들은 동일한 환경 변수를 읽으므로, 대부분의 설정에서는 이 블록이 통합 작업의 전부입니다.

```bash
# 프로젝트 API 키로부터 인증 문자열을 생성합니다:
#   echo -n "pk-lf-...:sk-lf-..." | base64
OTEL_EXPORTER_OTLP_ENDPOINT="https://cloud.langfuse.com/api/public/otel" # 🇪🇺 EU 데이터 리전
# OTEL_EXPORTER_OTLP_ENDPOINT="https://us.cloud.langfuse.com/api/public/otel" # 🇺🇸 US 데이터 리전
OTEL_EXPORTER_OTLP_HEADERS="Authorization=Basic ${AUTH_STRING},x-langfuse-ingestion-version=4"
```

SDK가 시그널별 설정을 필요로 한다면, 트레이스 경로는 `/api/public/otel/v1/traces`입니다. 자세한 내용과 속성 매핑은 [OpenTelemetry 통합 문서](/integrations/native/opentelemetry)를 참고하세요.

스팬이 Langfuse에서 어떻게 표시되는지는 다음 두 가지 규칙으로 결정됩니다.

1. **`gen_ai.*` 속성을 가진 스팬은 생성(generation)이 됩니다.** 모델 이름, 토큰 사용량, 비용은 semantic convention에서 매핑됩니다.
2. **그 외의 모든 것은 트레이스 트리에서 일반 관측값(observation)이 됩니다.** 사용자 및 세션 컨텍스트를 연결하려면 스팬에 `user.id`와 `session.id` 속성을 설정하세요.

## Go

표준 OTel Go SDK를 HTTP 트레이스 익스포터와 함께 사용하면, 위의 환경 변수를 그대로 읽어들입니다.

```go
import (
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/exporters/otlp/otlptrace/otlptracehttp"
    "go.opentelemetry.io/otel/sdk/trace"
)

exp, err := otlptracehttp.New(ctx) // OTEL_EXPORTER_OTLP_* 환경 변수를 읽습니다
tp := trace.NewTracerProvider(trace.WithBatcher(exp))
otel.SetTracerProvider(tp)
defer tp.Shutdown(ctx) // 종료 전에 flush합니다; 그렇지 않으면 짧게 실행되는 프로그램은 스팬을 잃게 됩니다
```

LLM 호출을 계측하려면 해당 호출을 감싸는 스팬에 `gen_ai.*` 속성을 설정하세요. Go에서 LLM 자동 계측을 원한다면 [OpenLIT](/docs/opentelemetry/example-openlit)이 동일한 방식으로 Langfuse에 내보내는 라이브러리 지원을 제공합니다.

## Java

가장 간편한 방법은 OpenTelemetry Java 에이전트를 사용하는 것입니다: JVM에 에이전트를 연결하고 위의 환경 변수만으로 설정을 완료할 수 있습니다.

```bash
OTEL_TRACES_EXPORTER=otlp \
OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf \
java -javaagent:opentelemetry-javaagent.jar -jar app.jar
```

이 에이전트는 HTTP 호출을 자동으로 계측합니다. LLM 클라이언트 호출이 일반 HTTP 스팬이 아니라 생성(generation)으로 표시되도록 하려면 `gen_ai.*` 속성을 가진 스팬으로 감싸거나, Java에서 자동 LLM 계측을 확장하는 [OpenLLMetry](/docs/opentelemetry/example-openllmetry)를 사용하세요.

## C# / .NET

OpenTelemetry .NET SDK를 OTLP 익스포터의 프로토콜을 HTTP로 설정하여 사용하세요.

```csharp
using OpenTelemetry;
using OpenTelemetry.Trace;
using OpenTelemetry.Exporter;

using var tracerProvider = Sdk.CreateTracerProviderBuilder()
    .AddSource("my-llm-app")
    .AddOtlpExporter(o => o.Protocol = OtlpExportProtocol.HttpProtobuf) // OTLP 환경 변수를 읽습니다
    .Build();
```

LLM 호출을 위한 `ActivitySource`를 생성하고 해당 activity에 `gen_ai.*` 태그를 설정하세요. Semantic Kernel과 같은 프레임워크를 사용하는 팀은 [OpenTelemetry 문서](/integrations/native/opentelemetry)에 나열된 OTel 기반 계측 라이브러리를 통해 해당 텔레메트리를 Langfuse로 라우팅할 수도 있습니다.

## Ruby

OTel Ruby SDK는 기본적으로 OTLP를 HTTP로 내보내며, 공유 환경 변수를 그대로 따릅니다.

```ruby
require "opentelemetry/sdk"
require "opentelemetry/exporter/otlp"

OpenTelemetry::SDK.configure # OTEL_EXPORTER_OTLP_* 환경 변수를 읽습니다

tracer = OpenTelemetry.tracer_provider.tracer("my-llm-app")
tracer.in_span("chat-completion", attributes: {
  "gen_ai.system" => "openai",
  "gen_ai.request.model" => "gpt-4.1",
  "user.id" => user_id,
}) do |span|
  # 여기서 LLM 제공자를 호출한 다음, 사용량을 기록합니다:
  span.set_attribute("gen_ai.usage.input_tokens", usage.prompt_tokens)
  span.set_attribute("gen_ai.usage.output_tokens", usage.completion_tokens)
end
```

Ruby에는 아직 자동 LLM 계측 생태계가 없으므로, 위와 같은 수동 `gen_ai.*` 패턴이 표준적인 접근 방식입니다.

## 통합 검증하기

1. 트레이스가 기록되는 요청을 하나 보내고 익스포터를 flush하세요(짧게 실행되는 프로세스는 트레이서 프로바이더를 명시적으로 종료해야 합니다. 그렇지 않으면 마지막 배치가 프로세스를 벗어나지 못합니다).
2. Langfuse에서 트레이스를 열어 LLM 호출이 일반 스팬이 아니라 생성(generation)으로 표시되는지 확인하세요.
3. 스팬은 도착했지만 토큰 사용량이나 입력/출력이 누락된 경우, [속성 매핑 표](/integrations/native/opentelemetry)와 속성 이름을 비교해 보세요. 이름이 거의 비슷한 속성이 가장 흔한 원인입니다.

## FAQ

### 셀프 호스팅 Langfuse에서도 동작하나요?

네. 엔드포인트 호스트를 배포된 도메인으로 바꾸면 됩니다. OTLP 엔드포인트 경로는 동일합니다.

### 여러 언어에서 하나의 프로젝트로 트레이스를 보낼 수 있나요?

네. OTLP 엔드포인트는 어떤 SDK가 스팬을 생성했는지 신경 쓰지 않으므로, 다중 언어로 구성된 시스템이 Go, Java, Python 스팬을 동일한 프로젝트로 보낼 수 있습니다. 서비스 간에 W3C `traceparent` 헤더를 전파하면 분산 트레이스가 서비스 전체에 걸쳐 연결됩니다.

### gRPC는 왜 동작하지 않나요?

Langfuse의 OTLP 엔드포인트는 protobuf 또는 JSON 인코딩을 사용하는 HTTP만 지원합니다. 기본값이 gRPC인 SDK(Java 에이전트와 일부 컬렉터 파이프라인)에서는 `http/protobuf`를 명시적으로 설정하세요.

### 중간에 OpenTelemetry Collector를 사용해야 하나요?

선택 사항이지만 유용한 경우가 많습니다. 컬렉터를 사용하면 동일한 스팬을 Langfuse와 APM 백엔드로 동시에 분산시키거나, 샘플링을 적용하거나, 인터넷에 직접 접근할 수 없는 환경에서 브리지 역할을 할 수 있습니다.

# LLM-as-a-judge로 RAG 충실성(faithfulness)을 평가하는 방법

**TL;DR:** 충실성(faithfulness)은 RAG 답변이 검색된 컨텍스트에 근거하는지를 측정합니다. 답변이 만들어내는 사실적 주장(factual claim)들을 추출하고, 각 주장을 검색된 문서와 대조하여 확인한 뒤, 지지되는 주장의 비율을 점수로 산출하는 방식입니다. 이는 참조 없이(reference-free) 계산되므로 정답(ground-truth) 답변이 필요하지 않으며, 그 덕분에 어떤 데이터셋에서도, 그리고 실제 트래픽에서도 실행할 수 있는 RAG 메트릭이 됩니다. 이 페이지에서는 이 메트릭을 정의하고, 충실성 평가기(judge) 프롬프트를 설계하는 방법을 보여준 뒤, Langfuse Python SDK로 이를 처음부터 끝까지 구현합니다: `run_experiment`를 사용한 오프라인 실험(experiment)으로, 그리고 프로덕션 트레이스에 대한 관리형(managed) LLM-as-a-judge 평가기(evaluator)로 각각 구현합니다.

## RAG 충실성이 측정하는 것

충실성(그라운드성, groundedness라고도 부름)은 답변이 해당 답변을 위해 검색된 컨텍스트에 의해 얼마나 뒷받침되는지를 나타내는 정도입니다. 충실한 답변은 검색된 문서가 말하거나 직접적으로 암시하는 내용만을 서술합니다. 충실하지 않은 답변은 모델의 파라메트릭 지식(parametric knowledge)에서 온 주장을 섞어 넣거나, 문장이 매끄럽고 답변이 우연히 맞더라도 세부 사항을 아예 지어냅니다.

Ragas가 널리 알린 표준적인 정식화는 주장(claim) 기반입니다:

```
faithfulness = 검색된 컨텍스트에 의해 지지되는 주장의 수 / 답변에 포함된 전체 주장 수
```

점수 1.0은 모든 주장이 컨텍스트로 거슬러 올라갈 수 있음을 의미합니다. 점수 0.5는 답변의 절반이 지지되지 않음을 의미합니다. 비교 대상이 정답 답변이 아니라 검색된 컨텍스트이기 때문에, 충실성은 참조 없는(reference-free) 메트릭입니다. 즉 아무도 기대 답변을 작성하지 않은 질문을 포함해 어떤 질문에 대해서도 계산할 수 있습니다.

충실성은 RAG 평가에서 흔히 혼동되는 세 가지 메트릭 중 하나이며, 이들은 서로 다른 질문에 답합니다:

| 메트릭                             | 답하는 질문                                             | 비교 대상       |
| ---------------------------------- | ------------------------------------------------------- | --------------- |
| 충실성(Faithfulness)               | 답변의 모든 주장이 검색된 컨텍스트에 의해 지지되는가?   | 검색된 컨텍스트 |
| 답변 관련성(Answer relevance)      | 답변이 실제로 던져진 질문을 다루고 있는가?              | 사용자 질문     |
| 환각 탐지(Hallucination detection) | RAG 흐름 안팎을 막론하고 모델이 내용을 조작하고 있는가? | 모든 소스       |

답변은 완벽하게 충실하면서도 쓸모없을 수 있습니다: "저희 환불 정책은 주문을 보장합니다"는 환불 문서에 근거하지만 "이걸 반품하는 데 시간이 얼마나 걸리나요?"라는 질문에는 답하지 못합니다. 이러한 실패는 [답변 관련성 평가](/resources/engineering/answer-relevance-evaluation)의 영역입니다. 충실성은 환각 탐지보다도 범위가 좁습니다. 세상 지식이 아니라 여러분이 통제하는 문서를 기준으로 검증하는, RAG에 특화되고 컨텍스트로 범위가 제한된 문제 버전입니다.

## 충실성을 측정하는 방법

충실성은 독해력을 요구하므로 문자열 매칭이 아니라 LLM 평가기(judge)로 측정합니다. 인용 존재 여부나 어휘적 중첩(lexical overlap) 같은 결정론적 체크는 유용한 사전 필터가 될 수 있지만, 바꿔 말한 것(paraphrase)과 지어낸 것(fabrication)을 구분하지는 못합니다. 도구와 무관한 측정 파이프라인은 세 단계로 구성됩니다:

1. **답변을 주장(claim)으로 분해합니다.** 답변에 담긴 각각의 원자적인 사실 진술이 평가 단위 하나가 됩니다. "환불은 영업일 기준 5일이 걸리며 원래 결제 수단으로 이루어집니다"는 두 개의 주장입니다.
2. **각 주장을 검색된 컨텍스트와 대조하여 검증합니다.** 평가기 모델은 컨텍스트와 하나 이상의 주장을 받아, 컨텍스트가 그 주장을 명시하거나 직접적으로 암시하는지 주장별로 판단합니다. 세상 지식은 지지 근거로 인정되지 않습니다.
3. **집계합니다.** 지지되는 주장의 비율이 충실성 점수입니다. 지지되지 않는 주장 목록은 디버깅용 페이로드로, 어떤 문장을 고쳐야 하는지, 그리고 실패가 생성 단계(컨텍스트에 사실이 있었지만 모델이 무시함)에서 일어났는지 검색 단계(컨텍스트에 애초에 그 사실이 없었음)에서 일어났는지를 알려줍니다.

이를 실행하려면 평가 시점에 질문, 생성된 답변, 그리고 그 답변을 위해 실제로 검색된 컨텍스트라는 세 가지 입력이 손에 있어야 합니다. 이것이 충실성 평가와 트레이싱이 자연스럽게 결합되는 이유입니다. [LLM-as-a-judge](/docs/evaluation/evaluation-methods/llm-as-a-judge) 설정에서 생성 결과 옆에 검색된 문서를 볼 수 있다면 충실성 평가는 프롬프트 하나면 충분하지만, 생성 이후 컨텍스트를 버렸다면 대조할 대상 자체가 없는 것입니다.

## 충실성 평가기 프롬프트 설계하기

평가기 프롬프트는 충실성 점수를 신뢰할 수 있는지를 결정합니다. 가장 중요한 설계 결정은 두 가지입니다.

**주장 추출은 전체적 판단보다 낫습니다.** 전체적 평가기("이 답변의 충실성을 1에서 5까지 평가하라")는 실행할 때마다 흔들리는 점수를 만들어내고 무엇이 실패했는지를 숨깁니다. 주장 추출 평가기는 대신 여러 개의 작은 이진 결정을 내립니다: 주장을 나열하고, 각각을 지지됨 또는 지지되지 않음으로 표시합니다. 각 결정은 평가기 모델 입장에서 전역적인 등급 매기기보다 쉽고, 집계 점수는 명확한 의미(지지되는 주장의 비율)를 가지며, 지지되지 않은 주장 목록은 점수 코멘트를 읽는 사람에게 실패 지점을 특정해 줍니다.

**이진 결정은 등급형 척도보다 낫습니다.** Langfuse Academy의 [평가 모듈](/academy/evaluate)은 평가기 설계 전반에서 1~5점 척도보다 이진 통과/실패 점수를 권장하는데, 이진 방식은 무엇이 허용 가능하고 무엇이 허용 불가능한지를 정의하도록 강제하는 반면 등급형 척도는 "3점과 4점을 가르는 것이 무엇인가"를 정의하지 않은 채로 남기기 때문입니다. 주장 단위 충실성은 이 두 가지 속성을 모두 얻습니다: 평가기의 모든 개별 결정은 이진이고, 최종 숫자 점수는 평가기가 직접 매긴 등급이 아니라 집계에서 나옵니다.

이 두 결정 외에도, 흔한 실패 유형을 막는 프롬프트 규칙들이 있습니다:

- **무엇을 주장으로 볼지 정의합니다.** 추출 대상을 사실 진술로 제한합니다. 완곡한 표현, 사용자에게 되묻는 질문, 그리고 능력 부족을 나타내는 진술("문서에서 이를 찾을 수 없었습니다")은 주장이 아니며 감점 대상이 되어서는 안 됩니다. 그렇지 않으면 평가기는 RAG 시스템에게 바라는 정직한 행동을 오히려 벌하게 됩니다.
- **외부 지식 사용을 명시적으로 금지합니다.** 평가기는 기본적으로 참인 진술을 지지됨으로 표시하는 경향이 있습니다. 어떤 주장이 참이더라도, 컨텍스트가 명시하거나 직접적으로 암시하는 경우에만 지지된 것으로 간주한다고 명시하세요.
- **바꿔 말한 표현(paraphrase)을 허용합니다.** 지지 여부는 문자 그대로의 일치를 요구하지 않습니다. 이 규칙이 없으면 평가기는 컨텍스트를 정확하게 요약한 답변에도 낮은 점수를 매기게 됩니다.
- **구조화된 출력을 요구합니다.** 평가기가 주장별로 항목 하나씩 담긴 JSON을 반환하게 하고, 점수 계산은 코드에서 수행하세요. 평가기에게 산술 계산을 시키지 마세요.
- **평가기의 temperature를 0으로 고정합니다**. 그래야 동일한 답변을 반복 평가해도 결과가 일치합니다.

## Langfuse Python SDK로 충실성 실험 실행하기

아래 스크립트는 독립적으로 실행됩니다: 질문 데이터셋을 만들고, 그에 대해 작은 RAG 태스크를 실행한 뒤, Python SDK v4 실험(experiment) 러너를 사용해 주장 추출 방식의 충실성 평가기로 모든 답변을 채점합니다. 태스크는 답변과 함께 사용된 컨텍스트도 반환하므로, 평가기는 표준 [SDK를 통한 실험](/docs/evaluation/experiments/experiments-via-sdk) 계약을 통해 필요한 모든 것을 전달받습니다. 예시로 사용된 코퍼스와 검색 함수는 실제 벡터 스토어로 교체하세요.

```python
# pip install langfuse openai
import json

from langfuse import Evaluation, get_client
from langfuse.openai import OpenAI  # 자동으로 트레이싱되는 드롭인 OpenAI 클라이언트

# 환경 변수에서 LANGFUSE_PUBLIC_KEY, LANGFUSE_SECRET_KEY, LANGFUSE_HOST를 읽습니다
langfuse = get_client()
client = OpenAI()

# --- 벡터 스토어를 대신하는 최소한의 코퍼스와 검색기 ---

DOCUMENTS = [
    "Orders can be refunded within 30 days of delivery. Refunds are issued to "
    "the original payment method within 5 business days.",
    "Standard shipping takes 3 to 5 business days within the EU. Express "
    "shipping delivers within 24 hours for orders placed before 2pm.",
    "All devices include a 2-year manufacturer warranty covering hardware "
    "defects. Water damage is not covered by the warranty.",
]

def retrieve(question: str) -> list[str]:
    words = question.lower().split()
    ranked = sorted(DOCUMENTS, key=lambda d: -sum(w in d.lower() for w in words))
    return ranked[:2]

# --- 최초 1회 설정: Langfuse에 데이터셋 생성 ---

langfuse.create_dataset(name="rag-faithfulness-demo")
for question in [
    "How long do I have to return an order?",
    "When does express shipping arrive?",
    "Does the warranty cover water damage?",
]:
    langfuse.create_dataset_item(
        dataset_name="rag-faithfulness-demo",
        input={"question": question},
        # expected_output 없음: 충실성은 참조가 필요 없는(reference-free) 메트릭입니다
    )

# --- 평가 대상이 되는 RAG 태스크 ---

def rag_task(*, item, **kwargs):
    question = item.input["question"]
    contexts = retrieve(question)
    context_block = "\n\n".join(contexts)
    response = client.chat.completions.create(
        model="gpt-4.1-mini",
        messages=[
            {
                "role": "system",
                "content": "Answer using only the provided context. If the "
                "context is insufficient, say you do not know.",
            },
            {
                "role": "user",
                "content": f"Context:\n{context_block}\n\nQuestion: {question}",
            },
        ],
    )
    # 평가기가 대조 검증할 수 있도록 답변과 함께 컨텍스트도 반환합니다
    return {
        "answer": response.choices[0].message.content,
        "contexts": contexts,
    }

# --- 충실성 평가기(judge) ---

JUDGE_PROMPT = """You are verifying whether an answer is faithful to its source context.

Context:
{context}

Answer:
{answer}

Extract every factual claim the answer makes. Hedges, questions, and statements
of inability ("I don't know") are not claims. For each claim, decide whether the
context supports it. A claim is supported only if the context states or directly
implies it; paraphrase counts, outside knowledge does not, even if the claim is true.

Respond with JSON only, marking each claim as supported true or false:
{{"claims": [{{"claim": "<claim text>", "supported": true}}]}}"""


def faithfulness(*, input, output, **kwargs):
    context = "\n\n".join(output["contexts"])
    response = client.chat.completions.create(
        model="gpt-4.1",
        messages=[
            {
                "role": "user",
                "content": JUDGE_PROMPT.format(
                    context=context, answer=output["answer"]
                ),
            }
        ],
        response_format={"type": "json_object"},
        temperature=0,
    )
    claims = json.loads(response.choices[0].message.content)["claims"]
    if not claims:
        return Evaluation(
            name="faithfulness", value=1.0, comment="No factual claims made"
        )
    unsupported = [c["claim"] for c in claims if not c["supported"]]
    return Evaluation(
        name="faithfulness",
        value=1 - len(unsupported) / len(claims),
        comment=(
            f"Unsupported claims: {unsupported}"
            if unsupported
            else f"All {len(claims)} claims supported"
        ),
    )


def avg_faithfulness(*, item_results, **kwargs):
    values = [
        e.value
        for result in item_results
        for e in result.evaluations
        if e.name == "faithfulness" and isinstance(e.value, (int, float))
    ]
    return Evaluation(
        name="avg_faithfulness",
        value=sum(values) / len(values) if values else None,
    )

# --- 실험 실행 ---

dataset = langfuse.get_dataset("rag-faithfulness-demo")

result = dataset.run_experiment(
    name="Faithfulness baseline",
    description="Claim-level faithfulness judge on the RAG demo dataset",
    task=rag_task,
    evaluators=[faithfulness],
    run_evaluators=[avg_faithfulness],
)

print(result.format())
```

모든 태스크 실행은 트레이싱되고, 항목별 점수는 해당 항목의 트레이스에 첨부되며, `avg_faithfulness`는 데이터셋 실행(run)에 첨부되므로 연속된 실행들을 Langfuse UI에서 비교할 수 있습니다. 여기서부터 유용한 다음 단계는 검색이나 프롬프트를 변경하기 전과 후의 실행을 비교하는 것, 그리고 실행 단위 평균값을 충실성이 저하되면 풀 리퀘스트를 실패시키는 CI 게이트로 전환하는 것입니다.

## 관리형 LLM-as-a-judge 평가기로 프로덕션 트레이스 채점하기

오프라인 실험은 여러분이 테스트하기로 생각해 둔 입력들만 다룹니다. 동일한 충실성 판단을 Langfuse의 관리형(managed) LLM-as-a-judge 평가기를 통해 실제 트래픽에도 실행할 수 있습니다. 이 평가기는 여러분 자신의 인프라에 평가 작업을 두지 않고도 프로덕션 관측 데이터(observation)에 대해 서버 사이드에서 실행됩니다:

1. **대상 관측 데이터에 컨텍스트를 계측합니다.** 관측 단위(observation-level) 평가기는 매칭된 관측 데이터에 있는 데이터만 읽으며, 트레이스에서 형제 관측 데이터를 불러오지 않습니다. 평가기가 대조 검증할 대상을 가질 수 있도록, 검색된 문서를 생성(generation) 관측 데이터(또는 루트 관측 데이터)의 input이나 metadata에 기록하세요.
2. **평가기를 생성합니다.** Evaluators 페이지에서 카탈로그(Langfuse와 Ragas를 포함한 파트너들이 관리)에 있는 관리형 평가기를 선택하거나, `{{context}}`, `{{answer}}` 형식의 변수를 사용해 자신만의 충실성 프롬프트로 커스텀 평가기를 정의합니다. 평가기 모델은 LLM 연결(connection)을 통해 설정하며 구조화된 출력을 지원해야 합니다.
3. **실시간 관측 데이터를 대상으로 지정하고 샘플링합니다.** RAG 생성을 포함하는 관측 데이터로 필터링한 뒤 샘플링 비율을 설정합니다. 일반적으로 트래픽의 5~10%를 평가하는 것만으로도 충실성 추이를 파악하기에 충분하며, 전체를 평가하는 비용을 들이지 않아도 됩니다.
4. **변수를 매핑합니다.** 관측 데이터 필드를 프롬프트 변수에 매핑하며, 중첩된 데이터에는 JSONPath 표현식을 사용합니다. Langfuse는 저장하기 전에 최근 프로덕션 데이터를 대상으로 채워진 평가기 프롬프트를 미리보기로 보여줍니다.

점수는 응답이 이미 전송된 이후 비동기적으로 매칭된 관측 데이터에 반영됩니다. 이는 온라인 충실성 채점을 모니터링 및 트리아지(triage) 도구로 만듭니다: 평균이 하락하면 알림을 보내고, 낮은 점수를 받은 트레이스를 리뷰로 라우팅하고, 그것들을 위의 실험 데이터셋에 추가해 다음 릴리스가 지난주의 실패 사례들에 대해 테스트되도록 하세요. 사용자가 보기 전에 충실하지 않은 답변을 반드시 차단해야 하는 체크는 평가기가 아니라 애플리케이션 내 가드레일(guardrail)에 있어야 합니다.

## 새로운 인프라 없이 하는 충실성 평가

Langfuse가 이미 여러분의 RAG 파이프라인을 트레이싱하고 있다면, 이 페이지의 모든 내용은 이미 보유한 배포 환경에서 그대로 실행됩니다. 실험 러너는 트레이싱을 수행하는 것과 동일한 Python SDK의 일부이며, 동일한 API 키로 인증하고, 여러분이 이미 축적한 트레이스와 데이터셋 옆에 점수를 기록합니다. 관리형 평가기도 동일한 Langfuse 인스턴스 내부에서 실행됩니다. 별도로 배포해야 할 평가 서비스도, 두 번째 벤더 계정도, 추가로 운영해야 할 데이터 파이프라인도 없습니다. Langfuse는 오픈 소스이며 [셀프 호스팅](/self-hosting)이 가능하므로, 데이터 거주(residency) 요건이 있는 경우 평가기 입력과 점수를 전적으로 여러분의 인프라 내부에 유지할 수 있습니다.

## Ragas 충실성과 이를 사용해야 하는 경우

[Ragas](https://docs.ragas.io/)는 RAG 평가를 위한 정평 있는 오픈 소스 라이브러리이며, 그 충실성 메트릭은 위에서 설명한 주장 분해(claim-decomposition) 접근법의 잘 알려진 참조 구현입니다. 팀이 이미 Ragas로 평가하고 있다면 그 메트릭들을 다시 작성할 필요는 없습니다. [Ragas 통합](/integrations/frameworks/ragas)은 Ragas 평가를 실행하고 그 결과 점수를 Langfuse 트레이스에 첨부하는 방법을 보여주므로, Ragas 충실성 수치가 다른 모든 것과 동일한 점수 타임라인에 담기게 됩니다.

이 페이지에서 다룬 네이티브 방식은 트레이스와 동일한 플랫폼에서 평가기 프롬프트를 보고 편집할 수 있길 원할 때, 별도의 평가 작업을 운영하지 않고도 프로덕션 트래픽을 샘플링하는 관리형 평가기를 원할 때, 또는 충실성이 하나의 데이터셋, 하나의 실험 실행, 하나의 점수 테이블을 공유해야 하는 여러 메트릭(관련성, 정확성, 커스텀 정책 체크 등) 중 하나일 때 더 적합합니다. 두 접근법은 함께 사용할 수도 있습니다: 개발 단계에서는 Ragas를 오프라인으로, 프로덕션에서는 관리형 평가기를 온라인으로 사용하는 식입니다.

## FAQ

### RAG 충실성과 환각 탐지의 차이는 무엇인가요? [#faithfulness-vs-hallucination]

충실성은 RAG로 범위가 한정됩니다. 해당 요청을 위해 검색된 특정 컨텍스트를 기준으로 답변을 검사하며, 참이지만 컨텍스트로 뒷받침되지 않는 답변은 여전히 실패로 처리됩니다. 환각 탐지는 더 넓은 실패 범주로, 검색 여부와 무관하게 어떤 LLM 출력에서든 조작된 내용을 대상으로 하며 해당 상황에 적용되는 어떤 진실 소스와도 대조하여 확인합니다. RAG 시스템에서는 비교 대상이 여러분이 통제하는 문서이기 때문에 충실성 점수가 대개 가장 실행 가능성 높은 환각 신호가 됩니다. 더 넓은 문제에 대해서는 [환각 탐지](/resources/engineering/hallucination-detection)를 참고하세요.

### 충실성 평가에는 정답(ground-truth) 답변이 필요한가요? [#reference-free-vs-ground-truth]

아니요. 충실성은 답변을 검색된 컨텍스트와 비교하며, 이 둘은 모든 요청에 대해 존재하므로 참조가 필요 없는(reference-free) 메트릭입니다. 이것이 위 예제의 데이터셋에 `expected_output`이 없는 이유이며, 충실성이 샘플링된 프로덕션 트래픽에서도 실행될 수 있는 이유입니다. 답변 정확성(answer correctness) 같은 참조 기반 메트릭은 항목마다 정답 답변이 필요하며 레이블이 있는 데이터가 있는 곳에서만 실행됩니다. 완전한 RAG 평가는 보통 두 종류를 함께 사용합니다.

### 충실성 평가기로는 어떤 모델을 사용해야 하나요? [#judge-model-choice]

지시 사항 준수 능력이 뛰어나고 구조화된 출력을 지원하는 모델을 사용하고, 답변을 생성하는 모델보다 더 강력한 모델을 우선하세요. 긴 컨텍스트를 대상으로 한 주장 검증은 까다로운 독해 작업이기 때문입니다. 반복 가능성을 위해 temperature를 0으로 고정하세요. 사람이 20~50개의 답변에 레이블을 붙이고 평가기의 판정과 비교하는 방식으로 한 번 보정(calibrate)하세요. 일치도가 낮다면 모델을 바꾸기 전에 루브릭(rubric) 문구를 먼저 고치세요. 루브릭의 모호함이 더 흔한 원인이기 때문입니다.

### 충실성은 이진 점수여야 하나요, 아니면 등급형 점수여야 하나요? [#binary-vs-graded]

개별 평가기 결정은 이진으로 만들고 숫자 점수는 집계에서 자연스럽게 나오게 하세요. 각 주장은 컨텍스트에 의해 지지되거나 그렇지 않거나 둘 중 하나이며, 이진 결정은 LLM 평가기가 가장 일관적으로 판단하는 지점입니다. 지지되는 주장의 집계 비율은 대시보드와 임곗값(threshold)에 사용할 수 있는 0~1 사이의 메트릭을 제공하며, 단계 구분이 불분명한 등급형 평가를 평가기에게 요구하지 않아도 됩니다.

### 충실성을 평가하려면 Ragas가 필요한가요? [#ragas-required]

아니요. 충실성은 라이브러리 기능이 아니라 메트릭입니다. 주장 추출 프롬프트를 사용하는 어떤 LLM 평가기라도 이를 구현할 수 있으며, 이 페이지의 실험 코드도 Langfuse SDK만으로 이를 구현합니다. Ragas는 다른 RAG 메트릭들과 함께 그 평가기의 관리된 구현을 제공하며, 그 점수는 통합을 통해 Langfuse로 전송될 수 있습니다. 평가기 프롬프트를 버전 관리하에 여러분 소유로 둘 것인지, 아니면 임포트하는 라이브러리 안에 둘 것인지에 따라 선택하세요.

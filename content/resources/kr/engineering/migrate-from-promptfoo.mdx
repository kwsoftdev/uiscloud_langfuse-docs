# Promptfoo 평가 스위트를 Langfuse로 마이그레이션하기

이 가이드는 기존 [Promptfoo](https://www.promptfoo.dev) 평가 스위트를 [Langfuse](/)
데이터셋과 실험으로 옮기는 과정을 다룹니다: 테스트 케이스부터 시작해 어설션, 프롬프트/프로바이더
매트릭스, 그리고 CI 게이트 순서로 진행합니다.

**요약:** promptfooconfig.yaml은 Langfuse 개념에 깔끔하게 대응됩니다. `tests`와 그 `vars`는
데이터셋 아이템이 되고, 결정론적 어설션은 코드 평가기가 되며, 모델 채점(model-graded) 어설션은
LLM-as-a-judge 평가기가 되고, CI의 `promptfoo eval`은 `langfuse/experiment-action` GitHub
Action 내에서 `RegressionError`를 발생시키는 실험 스크립트가 됩니다. 구조적으로 달라지는 점은
Promptfoo 평가가 결과를 로컬 파일이나 공유 뷰어에 저장하는 선언적 YAML 실행인 반면, Langfuse
실험은 프로덕션 트레이스, 점수, 프롬프트가 이미 존재하는 플랫폼에서 데이터셋을 대상으로 실행되는
코드라는 것입니다. 마이그레이션은 점진적으로 진행하면서 Promptfoo를 나란히 계속 운영할 수 있고,
Promptfoo의 레드 티밍(red teaming) 기능은 Langfuse에 대응 기능이 없으므로 이 부분은 그대로
유지하면 됩니다.

## 팀들이 마이그레이션하는 이유

Promptfoo는 프롬프트 테스트를 위한 훌륭한 CI 하니스입니다: 선언적 설정, 폭넓은 어설션 라이브러리,
그리고 `promptfoo eval`과 `promptfoo view`를 통한 빠른 로컬 루프를 제공합니다. 팀들이 흔히
겪는 한 가지 한계는 평가 결과가 프로덕션과 분리되어 있다는 점입니다. Promptfoo 실행은 출력
파일과 결과 뷰어를 생성하며, promptfoo.app이나 셀프 호스팅 공유 서버로 공유할 수 있습니다
(2026년 7월 기준,
[Promptfoo 공유 문서](https://www.promptfoo.dev/docs/usage/sharing/) 참고). 프로덕션
텔레메트리는 완전히 다른 곳에 존재합니다.

스위트를 Langfuse로 옮기면 다음과 같이 달라집니다:

- **평가 결과가 프로덕션 트레이스 바로 옆에 위치합니다.** 모든 실험 아이템 실행은 완전한
  [트레이스](/docs/observability/data-model)를 생성하므로, 실패한 테스트 케이스도 프로덕션
  인시던트에 사용하는 것과 동일한 도구로 조사할 수 있고, 동일한 데이터셋을 실제 프로덕션
  트레이스로부터 구축할 수 있습니다.
- **비엔지니어도 평가를 확인하고 다듬을 수 있습니다.** 데이터셋, 실험 비교,
  [주석 큐](/docs/evaluation/evaluation-methods/annotation-queues)는 모두 Langfuse UI에
  있으므로, 제품 관리자와 도메인 전문가가 저장소를 체크아웃하지 않고도 출력을 검토하고 테스트
  케이스를 편집할 수 있습니다.
- **평가기는 관리형 플랫폼 평가기로 실행됩니다.** Langfuse는 Ragas와 같은 파트너와 함께 유지
  관리하는 [LLM-as-a-judge 평가기](/docs/evaluation/evaluation-methods/llm-as-a-judge)
  카탈로그(Hallucination, Context-Relevance, Toxicity, Helpfulness)를 제공하며, 동일한
  평가기가 오프라인 실행뿐 아니라 실제 프로덕션 트래픽도 채점합니다.
- **플랫폼 전체를 셀프 호스팅할 수 있습니다.** 두 도구 모두 MIT 라이선스 오픈 소스지만, 실행할
  수 있는 범위가 다릅니다. [Langfuse 셀프 호스팅](/self-hosting)은 평가 이력을 위한 영구적인
  시스템 오브 레코드로서 자체 인프라에 전체 플랫폼(UI, API, 스토리지)을 구축할 수 있게 해줍니다.

무엇을 포기하게 되는지도 명확히 짚어야 합니다. Promptfoo의 선언적 어설션 카탈로그는 폭이
넓습니다(`rouge-n`, `bleu`, `perplexity`, `latency`, `cost` 등 30여 종류의 타입이 있으며 모두
부정(negate) 및 가중치(weight) 설정이 가능합니다). Langfuse에서는 결정론적 검사를 평가기
함수 또는 [코드 평가기](/docs/evaluation/evaluation-methods/code-evaluators)로 직접
작성하며, [autoevals](https://github.com/braintrustdata/autoevals) 라이브러리가 많은
모델 채점 및 휴리스틱 검사를 기본으로 제공합니다. 코드 없는 YAML 워크플로는 Python 또는
TypeScript 스크립트로 바뀝니다. 그리고 `providers × prompts × tests`를 자동으로 확장하는
Promptfoo의 프로바이더 매트릭스는 직접 작성하는 루프가 됩니다. 선언적 하니스만으로 충분하고
결과를 파일로 남기는 방식이 괜찮다면 Promptfoo는 여전히 합리적인 선택입니다. 이 가이드는 트레이스가
존재하는 곳으로 평가를 통합하려는 팀을 위한 것입니다.

## 개념 매핑

| Promptfoo (promptfooconfig.yaml)                                      | Langfuse                                                                                                                                                            | 참고                                                           |
| --------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------- |
| `vars`를 가진 `tests`                                                 | [데이터셋 아이템](/docs/evaluation/experiments/datasets) (`input`, `expected_output`, `metadata`)                                                                   | 테스트 케이스 하나가 아이템 하나가 됩니다                      |
| 어설션 참조 값 (`contains`, `factuality` 등의 `value:`)               | 데이터셋 아이템의 `expected_output`                                                                                                                                 | 아이템별 정답(ground truth)이 아이템 안으로 이동합니다         |
| 결정론적 `assert` 타입 (`contains`, `regex`, `is-json`, `javascript`) | [실험 SDK](/docs/evaluation/experiments/experiments-via-sdk)의 평가기 함수, 또는 UI에서 작성하는 [코드 평가기](/docs/evaluation/evaluation-methods/code-evaluators) | 검사가 YAML 타입 대신 코드가 됩니다                            |
| 모델 채점 `assert` 타입 (`llm-rubric`, `factuality`, `context-*`)     | [LLM-as-a-judge 평가기](/docs/evaluation/evaluation-methods/llm-as-a-judge) 또는 `create_evaluator_from_autoevals`를 통한 autoevals 스코어러                        | 관리형 평가기는 프로덕션 트레이스에서도 실행됩니다             |
| `defaultTest`                                                         | `run_experiment`에 전달되는 `evaluators`                                                                                                                            | 실험 평가기는 기본적으로 모든 아이템에 적용됩니다              |
| `prompts` (인라인 또는 `file://`)                                     | 버전과 라벨을 가진 [프롬프트 관리](/docs/prompt-management)                                                                                                         | Promptfoo도 `langfuse://` 참조를 통해 이를 가져올 수 있습니다  |
| `providers` × `prompts` 매트릭스                                      | 동일한 데이터셋에 대해 변형(variant)당 하나의 실험 실행                                                                                                             | 실행들은 Langfuse UI에서 나란히 비교됩니다                     |
| CI의 `promptfoo eval` (실패 시 종료 코드 `100`)                       | `RegressionError`를 발생시키는 실험 스크립트 + [langfuse/experiment-action](/docs/evaluation/experiments/experiments-ci-cd)                                         | 둘 다 작업을 실패시키고 PR에 코멘트를 남깁니다                 |
| `outputPath`, `promptfoo view`, `promptfoo share`                     | Langfuse에 영구 저장되는 실험 실행 결과                                                                                                                             | 이력과 비교가 플랫폼 안에 있어 아티팩트를 옮길 필요가 없습니다 |

## 예시 스위트

아래 단계들은 다음과 같이 작지만 대표성이 있는 설정을 마이그레이션합니다: 테스트 케이스 두 개, 각각의
결정론적 어설션 하나, 그리고 `defaultTest`를 통해 모든 테스트에 적용되는 모델 채점 루브릭입니다.

```yaml
# promptfooconfig.yaml (마이그레이션 전)
prompts:
  - file://prompts/support-agent.txt
providers:
  - openai:gpt-4.1
defaultTest:
  assert:
    - type: llm-rubric
      value: "Is polite and offers a concrete next step"
tests:
  - vars:
      question: "How do I reset my password?"
    assert:
      - type: icontains
        value: "reset link"
  - vars:
      question: "What is your refund window?"
    assert:
      - type: icontains
        value: "30 days"
```

아래의 모든 Langfuse 코드 스니펫은 Python SDK v4(`pip install langfuse`)를 사용합니다. JS/TS
SDK(`@langfuse/client`)도 동일한 API를 전반적으로 제공합니다.

## 1단계: 테스트를 데이터셋으로 전환하기

`tests`의 각 항목은 데이터셋 아이템이 됩니다. `vars`는 `input`으로 매핑되고, 테스트별 참조 값(여기서는
`icontains` 문자열)은 `expected_output`으로 이동합니다:

```python
from langfuse import get_client

langfuse = get_client()  # LANGFUSE_PUBLIC_KEY, LANGFUSE_SECRET_KEY, LANGFUSE_BASE_URL를 읽습니다

langfuse.create_dataset(name="support-agent-regression")

test_cases = [
    {"question": "How do I reset my password?", "must_contain": "reset link"},
    {"question": "What is your refund window?", "must_contain": "30 days"},
]

for case in test_cases:
    langfuse.create_dataset_item(
        dataset_name="support-agent-regression",
        input={"question": case["question"]},
        expected_output=case["must_contain"],
    )
```

저장소 안의 YAML 테스트와 달리, [데이터셋 아이템](/docs/evaluation/experiments/datasets)은
UI에서 편집 가능하고 버전이 관리되며, 흥미로운 프로덕션 트레이스로부터 직접 생성할 수 있습니다. 이는
마이그레이션 이후 회귀 스위트가 성장하는 일반적인 방식이기도 합니다.

## 2단계: 어설션을 평가기로 이식하기

평가기는 아이템의 `input`, `output`, `expected_output`을 받아 `Evaluation`을 반환하는 일반
함수입니다. `icontains` 어설션은 몇 줄의 Python 코드가 되고, `llm-rubric` 어설션은 모델을
호출하는 채점 함수가 됩니다. `defaultTest`와 마찬가지로, 실험에 전달된 평가기는 모든 아이템에
적용됩니다.

```python
from langfuse import Evaluation
from langfuse.openai import OpenAI  # 트레이싱이 적용된 드롭인 대체재

def contains_expected(*, output, expected_output, **kwargs):
    # 대체 대상: type: icontains
    passed = (expected_output or "").lower() in (output or "").lower()
    return Evaluation(name="contains_expected", value=1.0 if passed else 0.0)

RUBRIC = "Is polite and offers a concrete next step"

def rubric_judge(*, input, output, **kwargs):
    # 대체 대상: type: llm-rubric
    response = OpenAI().chat.completions.create(
        model="gpt-4.1",
        messages=[{
            "role": "user",
            "content": (
                f"Grade this response against the rubric: {RUBRIC}\n\n"
                f"Question: {input['question']}\n\nResponse: {output}\n\n"
                'Reply "PASS" or "FAIL", then a one-sentence reason.'
            ),
        }],
    )
    verdict = response.choices[0].message.content or ""
    return Evaluation(
        name="rubric",
        value=1.0 if verdict.strip().upper().startswith("PASS") else 0.0,
        comment=verdict,
    )
```

Promptfoo의 다른 모델 채점 어설션들의 경우, 직접 채점기를 작성할 필요가 거의 없습니다:

- [autoevals](https://github.com/braintrustdata/autoevals) 라이브러리는 `Factuality`(OpenAI의
  evals 프로젝트에서 파생되었으며, Promptfoo의 `factuality` 어설션과 같은 계보입니다),
  `ClosedQA`, 임베딩 유사도, 그리고 컨텍스트 정밀도·재현율·충실도를 다루는 RAG 메트릭을
  제공합니다. Langfuse는 `create_evaluator_from_autoevals`를 통해 이들을 평가기로 변환합니다
  (자세한 내용은 [실험 SDK 문서](/docs/evaluation/experiments/experiments-via-sdk) 참고).
- Langfuse 관리형 [LLM-as-a-judge 평가기](/docs/evaluation/evaluation-methods/llm-as-a-judge)는
  UI에서 설정되고 플랫폼이 실험 또는 실제 트래픽에 대해 실행하므로, 채점 프롬프트가 애플리케이션
  코드에 남아있지 않게 됩니다.

비엔지니어가 확인하고 재사용하기를 원하는 결정론적 검사도 Langfuse UI에서
[코드 평가기](/docs/evaluation/evaluation-methods/code-evaluators)로 직접 작성할 수 있으며,
Langfuse가 이를 실험 관측(observation)에 대해 실행합니다.

## 3단계: 실험 실행하기 (그리고 프로바이더 매트릭스)

실험 러너는 애플리케이션을 데이터셋에 대해 반복 실행하면서 모든 실행을 트레이싱하고 평가기를
적용합니다. Promptfoo가 프로바이더 호출을 대신 처리해주는 것과 달리, Langfuse의 태스크 함수는
여러분 자신의 코드이므로, 평가가 단순한 모델 호출이 아니라 실제 검색(retrieval), 도구, 연결
로직까지 검증하게 됩니다.

```python
def support_agent_task(*, item, **kwargs):
    return answer_question(item.input["question"])  # 여러분의 애플리케이션 로직

dataset = langfuse.get_dataset("support-agent-regression")

result = dataset.run_experiment(
    name="post-migration-baseline",
    task=support_agent_task,
    evaluators=[contains_expected, rubric_judge],
)
print(result.format())
```

Promptfoo의 `providers`와 `prompts` 목록은 자동으로 그리드 형태로 확장됩니다. Langfuse에서는
변형(variant) 비교를, 동일한 데이터셋에 대해 변형별로 하나씩 실험을 실행하는 방식으로
표현합니다. 보통 [프롬프트 관리](/docs/prompt-management)의 버전이나 라벨을 순회합니다:

```python
for label in ["production", "candidate"]:
    prompt = langfuse.get_prompt("support-agent", label=label)

    def task(*, item, prompt=prompt, **kwargs):
        compiled = prompt.compile(question=item.input["question"])
        response = OpenAI().chat.completions.create(
            model=prompt.config.get("model", "gpt-4.1"),
            messages=[{"role": "user", "content": compiled}],
        )
        return response.choices[0].message.content

    dataset.run_experiment(
        name=f"support-agent ({label})",
        task=task,
        evaluators=[contains_expected, rubric_judge],
    )
```

이 실행들은 데이터셋의 실험 비교 뷰에 나란히 표시되며, 이는 `promptfoo view`를 대체해 변형
결과를 비교하는 자리가 됩니다. 차이점은 모든 결과가 완전한 트레이스로 연결된다는 것입니다.

## 4단계: 회귀에 대해 CI를 계속 실패시키기

`promptfoo eval`은 테스트 케이스가 실패하면 종료 코드 `100`으로 종료되며(2026년 7월 기준,
[Promptfoo CLI 문서](https://www.promptfoo.dev/docs/usage/command-line/) 참고),
`promptfoo/promptfoo-action`은 풀 리퀘스트에 결과 링크를 게시합니다. Langfuse에서 이에
대응하는 것은 [langfuse/experiment-action](/docs/evaluation/experiments/experiments-ci-cd)
GitHub Action입니다: 실험 스크립트가 메트릭이 임계값을 위반할 때 `RegressionError`를
발생시키면, 액션이 작업을 실패시키고 실행 수준 점수, 아이템 수준 표, Langfuse 내 실험으로의 링크가
포함된 PR 코멘트를 게시합니다.

```python
# experiments/support-agent-gate.py
# support_agent_task, contains_expected, rubric_judge는 2, 3단계에서 정의한 대로 사용합니다
from langfuse import Evaluation, RegressionError, RunnerContext

THRESHOLD = 0.9

def pass_rate(*, item_results, **kwargs):
    scores = [
        e.value
        for item in item_results
        for e in item.evaluations
        if e.name == "contains_expected"
    ]
    return Evaluation(name="pass_rate", value=sum(scores) / len(scores) if scores else 0.0)

def experiment(context: RunnerContext):
    result = context.run_experiment(
        name="PR gate: support agent",
        task=support_agent_task,
        evaluators=[contains_expected, rubric_judge],
        run_evaluators=[pass_rate],
    )

    rate = next(
        (e.value for e in result.run_evaluations if e.name == "pass_rate"), None
    )

    if not isinstance(rate, (int, float)) or rate < THRESHOLD:
        raise RegressionError(
            result=result,
            metric="pass_rate",
            value=float(rate) if isinstance(rate, (int, float)) else 0.0,
            threshold=THRESHOLD,
        )

    return result
```

## 계속 유지할 수 있는 것들

마이그레이션이 반드시 일괄 전환일 필요는 없으며, 다음 두 가지 Promptfoo 워크플로는 Langfuse와
함께 계속 유용하게 사용할 수 있습니다:

- **Langfuse가 관리하는 프롬프트에 대한 Promptfoo 평가.** Promptfoo는 `prompts` 목록에서
  `langfuse://prompt-name@production` 참조를 네이티브로 해석하므로, 프롬프트를 먼저
  Langfuse로 옮기면 전환 기간 동안 두 시스템이 동일한 소스 오브 트루스를 테스트할 수 있습니다.
  [Promptfoo 통합 페이지](/integrations/other/promptfoo)에서 설정 방법을 다루며, 이
  마이그레이션 가이드는 그 경로의 선택적 종착점입니다.
- **레드 티밍.** Promptfoo의 `redteam` 및 모델 스캐닝 기능은 보안 프로빙을 대상으로 하며,
  Langfuse는 이를 지원하지 않습니다. 품질 평가를 마이그레이션한 팀들도 적대적(adversarial)
  테스트를 위해 흔히 Promptfoo를 툴체인에 그대로 유지합니다.

과거 Promptfoo 결과는 가져오기(import)보다는 그대로 보관해 두는 편이 낫습니다: 점수는 마이그레이션된
데이터셋에 대해 다시 생성하는 비용이 저렴하며, 최신 스위트를 `post-migration-baseline` 실험으로
다시 실행하면 이후 실행들을 위한 깔끔한 기준점을 얻을 수 있습니다.

## FAQ

### Langfuse와 함께 Promptfoo를 계속 실행할 수 있나요? [#keep-both-tools]

네, 대부분의 팀이 전환 기간 동안 그렇게 합니다. Promptfoo는 `langfuse://` 참조를 통해 Langfuse에서
프롬프트를 가져올 수 있으므로, 테스트 케이스와 어설션을 점진적으로 이식하는 동안 두 도구가 동일한
프롬프트 버전을 평가할 수 있습니다. 많은 팀이 Langfuse가 대체하지 않는 레드 티밍을 위해 Promptfoo를
영구적으로 유지하기도 합니다.

### Langfuse 평가기가 Promptfoo의 어설션 타입을 모두 커버하나요? [#assertion-parity]

대부분 커버하지만, 일대일 카탈로그 형태는 아닙니다. 결정론적 어설션(`contains`, `regex`,
`is-json`, 커스텀 `javascript`/`python`)은 짧은 평가기 함수나 UI에서 작성한 코드 평가기가
됩니다. 모델 채점 어설션은 autoevals 스코어러(factuality, closed QA, similarity, RAG 컨텍스트
메트릭) 또는 Langfuse 관리형 LLM-as-a-judge 평가기로 매핑됩니다. Promptfoo의 어설션별
`weight`와 `threshold` 채점 방식에는 직접적인 대응이 없으므로, 대신 실행 수준 평가기에서 합성
점수를 계산해야 합니다. `rouge-n`이나 `perplexity` 같은 통계적 메트릭은 평가기 함수 내부에서
라이브러리를 호출해야 합니다.

### Langfuse도 `promptfoo eval`처럼 CI 파이프라인을 실패시킬 수 있나요? [#ci-parity]

네. `promptfoo eval`이 실패한 테스트 케이스에 대해 종료 코드 `100`으로 종료되는 것처럼, Langfuse
실험 스크립트는 메트릭이 임계값 아래로 떨어질 때 `RegressionError`를 발생시키고,
`langfuse/experiment-action` GitHub Action이 작업을 실패시키며 점수와 실행 링크가 포함된 PR
코멘트를 게시합니다. GitHub Actions 밖에서는 동일한 실험 코드가 일반 어설션을 사용하는 pytest
또는 Vitest 스위트로 실행됩니다.

### 마이그레이션 이후 비엔지니어는 결과에 어떻게 접근하나요? [#team-access]

조직 및 프로젝트 수준 역할이 적용된 Langfuse UI를 통해 접근합니다. 실험 실행, 점수 비교, 개별
트레이스를 저장소를 건드리지 않고 열람할 수 있으며, 주석 큐를 통해 리뷰어가 체계적인 워크플로
안에서 출력을 채점할 수 있습니다. 이는 뷰어 링크나 결과 파일을 공유하던 Promptfoo의 방식을
대체하며, 셀프 호스팅된 Langfuse 배포에서도 동일하게 적용됩니다.

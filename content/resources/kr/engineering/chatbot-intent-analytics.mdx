# 챗봇 분석: AI 챗봇에 사용자가 무엇을 묻는지 분석하기

AI 챗봇이 프로덕션에 배포되고 나면, 질문의 초점은 "작동하는가"에서 "사용자가 실제로 무엇을 묻고 있고, 우리가
그것을 얼마나 잘 처리하고 있는가"로 옮겨갑니다. 이 질문에 답하려면 클릭 이벤트가 아니라 대화 콘텐츠가
필요하며, 그래서 챗봇 인텐트 분석은 LLM 트레이스를 기반으로 구축됩니다.

**요약:** 모든 대화를 인텐트로 분류하고 그 레이블을 Langfuse에 카테고리형 점수(categorical score)로
기록합니다. 이는 오프라인 배치 작업으로 수행할 수도 있고, 실시간 트래픽에서 실행되는 내장 LLM-as-a-judge
평가기로 수행할 수도 있습니다. 숫자형, 카테고리형, 불리언 점수는 커스텀 대시보드와 점수 분석에서 기본적으로
차트화되지만, 자유 형식 텍스트 점수는 그렇지 않으므로 차트로 표현하고 싶은 값은 이 세 가지 차트 가능 타입 중
하나로 저장해야 합니다. "이번 주에 새로 등장한 토픽은 무엇인가"와 같은 개방형 질문에는 Langfuse MCP 서버나
API를 통해 추론 모델(reasoning model)을 트레이스에 연결하는 방식이 적합합니다.

## 챗봇 분석에 트레이스 데이터가 필요한 이유

제품 분석 도구는 몇 개의 세션이 발생했는지, 사용자가 어디서 이탈했는지를 알려줍니다. 하지만 사용자가 무엇을
물었는지, 모델이 무엇을 답했는지, 그 답이 좋았는지는 알려주지 못합니다. 대화 콘텐츠 자체가 해당 도구에
도달하지 않기 때문입니다. 인텐트 분석에는 LLM 관측성(observability) 플랫폼에만 존재하는 다음 세 가지가
필요합니다.

1. **전체 대화 콘텐츠.** Langfuse의 각 트레이스는 사용자 입력, 모델 출력, 그리고 모든 중간 단계(검색, 도구
   호출)를 담고 있습니다. 이것이 인텐트 분류가 동작하는 원재료입니다.
2. **대화 그룹화.** 멀티턴 대화는 [세션](/docs/observability/features/sessions)으로 그룹화되므로, 개별
   메시지가 아니라 대화 전체를 분석할 수 있습니다.
3. **레이블을 저장할 장소.** Langfuse 점수(score)는 분류 결과(인텐트, 범위 이탈 플래그, 품질 평가)를 해당
   트레이스나 세션에 정확히 연결합니다. 이것이 나중에 같은 대화에 대한 비용, 지연 시간, 사용자 피드백과
   나란히 필터링하고 차트화할 수 있게 만드는 핵심입니다.

아래 네 가지 패턴은 서로를 기반으로 발전하지만, 각각 독립적으로도 사용할 수 있습니다.

## 패턴 1: 트레이스에 대한 오프라인 인텐트 분류

가장 일반적인 구성은 예약된 배치 작업입니다. API를 통해 어제의 챗봇 트레이스를 가져와, 소형 LLM으로 각
사용자 메시지를 분류하고, 그 레이블을 **카테고리형 점수(categorical score)**로 기록합니다. 점수는 트레이스가
생성된 한참 뒤에도 추가할 수 있고 대시보드에서 기본적으로 집계되므로, 이 용도에 적합한 기록 대상입니다.

아래 스크립트는 완전하고 실행 가능하며, 트레이스 이름과 입력 추출 방식은 여러분 애플리케이션의 트레이스
구조에 맞게 조정하면 됩니다.

```python
import os
from datetime import datetime, timedelta, timezone

from langfuse import get_client
from openai import OpenAI

# LANGFUSE_PUBLIC_KEY, LANGFUSE_SECRET_KEY, LANGFUSE_BASE_URL로 인증
langfuse = get_client()
openai_client = OpenAI()  # OPENAI_API_KEY 사용

INTENTS = [
    "order_status",
    "returns_and_refunds",
    "product_question",
    "account_issue",
    "pricing_and_plans",
    "other",
]


def classify_intent(user_message: str) -> str:
    response = openai_client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {
                "role": "system",
                "content": (
                    "Classify the user message into exactly one of these intents: "
                    + ", ".join(INTENTS)
                    + ". Respond with the intent label only."
                ),
            },
            {"role": "user", "content": user_message},
        ],
    )
    label = response.choices[0].message.content.strip()
    return label if label in INTENTS else "other"


# 지난 24시간 동안의 모든 챗봇 트레이스를 분류
to_ts = datetime.now(timezone.utc)
from_ts = to_ts - timedelta(days=1)

page = 1
while True:
    traces = langfuse.api.trace.list(
        name="chat-turn",  # 챗봇에서 사용하는 트레이스 이름
        from_timestamp=from_ts,
        to_timestamp=to_ts,
        page=page,
        limit=50,
    )
    if not traces.data:
        break
    for trace in traces.data:
        # 트레이스 입력 구조에 맞게 추출 방식을 조정
        message = (
            trace.input.get("message")
            if isinstance(trace.input, dict)
            else str(trace.input or "")
        )
        if not message:
            continue
        langfuse.create_score(
            trace_id=trace.id,
            name="intent",
            value=classify_intent(message),
            data_type="CATEGORICAL",
            score_id=f"intent-{trace.id}",  # 멱등성 키, 같은 날 재실행 시 덮어씀
        )
    page += 1

langfuse.flush()
```

제대로 챙겨야 할 세 가지 세부 사항이 있습니다.

- **안정적인 `score_id`를 사용하세요.** `intent-{trace_id}`처럼 결정적인(deterministic) id를 사용하면
  (예를 들어 분류기를 개선한 뒤) 재실행 시 기존 점수를 중복 생성하지 않고 덮어씁니다. 덮어쓰기는 id, name,
  점수의 날짜가 모두 일치해야 하므로, 다른 날짜에 재실행하면 첫 번째 점수를 대체하는 대신 두 번째 점수가
  추가됩니다.
- **대화 단위 인텐트는 세션에 점수를 매기세요.** 하나의 대화가 여러 트레이스에 걸쳐 있다면, `create_score`에
  `trace_id` 대신 `session_id`를 전달하고 대화 전체를 한 번만 분류하세요.
- **고정된 레이블 집합으로 시작하세요.** 미리 정의된 분류 체계는 주 단위로 차트를 안정적으로 유지해 줍니다.
  아직 인텐트를 파악하지 못했다면 먼저 임베딩 후 클러스터링 과정을 실행해 보세요.
  [인텐트 분류 쿡북](/guides/cookbook/example_intent_classification_pipeline)에서 Langfuse 트레이스에 대한
  지도 학습 분류기와, LLM이 생성한 클러스터 레이블을 사용하는 비지도 클러스터링을 모두 보여줍니다.

## 패턴 2: 인텐트 및 범위 이탈 탐지를 위한 LLM-as-a-judge

직접 파이프라인을 운영하고 싶지 않다면, Langfuse에 내장된
[LLM-as-a-judge 평가기](/docs/evaluation/evaluation-methods/llm-as-a-judge)가 Langfuse 내부에서 관리되는
방식으로 실시간 프로덕션 트래픽에 대해 동일한 분류를 수행합니다. `{{input}}`과 `{{output}}` 변수를 사용해
judge 프롬프트를 작성하고 점수 타입을 선택하면, Langfuse가 새로운 트래픽에 대해 자동으로 이를 실행합니다.

일반적인 챗봇 요구 사항은 두 가지 평가기 구성으로 다룰 수 있습니다.

- **인텐트 분류**는 **카테고리형** 점수 타입을 가진 커스텀 평가기입니다. 허용되는 카테고리(패턴 1과 동일한
  분류 체계)를 정의하면 Langfuse가 judge의 출력이 그중 하나로 매핑되도록 강제합니다. 하나의 메시지가 두
  가지 인텐트를 가질 수 있다면 "복수 매칭 허용(Allow multiple matches)"을 활성화하세요.
- **범위 이탈 요청 탐지**는 **불리언** 점수 타입을 가진 커스텀 평가기이며, "사용자 요청이 어시스턴트의 문서화된
  범위를 벗어나면 true를 반환하라"와 같은 루브릭을 사용합니다. judge의 추론 과정도 점수와 함께 저장되므로,
  범위 이탈 플래그가 급증할 때 그 이유를 바로 확인할 수 있습니다.

운영 측면에서, 평가기는 샘플링 비율을 지원합니다(트래픽의 10퍼센트만 분류해도 분포 모니터링에는 보통 충분하며
judge 비용을 10배 절감합니다). 그리고 전체 트레이스가 아니라 관측(observation) 단위로 평가기를 실행하는 것이
권장되는, 더 빠른 방법입니다. 챗봇의 최종 응답 관측으로 필터링하면 내부의 모든 단계가 아니라 턴당 하나의
메시지만 분류하게 됩니다.

## 패턴 3: 인텐트와 품질 점수에 대한 대시보드

점수 기록은 차트에서 그 진가를 발휘합니다. [커스텀 대시보드](/docs/metrics/features/custom-dashboards)에서
카테고리형 점수 데이터 소스를 점수 값별로 그룹화한 위젯은 시간에 따른 카테고리별 인텐트 볼륨을 표시하고,
불리언 `out_of_scope` 점수는 비율(rate) 라인 차트가 됩니다. 다른 데이터 소스에 대한 위젯도 점수 필터를
지원하므로, 비용이나 지연 시간 위젯을 특정 점수와 일치하는 대화로 좁힐 수 있습니다. Scores 섹션에는 설정이
필요 없는 분석 탭도 있어, 점수별 분포와 추세 차트, 그리고 두 점수를 비교할 때(예: 동일한 트레이스에 대한
오프라인 분류기와 LLM judge 비교) 사용하는 일치도(agreement) 메트릭도 제공합니다.

점수 타입은 무엇을 차트화할 수 있는지를 결정하므로 신중하게 선택해야 합니다. 2026년 7월 기준:

| 점수 타입     | 예시 값                 | 대시보드 및 점수 분석에서 차트화 |
| ------------- | ----------------------- | -------------------------------- |
| `NUMERIC`     | `0.87`                  | 가능                             |
| `CATEGORICAL` | `"returns_and_refunds"` | 가능                             |
| `BOOLEAN`     | `1` (true)              | 가능                             |
| `TEXT`        | 자유 형식 문자열        | 불가능                           |

텍스트 점수(최대 500자의 자유 형식 문자열)는 정성적인 메모를 위한 것이며, 의미 있게 집계할 수 없기 때문에
점수 분석, 실험(experiments), LLM-as-a-judge에서 지원되지 않습니다. 실무적인 함의는 이렇습니다. 분류기가
구조화된 JSON 결과를 반환한다면 그 덩어리를 하나의 텍스트 점수로 저장하지 마세요. 대신 카테고리형 `intent`,
불리언 `out_of_scope`, 숫자형 `classification_confidence`처럼 별도의 점수로 분리하면, 각각이 독립적으로
차트화됩니다.

## 패턴 4: 추론 모델을 이용한 애드혹 분석

분류 파이프라인은 미리 정의해 둔 질문에만 답합니다. "가격 정책 변경 이후 어떤 새로운 불만 테마가 등장했는가",
"이번 주에 사용자들이 어떤 부분에서 어려움을 겪었는지 요약하라"처럼 개방형 질문에는, 성능이 뛰어난 모델에게
트레이스 데이터에 직접 접근할 권한을 주고 자연어로 물어보는 방식이 적합합니다. Langfuse는
`https://cloud.langfuse.com/api/public/mcp`(셀프 호스팅 배포에서는 `/api/public/mcp`)에서 네이티브
[MCP 서버](/docs/api-and-data-platform/features/mcp-server)를 제공하며, AI 어시스턴트와 에이전트는 프로젝트
범위의 API 키로 이 서버에 연결합니다. CLI 도구를 실행할 수 있는 코딩 에이전트라면 Langfuse 에이전트 스킬을
사용하는 것이 권장되는 구성입니다.

```bash
# Claude Code에 Langfuse MCP 서버 등록 (EU 리전)
claude mcp add --transport http langfuse https://cloud.langfuse.com/api/public/mcp \
    --header "Authorization: Basic {base64 of pk-lf-...:sk-lf-...}"
```

챗봇 트레이스에 잘 통하는 프롬프트: "out_of_scope로 플래그된 최근 대화 50개를 가져와 테마별로 그룹화하라",
또는 "월요일 배포 전후로 사용자들이 무엇을 물었는지 비교하라". 추론 모델은 대화 샘플을 읽고 몇 분 만에
패턴을 보고해 주며, 이는 분류 체계에 새 레이블을 추가하기 전에 거치는 탐색 단계입니다.

솔직한 제약도 있습니다. 에이전트는 컨텍스트 윈도우에 들어가는 만큼만 읽을 수 있으므로, 이 패턴은 집계가
아니라 표본 추출입니다. 발견과 가설 수립에 사용한 뒤, 반복적으로 나타나는 결과는 패턴 1이나 패턴 2 분류기로
승격시켜 정확하고 차트화 가능한 수치로 만드세요.

## 인텐트 데이터에서 더 나은 챗봇으로

인텐트 분석은 그것이 실제로 제품을 바꿀 때만 구축할 가치가 있습니다. 실제로 팀들이 실행에 옮기는 세 가지
후속 조치는 다음과 같습니다.

1. **가장 많은 인텐트를 챗봇이 처리하기 전에 먼저 해결하세요.** 인텐트 분포는 사용자가 원하는 것의 순위
   목록입니다. 가장 큰 클러스터는 FAQ 페이지, 문서, 또는 자유 형식 채팅보다 그 작업을 더 잘 처리하는 전용
   플로우의 후보입니다.
2. **특정 인텐트에서 성능이 떨어지는 프롬프트를 수정하세요.** 부정적인 사용자 피드백이나 실패한 judge
   점수를 가진 트레이스를 하나의 인텐트로 필터링하고, 대화를 읽고, 그 케이스에 맞게 시스템 프롬프트나
   검색(retrieval)을 조정하세요. 세분화된 점수는 "봇이 예전보다 나쁜 것 같다"를 "returns_and_refunds
   만족도가 이번 주에 하락했다"로 바꿔 줍니다.
3. **실패한 대화를 테스트 케이스로 전환하세요.** 실패한 트레이스를 [데이터셋](/docs/evaluation/experiments/datasets)에
   추가하고 프롬프트 변경 사항을 배포하기 전에 실험(experiment)을 실행하세요. 그러면 한 번 실패했던 대화가,
   같은 실패가 반복되지 않도록 막아 주는 회귀 테스트 스위트가 됩니다.

## FAQ

### AI 챗봇에 사용자가 무엇을 묻는지 어떻게 분석하나요? [#analyze-user-questions]

LLM 관측성 플랫폼에서 모든 대화를 트레이싱한 다음, 각 사용자 메시지를 인텐트로 분류하고 그 레이블을
트레이스에 카테고리형 점수로 저장하세요. 분류는 트레이스 API를 통한 오프라인 배치 작업으로 실행하거나,
실시간 트래픽에 대한 LLM-as-a-judge 평가기로 실행할 수 있습니다. 그러면 인텐트 분포, 시간에 따른 추세,
인텐트별 품질 메트릭을 점수 데이터에 대한 대시보드 위젯에서 얻을 수 있습니다.

### 인텐트 레이블은 태그로 저장해야 하나요, 점수로 저장해야 하나요? [#tags-vs-scores]

사후에 대화를 분류하는 경우(인텐트 분석의 일반적인 경우)에는 점수를 사용하세요. 점수는 언제든 기존
트레이스에 추가할 수 있고, 데이터 타입을 가지며, 대시보드와 점수 분석에서 집계됩니다. 태그는 트레이싱 시점에
설정되며, 대화를 트리거한 앱 기능처럼 요청 시점에 이미 알고 있는 속성에 적합합니다.

### 모든 대화를 분류해야 하나요? [#classification-sampling]

아니요. 인텐트 분포와 범위 이탈 비율을 모니터링하는 데에는 일반적으로 10~25퍼센트 샘플이면 충분히
대표성을 가지며 비용도 그만큼 절감됩니다. LLM-as-a-judge 평가기에는 바로 이런 용도를 위한 내장 샘플링
비율 기능이 있습니다. 모든 범위 이탈 요청을 검토 큐로 라우팅하는 것처럼 대화 단위 후속 조치가 필요한
경우에만 모든 대화를 분류하세요.

### 챗봇 분석과 제품 분석의 차이는 무엇인가요? [#chatbot-vs-product-analytics]

제품 분석은 챗봇 주변의 사용자 행동을 측정합니다. 세션, 리텐션, 퍼널, 이탈 등입니다. 트레이스 기반의 챗봇
분석은 대화 자체를 측정합니다. 사용자가 무엇을 물었는지, 모델이 무엇을 했는지, 비용이 얼마였는지, 답변이
좋았는지를 측정하며, 종종 채팅 UI에서 수집된
[사용자 피드백](/docs/observability/features/user-feedback)과 결합됩니다. 대부분의 팀은 두 가지를 모두
운영하며 공유되는 사용자 및 세션 식별자를 통해 이 둘을 연결합니다.

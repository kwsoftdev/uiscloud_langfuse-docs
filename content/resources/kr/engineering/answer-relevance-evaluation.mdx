# LLM 애플리케이션을 위한 답변 관련성 평가

답변 관련성(answer relevance)은 생성된 답변이 실제로 던져진 질문을 다루고 있는지를 측정합니다. 질의응답 시스템, RAG 파이프라인, 지원 챗봇에서 핵심적인 품질 메트릭 중 하나인데, 이러한 애플리케이션에서 가장 흔한 실패는 잘못된 사실이 아니라 답변이 아닌 답변이기 때문입니다. 즉 모델이 유창하게 관련된 다른 질문에 답하거나, 두 부분으로 구성된 질문 중 절반만 다루거나, 아예 회피하는 경우입니다. 관련성은 질문과 답변만으로 판단되므로, 정답 레이블(ground truth) 없이도 계산할 수 있습니다.

**요약:** 답변 관련성 평가는 사실적 정확성이나 검색된 컨텍스트에 대한 근거성(groundedness)이 아니라 질문-답변의 적합성을 확인합니다. 참조 답변이 필요 없기 때문에 오프라인 실험과 실제 프로덕션 트래픽 모두에서 사용할 수 있습니다. 아래 레시피에서는 이 메트릭의 경계를 정의하고, 어려운 케이스(부분 답변과 거부)를 위한 심사 프롬프트 설계를 살펴본 다음, 이진(binary) LLM-as-a-judge 평가기를 사용해 Langfuse 실험으로 처음부터 끝까지 구현하고, 프로덕션 트레이스를 위한 온라인 평가기 설정까지 다룹니다.

## 답변 관련성이 충실성(faithfulness) 및 환각 탐지와 다른 점

답변 관련성은 정확히 한 가지 질문에만 답합니다. 바로 "응답이 사용자가 물어본 것을 다루고 있는가?"입니다. 응답은 완벽하게 관련성이 있으면서도 사실적으로 틀릴 수 있고, 검색된 문서에 완전히 근거하면서도 질문을 무시할 수도 있습니다. 이 두 가지를 하나로 뭉뚱그린 "품질" 점수로 평가하면 어떤 실패인지 알 수 없게 되므로, 각 차원은 각자의 평가기를 갖습니다.

| 출력에 대한 질문                              | 메트릭      | 레시피                                                                |
| --------------------------------------------- | ----------- | --------------------------------------------------------------------- |
| 실제로 물어본 질문을 다루고 있는가?           | 답변 관련성 | 이 페이지                                                             |
| 검색된 컨텍스트에 의해 뒷받침되는가?          | RAG 충실성  | [RAG 충실성 평가](/resources/engineering/rag-faithfulness-evaluation) |
| 거짓이거나 뒷받침되지 않는 사실을 주장하는가? | 환각 탐지   | [환각 탐지](/resources/engineering/hallucination-detection)           |

세 메트릭은 실제로 독립적으로 실패합니다. 검색기(retriever)가 약한 RAG 시스템은 충실하지만 관련성이 없는 답변을 만듭니다(엉뚱한 문서를 정확하게 요약하는 셈입니다). 반대로 검색 없이 동작하는 강력한 모델은 관련성은 있지만 근거가 없는 답변을 만듭니다. 관련성을 별도로 점수화하면 프롬프트와 질문 처리 로직을 고쳐야 하는지, 아니면 검색과 근거 부여 로직을 고쳐야 하는지 알 수 있습니다.

## 답변 관련성에는 정답 레이블이 필요 없습니다

답변 관련성은 참조가 필요 없는(reference-free) 메트릭입니다. 심사자는 답변을 기대 출력이 아니라 질문과 비교하므로, 이를 위한 평가 데이터셋은 질문 목록만 있으면 됩니다. 여기에는 두 가지 실용적인 함의가 있습니다.

1. 정답 답변을 아무도 작성하지 않은 상태에서도 평가를 시작할 수 있습니다. 실제 사용자 질문 덤프 자체가 완전한 관련성 데이터셋이 됩니다.
2. 정답 레이블이 존재하지 않는 프로덕션 트래픽에서도 동일한 평가기를 실행할 수 있습니다. answer correctness 같은 참조 기반 메트릭은 정제된 테스트 세트에 한정되지만, 관련성은 그렇지 않습니다.

이 트레이드오프는 범위(scope)에 있습니다. 참조가 없는 심사자는 답변이 옳은지는 알려주지 못합니다. 팀들은 보통 레이블이 있는 데이터의 하위 집합에서는 관련성 평가기와 정확성 평가기를 함께 사용하고, 레이블이 없는 나머지에서는 관련성 평가만 의존합니다.

## 답변 관련성 심사 프롬프트 설계하기

관련성 판단에는 독해력이 필요하므로, 평가기는 코드 검사가 아니라 LLM-as-a-judge여야 합니다. 세 가지 설계 결정이 점수의 유용성을 좌우합니다.

**심사자를 질문에 집중시키고, 품질에는 집중시키지 마세요.** 심사자는 답변이 길고, 자신감 있고, 형식이 잘 갖춰져 있으면 "좋다"고 평가하는 쪽으로 쏠리는 경향이 있습니다. 프롬프트는 문체, 정확성, 설명의 완전성은 평가 범위 밖이며, 유일한 테스트는 구체적으로 물어본 질문이 다뤄졌는지 여부라는 점을 명시해야 합니다. 관련은 있지만 다른 질문에 답하는 것은 실패입니다.

**부분 답변을 어떻게 점수화할지 명시적으로 결정하세요.** 여러 부분으로 구성된 질문("환불 정책은 무엇이고, 연간 플랜의 경우 다른가요?")의 경우, 한 부분만 다루는 답변은 이 메트릭이 정확히 잡아내려는 실패 유형입니다. 아래 루브릭은 부분 답변을 실패로 처리합니다. 부분적인 커버리지를 추적하고 싶다면, 관련성 정의를 완화하는 대신 별도의 범주형(categorical) 점수로 만드세요.

**거부(refusal)를 정직하게 점수화하세요.** "그 부분은 도와드릴 수 없습니다"는 거부하는 것이 옳은 행동이었더라도 질문에 대해 관련성이 없으며, 심사자는 그렇게 판단해야 합니다. 거부를 관련성 있는 것으로 통과시키면 모델이 점점 더 조심스러워질수록 메트릭이 조용히 부풀려집니다. 정직한 설정은 거부를 관련성 없음으로 점수화하고, 거부 자체를 별도의 점수로 추적하여, 거부율 상승이 관련성 수치에 세탁되어 감춰지지 않고 눈에 보이도록 합니다.

[Langfuse Academy](/academy/evaluate)에서 나온 한 가지 추가 권장 사항은, 1~5점 척도보다 이진(binary) 합격/불합격 점수를 선호하라는 것입니다. 이진 루브릭은 무엇이 허용 가능하고 무엇이 그렇지 않은지에 대한 명확한 정의를 강제하는 반면, 등급 척도는 3점과 4점의 차이를 심사자의 그때그때 판단에 맡기게 되어 평가기 간, 그리고 시간에 따라 점수가 흔들리게 됩니다.

## Langfuse Python SDK로 답변 관련성 실험 실행하기

아래 스크립트는 자체 완결적(self-contained)입니다: 질문에 답하는 태스크, 이진 관련성 심사자, 실행(run) 수준 집계, 그리고 예상 출력이 없는 순수 질문으로 구성된 데이터셋입니다. Langfuse Python SDK(v4)의 실험 러너를 사용하며, 이 러너는 모든 태스크 실행을 트레이싱하고, 평가기를 동시에 실행하고, 점수를 자동으로 기록합니다. 이미 애플리케이션이 Langfuse로 트레이싱되고 있다면, 새로운 서비스나 새로운 자격 증명이 추가로 필요하지 않습니다. 이 실험은 트레이싱 설정과 동일한 SDK와 동일한 API 키를 사용하며, Langfuse Cloud 또는 셀프 호스팅 인스턴스를 대상으로 동작합니다.

```python
# pip install langfuse openai
# env: LANGFUSE_PUBLIC_KEY, LANGFUSE_SECRET_KEY, LANGFUSE_HOST, OPENAI_API_KEY

import json

import openai
from langfuse import Evaluation, get_client
from langfuse.openai import OpenAI  # drop-in wrapper, traces the task's model calls

langfuse = get_client()
task_client = OpenAI()          # application calls, traced into each experiment item
judge_client = openai.OpenAI()  # judge calls, kept out of the application traces

JUDGE_PROMPT = """You are evaluating whether an answer is relevant to the question asked.

Question:
{question}

Answer:
{answer}

The answer is relevant only if it addresses the specific question asked. Apply these rules:
- Answering a related but different question is not relevant.
- Addressing only some parts of a multi-part question is not relevant.
- A refusal, deflection, or "I don't know" is not relevant, even if refusing was appropriate.
- Style, length, and factual correctness are out of scope; judge only question-answer fit.

Respond in JSON: {{"relevant": true or false, "refusal": true or false, "reasoning": "one sentence"}}"""


def answer_question(*, item, **kwargs):
    """The application under test: answer a user question."""
    response = task_client.chat.completions.create(
        model="gpt-4.1",
        messages=[
            {"role": "system", "content": "You are a support assistant for a SaaS product."},
            {"role": "user", "content": item["input"]},
        ],
    )
    return response.choices[0].message.content


def answer_relevance(*, input, output, **kwargs):
    """Binary LLM-as-a-judge for answer relevance. Reference-free: no expected output used."""
    judgment = judge_client.chat.completions.create(
        model="gpt-4.1",
        messages=[
            {
                "role": "user",
                "content": JUDGE_PROMPT.format(question=input, answer=output),
            }
        ],
        response_format={"type": "json_object"},
        temperature=0,
    )
    verdict = json.loads(judgment.choices[0].message.content)
    return [
        Evaluation(
            name="answer_relevance",
            value=1.0 if verdict["relevant"] else 0.0,
            comment=verdict["reasoning"],
        ),
        Evaluation(name="refusal", value=1.0 if verdict["refusal"] else 0.0),
    ]


def relevance_rate(*, item_results, **kwargs):
    """Run-level aggregate: share of relevant answers across the dataset."""
    values = [
        e.value
        for r in item_results
        for e in r.evaluations
        if e.name == "answer_relevance"
    ]
    return Evaluation(
        name="relevance_rate",
        value=sum(values) / len(values) if values else None,
    )


# A relevance dataset is just questions; no ground truth required.
data = [
    {"input": "How do I rotate an API key without downtime?"},
    {"input": "What is the refund policy, and does it differ for annual plans?"},
    {"input": "Can I export my usage data to CSV?"},
    {"input": "Why was my account charged twice this month?"},
]

result = langfuse.run_experiment(
    name="Answer relevance baseline",
    description="Binary relevance judge over the support question set",
    data=data,
    task=answer_question,
    evaluators=[answer_relevance],
    run_evaluators=[relevance_rate],
)
print(result.format())
```

각 아이템은 태스크 실행과 두 개의 점수(`answer_relevance`, `refusal`)를 가진 트레이스를 생성하며, 실행(run)에는 집계된 `relevance_rate`가 함께 기록됩니다. 심사자의 한 문장짜리 근거(reasoning)는 점수의 comment 필드에 남으며, 이는 특정 아이템이 왜 실패했는지 감사(audit)할 때 읽게 되는 내용입니다.

실행 결과를 시간에 따라 비교 가능하게 만들려면, 질문들을 Langfuse에 호스팅되는 데이터셋으로 옮기고 동일한 평가기를 그 데이터셋에 대해 실행하세요. 이렇게 하면 매 실행이 UI에서 이전 프롬프트나 모델 버전과 비교(diff)할 수 있는 데이터셋 실행(dataset run)이 됩니다. 한 가지 차이점에 유의하세요. 호스팅되는 데이터셋의 아이템은 딕셔너리가 아니라 `input` 속성을 가진 객체이므로, 태스크는 `item["input"]` 대신 `item.input`을 읽어야 합니다.

```python
dataset = langfuse.get_dataset("support-questions")


def answer_dataset_question(*, item, **kwargs):
    """Same application task; hosted dataset items expose attributes, not dict keys."""
    return answer_question(item={"input": item.input}, **kwargs)


result = dataset.run_experiment(
    name="Answer relevance, prompt v2",
    task=answer_dataset_question,
    evaluators=[answer_relevance],
    run_evaluators=[relevance_rate],
)
```

동시성 제어와 비동기 태스크를 포함한 전체 러너 API는 [SDK를 통한 실험](/docs/evaluation/experiments/experiments-via-sdk)에 문서화되어 있습니다.

## 프로덕션 트레이스에서 답변 관련성 평가하기

관련성은 참조가 필요 없는 메트릭이므로, 동일한 루브릭을 실제 트래픽에도 그대로 적용할 수 있습니다. Langfuse에서는 [LLM-as-a-judge 평가기](/docs/evaluation/evaluation-methods/llm-as-a-judge)를 통해 플랫폼이 대신 실행해 주므로 SDK 코드가 필요 없습니다. 위 루브릭을 프롬프트로 하는 커스텀 평가기를 만들고, `{{question}}`과 `{{answer}}` 변수를 판단하고자 하는 관측(observation)의 입력과 출력에 매핑한 다음(중첩된 필드는 JSONPath로, 최근 24시간 데이터를 대상으로 한 실시간 미리보기도 제공됩니다), 샘플링 비율을 지정해 실제(live) 관측을 대상으로 지정하세요. 트래픽의 5~10퍼센트 정도를 샘플링하면 모든 요청을 판단하지 않고도 관련성 추이를 파악하기에 보통 충분합니다.

점수는 매칭된 관측에 비동기적으로 붙기 때문에, 이는 실시간(in-flight) 가드레일이 아니라 모니터링 시그널입니다. 여기서 얻는 가치는 추세선입니다. 프롬프트 변경, 모델 교체, 또는 검색기 회귀 이후 관련성이 떨어지는 현상이 지원 티켓으로 나타나기 전에 점수 타임라인에서 먼저 드러나며, 플래그가 붙은 트레이스는 다음 오프라인 반복 작업을 위해 실험 데이터셋에 그대로 가져올 수 있는 항목이 됩니다.

## Ragas answer relevancy와 Langfuse

[Ragas](https://docs.ragas.io/)는 이 메트릭을 `answer_relevancy`(클래스 `ResponseRelevancy`)라는 이름으로 대중화했으며, 라이브러리를 사용하지 않더라도 그 구현 방식을 이해해 둘 가치가 있습니다. Ragas는 심사자에게 직접 묻는 대신, 답변으로부터 몇 개의 후보 질문(기본값 3개)을 생성한 다음 그 질문들의 임베딩과 원래 질문의 임베딩 간 평균 코사인 유사도를 점수로 계산합니다. 회피적이거나 확답을 하지 않는 답변은 강제로 0점 처리됩니다. 이는 위의 심사자 방식과 마찬가지로 참조가 필요 없으며, LLM과 임베딩 모델 둘 다 필요합니다. 임베딩 기반 방식은 0에서 1 사이의 등급화된 신호를 제공하며 Ragas 프로젝트가 유지 관리하는 프롬프트가 함께 제공됩니다. 반면 직접적인 이진 심사자 방식은 실행하기 더 간단하고(임베딩 호출이 없어) 더 저렴하며, 점수가 이상해 보일 때 직접 읽고 수정할 수 있는 루브릭을 제공합니다.

두 방식 모두 Langfuse와 함께 사용할 수 있습니다. 자체 파이프라인에서 계산한 Ragas 점수를 Langfuse의 트레이스와 실험으로 전송할 수 있으며, 관리형 평가기 카탈로그에는 Ragas 같은 파트너와 함께 구축된 평가기도 포함되어 있습니다. 전체 설정 방법은 [Ragas 통합](/integrations/frameworks/ragas)을 참고하세요.

## FAQ

### 답변 관련성과 답변 정확성(answer correctness)의 차이는 무엇인가요? [#relevance-vs-correctness]

답변 관련성은 응답이 물어본 질문을 다루고 있는지를 확인하고, 답변 정확성은 응답이 사실적으로 옳은지를 보통 참조 답변과 비교하여 확인합니다. 응답은 완전히 관련성이 있으면서도 틀릴 수 있고, 사실은 정확하면서도 다른 질문에 답할 수도 있습니다. 관련성은 참조가 필요 없어 어디서나 실행할 수 있지만, 정확성은 정답 레이블이 필요하므로 레이블이 있는 데이터셋에 한정됩니다.

### 정답 레이블 없이 답변 관련성을 평가할 수 있나요? [#no-ground-truth]

네. 답변 관련성은 질문과 답변만으로 판단되므로 기대 출력이 필요하지 않습니다. 그래서 실제 사용자 질문의 단순한 목록만으로도 완전한 관련성 데이터셋이 되며, 정답 레이블이 존재하지 않는 프로덕션 트레이스에서도 동일한 평가기를 실행할 수 있습니다.

### 답변 관련성 평가에서 거부(refusal)는 어떻게 점수화해야 하나요? [#handling-refusals]

거부는 관련성 없음으로 점수화하고, 일반적인 실패와 구분할 수 있도록 별도의 거부 점수를 기록하세요. 거부하는 것이 올바른 행동이었더라도 거부는 질문을 다루지 않은 것이며, 거부를 관련성 있는 것으로 집계하면 모델의 조심스러움이 증가하는 추세가 평평한 메트릭 뒤에 숨겨집니다. Ragas도 동일한 방식으로 처리합니다. 확답하지 않는 답변은 0점으로 처리됩니다.

### 답변 관련성은 Ragas answer relevancy와 같은 것인가요? [#ragas-answer-relevancy]

두 방식은 동일한 속성을 서로 다른 메커니즘으로 측정합니다. Ragas의 `answer_relevancy`는 답변으로부터 질문들을 생성하고 원래 질문과의 임베딩 유사도를 점수화하여 0에서 1 사이의 등급화된 값을 산출합니다. 반면 직접적인 LLM-as-a-judge 방식은 루브릭에 따라 질문-답변 쌍을 읽고, 보통 근거와 함께 이진 판정을 반환합니다. 둘 다 참조가 필요 없으며, 둘 다 Langfuse에 점수를 기록할 수 있습니다.

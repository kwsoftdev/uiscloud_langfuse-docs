# LLM 애플리케이션을 위한 PII 마스킹 패턴

LLM 애플리케이션은 사용자 텍스트를 다른 어떤 소프트웨어보다도 더 많은 시스템을 거쳐 이동시킵니다: 프롬프트, 검색된 문서, 모델 제공자, 관측성 플랫폼, 그리고 로그. 개인 데이터는 이 중 어디로든 유출될 수 있습니다. 이를 잘 처리하는 팀들은 하나의 마스킹 도구만 선택하지 않습니다. 대신 **각 종류의 데이터를 어느 계층에서 차단할지**를 결정하고, 소수의 메커니즘을 의도적으로 사용합니다.

**요약:** LLM 애플리케이션에서 PII를 통제할 수 있는 지점은 네 곳입니다: 데이터가 프로세스를 떠나기 전 클라이언트 SDK에서, 데이터가 저장되기 전 수집 서버에서, 데이터가 모델 제공자에 도달하기 전 게이트웨이에서, 그리고 사후적으로 탐지와 대응을 통해서입니다. 클라이언트 측에서 시작하십시오(가장 적은 장치로 가장 흔한 요구사항을 커버합니다). 클라이언트를 신뢰하여 일관되게 마스킹한다고 보장할 수 없다면 서버 측 강제를 추가하고, 어떤 경우든 탐지용 평가기를 운영하십시오. 모든 예방 계층은 언젠가는 놓치기 때문입니다.

## 계층 1: 데이터가 프로세스를 떠나기 전 SDK에서 마스킹

[Langfuse 마스킹 기능](/docs/observability/features/masking)은 트레이스 데이터가 전송되기 전에 사용자가 정의한 함수를 실행하므로, 민감한 값이 애플리케이션 바깥으로 나가지 않습니다. 2026년 6월부터 여기에는 `mask_otel_spans`가 포함되는데, 이는 Python SDK가 내보내는 모든 스팬에 마스킹을 적용합니다. 여기에는 사용자가 제어하지 않는 서드파티 OpenTelemetry 계측이 생성한 스팬도 포함됩니다.

이것이 올바른 기본 계층인 이유는 안전하게 실패하고(소스에서 마스킹된 데이터는 이후 단계로 유출될 수 없습니다), 추가 인프라가 필요 없으며, 클라우드든 셀프 호스팅이든 어떤 배포 모델과도 조합할 수 있기 때문입니다.

실무에서는 두 가지 패턴이 주로 쓰입니다.

- **구조화된 편집(redaction).** 민감한 필드의 위치를 알고 있는 경우(`user_email` 인자, 프롬프트 컨텍스트 내 고객 레코드)입니다. 필드 이름이나 경로를 기준으로 편집합니다. 비용이 적게 들고 정확하며, 마스킹된 트레이스의 주석도 디버깅 가능한 상태로 유지됩니다.
- **패턴 스크러빙.** 이메일, 전화번호, 카드 번호 형태의 숫자열, 알려진 형식의 ID에 대한 정규식을 모든 문자열 값에 적용합니다. [마스킹 문서](/docs/observability/features/masking)에 바로 사용할 수 있는 예제가 포함되어 있습니다. 이는 거친 안전망으로, 자유 텍스트에서는 오탐지(false negative)가 발생할 수 있습니다.

마스킹 함수는 빠르게 유지하십시오. 스팬의 내보내기(export) 경로에서 실행되므로, 필드 조회와 컴파일된 정규식보다 무거운 작업은 신중하게 검토해야 합니다. 느린 분석 작업은 핫 패스가 아닌 계층 4에 두십시오.

## 계층 2: 수집 시점에 마스킹 강제(서버 측)

클라이언트 측 마스킹은 모든 팀이 이를 올바르게 배선했다는 전제에 의존합니다. 이 전제가 무너지는 경우(여러 팀, 여러 SDK, 사용자가 제어할 수 없는 서드파티 서비스), 셀프 호스팅 Langfuse 배포는 [서버 측 수집 마스킹](/self-hosting/security/data-masking#server-side-ingestion-masking-ee)(Enterprise)을 강제할 수 있습니다. 이는 사용자 인프라가 노출하는 콜백으로, 저장 전에 모든 이벤트를 검사하고 다시 쓰며, 어느 클라이언트가 보냈는지와 무관하게 동작합니다.

이 계층은 마스킹이 위생 관리 차원의 선호가 아니라 컴플라이언스 요구사항일 때 사용하십시오. "모든 클라이언트가 마스킹해야 한다"를 "마스킹되지 않은 데이터는 저장될 수 없다"로 바꿔주기 때문입니다. 트레이드오프는 다음과 같습니다: 이는 Enterprise 셀프 호스팅 기능이고, 사용자의 콜백이 수집 경로의 일부가 되며(처리량과 가용성을 고려해 구축해야 합니다), 중앙화된 규칙은 애플리케이션만큼 필드의 의미를 알지 못합니다. 대부분의 팀은 이를 클라이언트 측 마스킹 *대신*이 아니라 _추가로_ 운영합니다.

## 계층 3: 게이트웨이에서, 모델 제공자에 도달하기 전에 필터링

관측성을 위한 마스킹은 모델 제공자가 보는 내용을 바꾸지 않습니다. 요구사항이 "특정 데이터가 LLM 벤더에 도달해서는 안 된다"는 것이라면, 통제 지점은 애플리케이션과 제공자 사이입니다: 요청을 실시간으로 필터링하거나 가명화하는 [LLM 게이트웨이](/resources/engineering/llm-gateway)나 미들웨어입니다.

비정형 PII(이름, 주소, 자유 텍스트 내 의료 정보)의 경우 이 계층에서는 정규식만으로는 충분하지 않으며, 팀들은 [보안 및 가드레일 생태계](/docs/security-and-guardrails)의 라이브러리를 포함해 NER 기반 또는 모델 기반 탐지기를 활용하게 됩니다. 실제 배포에서 나온 두 가지 주의사항이 있습니다.

- **의존하기 전에 필터링 라이브러리의 유지 관리 상태를 확인하십시오.** 이 생태계는 빠르게 변화하며, 한때 인기 있던 여러 프로젝트가 활동을 멈췄습니다. 오래된 탐지기는 조용한 컴플라이언스 구멍이 됩니다.
- **모델 기반 탐지는 모든 요청에서 지연 시간(latency) 비용을 발생시킵니다.** 지연 시간에 민감한 경로에서는 동기식 자유 텍스트 NER보다 알려진 엔터티의 가명화와 비동기식 심층 스캔을 우선하십시오.

## 계층 4: 빠져나간 것을 탐지하고 대응

모든 예방 계층은 언젠가는 놓치므로, 탐지를 선택 사항이 아니라 필수로 다루어야 합니다. Langfuse에는 이를 다루는 두 가지 메커니즘이 있습니다.

- **[코드 평가기](/docs/evaluation/evaluation-methods/code-evaluators)로 구현한 PII 스크린**을 프로덕션 트레이스에서 실행하면 유출이 점수화되고 조회 가능한 이벤트로 전환됩니다(바로 사용할 수 있는 패턴은 [코드 평가기 예제](/resources/engineering/code-evaluator-examples)에서 확인하십시오). 대시보드에서 점수를 추적하십시오. 상승 추세는 상위 계층 중 하나가 저하되었음을 의미합니다.
- **보존과 삭제를 최종 방어선으로 활용.** 짧은 [데이터 보존](/docs/administration/data-retention) 기간은 빠져나간 어떤 것이든 그 피해 범위를 제한하며, 인시던트 데이터를 포함한 트레이스는 삭제할 수 있습니다. 보존 정책은 마스킹 설계의 일부로 미리 결정해야 하며, 첫 인시던트 이후에 결정해서는 안 됩니다.

## 조합 선택하기

| 요구사항                                                     | 계층                                              |
| ------------------------------------------------------------ | ------------------------------------------------- |
| 명백한 PII를 관측성 데이터에서 배제                          | 1 (필드 편집 + 패턴 스크러빙을 사용한 SDK 마스킹) |
| 마스킹되지 않은 데이터가 저장되지 않는다는 컴플라이언스 보장 | 1 + 2 (서버 측 강제, 셀프 호스팅 EE)              |
| 제공자가 특정 데이터를 절대 볼 수 없어야 함                  | 3 (게이트웨이/미들웨어 필터링) + 관측성을 위한 1  |
| 회귀와 인시던트를 탐지                                       | 4 (평가기 스크린 + 보존 정책), 항상               |

이번 주에 계층 1을 시작하고, 같은 주에 계층 4를 추가하십시오(평가기 하나만 있으면 됩니다). 그리고 명확한 요구사항이 생길 때 계층 2와 3을 도입하십시오.

## FAQ

### 모델 호출 전에 마스킹해야 하나요, 후에 해야 하나요?

두 가지 관심사는 서로 독립적으로 존재합니다. 관측성을 위한 마스킹(계층 1과 2)은 Langfuse에 저장되는 내용을 통제하고, 제공자를 위한 필터링(계층 3)은 모델이 보는 내용을 통제합니다. 프롬프트에는 모델이 필요로 하는 PII가 정당하게 포함될 수 있는 동시에, 저장된 트레이스에서는 이를 편집할 수 있습니다.

### 마스킹이 트레이스 디버깅 능력을 저해하나요?

필드 수준 편집은 구조를 보존하므로 트레이스는 읽을 수 있는 상태를 유지합니다("카드 `****`"는 여전히 카드가 언급되었다는 *사실*을 보여줍니다). 공격적인 전체 텍스트 스크러빙은 디버깅 편의성을 해칩니다. 모든 것을 지우기보다는 표적화된 편집과 짧은 보존 기간을 함께 사용하는 것을 우선하십시오.

### 서드파티 프레임워크가 생성하는 데이터도 마스킹할 수 있나요?

가능합니다. `mask_otel_spans` 옵션은 Langfuse Python SDK를 통해 내보내지는 서드파티 OpenTelemetry 계측 스팬에 사용자의 마스킹 함수를 적용하여, 프레임워크가 생성한 스팬이 애플리케이션 수준 마스킹을 우회하던 공백을 메웁니다. 동작 세부사항은 [마스킹 문서](/docs/observability/features/masking)를 참고하십시오.

### 프롬프트 수준의 지시("개인 데이터를 반복하지 마시오")도 마스킹 전략인가요?

아닙니다. 지시는 모델 동작에 영향을 줄 뿐, 시스템이 저장하거나 전송하는 내용을 통제하지 않습니다. 이를 UX 상의 편의 요소로 취급하되, 개인정보 보호 통제 수단으로 여겨서는 안 됩니다.

# AI 에이전트 평가

AI 에이전트 평가는 에이전트가 작업을 완수했는지, 그리고 그 과정이 적절했는지를 측정하는 작업입니다. 단일 LLM 호출과 달리 에이전트는 여러 단계로 이루어진 실행 과정을 만들어냅니다. 계획을 세우고, 도구를 호출하고, 도구 결과에 반응하며, 여러 대화 턴에 걸쳐 진행되는 경우도 많습니다. 최종 답변만 평가하면 잘못될 수 있는 지점의 대부분을 놓치게 됩니다.

**요약:** 에이전트는 네 가지 차원으로 평가합니다: 궤적(에이전트가 거친 단계들의 경로), 도구 사용(올바른 도구를 올바른 인자로 호출했는지), 작업 완료(사용자의 목표가 충족되었는지), 멀티턴 품질(대화 전반에 걸쳐 성능이 유지되는지). 변경 사항을 배포하기 전에는 데이터셋으로 오프라인 평가를 실행하고, 배포 후에는 샘플링된 프로덕션 트래픽으로 온라인 평가를 실행합니다. 판별 가능한 항목에는 결정론적(deterministic) 코드 검사를, 의미론적 판단에는 LLM-as-a-judge를, 그리고 두 방식 모두를 보정하기 위해 사람이 직접 주석을 다는 방식을 사용합니다. 이 가이드는 Langfuse에서의 실제 패턴과 함께 각 차원을 다룹니다.

## AI 에이전트 평가가 측정하는 것

에이전트 품질은 네 가지 차원으로 나뉘며, 성숙한 평가 체계는 단일한 "전체 품질" 점수가 아니라 이 네 가지 모두를 측정합니다. 종합 점수 하나만으로는 에이전트가 나빠졌다는 사실만 알 수 있지만, 차원별 점수는 어디가 나빠졌는지를 알려줍니다.

| 차원      | 답하는 질문                                              | 대표적인 메트릭                                                              |
| --------- | -------------------------------------------------------- | ---------------------------------------------------------------------------- |
| 궤적      | 에이전트가 결과에 도달하기까지 합리적인 경로를 거쳤는가? | 단계 수, 불필요한 도구 호출, 루프와 재시도, 필수 단계 존재 여부, 올바른 순서 |
| 도구 사용 | 올바른 도구를 정확하게 호출했는가?                       | 올바른 도구 선택 여부, 인자 유효성, 도구 오류율, 실패한 호출 이후의 복구     |
| 작업 완료 | 사용자가 요청한 것을 얻었는가?                           | 목표 달성 여부(정답 유무와 무관), 답변 정확성, 해결율                        |
| 멀티턴    | 대화 전반에 걸쳐 품질이 유지되는가?                      | 턴 간 컨텍스트 유지, 목표 이탈(drift), 해결까지의 턴 수, 세션별 결과         |

이 차원들은 각각 독립적으로 실패할 수 있습니다. 에이전트는 두 번이면 충분한 도구 호출을 열두 번이나 하며 낭비적인 궤적으로도 작업을 완료할 수 있으며(이는 지연 시간과 비용으로 드러납니다), 반대로 깔끔한 궤적을 실행하고도 최종 종합(synthesis)이 잘못되어 목표를 놓칠 수도 있습니다. 도구 사용 오류는 최종 답변에서는 보이지 않는 경우가 많습니다. 에이전트가 복구에 성공하더라도, 재시도 과정에서 소모된 토큰과 시간은 결국 비용으로 지불하게 됩니다.

## 에이전트 평가가 LLM 평가와 다른 점

표준 LLM 평가는 입력-출력 쌍을 점수화합니다. 하나의 프롬프트, 하나의 완료(completion), 하나의 판단입니다. 이 모델은 에이전트에는 세 가지 측면에서 맞지 않습니다.

**1.** 평가 단위는 완료(completion)가 아니라 트레이스입니다. 에이전트 실행은 옵저베이션(LLM 호출, 도구 실행, 검색 등)으로 이루어진 트리이며, 서로 다른 품질 질문은 서로 다른 노드에 결부됩니다. 도구 인자 검사는 도구 호출 옵저베이션에, 작업 완료 평가는 트레이스의 루트에 속합니다.

**2.** 중간 단계들은 각기 독립적인 실패 유형을 가집니다. 검색은 잘못된 문서를 반환할 수 있고, 도구는 형식이 잘못된 인자로 호출될 수 있으며, 루프가 동일한 실패 호출을 반복할 수도 있습니다. 최종 텍스트만 점수화하면 이런 것들은 전혀 보이지 않습니다.

**3.** 많은 에이전트는 대화형이므로, 사용자가 실제로 경험하는 단위는 트레이스가 아니라 세션입니다. 챗봇은 개별적으로는 각각 합리적인 다섯 개의 턴을 생성하고도, 전체적으로는 사용자의 문제를 해결하지 못할 수 있습니다.

에이전트 내부의 LLM 호출에 여전히 적용되는 입력-출력 방법론에 대해서는 [LLM 애플리케이션 평가](/blog/2025-11-12-evals)에 대한 로드맵을 참고하세요. 이 가이드는 에이전트에 특화된 내용을 다룹니다.

## 오프라인 대 온라인 에이전트 평가

오프라인 평가는 배포 전에 고정된 데이터셋을 대상으로 에이전트를 실행하는 것이고, 온라인 평가는 배포 후 샘플링된 프로덕션 트래픽을 점수화하는 것입니다. 두 방식이 잡아내는 실패 유형이 다르기 때문에 둘 다 필요합니다.

|              | 오프라인 (실험)                          | 온라인 (프로덕션 평가기)                             |
| ------------ | ---------------------------------------- | ---------------------------------------------------- |
| 실행 시점    | 변경 사항 배포 전, 필요 시 또는 CI에서   | 실시간 트래픽에 대해 지속적으로                      |
| 데이터       | 큐레이션된 데이터셋, 보통 예상 출력 포함 | 실제 사용자 입력, 정답 데이터 없음                   |
| 잡아내는 것  | 프롬프트, 모델, 도구 변경으로 인한 회귀  | 드리프트, 새로운 입력, 실제 환경의 도구 실패         |
| 비용 구조    | 데이터셋 크기에 따라 제한됨              | 샘플링 비율로 제어                                   |
| Langfuse에서 | CI에 게이트로 걸린 데이터셋과 실험       | 필터와 샘플링을 적용한 실시간 옵저베이션 대상 평가기 |

표준적인 루프는 다음과 같습니다: 실패한 프로덕션 트레이스를 수집하고, 이를 데이터셋 항목으로 전환하고, 실험에서 그 실패를 재현하고, 수정한 뒤, 해당 항목을 영구적인 회귀 테스트로 남깁니다. Langfuse는 프로덕션 [트레이스](/docs/observability/data-model)와 실험 실행이 동일한 데이터 모델과 동일한 점수 체계를 공유하기 때문에 이 루프 전체를 지원합니다.

## 평가 방법 선택하기

세 가지 방법군이 에이전트 평가를 포괄하며, 어떤 것을 선택할지는 해당 검사가 코드로 판별 가능한지, 읽기가 필요한지, 아니면 도메인 전문 지식이 필요한지에 달려 있습니다.

| 방법                        | 사용 대상                  | 비용 및 지연 시간         | 에이전트 예시                                                                                    |
| --------------------------- | -------------------------- | ------------------------- | ------------------------------------------------------------------------------------------------ |
| 코드 평가기                 | 결정론적이고 객관적인 검사 | 실행당 무료, 밀리초 단위  | 필수 도구가 호출되었는지, 인자가 스키마와 맞는지, 단계 예산을 준수했는지, 출력이 유효한 JSON인지 |
| LLM-as-a-judge              | 대규모의 의미론적 판단     | 평가당 모델 비용, 초 단위 | 정답 없이 작업 완료 판단, 추론 품질, 최종 답변의 근거성(groundedness)                            |
| 사람 주석(human annotation) | 정답 데이터 확보 및 보정   | 전문가 시간               | 모호한 궤적에 라벨링, 판정기를 보정하는 데 쓰이는 참조 세트 구축                                 |

이 방법들은 대안이 아니라 계층입니다. 실전 구성에서는 샘플링된 모든 트레이스에 저렴한 코드 검사를 실행하고, 의미론적 판단이 중요한 하위 집합에는 LLM 판정기를 적용하며, 의견이 엇갈리거나 신뢰도가 낮은 사례는 사람의 검토를 위해 [주석 큐](/docs/evaluation/evaluation-methods/annotation-queues)로 라우팅합니다. 이렇게 얻은 사람의 라벨은 다시 판정기를 재보정하여 자동화 계층을 정확하게 유지합니다.

Langfuse에서 이 세 가지 방법은 모두 네 가지 데이터 타입 중 하나를 가진 [점수](/docs/evaluation/scores/overview)를 생성합니다: 단계 수와 같은 연속적인 측정치에는 numeric, "필수 도구를 사용했는가"와 같은 통과/실패 검사에는 boolean, `resolved` / `escalated` / `abandoned`와 같은 라벨에는 categorical, 자유 형식 리뷰어 메모에는 text가 사용됩니다. 점수는 트레이스, 옵저베이션, 세션, 또는 실험 실행에 결부되므로 에이전트 품질의 모든 차원이 하나의 조회 가능한 시스템에 담기게 됩니다.

## 도구 호출 평가하기

도구 호출 평가는 에이전트가 올바른 도구를 선택하고, 유효한 인자를 전달했으며, 결과를 적절히 처리했는지를 검증합니다. 2026년 7월 기준으로 Langfuse는 기록된 도구 호출을 구조화된 필드로 코드 평가기와 LLM-as-a-judge 평가기 양쪽에 노출하므로, 원시 옵저베이션 입력을 파싱하는 대신 이를 직접 검증할 수 있습니다.

[코드 평가기](/docs/evaluation/evaluation-methods/code-evaluators)에서는 대상 옵저베이션의 각 호출이 `id`, `name`, `arguments`, `type`, `index`를 가지며, JSON 인자는 유효하게 파싱되어 있습니다:

```python
def evaluate(ctx):
    tool_calls = ctx.observation.tool_calls
    used_search = any(tc.name == "search" for tc in tool_calls)
    return EvaluationResult(scores=[
        Score(name="used_search_tool", value=used_search, data_type="BOOLEAN"),
        Score(name="tool_call_count", value=len(tool_calls), data_type="NUMERIC"),
    ])
```

동일한 계약이 TypeScript에서도 `ctx.observation.toolCalls`로 존재합니다. 코드 평가기는 Langfuse 내부에서 실시간 프로덕션 옵저베이션 또는 실험 출력에 대해 실행되며, 네트워크 접근 없이 표준 라이브러리 코드만 실행하고, 한 번의 실행에서 여러 점수를 반환할 수 있으므로 하나의 평가기가 도구 선택, 인자 유효성, 호출 횟수 예산을 함께 다룰 수 있습니다. 셀프 호스팅 배포에서도 사용할 수 있습니다.

선택한 도구가 사용자 요청에 적합했는지와 같은 의미론적 질문에는 LLM-as-a-judge 평가기에서 **Tool Calls** 변수를 매핑하세요. `$[*].name`과 같은 JSONPath 선택자를 사용하면 판정기에는 도구 이름만 전달되며, 배열 전체를 매핑하면 인자까지 포함됩니다.

관련된 두 가지 패턴이 있습니다: 실험에서는 도구 호출 검증이 좋은 회귀 테스트가 됩니다("이 입력이 주어지면 에이전트는 반드시 `search`를 호출해야 한다"). 프로덕션에서는 boolean 도구 호출 점수가 샘플링된 모든 트레이스에 대해 무료로 계산할 수 있으므로 좋은 알림 신호가 됩니다. 궤적 예산 평가기를 포함해 바로 붙여 쓸 수 있는 열 가지 검사는 [코드 평가기 예제](/resources/engineering/code-evaluator-examples)에 정리되어 있습니다.

## 작업 완료 판단하기

작업 완료는 에이전트가 사용자의 목표를 달성했는지를 묻는 것이며, "목표를 달성했는가"는 전체 실행 과정에 대한 의미론적 판단이기 때문에 대개 LLM 판정기가 필요합니다. 판정기에는 전체 실행의 입력과 최종 출력이 필요하며, 중간 단계는 필요하지 않습니다.

Langfuse에서 [LLM-as-a-judge](/docs/evaluation/evaluation-methods/llm-as-a-judge) 평가기는 옵저베이션 수준에서 실행되며, 이는 2026년 7월 기준 권장 대상입니다(트레이스 수준 판정기 평가기는 레거시입니다). 전체 실행에 대한 판단을 내리는 패턴은 에이전트의 루트 옵저베이션, 즉 전체 요청과 최종 응답을 기록하는 스팬을 대상으로 지정하고, 그 입력과 출력을 판정기 프롬프트에 매핑하는 것입니다. 계측(instrumentation)이 아직 실행 요약을 루트 옵저베이션에 기록하지 않는다면 먼저 이를 추가하세요. 평가기는 대상으로 지정된 옵저베이션만 보고, 그 자식 옵저베이션은 보지 않습니다.

구체적으로는 다음과 같습니다:

1. 평가기를 이름이나 타입으로 루트 옵저베이션에 필터링하고, 필요하면 태그나 사용자 ID 같은 트레이스 속성으로 범위를 더 좁힙니다.
2. `{{input}}`과 `{{output}}`을 루트 옵저베이션의 필드에서 매핑하고, 활성화하기 전에 최근 24시간 동안의 매칭된 데이터로 채워진 프롬프트를 미리 확인합니다.
3. 프로덕션에서 판정기 비용을 제어할 샘플링 비율을 설정하고, 점수 타입을 선택합니다: "작업 완료 여부"에는 boolean, 완료 정도에는 numeric, 결과 라벨에는 categorical.

일반적인 차원에 대해서는 프롬프트를 직접 작성하지 않아도 됩니다. Langfuse는 Ragas와 같은 파트너와 함께 구축한 관리형 평가기 카탈로그를 제공하며, 여기에는 환각(hallucination), 컨텍스트 관련성, 유해성(toxicity), 유용성 같은 차원이 포함됩니다. 정답이 있는 실험에서는 데이터셋 항목의 예상 출력을 판정기의 참조값으로 매핑하면 됩니다. 평가기와 그 규칙은 공개 API를 통해서도 생성할 수 있으며, 이는 평가 구성을 버전 관리하는 데 유용합니다. 다만 2026년 7월 기준 해당 엔드포인트는 아직 unstable로 표시되어 있습니다.

## 궤적 점수화하기

궤적 평가는 목적지가 아니라 경로를 검사합니다: 어떤 단계가 실행되었는지, 어떤 순서로, 어떤 비용으로. 이는 정량적 절반과 정성적 절반으로 나뉩니다.

정량적 절반은 코드 평가기의 영역입니다. 단계 수, 루프 감지, 필수 단계 존재 여부, 예산 검사는 모두 판별 가능하며, 구조화된 도구 호출 필드와 옵저베이션 메타데이터가 평가기에 원재료를 제공합니다. 샘플링된 모든 프로덕션 트레이스에 대한 `within_step_budget`과 같은 궤적 점수는 "에이전트가 느려지고 비용이 더 들게 되었다"는 이야기를 일화가 아니라 대시보드의 한 줄로 바꿔줍니다.

정성적 절반은 궤적을 읽는 작업이며, 시각적 도구를 사용하면 더 빠릅니다. Langfuse의 [에이전트 그래프 뷰](/docs/observability/features/agent-graphs)는 트레이스를 두 가지 모드(2026년 7월 기준 베타)로 그래프로 렌더링합니다. **Aggregated** 모드는 반복되는 단계를 카운터가 붙은 단일 노드로 축약하므로, `retrieve_docs (3/3)`와 사이클 에지를 통해 에이전트의 형태와 루프를 한눈에 파악할 수 있습니다. **Expanded** 모드는 모든 호출을 실행 순서대로 펼쳐 DAG로 보여주어 특정 실행 하나를 단계별로 살펴보는 데 적합합니다. Aggregated 모드는 오류 분석 중 병적인 궤적을 빠르게 찾아내는 데 유용하고, Expanded 모드는 정확히 어디서 잘못되었는지 짚어내는 데 유용합니다.

잘못된 궤적을 발견하면 이를 규칙화하세요: 해당 입력을 데이터셋에 추가하고, 위반된 속성("결제 도구를 두 번 호출해서는 안 된다")을 코드 평가기로 작성하여 이후의 모든 실험에서 실행되도록 합니다.

## 멀티턴 대화 평가하기

멀티턴 평가는 개별 요청이 아니라 대화 전체를 점수화합니다. 사용자가 실제로 경험하는 단위가 세션이기 때문입니다. Langfuse에서는 모든 트레이스가 세션 ID를 가질 수 있고, 점수는 세션에 직접 결부될 수 있으므로 대화 수준의 판단이 그것이 설명하는 바로 그 수준에 존재하게 됩니다.

세션 수준 점수는 세 가지 방식으로 생성할 수 있습니다:

- 도메인 전문가가 주석 큐에서 대화 전체를 검토합니다. 주석 큐는 트레이스와 옵저베이션뿐 아니라 세션도 지원하며, `resolution`이나 `conversation_quality` 같은 점수를 부여합니다(키보드 단축키 기반 워크플로 덕분에 대량 검토도 실용적입니다).
- 애플리케이션이나 평가 파이프라인이 [SDK 또는 API를 통해](/docs/evaluation/evaluation-methods/scores-via-sdk) `session_id`만 전달해 세션 점수를 기록합니다. 예를 들어 대화 종료 시점에 전체 트랜스크립트를 읽고 목표 달성 여부와 컨텍스트 유지를 점수화하는 판정기가 있습니다.
- 리뷰어가 UI에서 세션을 살펴보며 직접 수동으로 점수를 매깁니다.

효과적인 역할 분담 방법: 옵저베이션 수준 평가기로는 턴별 품질(각 턴의 정확성과 도구 사용)을 평가하고, 세션 점수로는 대화 결과(해결 여부, 숫자 점수로서의 해결까지의 턴 수, 카테고리 점수로서의 이탈(abandonment))를 평가합니다. Boolean 및 categorical 세션 점수는 Langfuse UI 전반에서 필터링할 수 있으므로 "지난주에 해결되지 않은 모든 세션 보여줘"는 단순한 테이블 필터가 되며, 그런 세션은 곧바로 주석 큐나 데이터셋으로 이어집니다. 턴에 걸친 판정기 전략을 더 깊이 다룬 내용은 [멀티턴 대화 평가하기](/blog/2025-10-09-evaluating-multi-turn-conversations)를 참고하세요.

## CI에서 에이전트 변경 사항 게이트하기

코드에서 테스트가 병합 전에 게이트 역할을 하듯, 에이전트 변경 사항도 병합 전에 실험이 반드시 통과하도록 하면 안전하게 배포할 수 있습니다. 프롬프트 수정, 모델 교체, 도구 스키마 변경은 모두 타입 검사기가 잡아낼 수 없는 방식으로 에이전트 동작을 바꾸므로, 회귀 테스트 스위트는 데이터셋과 평가기로 구성됩니다.

Langfuse의 [실험 CI/CD 통합](/docs/evaluation/experiments/experiments-ci-cd)을 사용하면 GitHub Actions 단계가 데이터셋을 대상으로 실험 스크립트를 실행하고, 그 결과를 풀 리퀘스트에 게시하며, 점수가 임계값에 미달하면 워크플로를 실패시킵니다:

```yaml
- uses: langfuse/experiment-action@v1.0.0
  with:
    langfuse_public_key: ${{ secrets.LANGFUSE_PUBLIC_KEY }}
    langfuse_secret_key: ${{ secrets.LANGFUSE_SECRET_KEY }}
    langfuse_base_url: https://cloud.langfuse.com
    experiment_path: experiments/support-agent-gate
    dataset_name: support-agent-regression-set
    github_token: ${{ github.token }}
```

이 실험 스크립트는 SDK의 실험 러너(Python에서는 `run_experiment`, JS/TS에서는 `experiment.run`)를 사용하며, 개별 사례 검사에는 항목 수준 평가기를, 집계에는 실행 수준 평가기를 사용하고, 임계값을 위반하면 `RegressionError`를 발생시킵니다. `dataset_version`을 고정하면 게이트를 재현 가능하게 만들 수 있습니다. 이 액션은 Python SDK v4.6.0 이상 또는 JS SDK v5.3.0 이상을 필요로 합니다.

실험 결과는 프로그래밍 방식으로도 접근할 수 있습니다: 공개 API는 `GET /api/public/experiments`와 `GET /api/public/experiment-items`를 노출하며, Langfuse MCP 서버는 이를 `listExperiments`와 `listExperimentItems` 도구로 노출하므로, 커스텀 파이프라인과 코딩 에이전트가 UI 없이도 점수를 가져와 회귀에 대응할 수 있습니다. 우리는 이 데이터셋-실험 루프를 우리 자신의 에이전트 스킬을 개선하는 데도 활용했습니다. [AI 에이전트 스킬 평가하기](/blog/2026-02-26-evaluate-ai-agent-skills)에 실제 엔드투엔드 사례가 정리되어 있습니다.

## FAQ

### AI 에이전트 평가란 무엇인가요? [#what-is-ai-agent-evaluation]

AI 에이전트 평가는 자율적이고 여러 단계로 이루어진 LLM 시스템의 품질을 네 가지 차원, 즉 궤적(단계들의 경로), 도구 사용(올바른 도구 선택과 인자), 작업 완료(사용자의 목표가 충족되었는지), 멀티턴 품질(대화 전반에 걸쳐 성능이 유지되는지)에 걸쳐 측정하는 작업입니다. 평가 단위가 단일 입력-출력 쌍이 아니라 트레이스나 세션이라는 점에서 표준 LLM 평가와 다릅니다.

### AI 에이전트를 평가할 때 어떤 메트릭을 사용해야 하나요? [#agent-evaluation-metrics]

각 차원마다 하나의 메트릭으로 시작하세요: 각 실행의 루트에 대한 boolean 작업 완료 판단, 도구 호출 옵저베이션에 대한 도구 선택 정확성 검사, 궤적 효율성을 위한 예산 임계값을 가진 numeric 단계 수, 대화형 에이전트를 위한 세션 수준 해결 점수. 여기에 트레이싱이 추가 평가기 없이 제공하는 실행당 비용과 지연 시간을 더하세요. 어떤 메트릭이 실제 회귀를 잡아냈을 때만 확장하세요. 사용되지 않는 메트릭은 잡음일 뿐입니다.

### 정답 데이터 없이 에이전트를 어떻게 평가하나요? [#evaluate-without-ground-truth]

참조 없이도 사용할 수 있는 방법들을 활용하세요: 루브릭을 사용하는 LLM-as-a-judge(최종 답변이 명시된 목표를 다루는가, 에이전트가 검색한 도구 결과에 근거하고 있는가), 참조가 필요 없는 결정론적 속성 검사(유효한 출력 형식, 필수 도구 호출 여부, 단계 예산 준수 여부), 그리고 판정기를 보정하기 위한 샘플에 대한 사람의 주석 작업입니다. 환각과 같은 차원에 대한 관리형 평가기는 예상 답변이 아니라 트레이스에 제공된 컨텍스트를 기준으로 출력을 판단합니다. 정답 데이터는 실험에서의 정확 일치(exact-match) 방식 검사에만 필요합니다.

### 트레이스 수준 평가와 옵저베이션 수준 평가의 차이는 무엇인가요? [#trace-vs-observation-evals]

옵저베이션 수준 평가기는 트레이스 내부의 한 작업(LLM 호출, 도구 실행, 검색 등)을 점수화하므로 빠르고 정확합니다. Langfuse에서는 이것이 권장되는 대상이며, 트레이스 수준 판정기 평가기는 레거시입니다. 전체 에이전트 실행을 평가하려면 전체 입력과 최종 출력을 기록하는 루트 옵저베이션을 대상으로 지정하고, 계측이 실행 요약을 그 위에 기록하도록 해야 합니다. 평가기는 대상으로 지정된 옵저베이션만 보기 때문입니다.

### 멀티턴 대화는 어떻게 평가하나요? [#multi-turn-evaluation]

두 수준에서 점수화하세요: 개별 응답의 품질을 위한 턴별 평가기와, 해결 여부, 목표 이탈, 해결까지의 턴 수와 같은 대화 결과를 위한 세션 수준 점수입니다. Langfuse에서는 트레이스가 세션 ID를 공유하고, 점수는 UI, SDK, 또는 주석 큐를 통해 세션에 직접 결부되며, boolean 또는 categorical 세션 점수는 필터링이 가능하므로 실패한 대화를 검토 큐나 데이터셋으로 끌어올 수 있습니다.

### 에이전트 평가는 CI에서 실행해야 하나요? [#agent-evals-in-ci]

네, 에이전트 동작을 바꾸는 모든 변경 사항, 즉 프롬프트, 모델, 도구 스키마, 오케스트레이션 로직에 대해서는 그렇습니다. 버전 관리된 회귀 사례 데이터셋을 대상으로 실험을 실행하고, 테스트 스위트와 마찬가지로 점수가 임계값에 미달하면 파이프라인을 실패시키세요. CI 데이터셋은 모든 풀 리퀘스트에서 실행할 수 있을 만큼 작게(수십에서 수백 개 항목 정도) 유지하고, 더 큰 규모의 스윕은 일정에 따라 또는 릴리스 전에 실행하세요.

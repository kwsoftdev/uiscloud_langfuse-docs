# Braintrust에서 Langfuse로 마이그레이션

이 가이드는 LLM 관측성과 평가를 [Braintrust](https://www.braintrust.dev)에서
[Langfuse](/)로 마이그레이션하는 과정을 다룹니다: 먼저 계측을 교체한 뒤, 데이터셋, 실험,
프롬프트, 온라인 스코어링 순서로 진행합니다.

**요약:** Braintrust와 Langfuse는 동일한 핵심 개념(프로젝트, 로그/트레이스, 데이터셋, 실험,
스코어러)을 공유하므로, 이는 재설계라기보다 연결 대상을 바꾸는 작업에 가깝습니다. 계측은
래퍼 교체만으로 충분합니다: `wrap_openai`는 Langfuse의 OpenAI 드롭인으로, `@traced`는
`@observe`로 바뀝니다. 데이터셋은 Braintrust의 REST API나 SDK를 통해 내보낸 뒤 Langfuse
SDK로 가져옵니다. Langfuse가 autoevals 라이브러리를 직접 통합하고 있기 때문에, 기존
autoevals 스코어러는 재작성 없이 Langfuse 실험에서 그대로 동작합니다. 과거 로그는 보통 그대로
남겨두며, 실험은 마이그레이션된 데이터셋에 대해 다시 실행합니다.

## 팀들이 마이그레이션하는 이유

팀들이 Braintrust에서 Langfuse로의 전환을 검토하는 데는 몇 가지 반복적인 이유가 있습니다:

- **오픈소스와 완전한 셀프 호스팅.** Langfuse의 코어는 MIT 라이선스 오픈소스이며,
  [셀프 호스팅](/self-hosting)은 전체 스택(UI, API, 데이터)을 자체 인프라에서 실행합니다.
  Braintrust의 셀프 호스팅 옵션은 하이브리드 모델입니다: Terraform을 통해 데이터 플레인을
  자체 AWS 또는 Azure 계정에 배포하지만, 컨트롤 플레인(웹 앱과 메타데이터)은 Braintrust가
  호스팅합니다. 이는 Braintrust의
  [아키텍처 문서](https://www.braintrust.dev/docs/reference/platform/architecture)에
  설명되어 있습니다(2026년 7월 기준). 하이브리드 모델은 이벤트 데이터를 사용자 환경에
  유지하지만, 완전히 자체 완결적이거나 에어갭(air-gapped) 배포는 처음부터 끝까지 직접 실행할
  수 있는 소프트웨어에서만 가능합니다.
- **요청 수에 따라 결정되는 청구.** Langfuse Cloud는 집계된 단위(트레이스, 관측, 점수)를
  기준으로 청구하며, [공개 가격 정책](/pricing)은 월 5만 단위까지 무료로 시작합니다.
  Braintrust는 주로 로그, 실험, 데이터셋 전반에서 GB 단위로 측정되는 처리 데이터를 기준으로
  청구하며, 유료 플랜은 월 $249부터 시작합니다
  ([Braintrust 가격 정책](https://www.braintrust.dev/pricing), 2026년 7월 기준). 바이트
  기반 청구는 요청 수가 아니라 페이로드 크기에 따라 확장되므로, 트레이스에 큰 컨텍스트나
  첨부 파일이 포함되어 있다면 이 차이가 중요해집니다.
- **수집 단계의 샘플링.** Langfuse SDK는 소스 단에서 트레이스의 일정 비율을 유지하는 내장
  [샘플 레이트](/docs/observability/features/sampling)(`LANGFUSE_SAMPLE_RATE`)를
  제공하여, 트래픽이 많은 애플리케이션에서 볼륨과 비용을 직접 조절할 수 있는 다이얼 역할을
  합니다.

Braintrust는 여전히 촘촘한 평가 루프와 우수한 플레이그라운드, 관리형 하이브리드 배포를 갖춘
완성도 높은 플랫폼이며, 현재 팀에 잘 맞는다면 굳이 서둘러 옮길 이유는 없습니다. 이 가이드는
Langfuse로 통합하기로 결정한 팀을 위한 것입니다. 기능별 비교는
[Langfuse vs. Braintrust](/resources/engineering/best-braintrustdata-alternatives)를
참고하세요.

## 개념 매핑

Braintrust와 Langfuse는 거의 1:1로 대응되기 때문에 마이그레이션에 필요한 개념적 전환 폭이
작습니다:

| Braintrust                                | Langfuse                                                                                                                                      | 비고                                                               |
| ----------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| Project                                   | Project                                                                                                                                       | Langfuse는 프로젝트 상위에 조직(organization) 개념을 추가로 둡니다 |
| Logs (production traces and spans)        | [Traces / observations](/docs/observability/data-model)                                                                                       | 양쪽 모두 계층적인 스팬 트리 구조                                  |
| Experiments                               | [Experiments / dataset runs](/docs/evaluation/experiments/experiments-via-sdk)                                                                | 데이터셋 항목 및 점수와 연결된 실행                                |
| Datasets                                  | [Datasets](/docs/evaluation/experiments/datasets)                                                                                             | 양쪽 모두 `input` / `expected` / `metadata` 사용                   |
| Scorers (autoevals, LLM-as-a-judge, code) | [LLM-as-a-judge](/docs/evaluation/evaluation-methods/llm-as-a-judge) + [code evaluators](/docs/evaluation/evaluation-methods/code-evaluators) | autoevals 스코어러는 Langfuse 실험에서 그대로 동작                 |
| Human review                              | [Annotation queues](/docs/evaluation/evaluation-methods/annotation-queues)                                                                    | 트레이스에 대한 구조화된 사람 평가                                 |
| Playgrounds                               | [Playground](/docs/prompt-management/features/playground)                                                                                     | 실제 트레이스를 기반으로 프롬프트 반복 개선                        |
| Prompts                                   | [Prompt management](/docs/prompt-management)                                                                                                  | SDK를 통한 버전 및 배포 라벨 관리                                  |
| Gateway (formerly AI proxy)               | 필요하지 않음. 원한다면 [LiteLLM](/integrations/gateways/litellm) 같은 게이트웨이와 함께 사용                                                 | Langfuse는 요청 경로 밖에서 비동기적으로 관측                      |
| BTQL                                      | [Metrics API](/docs/metrics/features/metrics-api) 및 [custom dashboards](/docs/metrics/features/custom-dashboards)                            | REST API와 UI 내 대시보드를 통한 집계 메트릭                       |

시작하기 전에 알아둘 만한 아키텍처 차이가 하나 있습니다: Braintrust의 선택적
[게이트웨이](https://www.braintrust.dev/docs/deploy/gateway)(더 이상 사용되지 않는 AI
프록시를 대체한 기능)는 앱과 모델 제공업체 사이의 요청 경로에 위치합니다. Langfuse는 LLM 호출을
프록시하지 않습니다. SDK는 텔레메트리를 비동기적으로 배치 전송하므로, Langfuse에 장애가
발생해도 지연 시간이 늘어나거나 프로덕션 트래픽이 유실되지 않습니다.

## 1단계: 계측 전환

두 플랫폼 모두 동일한 방식으로 계측합니다: 모델 클라이언트를 래핑하고 함수를 데코레이트합니다.
마이그레이션은 패키지 교체 작업입니다.

### Python

```python
# Before (Braintrust)
from braintrust import init_logger, traced, wrap_openai
from openai import OpenAI

logger = init_logger(project="my-app")
client = wrap_openai(OpenAI())

@traced
def answer(question: str) -> str:
    response = client.chat.completions.create(
        model="gpt-4o", messages=[{"role": "user", "content": question}]
    )
    return response.choices[0].message.content
```

```python
# After (Langfuse)
from langfuse import observe
from langfuse.openai import OpenAI  # drop-in replacement, traces every call

client = OpenAI()

@observe()
def answer(question: str) -> str:
    response = client.chat.completions.create(
        model="gpt-4o", messages=[{"role": "user", "content": question}]
    )
    return response.choices[0].message.content
```

설정은 `BRAINTRUST_API_KEY`에서 Langfuse의 키 쌍(`LANGFUSE_PUBLIC_KEY`,
`LANGFUSE_SECRET_KEY`, `LANGFUSE_BASE_URL`)으로 옮겨갑니다. 별도로 호출해야 하는
`init_logger`에 상응하는 개념은 없습니다. 프로젝트는 API 키에 의해 결정됩니다. 중첩 방식은
Braintrust와 동일하게 동작합니다: 외부 함수에 붙인 `@observe`가 트레이스 계층 구조를 만들고,
래핑된 OpenAI 클라이언트는 활성 스팬에 제너레이션을 붙입니다.

### TypeScript

Langfuse JS/TS SDK는 OpenTelemetry 위에 구축되어 있으므로, 설정 시 스팬 프로세서를 한 번
등록한 뒤에는 Braintrust와 마찬가지로 클라이언트와 함수를 래핑합니다:

```typescript
// Before (Braintrust)
import { initLogger, wrapOpenAI, wrapTraced } from "braintrust";
import OpenAI from "openai";

const logger = initLogger({ projectName: "my-app" });
const client = wrapOpenAI(new OpenAI());

const answer = wrapTraced(async function answer(question: string) {
  const response = await client.chat.completions.create({
    model: "gpt-4o",
    messages: [{ role: "user", content: question }],
  });
  return response.choices[0].message.content;
});
```

```typescript
// After (Langfuse)
import { NodeSDK } from "@opentelemetry/sdk-node";
import { LangfuseSpanProcessor } from "@langfuse/otel";
import { observe } from "@langfuse/tracing";
import { observeOpenAI } from "@langfuse/openai";
import OpenAI from "openai";

const sdk = new NodeSDK({ spanProcessors: [new LangfuseSpanProcessor()] });
sdk.start();

const client = observeOpenAI(new OpenAI());

const answer = observe(
  async function answer(question: string) {
    const response = await client.chat.completions.create({
      model: "gpt-4o",
      messages: [{ role: "user", content: question }],
    });
    return response.choices[0].message.content;
  },
  { name: "answer" },
);
```

두 SDK 모두 OpenAI 이외의 범위까지 동일하게 다룹니다: Langfuse는 Anthropic, LangChain,
Vercel AI SDK, LiteLLM, Pydantic AI를 비롯한 100개 이상의 프레임워크와 제공업체에 대해
[네이티브 통합](/integrations)을 제공하므로, 래퍼 기반 계측(`wrap_anthropic`,
`wrap_litellm` 등)에도 직접 대응되는 기능이 있습니다.

### OpenTelemetry로 계측한 경우

Braintrust의 OTel 통합(Python의 `braintrust[otel]` 또는 JS의 `@braintrust/otel`)을
사용했거나 OTel 컬렉터에서 스팬을 내보냈다면, 계측 코드를 전혀 건드릴 필요가 없습니다. OTLP
익스포터를 [Langfuse의 엔드포인트](/integrations/native/opentelemetry)로 지정하기만 하면
됩니다:

```bash
OTEL_EXPORTER_OTLP_ENDPOINT="https://cloud.langfuse.com/api/public/otel"  # EU region
OTEL_EXPORTER_OTLP_HEADERS="Authorization=Basic ${AUTH_STRING},x-langfuse-ingestion-version=4"
```

`AUTH_STRING`은 base64로 인코딩한 프로젝트 키입니다
(`echo -n "pk-lf-...:sk-lf-..." | base64`). Langfuse는 protobuf와 JSON 인코딩의 HTTP를
통한 OTLP를 지원합니다(gRPC는 지원하지 않음). 검증 기간 동안 두 익스포터를 병렬로 운영하는 것이
전환 위험을 낮추는 일반적인 방법입니다.

SDK를 설정하는 김에 [샘플 레이트](/docs/observability/features/sampling)도
결정해두세요. `LANGFUSE_SAMPLE_RATE=0.1`로 설정하면 트레이스의 10%를 유지하는데, 이는
트래픽이 많은 엔드포인트를 모니터링하면서도 수집 비용을 비례적으로 줄이기에 충분한 경우가
많습니다. 기본값은 1.0(전체 유지)입니다.

## 2단계: 데이터셋 내보내기 및 재생성

Braintrust 데이터셋은 API나 SDK를 통해 깔끔하게 내보낼 수 있으며, 레코드 구조는 Langfuse
데이터셋 항목에 그대로 대응됩니다:

| Braintrust 필드 | Langfuse 필드                                     |
| --------------- | ------------------------------------------------- |
| `input`         | `input`                                           |
| `expected`      | `expected_output`                                 |
| `metadata`      | `metadata`                                        |
| `tags`          | `metadata` (전용 컬럼 없음; 메타데이터 키로 유지) |

일반적인 마이그레이션 방식은 각 Braintrust 데이터셋을 순회하며 행을 Langfuse에 삽입하는 짧은
스크립트를 작성하는 것입니다:

```python
from braintrust import init_dataset
from langfuse import get_client

langfuse = get_client()
langfuse.create_dataset(name="golden-set")

# init_dataset fetches an existing dataset; iterating yields all records
for row in init_dataset(project="my-app", name="golden-set"):
    langfuse.create_dataset_item(
        dataset_name="golden-set",
        input=row["input"],
        expected_output=row.get("expected"),
        metadata={**(row.get("metadata") or {}), "tags": row.get("tags")},
    )
```

Braintrust SDK가 없는 언어에서 REST API를 직접 사용하고 싶다면, 데이터셋 목록을 조회하는
`GET /v1/dataset`과 커서를 이용해 레코드를 페이지 단위로 가져오는
`GET /v1/dataset/{dataset_id}/fetch` 엔드포인트를 사용할 수 있습니다. 인증은
`https://api.braintrust.dev`에 대해 `Authorization: Bearer <api key>`로 이루어집니다
(Braintrust의 [OpenAPI 스펙](https://github.com/braintrustdata/braintrust-openapi) 기준,
2026년 7월 기준). fetch 엔드포인트는 과거 특정 시점의 데이터셋 스냅샷이 필요한 경우를 위해
`version` 파라미터도 지원합니다.

Langfuse [데이터셋](/docs/evaluation/experiments/datasets)은 폴더, `input`과
`expectedOutput`에 대한 JSON 스키마 검증, 미디어 첨부를 지원하므로, Braintrust의 관례에
암묵적으로 존재하던 구조를 가져오는 과정에서 명시적으로 정리할 수 있는 경우가 많습니다.

## 3단계: 실험 재실행 및 autoevals 스코어러 유지

Braintrust 실험은 `Eval(name, data, task, scores)`로 정의하고 `braintrust eval` CLI로
실행합니다. Langfuse에서 이에 대응하는 것은
[실험 러너 SDK](/docs/evaluation/experiments/experiments-via-sdk)입니다: 데이터셋에 대해
작업(task)을 반복하고 평가기를 적용하는 일반적인 스크립트이며, 동시성, 트레이싱, 오류 격리는
SDK가 처리합니다.

[autoevals](https://github.com/braintrustdata/autoevals)로 작성한 스코어러는 그대로
이전됩니다. Langfuse는 모든 autoevals 스코어러를 Langfuse 평가기로 변환해주는 컨버터를
제공합니다:

```python
# Before (Braintrust)
from braintrust import Eval
from autoevals.llm import Factuality

Eval(
    "my-app",
    data=load_examples,
    task=answer,
    scores=[Factuality],
)
```

```python
# After (Langfuse): same scorer, no rewrite
from langfuse import get_client
from langfuse.experiment import create_evaluator_from_autoevals
from autoevals.llm import Factuality

langfuse = get_client()
dataset = langfuse.get_dataset("golden-set")

def task(*, item, **kwargs):
    return answer(item.input)  # your existing application logic

result = dataset.run_experiment(
    name="post-migration-baseline",
    task=task,
    evaluators=[create_evaluator_from_autoevals(Factuality())],
)
print(result.format())
```

TypeScript SDK는 `@langfuse/client`의 `createEvaluatorFromAutoevals`를 통해 동일한
브리지를 제공합니다. 포팅할 때 유의할 차이는 두 가지입니다:

- **작업(task) 시그니처.** Braintrust의 작업은 원시 `input`을 받지만, Langfuse의 작업은
  전체 데이터셋 `item`(`item.input`, `item.expected_output`, `item.metadata`)을
  받습니다.
- **러너.** 전용 CLI는 없으며, 실험은 일반적인 Python/TS 스크립트입니다. 덕분에
  [CI/CD에서 실행](/docs/evaluation/experiments/experiments-ci-cd)해 회귀 게이트로
  활용하기가 수월합니다.

과거 Braintrust 실험 결과는 가져오기보다 참고용 아카이브로 남겨두는 편이 좋습니다. 점수는
마이그레이션된 데이터셋에 대해 다시 생성하는 비용이 적고, 애초에 도구 간 점수 비교 가능성도
불안정하기 때문입니다. 데이터셋 가져오기 직후 최신 베이스라인 실험을 Langfuse에서 다시
실행해두면 이후 실행의 기준점이 됩니다.

## 4단계: 프롬프트, 온라인 스코어러, 검토 워크플로 재구성

- **프롬프트.** Braintrust의 `GET /v1/prompt` 엔드포인트를 통해 프롬프트 정의를 내보낸 뒤,
  배포 대상을 지정하기 위한 라벨(예: `production`, `staging`)을 사용해
  [Langfuse 프롬프트 관리](/docs/prompt-management)에서 재구성합니다. 애플리케이션 코드는
  이름과 라벨로 프롬프트를 가져와 런타임에 변수를 컴파일합니다.
- **온라인 스코어러.** Braintrust의 온라인 스코어링(프로덕션 로그에서 샘플링해 실행하는
  스코어러)은 유입되는 트레이스에 대해 지속적으로 실행되며 자체 샘플링과 필터 제어 기능을 갖춘
  Langfuse의 [LLM-as-a-judge 평가기](/docs/evaluation/evaluation-methods/llm-as-a-judge)와
  [코드 평가기](/docs/evaluation/evaluation-methods/code-evaluators)에 대응됩니다.
- **사람 검토.** Braintrust의 사람 검토 루브릭은 Langfuse
  [주석 큐](/docs/evaluation/evaluation-methods/annotation-queues)의 점수 설정에
  대응됩니다: 숫자형, 범주형, 불리언 점수와 코멘트를 큐로 묶어 검토자에게 할당할 수 있습니다.

## 5단계: 과거 로그를 어떻게 처리할지 결정

대부분의 팀은 깔끔하게 전환합니다: 기존 로그는 보존 기간 동안 Braintrust에서 계속 조회할 수
있게 두고, 전환일부터 Langfuse를 시스템 오브 레코드로 삼습니다. 선택된 이력(예: 미해결 인시던트에
참조되는 트레이스)이 필요한 경우, Braintrust의
`GET /v1/project_logs/{project_id}/fetch` 엔드포인트로 로그 이벤트를 내보내고 Langfuse
수집 API로 백필할 수 있습니다. 전체를 일괄로 가져오는 것은 대부분의 경우 그럴 가치가
없습니다. 오래된 트레이스는 가치가 낮고, 그 노력은 새 파이프라인을 검증하는 데 쓰는 편이
낫습니다.

## 검증 체크리스트

- [ ] 트레이스가 예상된 계층 구조, 타이밍, 토큰/비용 데이터와 함께 Langfuse에 도착하는지 확인
- [ ] 사용자 및 세션 귀속이 동작하는지 확인 (Braintrust가 메타데이터를 사용하던 곳에 `user_id`, `session_id` 설정)
- [ ] 샘플 레이트를 의도적으로 설정했는지 확인 (기본값은 트레이스의 100% 유지)
- [ ] 데이터셋이 원본과 항목 수가 일치하도록 마이그레이션되었는지 확인
- [ ] Langfuse에서 autoevals 스코어러로 베이스라인 실험을 재실행해 점수가 산출되는지 확인
- [ ] 애플리케이션 코드에서 이름과 라벨로 프롬프트가 정상적으로 조회되는지 확인
- [ ] 온라인 평가기가 프로덕션 트레이스 샘플을 채점하고 있는지 확인
- [ ] 팀 접근 권한 설정 완료 (조직/프로젝트 역할, 해당되는 경우 SSO)
- [ ] Braintrust 익스포터를 제거했거나, 병렬 운영 기간에 종료 시점을 정해두었는지 확인

## 자주 묻는 질문

### autoevals 스코어러를 그대로 유지할 수 있나요? [#keep-autoevals]

가능합니다. autoevals는 Apache-2.0 오픈소스 라이브러리이며, Langfuse의 실험 SDK는
Langfuse 실험 내부에서 autoevals 스코어러를 변경 없이 실행할 수 있는 컨버터
(Python의 `create_evaluator_from_autoevals`, TypeScript의
`createEvaluatorFromAutoevals`)를 제공합니다. 일반 함수로 작성된 커스텀 스코어러도 Langfuse
평가기로 바로 이식됩니다.

### 과거 로그를 반드시 마이그레이션해야 하나요? [#historical-logs]

아니요, 대부분의 팀은 그렇게 하지 않습니다. 기존 로그는 보존 기간 동안 Braintrust에 그대로
남아 있고, Langfuse는 전환일부터의 모든 데이터를 기록합니다. Braintrust의 project-logs
fetch 엔드포인트와 Langfuse 수집 API를 통한 선택적 백필도 가능하지만, 실제로 참조하는
트레이스에 한해서만 하는 것이 대개 가치가 있습니다.

### Langfuse에도 Braintrust와 같은 LLM 게이트웨이나 프록시가 있나요? [#gateway]

없습니다. 이는 의도적인 설계입니다. Langfuse SDK는 텔레메트리를 요청 경로 밖에서 비동기적으로
전송하므로, 관측성이 모델 호출에 지연을 더하거나 장애 지점이 될 수 없습니다. 라우팅, 캐싱,
장애 조치 같은 게이트웨이 기능이 필요하다면, Langfuse는 [LiteLLM](/integrations/gateways/litellm)
같은 전용 게이트웨이와 통합하여 그로부터 트레이스를 받을 수 있습니다.

### Langfuse에서 수집량과 비용을 어떻게 조절하나요? [#sampling]

SDK에서 [샘플 레이트](/docs/observability/features/sampling)를 설정하세요
(`LANGFUSE_SAMPLE_RATE=0.1`로 설정하면 트레이스의 10%를 유지합니다). 이를 통해 소스
단에서 수집량을 제한할 수 있습니다. Langfuse는 처리된 데이터의 바이트가 아니라 집계된
단위로 청구하므로, 비용은 페이로드 크기가 아니라 보관하는 트레이스 수에 따라 확장됩니다.

### Braintrust와 달리 Langfuse는 오픈소스인가요? [#open-source]

Langfuse의 핵심 플랫폼은 MIT 라이선스이며 UI와 저장 계층을 포함해 완전히 셀프 호스팅할 수
있습니다. Braintrust의 SDK는 Apache-2.0 오픈소스이지만 플랫폼 자체는 독점 소프트웨어이며,
셀프 호스팅 옵션은 데이터 플레인이 사용자의 클라우드 계정에서 실행되고 컨트롤 플레인은
Braintrust가 호스팅하는 하이브리드 배포입니다(2026년 7월 기준). 특히 에어갭 운영이
필요하다면 두 모델 모두 컴플라이언스 요구사항에 비추어 확인해보시기 바랍니다.

# Langfuse로 코딩 에이전트 트레이싱하기

AI 코딩 에이전트는 이제 실제 엔지니어링 작업을 수행합니다. 파일을 편집하고, 터미널 명령을 실행하고,
MCP 도구를 호출하며, 그 과정에서 상당한 토큰 예산을 소비합니다. 이를 트레이싱하면 팀이 꾸준히 묻는
세 가지 질문에 답할 수 있습니다: **에이전트가 실제로 무엇을 했는가**, **개발자 또는 세션당 비용은
얼마인가**, 그리고 **어디에서 실패하거나 작업을 낭비하는가**입니다.

**요약:** Langfuse는 오늘날 주요 코딩 에이전트를 모두 트레이싱합니다. GitHub Copilot은
OpenTelemetry를 네이티브로 내보내므로, Langfuse의 OTLP 엔드포인트를 지정하기만 하면 끝입니다.
Claude Code와 OpenAI Codex는 각 세션을 Langfuse로 전송하는 라이프사이클 훅을 사용합니다. Cursor,
Kiro, OpenCode, Augment Code에도 전용 통합이 마련되어 있습니다. 설정은 개발자별로 몇 분이면
끝나며, 프록시나 게이트웨이가 필요하지 않습니다.

## 팀이 코딩 에이전트를 트레이싱하는 세 가지 이유

세 가지 요인이 꾸준히 등장합니다:

1. **에이전트 동작에 대한 가시성.** 코딩 에이전트의 세션은 모델 호출과 도구 실행이 길게 이어지는
   체인입니다. 잘못된 파일을 삭제하거나 실패하는 테스트에서 루프에 빠질 때, 트레이스는 어떤 도구가
   어떤 인자로, 어떤 모델 출력 이후에 실행되었는지 정확한 순서를 보여줍니다. 이는 프로덕션 앱에서
   LLM 트레이싱이 제공하는 것과 동일한 디버깅 가치를, 여러분 자신의 개발 워크플로우에 적용한
   것입니다.
2. **비용 및 사용량 계산.** 코딩 에이전트는 흔히 팀에서 가장 큰 단일 LLM 지출 항목입니다. 트레이스는
   턴 단위 토큰 사용량을 담고 있으므로, 개발자별·프로젝트별·모델별 비용이 추측이 아니라 대시보드
   쿼리로 확인됩니다.
3. **거버넌스.** 조직 전체에 에이전트를 배포하는 플랫폼 팀은 사용자별로 그룹화된 사용량, 어떤
   머신에서 무엇이 실행되었는지에 대한 감사 추적(audit trail), 그리고 프롬프트와 코드 콘텐츠를
   캡처할지 아니면 메타데이터만 캡처할지에 대한 명확한 정책을 필요로 합니다. 트레이싱은 이 세 가지
   모두를 위한 인프라를 제공합니다.

## 지원되는 코딩 에이전트

2026년 7월 기준, 아홉 개의 개발자 도구 통합 전체입니다:

| 에이전트                                                       | 메커니즘                                  | 설정 형태                                                     |
| -------------------------------------------------------------- | ----------------------------------------- | ------------------------------------------------------------- |
| [Claude Code](/integrations/developer-tools/claude-code)       | Stop 훅 스크립트(각 응답 후 실행)         | 훅 스크립트 + 환경 변수의 Langfuse 키                         |
| [OpenAI Codex](/integrations/developer-tools/codex)            | 플러그인 훅(턴마다 Stop 훅)               | `~/.codex/config.toml`의 플러그인; Node 22+, Codex 0.128+     |
| [GitHub Copilot](/integrations/developer-tools/github-copilot) | 네이티브 OpenTelemetry 내보내기           | OTLP 익스포터를 Langfuse로 지정; SDK 불필요, 코드 변경 불필요 |
| [Cursor](/integrations/developer-tools/cursor)                 | 에이전트 세션 트레이싱                    | 통합 페이지 참고                                              |
| [Kiro IDE](/integrations/developer-tools/kiro)                 | 에이전트 활동 트레이싱                    | 통합 페이지 참고                                              |
| [Kiro CLI](/integrations/developer-tools/kiro-cli)             | 터미널 세션 트레이싱                      | 통합 페이지 참고                                              |
| [OpenCode](/integrations/developer-tools/opencode)             | 세션 트레이싱(턴, 도구, 재시도, 추론)     | 통합 페이지 참고                                              |
| [Augment Code](/integrations/developer-tools/augment-code)     | 대화 + 도구 실행 트레이싱                 | 통합 페이지 참고                                              |
| [VS Code (MCP)](/integrations/developer-tools/vscode)          | Copilot 에이전트 모드의 Langfuse MCP 서버 | 에디터에서 프롬프트/트레이스/데이터셋 조회                    |

VS Code 항목은 반대 방향입니다. 에이전트를 트레이싱하는 대신, MCP를 통해 에이전트에게 여러분의
Langfuse 데이터에 대한 접근 권한을 부여합니다. 이는 트레이스가 이미 존재하고 에디터를 벗어나지 않고
분석하고 싶을 때 유용합니다.

## 트레이스로 할 수 있는 일

- **개발자별, 모델별 비용 대시보드.** 훅 및 OTel 기반 통합은 사용자 식별자를 첨부하므로,
  [대시보드](/docs/metrics/features/custom-dashboards)에서 토큰 지출을 개발자별, 프로젝트별,
  모델별로 세분화할 수 있습니다. 플랫폼 팀이 배포 결정과 예산 계획에 필요로 하는 수치입니다.
- **도구 사용 분석.** 모든 도구 호출은 하나의 관찰(observation)입니다. 어떤 도구가 가장 많이
  사용되는지, 어떤 도구가 실패하는지, 어떤 도구 다음에 세션이 중단되는지를 모두 쿼리할 수 있습니다.
- **실패한 세션 재구성.** 잘못된 방향으로 진행된 긴 에이전트 세션은 터미널 스크롤백만으로는
  재구성하기 어렵습니다. 트레이스 타임라인은 재시도와, 에이전트가 노출하는 경우의 추론 요약을
  포함해 전체 순서를 보존합니다.
- **세션 전체에 대한 전문 검색(full-text search)**으로 누가 특정 API, 패턴, 스킬을 사용했는지 찾을
  수 있습니다. Claude Code를 대규모로 운영하는 팀에서 가장 자주 요청하는 기능 중 하나입니다.

## 한계

- **세션 컨텍스트 파일은 캡처되지 않습니다.** `CLAUDE.md`, 스킬, 또는 자동으로 로드되는 컨텍스트가
  실제 시스템 프롬프트에 어떻게 기여했는지는 현재 훅이 읽을 수 있는 범위에 포함되지 않습니다. 대화와
  도구 호출은 볼 수 있지만, 조합된 컨텍스트 자체는 볼 수 없습니다.
- **훅은 머신 단위이며 사용자가 직접 관리할 수 있습니다.** 개발자는 로컬 설정에서 훅을 비활성화할 수
  있습니다. 컴플라이언스를 위해 보장된 캡처가 필요하다면, 훅 기반 트레이싱을 강제(enforcement)
  수단이 아니라 텔레메트리로 취급하세요. 강제는 게이트웨이나 프로바이더 수준에서 이루어져야 합니다.
- **웹 기반 에이전트는 대상 범위 밖입니다.** 훅이나 텔레메트리 내보내기가 없는 도구(예: 순수하게
  웹에서 호스팅되는 어시스턴트)는 이 방식으로 트레이싱할 수 없습니다.

## FAQ

### 어떤 Claude Code 스킬이나 규칙이 사용되었는지 추적할 수 있나요?

간접적으로 가능합니다. 도구 호출로 실행되는 스킬 호출은 트레이스에 나타나며, 세션 콘텐츠에 대한
전문 검색으로 스킬 언급을 찾을 수 있습니다. 조합된 컨텍스트 자체(어떤 스킬이 로드되었는지)는
캡처되지 않습니다. 위의 한계 항목을 참고하세요.

### 셀프 호스팅 Langfuse에서도 동작하나요?

예. 이 통합들은 표준 수집(ingestion) 및 OTLP 엔드포인트를 대상으로 합니다. 최소 버전 요구 사항은
각 통합 페이지에서 확인하세요. Claude Code 통합은 최신 Langfuse 버전을 필요로 하며, 이전 셀프
호스팅 배포(v1/v2)는 OTLP 엔드포인트가 도입되기 이전 버전입니다.

### 코딩 에이전트를 트레이싱하면 Langfuse 단위로 비용이 얼마나 드나요?

일반적인 트레이스와 동일합니다. 세션은 통합 방식에 따라 턴당 또는 세션당 하나의 트레이스를
생성하며, 모델 호출과 도구 실행에 대한 관찰(observation)을 포함합니다. 사용량이 많은 에이전트
사용은 예상치 못한 청구서로 이어지기 전에 [빌링 가능 단위
페이지](/docs/administration/billable-units)에서 확인할 수 있습니다.

### 개발자가 옵트아웃할 수 있나요?

예, 훅은 개발자의 로컬 설정에 있습니다. 팀 단위로 배포할 때는 트레이싱을 표준 dotfiles/설정의
일부로 포함시키고, 무엇이 캡처되는지 명확히 밝히세요. Copilot 통합의 메타데이터 전용 기본값은
프라이버시를 보수적으로 고려한 기준선을 위한 좋은 템플릿입니다.

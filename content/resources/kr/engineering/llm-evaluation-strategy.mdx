# LLM 평가 전략을 구축하는 방법

대부분의 LLM 평가 콘텐츠는 방법론을 설명합니다. 저지 프롬프트가 어떻게 생겼는지, 유사도를 어떻게 계산하는지, 정확 일치(exact match)만으로 충분한 경우는 언제인지와 같은 것들입니다. 하지만 팀 전체에 평가를 도입하는 것은 다른 문제입니다. 이 애플리케이션에서 어떤 품질 차원이 중요한지, 각 차원을 어떤 방법으로 측정할지, 어떤 점수가 릴리스를 막을지, 그리고 경영진이 시스템이 제대로 작동한다는 것을 어떻게 아는지 물었을 때 어떤 근거를 제시할지를 누군가는 결정해야 합니다. 이 가이드는 Langfuse를 실전 예시로 삼아 이러한 도입 과정을 단계별로 다룹니다.

**요약:** LLM 평가 전략이란 신경 쓰는 모든 품질 차원을 그것을 신뢰성 있게 측정할 수 있는 가장 저렴한 방법에 매핑한 다음, 의사결정이 이루어지는 지점, 즉 병합 전 실험, CI의 게이트, 프로덕션 트래픽에 대한 샘플링된 평가기, 자동화가 내릴 수 없는 판단을 위한 휴먼 리뷰 루프에 그 측정 결과를 연결하는 것입니다. 다음 여섯 단계로 구축합니다: 관찰된 실패 모드로부터 차원을 도출하고, 차원별로 방법을 선택하고, 데이터셋과 실험으로 오프라인 베이스라인을 수립하고, CI에서 변경 사항을 게이트하고, 샘플링된 평가기와 알림으로 프로덕션을 모니터링하고, 프로덕션 실패를 새로운 테스트 케이스로 전환해 루프를 닫습니다.

## 왜 팀에는 평가기가 아니라 전략이 필요한가

의사결정이 연결되지 않은 평가기는 아무도 조치를 취하지 않는 숫자를 만들어낼 뿐입니다. "평가를 하고 있다"고 말하면서도 점수가 떨어졌을 때 무슨 일이 일어나는지 말하지 못하는 팀은 전략이 아니라 모니터링을 하고 있는 것입니다. 전략은 모든 품질 차원에 대해 다음을 명시합니다.

1. **무엇을** 측정하는가: 두 사람이 동일한 트레이스에 동일한 점수를 매길 수 있을 만큼 정밀하게 정의되어야 합니다.
2. **어떻게** 측정하는가: 결정론적 코드, 모델 저지, 혹은 사람 리뷰어입니다.
3. **언제** 측정하는가: 릴리스 전 실험 단계인지, 프로덕션 트래픽에서인지, 혹은 둘 다인지입니다.
4. 측정값이 임계값을 넘었을 때 **무슨 일이 일어나는가**: 릴리스가 차단되는지, 알림이 발생하는지, 리뷰어에게 트레이스가 전달되는지입니다.

이 매핑은 트레이스를 직접 읽지 않는 사람들에게 보여줄 산출물이기도 합니다. 엔지니어링 경영진은 다음 릴리스가 품질을 저하시키는지 알고 싶어 하고, 규제 의무를 지는 팀은 누가 무엇을 확인했는지 보여줘야 합니다. "저희는 환각(hallucination) 저지를 돌리고 있습니다"라는 대답은 둘 중 어느 것에도 답이 되지 않습니다. 반면 차원-방법-임계값이 문서화된 매핑은 둘 다에 답이 됩니다. [평가 핵심 개념](/docs/evaluation/core-concepts) 페이지에서 기본 구성 요소를 다루고 있으며, 이 가이드의 나머지 부분은 그것들을 조립하는 방법을 다룹니다.

## 1단계: 실제 실패 모드로부터 품질 차원 정의하기

차원은 메트릭 카탈로그가 아니라 트레이스로부터 도출해야 합니다. "일관성(coherence)" 같은 일반적인 메트릭은 절대 변하지 않고 어떤 조치도 유발하지 않는 점수를 만들어냅니다. 대신 프로덕션이나 파일럿에서 실제 상호작용 50~100건을 읽고 실제로 무엇이 잘못되었는지 기록하세요. 반복적으로 나타나는 실패는 각각 차원 후보가 됩니다.

- 어시스턴트가 에스컬레이션했어야 할 질문에 직접 답했다 → 불리언 `out_of_scope_handled` 점수가 됩니다.
- 답변에 필수 고지 사항이 빠졌다 → 불리언 `disclosure_present` 점수가 됩니다.
- 응답이 검색된 문서와 모순됐다 → 수치형 `faithfulness` 점수가 됩니다.
- 에이전트가 잘못된 도구를 호출했거나 루프에 빠졌다 → `used_required_tool`, `within_step_budget`과 같은 궤적(trajectory) 체크가 됩니다.

복합 차원은 지양하세요. 1~5점짜리 단일 "컴플라이언스" 점수는 리뷰어에게 꼭 필요한 정보를 가려버립니다. 그러니 이를 각각 이름이 붙은 기준별 하위 점수로 분리하세요. 답변이 조언 경계 안에 머물렀는지, 필수 고지 사항이 표시되었는지, 어조가 대상 독자에게 맞았는지처럼 말입니다. Langfuse에서는 각 하위 점수가 개별 점수 설정(불리언, 범주형, 수치형)이며, 자동화된 평가기와 사람 주석자가 동일한 점수 설정에 기록하기 때문에 결과가 하나의 타임라인 안에서 비교 가능한 상태로 유지됩니다.

세 개에서 다섯 개 차원으로 시작하세요. 정의하는 차원마다 유지 관리해야 할 평가기와 논쟁해야 할 임계값이 늘어나며, 실제로 조치를 취하는 짧은 목록이 방치되는 긴 목록보다 낫습니다.

## 2단계: 차원별로 평가 방법 선택하기

해당 차원을 신뢰성 있게 측정할 수 있는 가장 저렴한 방법을 사용하세요. 검사가 판별 가능하다면 코드로, 읽어야 판단할 수 있다면 LLM 저지로, 도메인 책임이 필요하거나 그라운드 트루스를 만들어야 한다면 사람으로 판단합니다.

| 방법           | 적합한 경우                                                     | 비용 및 지연 시간             | 일관성                                                |
| -------------- | --------------------------------------------------------------- | ----------------------------- | ----------------------------------------------------- |
| 코드 평가기    | 형식 검증, 정책 패턴, 수치 허용 오차, 궤적 및 도구 호출 체크    | LLM 비용 없음, 밀리초 단위    | 결정론적, 항상 같은 판정                              |
| LLM-as-a-judge | 충실도(faithfulness), 유용성, 어조, 추론 품질                   | 호출당 LLM 비용, 초 단위      | 좋은 루브릭이 있으면 일관적이지만 결정론적이지는 않음 |
| 사람 주석      | 그라운드 트루스, 저지 캘리브레이션, 도메인 및 컴플라이언스 승인 | 리뷰어 시간, 시간에서 일 단위 | 권위 있음, 리뷰어마다 편차 있음                       |

가장 강력한 구성은 이 세 가지를 연쇄적으로 결합합니다. 무료 결정론적 스크리닝을 모든 항목에 실행하고, 스크리닝에서 플래그가 붙은 항목과 나머지 중 일부 샘플에 저지를 실행하며, 저지가 확신하지 못하거나 리스크가 큰 경우에는 사람이 리뷰합니다. [코드 평가기 예시](/resources/engineering/code-evaluator-examples) 페이지에는 바로 붙여넣어 사용할 수 있는 결정론적 체크 10가지가 정리되어 있습니다. 에이전트 애플리케이션의 경우, 평가기가 전용 구조화 필드(2026년 7월부터 제공)를 통해 기록된 도구 호출을 읽을 수 있으므로 "에이전트가 답변 전에 검색을 호출했는가" 같은 궤적 차원은 저지 없이도 코드로 판별할 수 있습니다.

## 3단계: 오프라인 기반 구축하기 - 데이터셋, 실험, CI 게이트

무언가를 개선하기 전에 현재 품질이 어느 수준인지 먼저 확인해야 합니다. 여기에는 세 가지 요소가 필요합니다.

1. **현실을 대표하는 데이터셋.** 지어낸 예시가 아니라 실제 트레이스로 구축하세요. Langfuse에서는 관측(observation) 테이블에서의 일괄 선택을 포함해, UI에서 바로 어떤 프로덕션 관측이든 [데이터셋](/docs/evaluation/experiments/datasets)에 추가할 수 있습니다. 아직 보지는 못했지만 우려되는 실패 케이스는 직접 작성해서 추가하세요. 기대 출력값(expected output)은 도메인 전문가가 실제 출력값을 교정한 것에서 나올 수 있습니다.
2. **비교 메커니즘으로서의 실험.** 실험은 애플리케이션을 데이터셋에 대해 실행하고 모든 출력에 평가기를 적용합니다. 현재 버전에 대한 첫 실행이 베이스라인이 되며, 이후의 모든 프롬프트 변경, 모델 교체, 검색(retrieval) 조정은 의견의 문제가 아니라 이 베이스라인과의 비교가 됩니다.
3. **CI의 게이트.** 실험이 마련되면 릴리스마다 실행해, 깨진 단위 테스트가 빌드를 실패시키는 것과 같은 방식으로 품질 저하가 빌드를 실패시키도록 합니다.

GitHub의 경우 [CI/CD에서의 실험](/docs/evaluation/experiments/experiments-ci-cd) 설정은 `langfuse/experiment-action`을 사용합니다. 이 액션은 데이터셋을 로드하고(재현 가능한 실행을 위해 특정 버전에 고정할 수 있습니다), 실험 스크립트를 실행하고, 점수를 PR 코멘트로 게시하며, 스크립트가 `RegressionError`를 발생시키면 작업을 실패시킵니다.

```python
from langfuse import Evaluation, RegressionError, RunnerContext

THRESHOLD = 0.95


def answer_support_question(item, **kwargs):
    ...  # call your application with the dataset item


def exact_match(*, output, expected_output, **kwargs):
    passed = output.strip() == (expected_output or "").strip()
    return Evaluation(name="exact_match", value=1.0 if passed else 0.0)


def avg_accuracy(*, item_results, **kwargs):
    scores = [
        e.value
        for r in item_results
        for e in r.evaluations
        if e.name == "exact_match" and isinstance(e.value, (int, float))
    ]
    return Evaluation(name="avg_accuracy", value=sum(scores) / len(scores) if scores else 0.0)


def experiment(context: RunnerContext):
    result = context.run_experiment(
        name="PR gate: support agent",
        task=answer_support_question,
        evaluators=[exact_match],
        run_evaluators=[avg_accuracy],
    )
    accuracy = next(
        (e.value for e in result.run_evaluations if e.name == "avg_accuracy"),
        None,
    )
    if not isinstance(accuracy, (int, float)) or accuracy < THRESHOLD:
        raise RegressionError(
            result=result,
            metric="avg_accuracy",
            value=float(accuracy) if isinstance(accuracy, (int, float)) else 0.0,
            threshold=THRESHOLD,
        )
    return result
```

다른 CI 시스템에서도 동일한 실험 러너를 Pytest나 Vitest 내부에서 사용할 수 있으며, 임계값 어서션이 테스트 스위트를 실패시키는 방식입니다. 모든 PR에서 실행할 수 있도록 게이트용 데이터셋은 작게(수십~수백 개 항목 이내) 유지하고, 전체 회귀 세트는 릴리스 브랜치용으로 남겨두세요.

## 4단계: 샘플링된 평가기와 알림으로 프로덕션 모니터링하기

오프라인 평가는 테스트할 생각을 했던 입력만 다룰 뿐이며, 프로덕션 트래픽은 오래지 않아 데이터셋의 분포와 어긋나게 됩니다. 전략의 온라인 절반은 실시간 트래픽에 대해 평가기를 실행하는 것입니다.

- **실시간 관측에 대한 샘플링 저지.** [LLM-as-a-judge 평가기](/docs/evaluation/evaluation-methods/llm-as-a-judge)는 필터에 일치하는 프로덕션 관측에 대해 실행되며, 비용을 통제하기 위한 샘플링 비율을 설정할 수 있습니다. 보통 트래픽의 5~10%만 샘플링해도 전체를 판단하는 비용을 들이지 않고 품질 추세를 파악하기에 충분합니다.
- **의도(intent)에 따른 평가기 라우팅.** 대화 유형마다 서로 다른 루브릭이 필요합니다. 제품 조언을 평가하도록 만든 평가기를 결제 관련 질문에 적용하는 것은 말이 안 됩니다. 애플리케이션에서 의도를 분류하고 이를 태그나 메타데이터로 트레이스에 기록한 다음, 의도마다 일치하는 필터를 가진 평가기를 하나씩 만드세요. 그러면 각 평가기는 자신의 루브릭이 작성된 트래픽만 판단하게 됩니다.
- **점수 임계값에 대한 알림.** (Langfuse Cloud의) [모니터](/docs/metrics/features/monitors)는 시간 창(window) 동안 수치형 또는 범주형 점수 메트릭을 경고 및 알림 임계값과 비교하고, 위반 시 Slack, 웹훅, 또는 GitHub Actions 트리거로 라우팅합니다. 충실도(faithfulness) 평균이 하락하면 사용자가 불만을 제기하기 전에 담당자에게 알림이 갑니다.

한 가지 지켜야 할 경계가 있습니다. 온라인 평가기 점수는 응답이 이미 전달된 후 비동기적으로 생성됩니다. 이는 회귀를 감지하고, 트레이스를 리뷰로 라우팅하고, 데이터셋을 구축하기 위한 것이지, 실시간으로 나쁜 답변을 막기 위한 것이 아닙니다. 요청 시점에 동작을 바꿔야 하는 체크가 있다면 그것은 애플리케이션 코드 안의 가드레일로 두어야 하며, 그 가드레일의 판단 역시 다른 모든 것과 마찬가지로 평가 대상이 됩니다.

## 5단계: 휴먼 리뷰 루프 운영하기

자동화된 점수는 추세를 알려주지만, "좋다"는 것이 무엇을 의미하는지 정의하고 루브릭이 놓치는 것을 잡아내는 것은 사람입니다. [주석 큐](/docs/evaluation/evaluation-methods/annotation-queues)는 이를 화면 공유가 아닌 하나의 워크플로로 만들어 줍니다. 큐가 사용할 점수 설정을 정의하고, 리뷰어를 배정하고, 트레이스나 관측, 혹은 세션 전체를 구조화된 채점을 위해 큐로 라우팅할 수 있습니다.

팀 환경에서 이 루프가 잘 작동하도록 만드는 네 가지 관행이 있습니다.

- **사람과 자동화 점수가 점수 설정을 공유합니다.** 주석자가 저지와 동일한 점수 차원에 기록하기 때문에, 두 결과를 하나의 차트에 함께 놓고 서로 어긋나는 지점을 찾아내며 사람의 점수를 캘리브레이션 기준으로 삼을 수 있습니다.
- **리뷰어 그룹마다 다른 큐를 사용합니다.** 엔지니어는 한 큐에서 플래그가 붙은 트레이스를 분류하고, 도메인 또는 컴플라이언스 리뷰어는 동일한 트래픽을 별도 큐에서 자신들만의 하위 점수 기준으로 채점할 수 있습니다. 배정, 점수 설정, 진행 상황은 큐 단위로 관리되므로 각 그룹은 자신에게 해당하는 리뷰 작업만 보게 됩니다.
- **리뷰어는 점수뿐 아니라 교정 내용도 기록합니다.** 출력이 잘못되었다고 표시한 리뷰어는 [교정(corrections)](/docs/observability/features/corrections) 기능을 통해 출력이 어떠했어야 했는지도 함께 기록할 수 있습니다. 교정된 출력은 데이터셋에 필요한 기대 출력값 그 자체입니다.
- **채점은 소급 적용이 가능합니다.** 리뷰어는 언제든지 과거 트레이스에 주석을 달 수 있고, 자동화된 평가기도 트레이스 테이블에서 과거 관측에 대해 점수를 소급 채울(backfill) 수 있습니다. 이번 주에 정의한 기준을 지난 분기의 트래픽에도 적용해 측정할 수 있습니다.

모든 팀이 결국 묻게 되는 질문은 휴먼 리뷰를 언제 자동화된 임계값으로 대체할 수 있는가입니다. 그 메커니즘은 캘리브레이션입니다. 사람과 저지가 동일한 항목을 채점하게 하고, 차원별로 일치도를 측정하며, 일치도가 안정될 때까지 루브릭을 반복 개선하세요. 저지가 신뢰를 얻은 영역에서는 사람의 역할을 모든 것을 리뷰하는 것에서 저지가 플래그를 붙인 항목, 의견이 갈리는 사례, 그리고 저지 자체가 드리프트하지 않았는지 검증하는 무작위 샘플을 리뷰하는 것으로 옮길 수 있습니다. 어느 정도의 잔여 사람 리뷰 범위가 필요한지는 조직의 의무에 따른 정책적 판단이며, 전략의 역할은 이 트레이드오프를 수사가 아니라 측정 가능한 것으로 만드는 데 있습니다.

## 6단계: 프로덕션 실패에서 다음 릴리스까지 루프 닫기

측정은 애플리케이션을 실제로 바꿀 때에만 의미가 있습니다. 앞선 단계들을 연결하는 루프는 다음과 같습니다.

1. 프로덕션 트레이스가 실패합니다: 점수가 임계값을 넘거나, 알림이 발생하거나, 리뷰어가 플래그를 붙입니다.
2. 해당 트레이스는 데이터셋에 들어가며, 리뷰어가 제공한 경우 교정된 기대 출력값도 함께 들어갑니다. 이제 이 실패는 회귀 세트에 영구적으로 남습니다.
3. 프롬프트(또는 검색, 도구 정의)를 반복 개선하고, 업데이트된 데이터셋에 대해 실험을 실행해 후보를 현재 베이스라인과 비교합니다.
4. 실험 결과가 더 우수하면 변경 사항을 승격(promote)합니다. Langfuse에서 관리되는 프롬프트의 경우, 승격이란 `production` 레이블을 새 버전으로 옮기는 것을 의미합니다. [보호된 레이블](/docs/prompt-management/features/prompt-version-control)은 이 레이블을 누가 옮길 수 있는지를 제한하므로, 실험 결과와 승인된 권한자의 조합이 승격 게이트가 됩니다. (레이블 보호 기능은 Teams 애드온, Enterprise 플랜, 또는 셀프 호스팅 EE가 필요합니다.)
5. 3단계의 CI 게이트가 이후 이 수정 사항이 조용히 회귀하는 것을 막아줍니다.

이 루프를 한 바퀴 돌 때마다 데이터셋은 커지고, 평가기는 정교해지며, 다음 인시던트는 더 짧아집니다. 기반이 되는 기능들을 처음 접하는 팀이라면, [Langfuse Academy](/academy)에서 트레이싱부터 데이터셋과 실험, 프로덕션 모니터링에 이르기까지 이 루프 전체를 처음부터 끝까지 다룹니다.

## 규제 환경에서 이 전략이 증거로서 만들어내는 것

금융 행위 규제, 소비자 보호, 또는 이와 유사한 의무를 지는 팀은 AI 기능이 품질 관리되고 있음을 주장이 아니라 입증해야 합니다. 위 전략은 다음 네 가지 속성이 충족될 때 부산물로서 그러한 증거를 만들어냅니다.

- **모든 것이 버전 관리됩니다.** 프롬프트는 레이블이 붙은 배포와 함께 버전 관리되고, 데이터셋도 버전 관리되며, 모든 실험은 어떤 프롬프트 버전이 어떤 데이터셋 버전에 대해 실행되었고 어떤 점수를 받았는지 기록합니다. "3월에 무엇이 라이브였고 그것이 안전하다는 것을 어떻게 알았는가"는 발굴 작업이 아니라 조회 작업이 됩니다.
- **가능한 한 게이트는 결정론적입니다.** 코드 평가기는 동일한 입력에 대해 항상 동일한 판정을 반환하므로, 임계값에 의해 릴리스가 차단되거나 통과되었다는 것은 재현 가능한 사실입니다. 결정론적 게이트는 "그날 저지의 판단이 달랐다"보다 훨씬 방어하기 쉽습니다.
- **모든 점수에는 작성자가 있습니다.** 점수는 그 출처를 기록합니다. 어떤 평가기인지, 어떤 파이프라인인지, 혹은 어떤 사람 리뷰어인지 말입니다. 특정 트레이스에 대해 이름이 명시된 리뷰어가 남긴 컴플라이언스 하위 점수는 기준별 증거이며, 단순한 승인 표시보다 강력합니다. Enterprise 플랜과 셀프 호스팅 EE에서는 [감사 로그](/docs/administration/audit-logs)가 추가로 누가 평가기, 프롬프트, 설정을 변경했는지를 변경 전후 상태와 함께 기록합니다.
- **리뷰 범위는 측정 가능합니다.** 사람 리뷰가 정의된 점수 설정을 가진 큐를 통해 이루어지기 때문에, 트래픽 중 몇 퍼센트가 누구에 의해 어떤 기준으로 리뷰되었는지, 그리고 리뷰어의 점수가 자동화된 점수와 얼마나 일치했는지를 명시할 수 있습니다.

특정 규제 기관이 어떤 산출물을 요구하는지는 엔지니어링 가이드가 아니라 컴플라이언스 팀이 답할 질문입니다. 엔지니어링이 할 일은 그 질문에 답하는 데 수작업 증거 수집 스크램블이 필요 없도록, 이러한 산출물이 구조적으로 존재하게 만드는 것입니다.

## 이미 사내 평가 루프를 운영하고 있다면

많은 팀이 자체 구축한 체계, 즉 출력을 채점하는 스크립트나 스스로의 답변을 비평하고 재시도하는 애플리케이션을 이미 갖춘 상태로 시작합니다. 잘 작동하는 부분은 그대로 유지하세요. 외부 평가 파이프라인도 [SDK를 통해 Langfuse에 점수를 기록](/docs/evaluation/evaluation-methods/scores-via-sdk)할 수 있으므로, 사내 평가기가 플랫폼이 관리하는 평가기와 공존하며 그 결과가 동일한 트레이스 위에 남게 됩니다.

이미 작동 중인 루프 위에 플랫폼이 추가로 제공하는 것은 대체로 조직적인 측면입니다.

- **공유된 가시성.** 점수는 한 사람만 소유한 노트북이 아니라, 엔지니어와 제품, 리뷰어가 모두 들여다보는 곳, 즉 판단 대상 트레이스 바로 옆에 존재합니다.
- **접착 코드 없이 프로덕션으로부터 구축되는 데이터셋.** 실패한 트레이스를 회귀 테스트로 바꾸는 것은 익스포트 스크립트가 아니라 UI 동작입니다.
- **설정으로 유지 관리되는 게이트.** CI 실험 게이트, 샘플링 비율, 임계값은 낡아가는 자체 제작 스크립트 안에 묻혀 있지 않고 선언되고 버전 관리됩니다.
- **직접 만들 필요가 없는 리뷰 도구.** 주석 큐, 리뷰어 배정, 점수 설정이 아무도 유지 관리하고 싶어 하지 않는 내부 라벨링 도구를 대체합니다.
- **감사 추적.** 점수 이력, 버전 관리된 프롬프트와 데이터셋, 그리고 (Enterprise의 경우) 감사 로그가 별도의 추가 작업 없이 존재합니다.

한 가지 짚어둘 구분이 있습니다. 모델이 자신의 출력을 비평하고 재시도하는, 애플리케이션 내부의 자가 수정(self-repair) 루프는 런타임 동작이지 평가 전략이 아닙니다. 이는 개별 답변을 바꿀 뿐, 품질이 상승 추세인지, 재시도 단계가 실제로 도움이 되는지, 혹은 리뷰어에게 무엇을 보여줘야 하는지는 알려주지 않습니다. 자가 수정을 포함해 시스템 전체에 대한 외부 측정은 여전히 필요합니다.

## FAQ

### 평가 전략은 몇 개의 품질 차원으로 시작해야 하나요? [#how-many-dimensions]

실제 트레이스에서 관찰한 실패 모드로부터 각각 도출한 세 개에서 다섯 개로 시작하세요. 모든 차원은 지속적인 비용, 즉 유지 관리해야 할 평가기, 방어해야 할 임계값, 분류해야 할 알림을 동반합니다. 새로운 실패 모드가 프로덕션에 나타났을 때 목록을 확장하고, 미리 확장하지는 마세요.

### 여러 사람이 동일한 트레이스를 리뷰하고 채점할 수 있나요? [#multiple-reviewers]

가능합니다. Langfuse의 주석 큐는 여러 리뷰어를 배정하는 것을 지원하며, 동일한 트레이스를 서로 다른 점수 설정을 가진 별도의 큐로 라우팅할 수 있습니다. 예를 들어 엔지니어링 분류(triage) 큐와, 자체 하위 점수 기준을 가진 컴플라이언스 리뷰 큐로 나눌 수 있습니다. 각 점수는 누가 작성했는지를 기록하므로 병렬 리뷰도 귀속 관계가 유지됩니다.

### 트레이스를 소급해서 채점할 수 있나요? [#retroactive-scoring]

가능하며, 두 가지 방식이 있습니다. 사람은 언제든지 과거의 트레이스, 관측, 세션에 주석을 달 수 있고, 자동화된 평가기는 트레이스 테이블에서 과거 관측을 선택해 해당 선택 항목에 평가를 실행함으로써 점수를 소급 채울(backfill) 수 있습니다(이 기능을 사용하려면 평가기의 Fast Mode 설정이 필요합니다). 즉 새로 정의한 기준을 처음부터 다시 시작하지 않고도 과거 트래픽에 대해 측정할 수 있습니다.

### 자동화된 임계값이 언제 휴먼 리뷰를 대체할 수 있나요? [#automated-thresholds]

캘리브레이션 이후이며, 보통은 완전히가 아니라 부분적으로 대체됩니다. 사람과 LLM 저지가 동일한 항목을 채점하게 하고, 차원별로 일치도를 측정하며, 일치도가 안정될 때까지 루브릭을 다듬으세요. 그런 다음 사람의 리뷰 범위를 저지가 플래그를 붙인 항목, 의견이 갈리는 사례, 검증용 샘플로 줄이세요. 리스크가 큰 흐름에 대해 어느 정도의 잔여 리뷰를 유지할지는 측정 결과가 뒷받침해야 할 정책적 판단입니다.

### 단일 응답이 아니라 멀티턴 대화는 어떻게 평가하나요? [#multi-turn]

Langfuse의 점수는 트레이스와 관측뿐 아니라 세션에도 붙일 수 있으며, 주석 큐도 세션을 받아들이므로 사람 리뷰어가 대화 전체를 채점할 수 있습니다. 자동화된 저지의 경우, 평가기는 대상에 존재하는 데이터를 읽으므로 판단하고자 하는 전체 대화 맥락이 기록된 관측을 대상으로 지정하세요.

### 오프라인 평가와 온라인 평가의 차이는 무엇인가요? [#offline-vs-online]

오프라인 평가는 변경 사항이 배포되기 전에 고정된 데이터셋에 대해 애플리케이션을 실험으로 실행하는 것으로, 베이스라인과 CI 게이트가 존재하는 곳입니다. 온라인 평가는 비용을 통제하기 위해 샘플링된 실시간 프로덕션 트래픽을 채점하는 것으로, 드리프트와 회귀, 예상치 못한 입력이 드러나는 곳입니다. 완전한 전략은 프로덕션 실패로부터 성장하는 데이터셋으로 연결된 이 두 가지를 모두 필요로 합니다.

# 골든 데이터셋 평가: LLM 테스트 세트 구축 및 유지 관리

골든 데이터셋은 의미 있는 변경을 적용하기 전에 LLM 애플리케이션을 대상으로 실행하는 고정된 테스트 케이스 집합입니다. 데이터셋을 구축하는 것 자체는 어렵지 않지만, 프롬프트와 모델, 사용자가 모두 변화하는 상황에서 6개월 동안 대표성을 유지하는 것은 대부분의 팀이 놓치는 부분입니다. 이 가이드에서는 좋은 골든 데이터셋이 어떤 모습인지, 이미 보유한 데이터로부터 데이터셋을 구축하는 방법, 프로덕션이 진화함에 따라 데이터셋을 유지 관리하는 방법, 그리고 이를 사용해 시간에 따라 프롬프트 버전을 비교하는 방법을 다룹니다. 개념 자체는 도구에 종속되지 않지만, 구현에는 데이터셋과 실험, 트레이스가 한곳에 모여 있는 Langfuse를 사용합니다.

**요약:** 골든 데이터셋은 애플리케이션이 처리해야 하는 것을 대표하는, 검토된 참조 출력(또는 명시적인 통과 기준)이 포함된 입력들을 선별한 컬렉션입니다. 좋은 골든 데이터셋은 하나의 컴포넌트나 동작에 범위가 한정되어 있고, 용도에 맞는 크기(빠른 PR 게이트용으로는 수십 개, 완전한 회귀 세트용으로는 수백에서 천 개)를 가지며, 알려진 실패 모드를 다루고, 고정된 상태로 머무르지 않고 프로덕션 실패로부터 성장합니다. 먼저 실제 트레이스로부터 구축하고, 그다음으로 CSV 가져오기와 합성 생성을 활용하세요. 스키마 검증, 중복 제거, 아이템 버전 관리로 유지 관리합니다. 프롬프트 버전을 비교할 때는 동일하게 고정된 데이터셋 버전을 대상으로 버전별로 하나씩 실험을 실행하고, 지정된 기준선(baseline)에 대한 점수 차이를 확인합니다.

## 골든 데이터셋이란

골든 데이터셋(골드 세트 또는 참조 세트라고도 함)은 애플리케이션이 처리해야 하는 입력과, 대개는 올바른 응답이 어떤 모습인지에 대한 검토된 정의를 각 아이템이 기록하는 테스트 케이스 컬렉션입니다. "골든(golden)"이라는 것은 아이템이 선별되고 신뢰할 수 있다는 의미입니다. 즉, 도메인 지식을 가진 누군가가 예상 출력을 확인했거나, 참조 없는 검사(형식, 안전성, 톤)가 통과 기준이 되므로 참조 출력이 필요 없다고 의도적으로 결정한 것입니다.

이러한 신뢰가 전체 가치의 핵심입니다. 골든 데이터셋을 대상으로 한 실험에서 점수가 하락했을 때, 그 결론이 "23번 아이템의 레이블이 잘못됐다"가 아니라 "이 변경이 품질을 저하시켰다"이기를 원할 것입니다. 검토되지 않은 예시 더미는 아무도 조치하지 않는 점수를 만들어내지만, 골든 데이터셋은 릴리스를 차단하거나 승인하는 점수를 만들어냅니다.

팀에게 골든 데이터셋이 필요한 이유는 비공식적인 육안 점검이 반복 작업을 견디지 못하기 때문입니다. 첫 번째 프롬프트 수정은 눈으로 확인할 수 있지만, 열 번째쯤 되면 어떤 이전 동작을 포기했는지 기억할 수 없습니다. 고정되고 신뢰할 수 있는 테스트 세트는 "새 프롬프트가 더 나은 느낌인지"를 측정 가능한 비교로 바꾸며, 이는 이후의 모든 작업, 즉 CI에서의 회귀 테스트, 모델 마이그레이션 결정, 이해관계자를 위한 정직한 전후 비교 수치의 전제 조건이 됩니다.

## 좋은 골든 데이터셋이란

좋은 골든 데이터셋은 프로덕션에서 시스템이 마주하는 상황을 반영하며, 실패가 원인을 가리킬 수 있을 만큼 범위가 좁게 설정되어 있습니다. 세 가지 특성이 가장 중요합니다.

**범위: 컴포넌트 또는 동작당 하나의 데이터셋.** 데이터셋은 엔드투엔드 흐름을 대상으로 할 수도 있고, 검색(retrieval), 라우팅, 요약과 같은 단일 단계를 대상으로 할 수도 있지만, 이를 섞어서는 안 됩니다. 결국 각각 하나의 질문에 답하는 여러 데이터셋을 갖게 될 것입니다. Langfuse는 이를 체계적으로 정리할 수 있도록 폴더 형태의 이름(`support-agent/golden`, `support-agent/adversarial`)을 지원합니다.

**크기: 최대화가 아니라 용도에 맞게.** 아이템이 많을수록 커버리지는 넓어지지만 실행은 느려지고 비용도 늘어납니다. [Langfuse Academy 데이터셋 모듈](/academy/datasets)은 다음과 같은 실용적인 크기를 제안합니다.

| 목적                       | 일반적인 크기                                              |
| -------------------------- | ---------------------------------------------------------- |
| 단일 이슈 탐색             | 약 10개 아이템                                             |
| 모델 역량 한계 테스트      | 아직 해결되지 않은 복잡한 예시 약 10개                     |
| 대규모 변경에 대한 CI 검사 | 프로덕션 분포를 커버하는 100~1000개 아이템                 |
| 가드레일 침투 테스트       | 크고 계속 증가하며, 프로덕션에서 발견될 때마다 케이스 추가 |

풀 리퀘스트마다 실행되는 게이트의 경우, 검사 속도를 유지하기 위해 수십에서 수백 개 정도의 하위 집합을 유지하고, 전체 세트는 릴리스 브랜치나 야간 실행을 위해 남겨두세요.

**해피 패스뿐 아니라 실패 모드도 커버.** 프로덕션에서 관찰된 모든 실패 모드는 최소 몇 개의 아이템으로 다뤄져야 합니다. 모호한 질문, 여러 부분으로 구성된 요청, 잘못된 언어로 작성된 입력, 프롬프트 인젝션 시도 등입니다. 쉬운 케이스로만 이루어진 데이터셋은 언제나 95%의 점수를 기록하지만 아무것도 알려주지 않습니다. 유용한 경험칙은 다음과 같습니다. 특정 유형의 입력이 인시던트나 지원 티켓을 유발한 적이 있다면, 그 입력은 데이터셋에 포함되어야 합니다.

신선도(freshness)는 조용히 감소하는 특성입니다. 1월에 구성된 골든 데이터셋은 1월의 트래픽을 반영합니다. 7월이 되면 사용자는 다른 질문을 하고, 제품에는 새로운 기능이 생깁니다. 데이터셋을 프로덕션을 추적하는 추가 위주(append-mostly)의 로그로 다루고(아래 유지 관리 섹션에서 방법을 다룹니다), 아이템에 날짜를 메타데이터로 남겨 데이터셋이 얼마나 오래됐는지 한눈에 볼 수 있게 하세요.

## Langfuse에서 골든 데이터셋 구축하기

이 섹션의 모든 내용은 이미 보유한 데이터를 대상으로 한 UI 작업 또는 SDK 호출입니다. 배포하거나 동기화 상태를 유지해야 할 별도의 평가 서비스는 없습니다. 데이터셋은 트레이스와 동일한 Langfuse 프로젝트에 존재하고, SDK 호출은 기존 코드베이스 내에서 실행되며, Langfuse를 셀프 호스팅하는 경우 아무 데이터도 인프라 밖으로 나가지 않습니다. [데이터셋 문서](/docs/evaluation/experiments/datasets)가 전체 기능 레퍼런스이며, 이 섹션은 워크플로우를 다룹니다.

### 프로덕션 트레이스로부터 시작하기

실제 트래픽이 만들어낸 예시가 지어낸 예시보다 낫기 때문에 여기서부터 시작하세요. Langfuse UI에서는 모든 트레이스와 관측(observation)에 "+ Add to dataset" 액션이 있으며, 이는 실패를 하나씩 검토할 때 사용하기 적합한 도구입니다. 대량으로 수집하려면 관측 테이블을 (점수, 사용자 피드백, 태그, 또는 다른 임의의 열로) 필터링하고, 행을 선택한 뒤 **Actions** → **Add to dataset**을 사용하세요. 필드 매핑 단계에서 관측 데이터가 데이터셋 아이템으로 변환되는 방식을 제어할 수 있습니다. 전체 필드를 그대로 사용하거나, JSON path 표현식으로 값을 추출하거나, 여러 필드로부터 커스텀 객체를 구성할 수 있습니다. 대량 추가는 백그라운드에서 부분 성공(partial success) 방식으로 실행되므로, 일부 잘못된 형식의 행이 나머지를 막지 않습니다.

프로그래밍 방식으로는 동일한 작업이 SDK 호출 한 번으로 처리되며, 소스 트레이스를 연결해두면 나중에 아이템을 검토하는 사람이 전체 실행 컨텍스트를 확인할 수 있습니다.

```python
from langfuse import get_client

langfuse = get_client()

langfuse.create_dataset_item(
    dataset_name="support-agent/golden",
    input={"question": "How do I cancel my subscription?"},
    expected_output={"answer": "Go to Settings > Billing > Cancel plan. The plan stays active until the end of the billing period."},
    source_trace_id="<trace_id>",
    source_observation_id="<observation_id>",  # 선택 사항: 특정 span 또는 generation을 연결
)
```

예상 출력(expected output)은 대개 모델의 실제 프로덕션 출력에서 시작해 도메인 전문가가 수정한 형태입니다. 이 검토 단계가 아이템을 단순히 로그된 것이 아니라 골든으로 만드는 요소입니다.

### CSV로 기존 테스트 케이스 가져오기

테스트 케이스가 이미 스프레드시트에 존재한다면, 데이터셋 UI에서 CSV를 업로드하세요. 헤더 행이 필요하며, 이후 각 열을 `input`, `expected output`, `metadata`로 드래그하면 됩니다. 동일한 필드에 여러 열을 드롭하면 하나의 구조화된 JSON 객체로 결합됩니다. 매핑되지 않은 열은 무시됩니다. CSV 가져오기는 텍스트와 구조화된 JSON 아이템용이며, 미디어 첨부 파일이 있는 아이템에는 아이템 편집기나 SDK를 사용하세요.

2026년 6월부터 데이터셋 아이템은 `input`, `expected_output`, `metadata`에 미디어 첨부 파일(이미지, 오디오, 문서)을 포함할 수 있게 되어, 비전 및 문서 워크플로우도 골든 데이터셋을 활용할 수 있습니다. SDK 기반 실험은 Python SDK 4.10.0 이상 및 JS/TS `@langfuse/client` 5.6.0 이상에서 이러한 아이템을 지원하며, UI 기반 실험은 아직 지원하지 않습니다.

### 실제 데이터가 부족한 부분을 합성 데이터로 보강하기

합성 데이터 생성은 아직 프로덕션에서 발생하지 않은 커버리지 공백을 채워줍니다. 알려진 어려운 입력의 패러프레이즈, 막 출시된 기능의 엣지 케이스, 동일한 의도를 가진 적대적(adversarial) 표현 등입니다. LLM으로 변형을 생성한 뒤, 실제 아이템과 동일한 엄격함으로 검토하세요. 검토되지 않은 합성 아이템은 데이터셋을 골든으로 만드는 신뢰를 희석시킵니다. [합성 데이터셋 쿡북](/guides/cookbook/example_synthetic_datasets)에는 바로 실행 가능한 파이프라인이 있습니다. 비율을 적절히 유지하세요. 합성 아이템은 프로덕션에 기반한 핵심을 확장하는 용도이지, 이를 대체하는 용도가 아닙니다.

### 데이터셋 스키마로 구조를 강제하기

여러 사람과 파이프라인이 하나의 데이터셋에 쓰기 시작하면, 형식이 잘못된 아이템이 슬며시 유입되어 최악의 시점에 실험 실행을 망가뜨립니다. Langfuse 데이터셋은 `input`과 `expected_output`에 대해 선택적으로 JSON 스키마를 받을 수 있으며, 설정되면 생성, 수정, CSV 가져오기 시마다 모든 아이템이 검증되고, 유효하지 않은 아이템은 정확한 검증 오류와 함께 거부됩니다.

```python
langfuse.create_dataset(
    name="support-agent/golden",
    input_schema={
        "type": "object",
        "properties": {"question": {"type": "string"}},
        "required": ["question"],
    },
    expected_output_schema={
        "type": "object",
        "properties": {"answer": {"type": "string"}},
        "required": ["answer"],
    },
)
```

스키마는 첫날부터 정의하세요. 이미 천 개의 일관되지 않은 아이템이 쌓인 뒤에 스키마를 적용하는 것은 별도의 정리 프로젝트가 되지만, 처음부터 강제하는 것은 아무런 추가 비용이 들지 않습니다.

## 골든 데이터셋 유지 관리하기

골든 데이터셋은 테스트 스위트와 동일한 방식으로 유지 관리됩니다. 발견을 피해간 모든 버그는 테스트가 되고, 중복은 병합되며, 오래된 케이스는 조용히가 아니라 의도적으로 폐기됩니다.

### 발생하는 프로덕션 실패를 아이템으로 전환하기

가장 가치 있는 새 아이템은 애플리케이션이 방금 실패한 케이스입니다. 실패가 아직 생생할 때 플래그가 지정된 프로덕션 트레이스가 데이터셋으로 흘러 들어가도록 파이프라인을 구성하세요. 주석 큐(annotation queue)를 검토하는 검토자는 트레이스를 데이터셋에 직접 추가할 수 있고, 예약된 스크립트는 부정적인 신호를 자동으로 스캔할 수 있습니다. 아래 스크립트는 어제 발생한 부정적(thumbs-down) 피드백을 골든 데이터셋으로 수집합니다.

```python
from datetime import datetime, timedelta, timezone
from langfuse import get_client

langfuse = get_client()

scores = langfuse.api.scores.get_many(
    name="user-feedback",
    from_timestamp=datetime.now(timezone.utc) - timedelta(days=1),
    limit=100,
)

for score in scores.data:
    if score.value == 0 and score.trace_id:  # thumbs-down
        trace = langfuse.api.trace.get(score.trace_id)
        langfuse.create_dataset_item(
            dataset_name="support-agent/golden",
            input=trace.input,
            expected_output=None,  # 검토자가 수정된 답변을 제공
            metadata={"source": "production-failure", "added": str(datetime.now(timezone.utc).date())},
            source_trace_id=score.trace_id,
        )
```

이런 식으로 유입된 아이템은 아직 골든이 아니라 후보입니다. 실험에 반영되기 전에 전문가 검토를 거쳐 수정된 예상 출력을 채워 넣어야 합니다. 더 넓은 범위의 [LLM 평가 전략](/resources/engineering/llm-evaluation-strategy)은 이와 동일한 루프를 기반으로 프로덕션 모니터링을 오프라인 테스트로 다시 연결합니다.

### 아이템이 쌓이기 전에 중복 제거하기

동일한 질문의 거의 동일한 표현이 스무 개 있으면 모든 평균 점수에서 해당 케이스에 과도한 가중치가 실리고 모든 실행 속도가 느려집니다. 유입 시점에 중복을 제거하세요. 아이템을 추가하기 전에 데이터셋이 이미 동일한 의도를 다루고 있는지 확인합니다. `langfuse.api.dataset_items.list`로 가져온 정규화된 입력을 정확히 일치시키거나, 패러프레이즈의 경우 임베딩 유사도를 사용합니다. 거의 동일한 아이템의 클러스터를 발견하면 가장 어려운 한두 개의 변형만 유지하고 나머지는 삭제하세요. 데이터셋 스키마는 일관된 구조 덕분에 중복을 더 쉽게 탐지할 수 있게 해주므로 여기서도 도움이 됩니다.

### 데이터셋 버전을 관리하고, 고정된 버전을 대상으로 실험 실행하기

Langfuse 데이터셋은 자동으로 버전이 관리됩니다. 아이템의 모든 추가, 수정, 삭제, 보관(archival)은 타임스탬프로 식별되는 새 버전을 생성하며, 아이템 수준의 diff도 함께 기록됩니다. 이는 살아있는 골든 데이터셋의 핵심적인 관리 문제를 해결합니다. 즉, 서로 다른 날짜의 점수는 각 실행이 어떤 데이터셋 상태를 대상으로 했는지 알 때만 비교 가능하다는 문제입니다. 실험은 실행 대상이 된 데이터셋 상태를 기록하며, 특정 시점의 데이터셋을 가져와 과거 비교를 재현하거나 확장할 수 있습니다.

```python
from datetime import datetime, timezone
from langfuse import get_client

langfuse = get_client()

# 기준선(baseline) 실험이 실행됐을 당시 그대로의 데이터셋
baseline_version = datetime(2026, 7, 1, 9, 0, 0, tzinfo=timezone.utc)
dataset = langfuse.get_dataset("support-agent/golden", version=baseline_version)
```

버전 관리는 데이터셋 아이템에만 적용되며, 데이터셋의 스키마를 변경하는 것은 새 버전을 생성하지 않는다는 점에 유의하세요.

### 아이템을 의도적으로 폐기하기

아이템이 테스트하는 동작이 더 이상 존재하지 않을 때(제거된 기능, 폐기된 흐름) 또는 예상 출력이 잘못됐을 때(정책 변경으로 참조 답변이 부정확해진 경우) 해당 아이템을 폐기하세요. 삭제하지 말고 보관(archive)하세요. 보관된 아이템은 이후 실험 실행에서는 제외되지만 버전 히스토리에는 남아 있어, 과거 실행 결과를 계속 해석할 수 있습니다.

```python
langfuse.create_dataset_item(
    dataset_name="support-agent/golden",
    id="<item_id>",
    status="ARCHIVED",
)
```

단순히 애플리케이션이 계속 통과시킨다는 이유로 아이템을 폐기하지 마세요. 통과하는 아이템은 그 동작이 여전히 유지되고 있음을 증명한다는 제 역할을 하고 있는 것입니다. 합리적인 주기는 6개월이 지난 아이템에 대해 분기별로 검토를 진행하며, 각각을 현재 제품 동작과 대조해 확인하는 것입니다.

## 시간에 따라 프롬프트 버전 비교하기

신뢰할 수 있는 데이터셋이 갖춰지면 프롬프트 버전 비교는 기계적인 작업이 됩니다. 동일한 데이터셋 버전을 대상으로 프롬프트 버전별로 하나씩 [실험](/docs/evaluation/experiments/experiments-via-sdk)을 실행하고, 각 실행을 동일한 평가기(evaluator)로 채점한 뒤, 차이를 확인하면 됩니다.

### 프롬프트 버전별로 실험 하나씩 실행하기

프롬프트 관리에서 각 프롬프트 버전을 가져와 SDK의 실험 러너를 통해 실행하면, 동시성, 트레이싱, 점수 연결이 자동으로 처리됩니다. 실행 이름과 메타데이터에 프롬프트 버전을 기록해두면 몇 달 후에도 UI에서 실행을 쉽게 식별할 수 있습니다.

```python
from langfuse import get_client, Evaluation
from langfuse.openai import OpenAI

langfuse = get_client()
dataset = langfuse.get_dataset("support-agent/golden")

def correctness(*, output, expected_output, **kwargs):
    match = expected_output and expected_output["answer"].lower() in output.lower()
    return Evaluation(name="correctness", value=1.0 if match else 0.0)

for version in [7, 8]:
    prompt = langfuse.get_prompt("support-agent", version=version)

    def task(*, item, prompt=prompt, **kwargs):
        compiled = prompt.compile(question=item.input["question"])
        response = OpenAI().chat.completions.create(
            model="gpt-4.1",
            messages=[{"role": "user", "content": compiled}],
        )
        return response.choices[0].message.content

    result = dataset.run_experiment(
        name=f"support-agent prompt v{version}",
        description=f"Prompt version {version} on the golden dataset",
        task=task,
        evaluators=[correctness],
        metadata={"prompt_version": version},
    )
    print(result.format())
```

실험은 기본적으로 최신 데이터셋 버전을 대상으로 실행되므로, 비교가 데이터셋이 성장하는 며칠 또는 몇 주에 걸쳐 이루어진다면 (이전 섹션에서 다룬 대로) 버전을 고정하세요. 그렇지 않으면 두 실행이 서로 다른 아이템 집합을 대상으로 테스트되는 것입니다. 테스트 대상 변경이 프롬프트뿐이라면 코드를 완전히 생략할 수도 있습니다. [UI를 통한 실험](/docs/evaluation/experiments/experiments-via-ui)은 데이터셋 페이지에서 바로, 구성된 모델과 선택적 평가기를 사용해 선택한 프롬프트 버전을 데이터셋에 대해 실행합니다.

### 기준선(baseline) 대비 차이 확인하기

실험 테이블에서 두 실행을 선택하고 **Compare**를 클릭한 다음, 하나를(일반적으로 현재 프로덕션 버전을) 기준선으로 지정하세요. 비교 화면은 데이터셋 아이템별로 행을 매칭하므로, 각 행은 동일한 입력에 대한 기준선과 후보의 출력을 보여주며, 점수와 비용, 지연 시간에 대해 초록색과 빨간색으로 차이를 표시합니다. Charts 탭은 집계된 전체 그림을 요약해, "품질이 변화했는가, 그리고 어떤 비용으로?"라는 첫 번째 질문에 답합니다.

아이템 수준 테이블은 더 중요한 두 번째 질문, 즉 정확히 무엇이 나빠졌는지에 답합니다. 열을 필터링해 회귀 작업 목록을 구성하세요. 예를 들어 후보의 정확도(correctness)가 하락했거나 비용이 10% 이상 상승한 모든 아이템을 찾은 뒤, 각 행의 트레이스를 열어 원인을 확인합니다. 두 가지 검토 원칙이 차이를 신뢰할 수 있게 유지해줍니다. 하나는 집계 지표를 신뢰하기 전에 몇 개의 무작위 표본 행에서 평가기의 판정이 실제 출력 품질과 일치하는지 확인하는 것이고, 다른 하나는 큰 아이템 수준의 등락이 서로 상쇄되어 작은 집계 차이로 나타나는 경우를 무승부가 아니라 살펴볼 가치가 있는 동작 변화로 취급하는 것입니다.

더 긴 시간 범위에서는 데이터셋의 실험 테이블이 타임라인 역할을 합니다. 모든 실행이 프롬프트 버전 메타데이터와 점수, 데이터셋 버전을 기록하므로, "5번부터 12번 프롬프트 버전에 걸쳐 품질이 어떻게 변화했는가"라는 질문에는 테이블 하나를 훑어보는 것으로 답할 수 있습니다.

### CI에서 비교 자동화하기

수동 루프가 잘 작동한다면, 이를 CI로 옮겨 프롬프트를 변경하는 모든 풀 리퀘스트에서 골든 데이터셋이 자동으로 실행되고 회귀가 발생하면 실패하도록 만드세요. Langfuse는 GitHub Action(`langfuse/experiment-action`)을 제공하며, 이는 데이터셋을 로드하고(선택적으로 특정 버전에 고정), 실험 스크립트를 실행하고, PR 댓글로 점수를 게시하며, 회귀가 감지되면 작업을 실패시킵니다. 임계값 설정과 GitHub 외 CI 시스템을 포함한 전체 설정은 [LLM 회귀 테스트 가이드](/resources/engineering/llm-regression-testing)에서 다룹니다.

## 자주 묻는 질문

### 골든 데이터셋은 얼마나 커야 하나요? [#dataset-size]

실패 모드를 커버할 만큼 충분히 크면서, 실제로 실행할 수 있을 만큼 작아야 합니다. 가장 중요한 동작을 다루는 검토된 아이템 20~50개로 시작하면 이미 명백한 회귀를 잡아낼 수 있습니다. 프로덕션 분포를 반영하는 완전한 회귀 세트를 위해서는 100~1000개로 늘려가고, PR별 CI 게이트를 위해서는 수십에서 수백 개의 하위 집합을 유지하세요. 서로 다른 실패 모드를 커버하는 것이 단순한 개수보다 중요합니다. 다양한 100개의 아이템이 거의 동일한 1000개의 아이템보다 낫습니다.

### 모든 아이템에 정답(ground-truth) 예상 출력이 필요한가요? [#ground-truth]

아닙니다. 예상 출력은 참조 기반 평가기(정확 일치, 유사도, 사실 커버리지)를 지원하기 위해 존재하며, 문자 그대로의 답변, 골드 참조 응답, 또는 출력이 만족해야 할 기준 목록의 형태를 취할 수 있습니다. 형식 유효성, 안전성, 톤과 같은 참조 없는 평가기로만 검사되는 아이템은 예상 출력이 전혀 필요하지 않습니다. 많은 골든 데이터셋이 두 종류를 혼합해서 사용하며, 하나의 아이템의 예상 출력 필드가 JSON 형태로 여러 참조 유형을 담을 수도 있습니다.

### 데이터셋 드리프트를 어떻게 감지하고 처리하나요? [#dataset-drift]

데이터셋 드리프트란 골든 데이터셋이 더 이상 프로덕션 트래픽을 닮지 않게 되어, 이를 통과하는 것이 프로덕션 품질을 더 이상 예측하지 못하게 되는 것을 의미합니다. 가장 뚜렷한 증상은 강한 실험 점수와, 사용자 피드백이나 온라인 평가기 점수 같은 약화되는 프로덕션 신호 사이의 괴리입니다. 구조적으로 이를 방지하려면 최근 트레이스 표본과 새로운 실패를 지속적으로 추가하고, 아이템 메타데이터에 `added` 날짜를 기록해 데이터셋의 연령 분포를 측정할 수 있게 하며, 더 이상 존재하지 않는 동작에 대한 아이템은 폐기하세요. 데이터셋 버전 관리는 데이터셋이 진화하는 동안에도 이전 비교의 유효성을 유지해줍니다.

### 프로젝트 간에 골든 데이터셋을 공유할 수 있나요? [#share-across-projects]

Langfuse의 데이터셋은 단일 프로젝트로 범위가 한정되어 있으며, 현재(2026년 7월 기준)로서는 프로젝트 간 공유 기능이 내장되어 있지 않습니다. 다른 곳에서 데이터셋을 재사용하려면 UI에서 아이템을 CSV나 JSON으로 내보내거나, SDK를 통해 프로그래밍 방식으로 복사해 대상 프로젝트에서 `create_dataset_item`으로 아이템을 다시 생성하세요. [데이터 마이그레이션 쿡북](/guides/cookbook/example_data_migration)에는 프로젝트 간에 데이터셋, 아이템, 실행을 아이템 ID를 유지한 채로 복사하는 전체 스크립트가 포함되어 있습니다.

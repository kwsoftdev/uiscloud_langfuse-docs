# Scores

점수는 평가 결과를 저장하는 Langfuse의 범용 데이터 객체입니다. [사람이 직접 주석을 작성하거나](/docs/evaluation/evaluation-methods/scores-via-ui), [LLM 심사자](/docs/evaluation/evaluation-methods/llm-as-a-judge)를 사용하거나, [프로그래밍 방식의 검사](/docs/evaluation/evaluation-methods/scores-via-sdk)를 수행하거나, 최종 사용자 피드백을 받는 등 LLM 출력물에 품질 판단을 부여하고자 할 때마다 그 결과는 점수로 저장됩니다.

모든 점수는 **이름**(예: `"correctness"` 또는 `"helpfulness"`), **값**, 그리고 **[데이터 유형](#score-types)**을 가집니다. 점수는 추가 맥락을 위한 선택적 **[코멘트](#score-comments)**도 지원합니다.

점수는 [트레이스](/docs/observability/data-model#traces), [관측치](/docs/observability/data-model#observations), [세션](/docs/observability/data-model#sessions), 또는 [데이터셋 실행](/docs/evaluation/experiments/data-model)에 연결할 수 있습니다. 가장 일반적으로는 하나의 엔드투엔드 상호작용을 평가하기 위해 트레이스에 점수를 연결합니다.

점수를 생성하고 나면 [점수 분석](/docs/evaluation/scores/score-analytics)에 표시되고, [커스텀 대시보드](/docs/metrics/features/custom-dashboards)에서 시각화할 수 있으며, [API](/docs/api)를 통해 조회할 수 있습니다.

## 점수를 언제 사용할까요 [#when-to-use-scores]

애플리케이션이 무엇을 하는지 관찰하는 수준을 넘어, 그것을 얼마나 잘 수행하는지 측정하고자 할 때 점수가 유용해집니다. 일반적인 사용 사례는 다음과 같습니다.

- **사용자 피드백 수집**: 사용자로부터 좋아요/싫어요 또는 별점을 받아 트레이스에 연결합니다. [사용자 피드백 가이드](/docs/observability/features/user-feedback)를 참고하세요.
- **프로덕션 품질 모니터링**: [LLM-as-a-Judge](/docs/evaluation/evaluation-methods/llm-as-a-judge)와 같은 자동화된 평가기를 설정하여 환각, 관련성, 어조 등에 대해 실시간 트레이스를 지속적으로 채점합니다.
- **가드레일 실행**: PII 탐지, 형식 검증, 콘텐츠 정책 준수와 같은 안전성 검사를 출력물이 통과하는지 채점합니다.
- **실험을 통한 변경 사항 비교**: 프롬프트, 모델, 또는 파이프라인을 변경할 때 [실험](/docs/evaluation/experiments)을 실행하여 새 버전을 데이터셋에 대해 채점합니다.

## 점수 유형 [#score-types]

Langfuse는 네 가지 점수 데이터 유형을 지원합니다.

| 유형          | 값                                                                     | 사용 시점                                                                                                                                                                                                  |
| ------------- | ---------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `NUMERIC`     | 실수 (예: `0.9`)                                                       | 정확도, 관련성, 유사도 점수와 같은 연속적인 판단                                                                                                                                                           |
| `CATEGORICAL` | 미리 정의된 카테고리의 문자열 (예: `"correct"`, `"partially correct"`) | 가능한 값의 집합을 미리 알고 있는 이산적 분류                                                                                                                                                              |
| `BOOLEAN`     | `0` 또는 `1`                                                           | 환각 탐지나 형식 검증과 같은 통과/실패 검사                                                                                                                                                                |
| `TEXT`        | 자유 형식 문자열 (1~500자)                                             | 검토자 노트나 정성적 피드백 같은 개방형 주석. [정량화 가능한 점수](https://en.wikipedia.org/wiki/Axial_coding)로 공식화하기 전에 [개방 코딩](https://en.wikipedia.org/wiki/Open_coding)에 흔히 사용됩니다. |

<Callout type="info">
텍스트 점수는 정성적이고 개방형인 채점을 위해 설계되었습니다. 자유 형식 텍스트는 의미 있게 집계하거나 비교할 수 없기 때문에, 텍스트 점수는 [실험](/docs/evaluation/core-concepts#experiments), [LLM-as-a-Judge](/docs/evaluation/evaluation-methods/llm-as-a-judge), [점수 분석](/docs/evaluation/scores/score-analytics)에서 지원되지 않습니다.
</Callout>

## 점수 생성 방법 [#how-to-create-scores]

점수를 추가하는 방법은 다섯 가지입니다.

- **LLM-as-a-Judge**: 커스텀 기준(예: 환각, 어조, 관련성)에 따라 트레이스를 채점하는 [자동화된 평가기](/docs/evaluation/evaluation-methods/llm-as-a-judge)를 설정합니다. 숫자형 또는 범주형 점수와 함께 근거를 반환할 수 있으며, 실시간 프로덕션 트레이스나 실험 결과에서 실행할 수 있습니다.
- **코드 평가기**: 정확 일치, JSON 검증, 커스텀 비즈니스 규칙과 같은 결정론적 검사를 위해 Langfuse에서 [커스텀 Python 또는 TypeScript 평가기](/docs/evaluation/evaluation-methods/code-evaluators)를 실행합니다.
- **UI를 통한 점수 부여**: 팀원이 Langfuse UI에서 직접 트레이스, 관측치, 세션을 [수동으로 채점](/docs/evaluation/evaluation-methods/scores-via-ui)합니다. 먼저 [점수 설정](/faq/all/manage-score-configs)이 필요합니다.
- **주석 큐**: 검토자가 여러 트레이스를 일괄 처리하는 [구조화된 검토 워크플로](/docs/evaluation/evaluation-methods/annotation-queues)를 설정합니다.
- **API/SDK를 통한 점수 부여**: 애플리케이션 코드에서 [프로그래밍 방식으로 점수를 추가](/docs/evaluation/evaluation-methods/scores-via-sdk)합니다. 사용자 피드백(좋아요/싫어요, 별점), 가드레일 결과, 또는 커스텀 평가 파이프라인에 적합한 방식입니다.

## 점수와 태그 중 무엇을 사용해야 할까요 [#scores-vs-tags]

|               | 점수                                                   | 태그                                  |
| ------------- | ------------------------------------------------------ | ------------------------------------- |
| **목적**      | 무언가가 _얼마나 좋은지_ 측정                          | 무언가가 _무엇인지_ 설명              |
| **데이터**    | 숫자, 범주형, 불리언, 또는 텍스트 값                   | 단순 문자열 레이블                    |
| **추가 시점** | 트레이스 생성 후 오랜 시간이 지나도 언제든지 추가 가능 | 트레이싱 중에 설정되며 이후 변경 불가 |
| **용도**      | 품질 측정, 분석, 실험                                  | 필터링, 세분화, 정리                  |

경험칙: 트레이싱 시점에 이미 카테고리를 알고 있다면(예: 어떤 기능이나 API 엔드포인트가 트레이스를 트리거했는지), [태그](/docs/observability/features/tags)를 사용하세요. 나중에 트레이스를 분류하거나 평가해야 한다면 점수를 사용하세요.

## 점수 코멘트 [#score-comments]

모든 점수는 선택적 **코멘트** 필드를 지원합니다. 이를 사용하여 근거(예: LLM 심사자가 특정 점수를 부여한 이유), 검토자 노트, 또는 다른 사람이 점수 값을 이해하는 데 도움이 되는 맥락을 기록하세요. 코멘트는 Langfuse UI에서 점수와 함께 표시됩니다.

독립적인 정성적 피드백을 기록하려면 코멘트 대신 [`TEXT` 점수](#score-types)를 사용하세요 -- 코멘트는 기존 점수에 대한 추가 근거를 남기는 데 가장 적합합니다.

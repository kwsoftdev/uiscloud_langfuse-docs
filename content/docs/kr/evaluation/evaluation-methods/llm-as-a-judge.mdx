# LLM-as-a-Judge

LLM-as-a-Judge는 LLM을 사용하여 다른 LLM 애플리케이션이 생성한 출력의 품질을 평가하는 평가 방법론입니다. 사람 검토자나 단순한 휴리스틱 메트릭에만 의존하는 대신, 성능이 뛰어난 모델("judge")에 프롬프트를 제공해 정의된 기준에 따라 애플리케이션 출력을 채점하고 그 근거를 추론하도록 합니다.

이 접근 방식은 사람의 판단이 지닌 미묘한 뉘앙스와 자동화된 평가의 확장성을 결합할 수 있어, LLM 애플리케이션을 평가하는 가장 널리 사용되는 방법 중 하나로 자리 잡았습니다.

## LLM-as-a-Judge 작동 방식

핵심 아이디어는 간단합니다. LLM에 입력, 애플리케이션의 출력, 채점 루브릭을 제시하고 출력을 평가하도록 요청하는 것입니다. judge 모델은 평가 근거를 설명하는 추론과 함께 [`score`](/docs/evaluation/scores/overview)를 생성합니다.

일반적인 LLM-as-a-Judge 프롬프트는 다음을 포함합니다:

1. **평가 기준** — "좋음"이 무엇인지 정의하는 루브릭(예: "답변이 사실과 다르면 1점, 완전히 정확하고 출처가 명확하면 5점")
2. **입력 컨텍스트** — 원본 사용자 쿼리 또는 프롬프트
3. **평가 대상 출력** — 애플리케이션의 응답
4. **선택적 참조 자료** — 비교를 위한 정답(ground truth) 또는 예상 출력

이렇게 생성된 judge 모델은 시간에 따라 추적, 집계, 분석할 수 있는 구조화된 점수와 추론 근거를 반환합니다. Langfuse에서 이 점수는 숫자형, 범주형, 불리언 중 하나일 수 있습니다. 도움이 되는 정도처럼 연속적인 판단에는 `0`에서 `1` 사이의 숫자형 점수를 사용합니다. `correct`, `partially_correct`, `incorrect`처럼 명시적인 레이블이 필요할 때는 범주형 점수를 사용합니다. 사용자가 어시스턴트에 동의하지 않는지, 요청이 범위를 벗어났는지, 답변이 정책을 위반했는지와 같이 결과가 `true` 또는 `false`인 이진 판단에는 불리언 점수를 사용합니다. 프로덕션 모니터링과 관련된 더 많은 예시는 [프로덕션 모니터링을 위한 LLM-as-a-Judge](/blog/2026-04-01-llm-as-a-judge-production-monitoring)를 참고하세요.

## LLM-as-a-Judge를 사용하는 이유

- **확장 가능:** 사람 주석자보다 훨씬 빠르게 수천 개의 출력을 채점할 수 있습니다.
- **사람과 유사한 판단:** 특히 루브릭을 기반으로 할 때, 단순한 메트릭보다 도움이 되는 정도, 독성, 관련성 같은 미묘한 뉘앙스를 더 잘 포착합니다.
- **재현 가능:** 고정된 루브릭을 사용하면 동일한 프롬프트를 재실행해 일관된 점수를 얻을 수 있습니다.

## LLM-as-a-Judge 사용 방법

LLM-as-a-Judge 평가기는 세 가지 유형의 데이터에 대해 실행할 수 있습니다: **관측치(Observations)**(개별 작업), **트레이스(Traces)**(전체 워크플로), **실험(Experiments)**(통제된 테스트 데이터셋). 어떤 것을 선택할지는 개발 환경에서 테스트하는지 아니면 프로덕션을 모니터링하는지, 그리고 어느 수준의 세분성이 필요한지에 따라 달라집니다.

#### 관측치 수준 평가기에서 사용 가능한 컨텍스트 [#observation-evaluator-context]

관측치 수준 평가기는 매칭된 관측치의 변수를 매핑합니다. 해당 관측치의 `input`, `output`, `metadata`와 같은 필드를 사용할 수 있습니다.

동일한 트레이스에 속한 형제 또는 자식 관측치는 로드하지 않습니다. 평가기가 애플리케이션이나 에이전트 호출 전체의 요청 및 응답을 필요로 한다면, 그 전체 입력과 출력을 기록하는 루트 관측치를 대상으로 지정하세요. 이 경우에도 평가기는 해당 루트 관측치의 데이터만 확인하며, 애플리케이션이 필요한 요약이나 컨텍스트를 루트 관측치에 직접 기록하지 않는 한 자식 관측치의 데이터를 자동으로 포함하지 않습니다.

**관측치를 대상으로 지정해야 하는 이유**

- **훨씬 빠른 실행 속도**: 평가가 몇 분이 아니라 몇 초 만에 완료됩니다. 평가 지연과 백로그를 없애줍니다. 비동기 아키텍처가 분당 수천 건의 평가를 처리합니다.
- **작업 단위의 정밀도**: 관측치 유형별로 필터링하여 전체 워크플로가 아닌 최종 LLM 응답이나 검색 단계만 평가할 수 있습니다. 특정 작업만 대상으로 지정해 평가량과 비용을 줄여줍니다.
- **조합형 평가**: 하나의 트레이스 내 서로 다른 작업에 서로 다른 평가기를 실행할 수 있습니다. LLM 출력에는 독성 평가를, 검색에는 관련성 평가를, 생성 결과에는 정확도 평가를 동시에 적용할 수 있습니다.
- **결합 필터링**: 관측치 필터(유형, 이름, 메타데이터)를 트레이스 필터(userId, sessionId, 태그, 버전)와 함께 조합할 수 있습니다. 예: "'customer-support' 태그가 붙은 프리미엄 사용자 대화의 모든 LLM 생성".

**데이터 흐름**

수집 시점에 각 관측치는 설정한 필터 기준에 따라 평가됩니다. 조건에 맞는 관측치는 평가 큐에 추가됩니다. 이후 평가 작업이 비동기적으로 처리됩니다. 점수는 해당 관측치에 첨부되며, 평가기당 관측치 하나에 점수 하나가 생성됩니다. 필터 기준에 따라 하나의 트레이스에서 여러 관측치가 매칭되어 트레이스당 여러 점수가 생성될 수 있습니다.

**사용 사례 예시**

- 사용자에게 전달되는 최종 챗봇 응답의 유용성만 평가
- 고객 대상 LLM 생성 전체의 독성 점수 모니터링
- 문서 검색 관측치를 대상으로 RAG 시스템의 검색 관련성 추적

</Callout>

<Callout type="info">

</Callout>

**트레이스를 대상으로 지정해야 하는 경우**

- 평가에 여러 작업에 걸친 전체 컨텍스트가 필요한 경우
- 레거시 SDK 버전(Python v2 또는 JS/TS v3)을 사용 중이며 업그레이드할 수 없는 경우

**데이터 흐름**

수집 시점에 각 트레이스는 설정한 필터 기준에 따라 평가됩니다. 조건에 맞는 트레이스는 평가 큐에 추가되어 비동기적으로 처리됩니다. 점수는 트레이스 자체에 첨부되며, 평가기당 트레이스 하나에 점수 하나가 생성됩니다.

**사용 사례 예시**

- 평가기가 여러 작업(예: 검색 → 재순위화 → 생성 → 인용)에 걸친 전체 컨텍스트를 필요로 하는 경우에 한해, 다단계 에이전트 워크플로의 정확도를 채점

**실험을 대상으로 지정해야 하는 이유**

- 의사 결정을 위한 재현 가능한 벤치마크가 필요한 경우
- 여러 프롬프트 버전이나 모델 구성을 비교하는 경우
- 예상 출력(정답)이 포함된 데이터셋이 있는 경우

**데이터 흐름**

각 실험 실행은 선택한 평가기에 의해 자동으로 채점되는 트레이스를 생성합니다. 각 실험 항목을 하나의 테스트 케이스로 생각하세요: 입력 → 실행 → 출력 → 평가.

1. 테스트 입력과 (선택적으로) 예상 출력이 포함된 데이터셋을 생성합니다. 테스트 데이터를 로컬에서 정의할 수도 있습니다.
2. UI 또는 SDK를 통해 실험을 실행합니다—이렇게 하면 각 데이터셋 항목에 대해 애플리케이션 코드가 실행됩니다. 자세한 내용은 [UI를 통한 실험](/docs/evaluation/experiments/experiments-via-ui) 또는 [SDK를 통한 실험](/docs/evaluation/experiments/experiments-via-sdk)을 참고하세요.
3. 선택한 평가기가 생성된 출력을 자동으로 채점합니다.
4. 실험 실행 간 결과를 비교하여 데이터 기반의 의사 결정을 내립니다.

**사용 사례 예시**

- 50개의 고객 지원 질문에 대해 GPT-4와 Claude Opus를 비교하고 정확도와 유용성을 모두 평가한 다음, 더 나은 성능을 보이는 모델을 배포

### 평가기 선택

<Frame fullWidth>![Evaluator select](/images/docs/evaluator-select.png)</Frame>

다음으로 평가기를 선택합니다. 주로 두 가지 방법이 있습니다:

<Tabs items={["관리형 평가기", "커스텀 평가기"]}>
<Tab>

Langfuse는 자체적으로, 그리고 **Ragas**와 같은 파트너와 함께 구축하고 유지 관리하는 평가기 카탈로그를 계속 확장하며 제공합니다. 각 평가기는 특정 품질 차원—예: _환각(Hallucination)_, _컨텍스트 관련성(Context-Relevance)_, _독성(Toxicity)_, _유용성(Helpfulness)_—에 대한 모범 사례 평가 프롬프트를 담고 있습니다.

- **바로 사용 가능**: 프롬프트를 직접 작성할 필요가 없습니다.
- **지속적으로 확장**: OSS 파트너가 유지 관리하는 평가기와, regex 기반과 같은 새로운 평가기 유형이 앞으로 추가됩니다.

</Tab>
<Tab>

라이브러리가 특정 요구 사항에 맞지 않는 경우, 직접 평가기를 추가할 수 있습니다:

1. `{{variables}}` 플레이스홀더(`input`, `output`, `ground_truth` 등)를 사용해 평가 프롬프트를 작성합니다.
2. **점수 유형**을 선택합니다: 유용성이나 충실도처럼 점진적인 값에는 **숫자형(Numeric)**을, 이산적인 레이블에는 **범주형(Categorical)**을, `User Disagreement`, `Out-of-Scope Request`, `Insufficient Answer`처럼 `true`/`false` 판단에는 **불리언(Boolean)**을 사용합니다.
3. **범주형**을 선택한 경우, 허용할 카테고리를 정의합니다. 여러 레이블이 동시에 적용될 수 있다면 선택적으로 **여러 항목 매칭 허용**을 활성화할 수 있습니다. Langfuse는 선택된 카테고리마다 하나의 점수를 생성합니다. **불리언** 평가기는 카테고리 목록이 필요하지 않습니다.
4. 선택 사항: 선택한 점수 유형에 맞춰 **점수 근거(score reasoning)** 프롬프트와 출력 프롬프트를 커스터마이즈합니다.
5. 선택 사항: 이 평가기 전용의 커스텀 모델을 지정합니다. 별도 모델을 지정하지 않으면 기본 평가 모델(2단계 참고)이 사용됩니다.
6. 저장하면 → 해당 평가기를 프로젝트 전반에서 재사용할 수 있습니다.

</Tab>
</Tabs>

### 평가할 데이터 선택

평가기와 모델을 선택했다면, 평가를 실행할 데이터를 설정합니다. 사용 사례에 맞는 옵션을 파악하려면 위의 [각 평가 대상 이해하기](#understanding-each-evaluation-target) 섹션을 참고하세요.

<Tabs items={["관측치 (권장)", "트레이스 (레거시)"]}>

**설정 단계**

1. 평가 대상으로 "Live Observations"를 선택합니다.
2. 관측치 유형, 트레이스 이름, 트레이스 태그, userId, sessionId, 메타데이터 및 기타 속성을 사용해 특정 관측치로 필터링합니다.
3. 평가 비용과 처리량을 관리하기 위해 샘플링 비율(예: 5%)을 설정합니다.

**요구 사항**

- **SDK 버전**: Python v3+(OTel 기반) 또는 JS/TS v4+(OTel 기반)

### 변수 매핑 및 평가 프롬프트 미리보기

이제 Langfuse에게 관측치, 트레이스, 실험 항목의 *어떤 속성*이 이러한 변수를 채우는 실제 데이터인지 알려주어, 타당한 평가가 이루어지도록 해야 합니다. 예를 들어 시스템에서 기록된 관측치 입력을 프롬프트의 `{{input}}` 변수에 매핑하고, LLM 응답(관측치 출력)을 프롬프트의 `{{output}}` 변수에 매핑할 수 있습니다. 이 매핑은 평가가 타당하고 관련성 있게 이루어지도록 하는 데 매우 중요합니다.

관측치 및 실험 대상의 경우, **도구 호출(Tool Calls)**은 대상 관측치에 기록된 호출을 매핑합니다. 각 호출에는 `id`, `name`, `arguments`, `type`, `index`가 포함됩니다. 유효한 JSON 인수는 파싱됩니다. 전체 배열을 매핑하거나 `$[*].name`과 같은 JSONPath 선택자를 사용할 수 있습니다.

<Tabs items={["실시간 프로덕션 데이터", "오프라인 실험 데이터"]}>
<Tab>

- **프롬프트 미리보기**: 매핑을 설정하는 동안 Langfuse는 **실제 데이터로 채워진 평가 프롬프트의 실시간 미리보기**를 보여줍니다. 이 미리보기는 필터에 매칭된 지난 24시간 동안의 과거 데이터를 사용합니다. 여러 예시를 살펴보며 각각의 데이터가 프롬프트를 어떻게 채우는지 확인함으로써 매핑이 올바른지 확신을 가질 수 있습니다.
- **JSONPath**: 데이터가 (예: JSON 객체 내부에) 중첩되어 있는 경우, `$.choices[0].message.content`와 같은 JSONPath 표현식을 사용해 정확히 위치를 지정할 수 있습니다.

<Frame fullWidth>![Filter preview](/images/docs/evaluator-mapping.png)</Frame>

</Tab>
<Tab>

- **제안된 매핑**: 시스템은 실험에서 흔히 사용되는 필드 이름을 기반으로 일반적인 매핑을 자동으로 완성해주는 경우가 많습니다. 예를 들어 정확도를 평가 중이고 프롬프트에 `{{input}}`, `{{output}}`, `{{ground_truth}}` 변수가 포함되어 있다면, 이들을 실험 항목의 input, output, expected_output에 각각 매핑하도록 제안할 가능성이 높습니다.
- **매핑 편집**: 실험 스키마가 다르다면 이러한 제안을 쉽게 편집할 수 있습니다. 실험 항목의 모든 속성(예: `input`, `expected_output`)을 매핑할 수 있습니다. 또한 실험은 내부적으로 트레이스를 생성하므로, 트레이스의 입력/출력을 평가 입력/출력으로 사용하는 방식이 일반적인 패턴입니다. 트레이스 출력을 실험 실행의 출력이라고 생각하면 됩니다.

</Tab>
</Tabs>

## API를 통한 프로그래밍 방식 설정 [#api]

UI 외에도 공개 API를 통해 프로그래밍 방식으로 LLM-as-a-Judge 평가를 설정하고 관리할 수 있습니다. 이는 평가 설정을 버전 관리하거나, 여러 프로젝트에 동일하게 적용하거나, 배포 파이프라인에서 롤아웃을 자동화하는 데 유용합니다.

이 설정은 두 가지 리소스로 나뉩니다:

- **평가기(Evaluators)**는 데이터를 채점하는 *방법*을 정의합니다: judge 프롬프트, 그 `{{variables}}`, 구조화된 출력 정의(숫자형, 불리언, 범주형), 선택적 모델 구성입니다. 평가기는 버전이 관리되며—기존 이름으로 새로 생성하면 다음 버전이 만들어지고, 활성화된 규칙은 자동으로 해당 버전으로 이동합니다.
- **평가 규칙(Evaluation rules)**은 _무엇을_ 평가할지를 정의합니다: 대상(실시간 관측치 또는 실험), 필터, 샘플링 비율, 그리고 데이터를 평가기의 변수에 매핑하는 방식입니다. 각 규칙은 `name`과 `scope`로 평가기 패밀리를 참조합니다. `tool_calls` 매핑 소스는 두 대상 모두에서 사용할 수 있습니다.

일반적인 흐름은 평가기를 생성하고, 그 `variables`와 `outputDefinition`을 읽어온 다음, 이를 참조하는 하나 이상의 평가 규칙을 생성하는 것입니다.

이 엔드포인트들은 코딩 에이전트가 탐색하고 활용하기 쉽도록 설계되어 있습니다. 프로그래밍 방식으로 평가기를 설정할 때 권장되는 방법은 에이전트에게 API 레퍼런스를 알려주고 평가기 생성과 평가 규칙 연결을 대신 처리하도록 하는 것입니다.

<Callout type="info">
  이 엔드포인트들은 현재 **불안정(unstable)** 상태이며, 기반이 되는 평가 데이터 모델이 재설계되는 동안 변경될 수 있습니다. 전체 요청/응답 스키마는 [Evaluators](https://api.reference.langfuse.com/#tag/unstableevaluators) 및 [Evaluation Rules](https://api.reference.langfuse.com/#tag/unstableevaluationrules) API 레퍼런스를 참고하세요.
</Callout>

## 고급 주제

### 트레이스 수준에서 관측치 수준 평가기로 마이그레이션

기존에 트레이스에서 실행 중인 평가기를 더 나은 성능과 안정성을 위해 관측치 수준으로 업그레이드하려면, 종합적인 [평가기 마이그레이션 가이드](/faq/all/llm-as-a-judge-migration)를 확인하세요.

### 관측치 수준 평가기 문제 해결

관측치 수준 평가기가 실행되지 않는다면, [관측치 수준 평가기가 실행되지 않는 이유는 무엇인가요?](/faq/all/observation-eval-not-executing)에서 일반적인 원인과 해결 방법을 확인하세요.

### 과거 관측치 점수 백필

관측치 테이블의 과거 데이터에 대해 관측치 수준 LLM-as-a-Judge를 실행할 수 있습니다. 이미 프로덕션 데이터를 수집해 두었고, 매칭되는 관측치를 새 평가기 또는 업데이트된 평가기로 소급하여 채점하고 싶을 때 유용합니다.

**사전 조건**: 평가기에서 **[패스트 모드(Fast Mode)](/docs/v4)** 토글을 활성화합니다. 새로 수집되는 데이터에 실시간으로 동일한 평가기를 사용하려면 최신 SDK(Python v4+ 또는 JS/TS v5+)로 업그레이드하거나, OTEL을 통해 직접 수집하는 경우 OTEL 스팬 익스포터에 `x-langfuse-ingestion-version: 4`를 설정하세요.

점수를 백필하려면:

1. **Traces** 테이블을 엽니다.
2. 백필하려는 기간과 트레이스 조건으로 필터링합니다. 평가기가 대상으로 삼는 것과 동일한 조건을 사용하세요.
3. 매칭되는 행을 선택합니다.
4. **Actions** → **Evaluate**를 클릭합니다.
5. 평가 흐름에 따라 선택한 트레이스에 평가기를 실행하고 매칭되는 관측치의 점수를 백필합니다.

<Frame fullWidth>
  ![Tracing table with filters applied before backfilling observation-level evaluations](/images/docs/llm-as-a-judge/observation-backfill.png)
</Frame>

이 백필 흐름은 트레이스 테이블에서 실행되지만, 생성되는 점수는 각 트레이스 내부의 매칭되는 관측치에 첨부됩니다.

## FAQ

<Details>
<Summary>LLM-as-a-Judge 평가란 무엇인가요?</Summary>

LLM-as-a-Judge는 대규모 언어 모델("judge")이 다른 LLM 애플리케이션의 출력 품질을 평가하는 평가 방법론입니다. judge 모델에게 입력, 애플리케이션의 출력, 채점 루브릭이 주어지면 근거와 함께 점수를 생성합니다. 사람과 유사한 뉘앙스와 자동화된 확장성을 결합할 수 있어, LLM 애플리케이션을 평가하는 가장 널리 사용되는 방법 중 하나입니다.

</Details>

<Details>
<Summary>LLM-as-a-Judge는 사람 평가와 비교했을 때 얼마나 정확한가요?</Summary>

연구에 따르면 (GPT-5 등급의 모델과 같은) 강력한 LLM judge는 많은 품질 차원에서 사람 평가자와 80~90%의 일치도를 보이며, 이는 사람 주석자 간의 일치도와 비슷한 수준입니다. 잘 설계된 루브릭과 명확한 평가 기준을 사용하면 정확도가 크게 향상됩니다. 최상의 결과를 얻으려면 소규모의 사람이 주석을 단 예시 데이터를 기준으로 LLM-as-a-Judge 설정을 보정하세요.

</Details>

<Details>
<Summary>LLM judge로 사용하기에 가장 적합한 모델은 무엇인가요?</Summary>

일반적으로 가장 성능이 뛰어난 모델이 가장 좋은 평가 결과를 냅니다. 지시 이행 능력과 추론 능력이 뛰어난 모델(예: GPT-4o, Claude Sonnet, Gemini Pro)이 흔히 사용됩니다. judge 모델은 점수를 안정적으로 파싱할 수 있도록 구조화된 출력을 지원해야 합니다. Langfuse에서는 [LLM Connections](/docs/administration/llm-connection)를 통해 judge 모델을 구성합니다.

</Details>

<Details>
<Summary>LLM-as-a-Judge를 사용하는 데 비용이 얼마나 드나요?</Summary>

비용은 judge 모델과 평가 대상 입력의 크기에 따라 달라집니다. 일반적인 평가 비용은 건당 $0.01~$0.10입니다. 비용은 다음과 같은 방법으로 관리할 수 있습니다: (1) 샘플링을 사용해 트레이스의 일부만 평가, (2) 전체 트레이스가 아닌 특정 관측치를 대상으로 지정, (3) 단순한 평가에는 비용 효율적인 judge 모델 선택.

</Details>

<Details>
<Summary>RAG 평가에 LLM-as-a-Judge를 사용할 수 있나요?</Summary>

네. LLM-as-a-Judge는 RAG 파이프라인 평가에 특히 효과적입니다. 충실도(답변이 검색된 컨텍스트에 근거하고 있는가?), 관련성(답변이 질문을 다루고 있는가?), 완전성(답변이 관련 정보를 모두 포함하는가?)을 평가할 수 있습니다. Langfuse는 특화된 RAG 평가 메트릭을 위해 [RAGAS](/guides/cookbook/evaluation_of_rag_with_ragas)와도 통합됩니다.

</Details>

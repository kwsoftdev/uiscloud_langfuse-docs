# 메트릭 API

```
GET /api/public/v2/metrics
```

**메트릭 API**를 사용하면 Langfuse 데이터로부터 커스터마이즈된 분석 결과를 조회할 수 있습니다.
이 엔드포인트를 사용하면 차원(dimension), 메트릭, 필터, 시간 단위를 지정하여 LLM 애플리케이션을 위한 강력한 커스텀 리포트와 대시보드를 구축할 수 있습니다.

<Callout type="info">

기존 메트릭 API는 계속 사용할 수 있지만, 대규모 환경에서는 성능이 상대적으로 낮기 때문에 신규 집계 데이터 추출 워크플로의 기본 선택지로는 권장하지 않습니다. 새로운 리포팅, 빌링, 대시보드, 모니터링 워크플로에는 메트릭 API v2를 사용하세요.

</Callout>

<Callout type="info">

**데이터 가용성:** `x-langfuse-ingestion-version: 4` 헤더를 전송하지 않는 이전 버전 SDK(`langfuse-python` < `4.0.0`, `langfuse-js` < `5.0.0`) 또는 OpenTelemetry 익스포터를 직접 사용하는 경우, v2 엔드포인트에서 데이터가 최대 10분까지 지연될 수 있습니다. 새 데이터를 실시간으로 확인하려면 [Python SDK v4.7.0+](/docs/observability/sdk/upgrade-path/python-v3-to-v4) 또는 [JS/TS SDK v5.4.0+](/docs/observability/sdk/upgrade-path/js-v4-to-v5)로 업그레이드하거나, OTEL 스팬 익스포터에 해당 헤더를 설정하세요.

</Callout>

```
GET /api/public/v2/metrics
```

v2 메트릭 API는 쿼리당 데이터베이스 작업을 최소화하는 새로운 이벤트 테이블 스키마 기반의 최적화된 데이터 아키텍처를 통해 상당한 성능 개선을 제공합니다.

### v1 대비 주요 변경 사항

**v2에서는 `traces` 뷰를 더 이상 사용할 수 없습니다.** 대신 v1보다 더 빠르고 강력한 `observations` 뷰를 사용하세요.

### v2에서 사용 가능한 뷰

| 뷰                   | 설명                                                                                               |
| -------------------- | -------------------------------------------------------------------------------------------------- |
| `observations`       | 옵저베이션 레벨 데이터를 트레이스 레벨 집계와 함께 조회                                            |
| `scores-numeric`     | 숫자형 점수 조회                                                                                   |
| `scores-categorical` | 범주형(문자열) 점수 조회                                                                           |
| `scores-boolean`     | 불리언 점수 조회. `booleanValue`로 그룹화 또는 필터링하거나, true 비율을 구하려면 `value`를 평균화 |

### 행 제한

v2 메트릭 API는 일관된 성능을 보장하기 위해 쿼리당 기본 `config.row_limit`을 100행으로 적용합니다. 쿼리에서 커스텀 `config.row_limit`을 지정하여 이 기본값을 재정의할 수 있으며, 최대 1,000행까지 지정할 수 있습니다.

### 카디널리티가 높은 차원

`id`, `traceId`, `userId`, `sessionId`와 같은 특정 차원은 v2 메트릭 API에서 그룹화에 사용할 수 없습니다. 이러한 카디널리티가 높은 필드로 그룹화하는 작업은 비용이 매우 크고 실제로 유용한 경우가 드뭅니다. 이 차원들은 필터링에는 계속 사용할 수 있습니다.

### 메트릭 기준 정렬

집계된 메트릭을 기준으로 정렬할 때는 반환되는 메트릭 필드 이름을 `{aggregation}_{measure}` 형식으로 사용하세요. 예를 들어 `{ "measure": "totalCost", "aggregation": "sum" }`의 경우 `sum_totalCost`를 사용합니다. 시간 차원을 기준으로 정렬할 때는 반환되는 필드 이름 `time_dimension`을 사용하세요.

### 예시: 옵저베이션에서 비용이 가장 높은 모델

```bash
curl \
  -H "Authorization: Basic <BASIC AUTH HEADER>" \
  -G \
  --data-urlencode 'query={
    "view": "observations",
    "metrics": [{"measure": "totalCost", "aggregation": "sum"}],
    "dimensions": [{"field": "providedModelName"}],
    "filters": [],
    "fromTimestamp": "2025-12-01T00:00:00Z",
    "toTimestamp": "2025-12-16T00:00:00Z",
    "orderBy": [{"field": "sum_totalCost", "direction": "desc"}],
    "config": {"row_limit": 1000}
  }' \
  https://cloud.langfuse.com/api/public/v2/metrics
```

<Callout type="info">

**API 레퍼런스:** 사용 가능한 모든 파라미터, 응답 스키마, 인터랙티브 예시는 전체 [v2 메트릭 API 레퍼런스](https://api.reference.langfuse.com/#tag/metricsv2/GET/api/public/v2/metrics)를 참고하세요.

</Callout>

## 레거시 메트릭 API v1 [#v1]

v1 `GET /api/public/metrics` 엔드포인트는 계속 사용할 수 있지만, 대규모 환경에서는 성능이 상대적으로 낮기 때문에 신규 집계 데이터 추출 워크플로의 기본 선택지로는 권장하지 않습니다. 새로운 리포팅, 빌링, 대시보드, 모니터링 워크플로에는 [메트릭 API v2](#v2)를 사용하세요.

기존 쿼리를 v2로 마이그레이션할 때:

- `GET /api/public/metrics`를 `GET /api/public/v2/metrics`로 교체하세요.
- `view: "traces"`를 `view: "observations"`로 교체하고, 지원되는 경우 `traceName`, `traceRelease`, `traceVersion`과 같은 트레이스 레벨 차원을 사용하세요. `userId`와 `sessionId`는 필터로는 계속 사용할 수 있지만, v2에서는 그룹화에 사용할 수 없습니다.
- 트레이스 뷰 쿼리를 마이그레이션할 때는 `metrics` 배열을 검토하세요. v2의 `observations` 뷰에서는 `count`, `latency`, `totalCost`, `totalTokens`와 같은 measure가 옵저베이션 행 단위로 계산됩니다. 트레이스 레벨 카운트나 트레이스 지속 시간이 필요한 경우 [옵저베이션 API v2](/docs/api-and-data-platform/features/observations-api#v2)를 사용하고 클라이언트 측에서 `traceId`로 그룹화하세요.
- 기본값인 100행보다 많은 결과를 반환해야 하는 쿼리를 마이그레이션할 때는 `config.row_limit`을 명시적으로 설정하세요.
- 행 단위의 스팬, 제너레이션, 이벤트가 필요한 경우에는 대신 [옵저베이션 API v2](/docs/api-and-data-platform/features/observations-api#v2)를 사용하세요.

이 섹션의 나머지 부분에서는 기존 통합을 위해 레거시 v1 엔드포인트를 계속 설명합니다.

v1 메트릭 API는 여러 뷰(traces, observations, scores)에 걸친 쿼리를 지원하며, 다음을 수행할 수 있습니다.

- 데이터를 그룹화할 특정 차원 선택
- 서로 다른 집계 방식으로 여러 메트릭 적용
- 메타데이터, 타임스탬프, 기타 속성을 기준으로 데이터 필터링
- 커스터마이즈 가능한 단위로 시간에 따라 데이터 분석
- 필요에 따라 결과 정렬

### 쿼리 파라미터

v1 API는 URL 인코딩된 파라미터로 전달되는 JSON 쿼리 객체를 받습니다.

| 파라미터 | 타입        | 설명                                        |
| -------- | ----------- | ------------------------------------------- |
| `query`  | JSON string | 조회할 메트릭을 정의하는 인코딩된 쿼리 객체 |

#### 쿼리 객체 구조

| 필드            | 타입   | 필수 여부 | 설명                                                                                                                                                            |
| --------------- | ------ | --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `view`          | string | Yes       | 조회할 데이터 뷰: `"traces"`, `"observations"`, `"scores-numeric"`, `"scores-categorical"`, `"scores-boolean"` 중 하나                                          |
| `dimensions`    | array  | No        | 그룹화할 차원 객체 배열, 예: `[{ "field": "name" }]`                                                                                                            |
| `metrics`       | array  | Yes       | 계산할 메트릭 객체 배열, 예: `[{ "measure": "latency", "aggregation": "p95" }]`                                                                                 |
| `filters`       | array  | No        | 결과를 좁히기 위한 필터 객체 배열, 예: `[{ "column": "metadata", "operator": "contains", "key": "customKey", "value": "customValue", "type": "stringObject" }]` |
| `timeDimension` | object | No        | 시간 기반 분석을 위한 설정, 예: `{ "granularity": "day" }`                                                                                                      |
| `fromTimestamp` | string | Yes       | 쿼리 기간의 시작을 나타내는 ISO 타임스탬프                                                                                                                      |
| `toTimestamp`   | string | Yes       | 쿼리 기간의 종료를 나타내는 ISO 타임스탬프                                                                                                                      |
| `orderBy`       | array  | No        | 결과 정렬 방식 지정, 예: `[{ "field": "name", "direction": "asc" }]`                                                                                            |

#### 차원 객체 구조

```json
{ "field": "name" }
```

#### 메트릭 객체 구조

```json
{ "measure": "count", "aggregation": "count" }
```

일반적인 measure 유형은 다음과 같습니다.

- `count` - 레코드 수
- `latency` - 지속 시간/지연 시간 메트릭

집계 유형은 다음과 같습니다.

- `sum` - 값의 합계
- `avg` - 값의 평균
- `count` - 레코드 수
- `max` - 최댓값
- `min` - 최솟값
- `p50` - 50번째 백분위수
- `p75` - 75번째 백분위수
- `p90` - 90번째 백분위수
- `p95` - 95번째 백분위수
- `p99` - 99번째 백분위수

#### 필터 객체 구조

```json
{
  "column": "metadata",
  "operator": "contains",
  "key": "customKey",
  "value": "customValue",
  "type": "stringObject"
}
```

#### 시간 차원 객체

```json
{
  "granularity": "day"
}
```

지원되는 단위: `hour`, `day`, `week`, `month`, `auto`.

### 데이터 모델

v1 메트릭 API는 각각 고유한 차원 및 메트릭 집합을 가진 여러 데이터 뷰에 대한 접근을 제공합니다. 이 섹션에서는 각 뷰에서 사용 가능한 옵션을 설명합니다.

#### 사용 가능한 뷰

| 뷰                   | 설명                        |
| -------------------- | --------------------------- |
| `traces`             | 트레이스 레벨 데이터 조회   |
| `observations`       | 옵저베이션 레벨 데이터 조회 |
| `scores-numeric`     | 숫자형 점수 조회            |
| `scores-categorical` | 범주형(문자열) 점수 조회    |
| `scores-boolean`     | 불리언 점수 조회            |

#### 트레이스 차원

| 차원              | 타입     | 설명                           |
| ----------------- | -------- | ------------------------------ |
| `id`              | string   | 트레이스 ID                    |
| `name`            | string   | 트레이스 이름                  |
| `tags`            | string[] | 트레이스 태그                  |
| `userId`          | string   | 트레이스와 연관된 사용자 ID    |
| `sessionId`       | string   | 트레이스와 연관된 세션 ID      |
| `release`         | string   | 릴리즈 태그                    |
| `version`         | string   | 버전 태그                      |
| `environment`     | string   | 환경 (예: production, staging) |
| `observationName` | string   | 연관된 옵저베이션의 이름       |
| `scoreName`       | string   | 연관된 점수의 이름             |

#### 트레이스 메트릭

| 메트릭              | 설명                           |
| ------------------- | ------------------------------ |
| `count`             | 트레이스 수                    |
| `observationsCount` | 트레이스 내 옵저베이션 수      |
| `scoresCount`       | 트레이스 내 점수 수            |
| `latency`           | 트레이스 지속 시간(밀리초)     |
| `totalTokens`       | 트레이스에서 사용된 총 토큰 수 |
| `totalCost`         | 트레이스의 총 비용             |

#### 옵저베이션 차원

| 차원                  | 타입   | 설명                           |
| --------------------- | ------ | ------------------------------ |
| `id`                  | string | 옵저베이션 ID                  |
| `traceId`             | string | 연관된 트레이스 ID             |
| `traceName`           | string | 상위 트레이스의 이름           |
| `environment`         | string | 환경 (예: production, staging) |
| `parentObservationId` | string | 상위 옵저베이션의 ID           |
| `type`                | string | 옵저베이션 유형                |
| `name`                | string | 옵저베이션 이름                |
| `level`               | string | 로그 레벨                      |
| `version`             | string | 버전                           |
| `providedModelName`   | string | 모델 이름                      |
| `promptName`          | string | 프롬프트 이름                  |
| `promptVersion`       | string | 프롬프트 버전                  |
| `userId`              | string | 상위 트레이스의 사용자 ID      |
| `sessionId`           | string | 상위 트레이스의 세션 ID        |
| `traceRelease`        | string | 상위 트레이스의 릴리즈         |
| `traceVersion`        | string | 상위 트레이스의 버전           |
| `scoreName`           | string | 연관된 점수 이름               |

#### 옵저베이션 메트릭

| 메트릭             | 설명                         |
| ------------------ | ---------------------------- |
| `count`            | 옵저베이션 수                |
| `latency`          | 옵저베이션 지속 시간(밀리초) |
| `totalTokens`      | 사용된 총 토큰 수            |
| `totalCost`        | 총 비용                      |
| `timeToFirstToken` | 첫 토큰까지의 시간(밀리초)   |
| `countScores`      | 연관된 점수 수               |

#### 점수 차원 (공통)

| 차원                       | 타입   | 설명                                       |
| -------------------------- | ------ | ------------------------------------------ |

| `environment`              | string | 환경                                       |

| `userId`                   | string | 트레이스의 사용자 ID                       |
| `sessionId`                | string | 트레이스의 세션 ID                         |

#### 점수 메트릭

##### 숫자형 점수

| 메트릭  | 설명           |
| ------- | -------------- |
| `count` | 점수 수        |
| `value` | 숫자형 점수 값 |

##### 불리언 점수

| 메트릭  | 설명                                                                                               |
| ------- | -------------------------------------------------------------------------------------------------- |
| `count` | 점수 수                                                                                            |
| `value` | 숫자형 점수 값, `0`은 false, `1`은 true를 나타냅니다. true 비율을 구하려면 `avg` 집계를 사용하세요 |

불리언 점수에는 추가 차원이 있습니다.

| 차원           | 타입    | 설명                                     |
| -------------- | ------- | ---------------------------------------- |
| `booleanValue` | boolean | 그룹화 및 필터링을 위한 점수의 불리언 값 |

##### 범주형 점수

| 메트릭  | 설명    |
| ------- | ------- |
| `count` | 점수 수 |

범주형 점수에는 추가 차원이 있습니다.

| 차원          | 타입   | 설명                    |
| ------------- | ------ | ----------------------- |
| `stringValue` | string | 범주형 점수의 문자열 값 |

## 일일 메트릭 API (레거시) [#daily-metrics]

<Callout type="warning">

이 엔드포인트는 레거시이며 더 이상 [공개 API 레퍼런스](https://api.reference.langfuse.com/)에 나열되지 않습니다. 하위 호환성을 위해 계속 사용할 수 있지만, 신규 사용 사례에는 위의 [메트릭 API v2](#v2)를 사용하세요. 더 높은 유연성을 제공하며, [API 제한 FAQ](/faq/all/api-limits)에 명시된 별도의 메트릭 API v2 레이트 리밋 버킷을 사용합니다.

</Callout>

```
GET /api/public/metrics/daily
```

**일일 메트릭 API**를 통해 분석, 빌링, 레이트 리밋 등 후속 활용을 위한 Langfuse의 집계된 일일 사용량 및 비용 메트릭을 조회할 수 있습니다. 이 API를 사용하면 애플리케이션 유형, 사용자, 태그로 필터링하여 원하는 데이터를 조회할 수 있습니다.

### 개요

반환되는 데이터에는 다음의 일일 시계열이 포함됩니다.

- USD 기준 [비용](/docs/model-usage-and-cost)
- 트레이스 및 옵저베이션 수
- 모델 이름별 분류
- 입력 및 출력 사용량으로 분류된 사용량(예: 토큰)
- USD 기준 [비용](/docs/model-usage-and-cost)
- 트레이스 및 옵저베이션 수

선택적 필터:

- `traceName` - 트레이스에서 `name`을 사용하는 방식에 따라 애플리케이션 유형별로 필터링할 때 일반적으로 사용
- `userId` - [사용자](/docs/observability/features/users)별로 필터링
- `tags` - [태그](/docs/observability/features/tags)별로 필터링
- `fromTimestamp`
- `toTimestamp`

필요한 주요 메트릭이나 필터가 없다면 [아이디어 보드](/ideas)를 통해 요청해 주세요.

GET /api/public/metrics/daily?traceName=my-copilot&userId=john&limit=2

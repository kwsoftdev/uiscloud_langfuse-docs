# 보안 FAQ

이 페이지는 Langfuse에 대해 자주 묻는 질문과 일반적인 보안 주제를 다룹니다. 컴플라이언스 주제에 대한 자세한 내용은 [컴플라이언스 FAQ](/security/compliance-faq)를 참조하세요.

여기서 원하는 답변을 찾지 못했다면 [Ask AI](/docs/ask-ai)를 사용해 즉시 답변을 받거나 security@clickhouse.com으로 문의하세요.

## 애플리케이션 보안

> **전송 중 및 저장 시 데이터는 어떻게 암호화되나요?**

TLS 1.2 이상이 트래픽을 보호하며, 저장된 모든 데이터는 AES‑256 암호화를 사용합니다. 자세한 내용은 [암호화 문서](/security/encryption) 및 [셀프 호스팅 암호화](/self-hosting/configuration/encryption)를 참조하세요.

> **Langfuse Cloud에서 데이터를 암호화할 때 자체 키를 사용할 수 있나요?**

고객이 관리하는 키(Customer-managed key)는 지원되지 않습니다. Langfuse Cloud는 모든 서비스에 대해 AWS 관리형 암호화(KMS를 통한 AES‑256)를 사용합니다. BYOK/CMEK 또는 HSM 통합은 제공되지 않습니다.

> **어떤 보존, 삭제, 내보내기 제어 기능이 있나요?**

각 프로젝트는 자체 보존 기간을 설정할 수 있으며, 해당 기간보다 오래된 데이터는 매일 밤 삭제됩니다. 사용자 또는 API를 통해 즉시 삭제나 내보내기를 실행할 수도 있습니다. 자세한 내용은 [데이터 보존](/docs/administration/data-retention) 및 [데이터 삭제](/docs/administration/data-deletion) 문서를 참조하세요.

> **Langfuse Cloud의 멀티테넌트 보안 모델을 어떻게 평가해야 하나요?**

멀티테넌시는 현대 SaaS 제품의 표준 운영 모델입니다. 관련된 보안 질문은 테넌트 격리가 명시적이고, 일관되게 적용되며, 독립적으로 테스트되는지 여부입니다. Langfuse Cloud는 프로젝트 단위 데이터 격리, 프로젝트 단위 API 키, RBAC 인가, 암호화된 인프라, 지속적인 모니터링, 그리고 연 1회 제3자 침투 테스트를 결합합니다. 대부분의 팀에게 이는 보안, 도입 속도, 제품 경험, 운영 안정성 사이의 강력한 균형을 제공합니다. Langfuse가 전체 애플리케이션 및 인프라 스택을 운영하기 때문입니다.

내부 정책상 인프라 수준의 격리, 고객이 운영하는 네트워킹, 또는 고객이 통제하는 데이터베이스 경계가 필요하다면 [셀프 호스팅 Langfuse](/self-hosting)를 사용하거나 Enterprise Cloud 옵션에 대해 저희와 상담하세요.

> **싱글 테넌트 환경에 Langfuse를 배포할 수 있나요?**

Langfuse Cloud는 멀티테넌트입니다. 엄격한 인프라 수준의 격리가 필요하다면 Langfuse를 셀프 호스팅하는 것을 권장합니다. 셀프 호스팅 Enterprise 배포는 선호하는 ClickHouse 배포(ClickHouse Cloud 포함)와 함께 사용할 수 있지만, 네트워킹, 확장, 업그레이드, 백업, 운영 제어는 여러분의 팀이 직접 소유하게 됩니다.

> **테넌트 격리는 어떻게 적용되나요?**

논리적 데이터 격리는 모든 데이터를 특정 `projectId`와 연결함으로써 기술적으로 구현됩니다. 이 고유 식별자는 데이터베이스 내에서 데이터를 분할하는 키 역할을 하여, 트레이스와 관찰(observation)부터 점수와 데이터셋에 이르기까지 모든 정보가 단일 프로젝트에 명시적으로 연결되도록 보장합니다. 다른 프로젝트의 데이터에 대한 접근은 프로젝트 멤버십을 기반으로 구축된 역할 기반 접근 제어(RBAC) 시스템을 통해 차단됩니다. 사용자는 이러한 멤버십을 통해 프로젝트에 대한 접근 권한을 부여받으며, 각 멤버십은 해당 특정 프로젝트 내에서의 역할과 권한을 정의합니다. 자세한 내용은 [RBAC 문서](/docs/administration/rbac)를 참조하세요. 이후 애플리케이션 로직은 사용자의 인증된 세션을 사용하여 모든 데이터베이스 쿼리를 사용자가 권한을 가진 프로젝트 ID로 필터링합니다. 이 테넌트 경계는 Langfuse의 연례 제3자 침투 테스트 범위에 포함됩니다.

> **Langfuse Cloud와 ClickHouse Cloud에서 직접 Langfuse를 운영하는 것은 어떻게 다른가요?**

Langfuse Cloud는 관리형 백엔드의 일부로 ClickHouse Cloud를 사용하므로, 고객은 ClickHouse Cloud의 인프라 보안 혜택을 누리는 동시에 Langfuse가 애플리케이션, 컨트롤 플레인, 큐/캐시, 오브젝트 스토리지, 데이터베이스 통합, 업그레이드, 백업, 인시던트 대응을 운영합니다. 가장 빠른 설정과 최소한의 운영 부담을 원한다면 이 방식이 권장됩니다.

자체 ClickHouse Cloud 환경과 함께 셀프 호스팅 Langfuse를 운영하면 여러분의 팀이 네트워크 토폴로지, 데이터베이스 구성, 인프라 정책을 직접 소유하게 됩니다. 고객 정책상 고객이 통제하는 VPC, 고객이 관리하는 프라이빗 커넥티비티, 또는 전용 데이터베이스 경계가 필요한 경우 이 모델이 적합할 수 있지만, 비용과 운영 부담이 더 커집니다.

> **고객이 데이터를 특정 리전에 고정할 수 있나요?**

네. EU, US, 일본, 그리고 HIPAA 대응이 가능한 US 리전을 이용할 수 있습니다. Langfuse Cloud 리전은 서로 분리되어 있으며, 셀프 호스팅 배포에서는 원하는 리전과 인프라를 자유롭게 선택할 수 있습니다.

## 아이덴티티 및 접근 관리

> **어떤 인증 옵션이 지원되나요?**

OIDC SSO, 이메일/비밀번호, SCIM 프로비저닝을 지원하며, MFA 또는 패스키는 IdP를 통해 강제할 수 있습니다. 자세한 내용은 인증 문서를 참조하세요.

> **SDK 및 API 접근에는 어떤 인증 방식이 사용되나요?**

REST API는 공개 키(사용자 이름)와 비밀 키(비밀번호) 한 쌍을 사용하는 Basic Auth를 사용합니다. 이 키들은 프로젝트 설정에서 생성되며, 저장 전에 해시 처리되고 회전(rotate)할 수 있습니다. JWT는 사용되지 않습니다.

> **세션 토큰은 만료되기까지 얼마나 유효한가요?**

세션 토큰은 86,400초(24시간) 동안 유효합니다. 서버 측에서 만료를 검증하는 JWT 토큰을 사용하여 토큰이 서버와 클라이언트 양쪽에서 모두 만료되도록 보장합니다.

> **최소 권한 원칙은 어떻게 적용되나요?**

RBAC를 통해 역할을 조직 또는 프로젝트 단위로 범위를 지정할 수 있습니다. 자세한 내용은 [RBAC 문서](/docs/administration/rbac)를 참조하세요.

## 인프라 및 네트워크 보안

> **Langfuse Cloud는 어디에 호스팅되며 경계는 어떻게 보호되나요?**

Langfuse는 격리된 VPC 내 AWS에서 실행되며, DDoS 완화를 위해 WAF와 AWS Shield를 사용합니다.

> **Redis와 같은 지원 서비스가 추가적인 테넌트 격리 문제를 일으키지는 않나요?**

Langfuse Cloud는 Postgres, ClickHouse, Redis/ElastiCache, S3를 포함한 여러 관리형 백엔드 서비스를 사용합니다. 이러한 서비스는 고객에게 직접 노출되지 않으며, Langfuse 서비스가 격리된 지역 환경 내에서 최소 권한 자격 증명을 사용하여 접근합니다. 저장된 데이터는 암호화되고, 활동은 모니터링되며, 이러한 서비스 제공업체들은 벤더 리스크 관리 및 컴플라이언스 검토 대상입니다.

테넌트 경계는 여전히 Langfuse 애플리케이션과 데이터 모델에 있습니다. 제품 데이터는 `projectId`로 범위가 지정되고, API 키는 프로젝트 단위로 지정되며, 모든 사용자 요청은 RBAC를 통해 인가됩니다. 지원 인프라는 별도의 고객 운영 영역이 아니라 이러한 통제된 서비스 아키텍처의 일부입니다.

> **Langfuse Cloud는 AWS PrivateLink 또는 프라이빗 커넥티비티를 지원하나요?**

Langfuse Cloud는 기본적으로 퍼블릭 지역 엔드포인트를 사용합니다. AWS PrivateLink는 계약을 체결한 Enterprise 고객에 한해 지역 가용성 및 온보딩 여부에 따라 활성화할 수 있습니다. 지역별 가용성을 논의하려면 [영업팀에 문의하세요](/talk-to-us). 여러분의 팀이 완전히 통제하는 인프라 내에서 프라이빗 커넥티비티가 필요하다면 셀프 호스팅을 통해 그러한 통제권을 확보할 수 있습니다.

> **Langfuse 엔드포인트에는 어떤 보안 헤더가 구현되어 있나요?**

[langfuse.com](https://domsignal.com/test/ea3amc5z08sh6fnovjek392ihxrb66q6)과 [cloud.langfuse.com](https://domsignal.com/test/spdnv4cx0949ah3nvz5b5eapab8o2390)을 포함한 모든 엔드포인트에서 CSP(Content Security Policy), HSTS(HTTP Strict Transport Security), X-Frame-Options를 적용합니다.

> **Langfuse Cloud는 환경을 어떻게 모니터링하나요?**

Langfuse는 DataDog과 Sentry를 사용하여 애플리케이션과 환경을 모니터링합니다. 모든 클라우드 감사 로그는 별도로 잠긴(locked-down) 계정에 자동으로 기록됩니다. 또한 이상 사용 패턴에 대해 알림을 보내는 자동화 시스템을 갖추고 있습니다.

> **어떤 이상 탐지 및 보안 알림 기능이 마련되어 있나요?**

AWS GuardDuty를 사용한 자동 이상 탐지가 활성화되어 있으며, 불가능한 이동 경로(impossible travel)나 서비스 계정 및 사용자의 비정상적인 API 사용에 대해 알림을 제공합니다.

> **Langfuse는 클라우드 플랫폼에서 활동을 어떻게 기록하나요?**

인증 실패 시도는 애플리케이션 로그에 기록되며, 인프라(AWS, ClickHouse 등)에서의 활동도 추적됩니다. 애플리케이션 및 시스템 로그는 핫 스토어(hot store)에 15일간 보관되며 콜드 스토리지(cold storage)에는 영구 보관됩니다.

> **Langfuse는 감사를 위해 시간 동기화를 사용하나요?**

네. 모든 서비스는 NTP(Network Time Protocol)와 PTP(Precision Time Protocol) 클록 장치를 모두 제공하는 AWS 클라우드에서 실행됩니다. 기본적으로는 NTP가 사용됩니다.

> **Langfuse는 얼마나 자주 API 키를 회전(rotate)하나요?**

Langfuse는 가능한 경우 단기(short-lived) API 키를 사용합니다. 장기(long-lived) API 키의 경우 90일마다 회전합니다.

## 소프트웨어 개발 수명 주기

> **어떤 보안 코딩 및 테스트 관행이 마련되어 있나요?**

모든 커밋은 엔드투엔드, 유닛, 보안 테스트로 구성된 CI 파이프라인을 통과합니다.

> **Langfuse는 SAST 또는 DAST를 수행하나요?**

메인 브랜치에 대한 커밋 시 Snyk로 컨테이너 이미지를 스캔하여 SAST(Static Application Security Testing)를 수행합니다([예시](https://github.com/langfuse/langfuse/actions/runs/19838910104/job/56842732608)). 또한 CodeQL을 사용한 정적 코드 스캔도 실행합니다([예시](https://github.com/langfuse/langfuse/actions/runs/19838910102/job/56842732711)).

> **소프트웨어 개발 수명 주기 정책은 어떻게 구성되어 있나요?**

저희의 소프트웨어 개발 수명 주기(SDLC) 정책은 개발의 모든 단계에 정보 보안이 통합되도록 보장하는 일관된 프로세스를 수립합니다. 이 프로세스는 계획 및 설계, 소프트웨어 구축, 시스템 준비 상태 평가, 안전한 배포 실행 등 정해진 단계를 따릅니다. 이 전체 수명 주기에 걸쳐 저희는 보안 엔지니어링 원칙을 강제하고, 개발자들에게 OWASP Top 10과 같은 관행에 대한 지속적인 교육을 제공하며, 위협으로부터 보호하기 위한 취약점 스캔을 수행합니다. 데이터 무결성을 보호하기 위해 개발 및 테스트 환경은 프로덕션 환경과 엄격히 분리되며, 명시적인 승인 없이 엄격한 통제 하에 테스트에 프로덕션 데이터를 사용하는 것은 금지됩니다. 저희는 암호화된 전송과 데이터 접근을 위한 안전한 API를 기본으로 제공함으로써 고객을 위한 안전한 데이터 상호 운용성과 이식성을 보장합니다.

## 인시던트 대응 및 비즈니스 연속성

> **인시던트 대응 프로세스는 어떻게 되나요?**

24시간 연중무휴 모니터링이 온콜 엔지니어를 호출하며, 영향을 받은 고객에게는 통지가 이루어지고 대규모 인시던트에 대해서는 사후 분석(post-mortem)이 공개됩니다. 자세한 내용은 [인시던트 및 침해](/security/incident-and-breach) 문서를 참조하세요.

> **RTO, RPO는 얼마이며 Langfuse는 백업을 어떻게 유지하나요?**

[여기의 세부 정보](/security/data-regions#business-continuity--availability)를 확인해 주세요.

> **재해 복구 계획은 어떻게 구성되어 있나요?**

저희의 재해 복구 계획은 대규모 사고 발생 후 서비스를 복구하기 위한 절차를 규정하며, 핵심 시스템이 장기간 사용 불가능할 것으로 예상될 때 경영진에 의해 공식적으로 발동됩니다. 복구 단계에서는 전체 인프라를 대체 환경에 체계적으로 복제하고, 엄격한 테스트를 수행한 후 DNS 레코드를 업데이트하여 새 환경을 프로덕션으로 배포합니다. 최종 재구성 단계는 재해 발생 후 24시간 이내에 전체 운영을 복원하는 것을 목표로 하며, 원활한 전환과 모든 임시 리소스의 안전한 처리를 보장합니다.

> **비즈니스 연속성 계획은 어떻게 구성되어 있나요?**

저희의 비즈니스 연속성 계획은 주요 장애 발생 후 운영 복원력과 복구를 보장하기 위한 프레임워크를 수립하며, 각 핵심 프로세스에 대한 복구 목표 시간(RTO)과 복구 목표 시점(RPO)을 정의하는 공식적인 비즈니스 영향 분석(Business Impact Analysis)을 기반으로 합니다. 클라우드 애플리케이션의 경우, 저희의 전략은 클라우드 제공업체의 복원력과 정기적인 데이터 백업에 의존하여 정의된 복구 목표를 충족합니다. 물리적 근무지에 장애가 발생할 경우 직원들은 원격으로 근무할 수 있도록 준비되어 있어 핵심 비즈니스 및 개발 운영이 중단 없이 지속될 수 있습니다. 이 계획은 의사 결정 권한에 대한 명확한 승계 체계를 정의하며, 최소 연 1회 테스트를 통해 그 유효성을 검증하고 신속한 대응을 보장합니다.

> **Langfuse는 변경 관리를 어떻게 수행하나요?**

변경 관리 정책은 인프라, 코드, 프로덕션 시스템에 대한 모든 변경이 리스크를 최소화하고 장애를 방지하기 위해 계획된 방식으로 처리되도록 보장합니다. 모든 변경은 승인된 사용자에 의한 제출, 보안 및 기능에 대한 자동화된 테스트, 그리고 구현 전 경영진의 공식 승인을 요구합니다. 저희는 구성을 표준화하기 위해 Git과 같은 도구를 사용하며, CTO 또는 지정된 경영진의 명시적인 승인 없이는 어떤 시스템도 배포되지 않습니다.

## 취약점 및 정보 보안

> **취약점 공개 프로그램은 어떻게 운영되나요?**

Langfuse는 취약점 공개를 위해 [ClickHouse Bugcrowd 프로그램](https://bugcrowd.com/engagements/clickhouse)에 참여하고 있으며, CVSS 점수에 따라 대응 SLA가 결정됩니다. 자세한 내용은 [침투 테스트](/security/penetration-testing)를 참조하세요.

> **정보 보안 정책은 어떻게 구성되어 있나요?**

저희의 정보 보안 정책은 전체 보안 프로그램의 포괄적인 프레임워크로, 모든 회사 및 고객 데이터의 기밀성, 무결성, 가용성(CIA)을 유지하는 것을 핵심 목표로 삼습니다. 이 정책은 모든 임직원, 계약자, 시스템에 적용되며, 리스크 관리, 벤더 관리, 안전한 소프트웨어 개발과 같은 영역을 다루는 더 구체적인 정책들의 포괄적인 모음을 관장합니다. 모든 인력에 대한 주요 요구 사항에는 채용 시 및 이후 매년 실시되는 필수 보안 인식 교육과 기밀 유지 계약 서명이 포함됩니다. 이 정책은 클린 데스크(clean desk) 기준과 업무에 사용되는 개인 기기에 대한 엄격한 보안 설정(필수 디스크 암호화 및 화면 잠금 등)을 포함한 일상적인 보안 관행을 의무화합니다. 전체 보안 프로그램은 최소 연 1회 검토되며, 저희는 지속적인 모니터링과 위반 시 공식적인 징계 절차를 통해 컴플라이언스를 강제합니다.

> **취약점 정책은 어떻게 구성되어 있나요?**

저희는 Snyk와 Dependabot과 같은 자동화된 도구를 사용하여 시스템의 보안 취약점을 지속적으로 모니터링하며, 발견된 모든 사항은 시정 조치를 위해 기록됩니다. 각 취약점은 24시간 이내에 분류(triage)되어 엄격한 서비스 수준 계약(SLA)에 따라 우선순위가 부여되며, 심각한 이슈는 24시간 이내에 수정되어야 합니다. 전체 수명 주기는 문서화되며, 해결은 직접적인 수정, 오탐(false positive)으로의 확인, 또는 완화 통제를 동반한 승인된 예외 처리를 통해 이루어집니다. 자세한 내용은 [취약점 관리](/security/vulnerability-management)를 참조하세요.

> **고객이 자체적으로 침투 테스트를 실행할 수 있나요?**

네. 셀프 호스팅 배포에서 침투 테스트를 실행해 주세요.

## 셀프 호스팅 Langfuse에 대한 고객 통제 및 책임 분담

> **셀프 호스팅 시 어떤 의무가 고객에게 남나요?**

엔드포인트 보안, 웹훅 엔드포인트 강화, 백업, 모니터링, 그리고 여러분 측의 IAM 위생 관리는 여러분의 책임으로 남습니다. 자세한 내용은 [셀프 호스팅 배포 전략](/self-hosting/security/deployment-strategies)을 참조하세요.

> **셀프 호스팅 시 백업 전략은 어떻게 구성해야 하나요?**

데이터 손실 없는 백업(zero dataloss backup)을 가능하게 하는 백업 관련 종합 가이드를 제공하고 있습니다. [여기의 백업 가이드](/self-hosting/configuration/backups)를 참조하세요.

> **Langfuse는 제 VPC 내에 관리형 Langfuse 인스턴스를 제공하나요?**

Enterprise 등급에서는 지원을 제공할 수 있지만, 저희는 고객의 인프라에 설치를 직접 운영하지는 않습니다. 셀프 호스팅이 불가능한 경우 Langfuse Cloud 사용을 권장합니다.

## AI / LLM 관련 우려 사항

> **Langfuse는 프롬프트에 포함된 PII나 영업 비밀을 저장하나요?**

Langfuse는 데이터를 있는 그대로 저장합니다. [데이터 마스킹](/docs/observability/features/masking)을 통해 민감한 데이터를 삭제(redact)할 수 있습니다.

> **장기 보존을 비활성화할 수 있나요?**

네. 기본적으로 데이터는 무기한 저장되지만, 프로젝트별로 사용자 지정 [데이터 보존](/docs/administration/data-retention) 정책을 구성할 수 있습니다.

> **프롬프트/트레이스 데이터가 벤치마킹이나 학습에 사용되는 경우가 있나요?**

아니요. Langfuse는 고객 데이터를 외부 벤치마크나 모델 학습에 재사용하지 않습니다. 선택적 AI 기반 기능이 데이터를 어떻게 처리하는지에 대한 자세한 내용은 [AI 기능](/security/ai-features)을 참조하세요.

> **고객 데이터를 모델이나 분석 학습에 사용하는 경우가 있나요?**

아니요. 고객의 트레이스와 프롬프트는 오직 Langfuse 서비스를 제공하기 위해서만 처리되며, 내부 또는 제3자 ML 모델 학습에는 절대 사용되지 않습니다. 선택적 AI 기반 기능이 데이터를 어떻게 처리하는지에 대한 자세한 내용은 [AI 기능](/security/ai-features)을 참조하세요.

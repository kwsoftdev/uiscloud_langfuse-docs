# 데이터 격리

Langfuse Cloud는 멀티 테넌트 SaaS 애플리케이션입니다. 고객 데이터는 프로젝트 범위, 인증, 인가를 통해 애플리케이션 계층에서 격리됩니다. 이러한 설계를 통해 Langfuse는 완전 관리형 서비스를 제공하면서도 프로젝트와 그 데이터를 서로 분리된 상태로 유지할 수 있습니다.

## 프로젝트 격리 방식

모든 프로젝트는 고유한 `projectId`를 가집니다. 트레이스, observation, 점수, 데이터셋, 프롬프트, 내보내기(export)를 포함한 제품 레코드는 프로젝트와 연결됩니다. 프로젝트 데이터에 접근하는 요청은 프로젝트 컨텍스트를 포함하며, 서버는 데이터를 반환하거나 변경하기 전에 인증된 사용자 또는 API 키가 해당 프로젝트에 접근할 권한이 있는지 확인합니다.

Langfuse는 Langfuse Cloud의 모든 프로젝트마다 별도의 애플리케이션 배포나 데이터베이스를 생성하지 않습니다. 대신 애플리케이션은 데이터 접근에 프로젝트 필터를 일관되게 적용합니다. 이는 많은 멀티 테넌트 SaaS 애플리케이션에서 사용하는 것과 동일한 논리적 격리(logical-isolation) 패턴입니다.

## 사용자 접근 및 RBAC

사용자는 조직 및 프로젝트 멤버십을 통해 접근 권한을 부여받습니다. [역할 기반 접근 제어(RBAC)](/docs/administration/rbac)는 트레이스 조회, 프롬프트 관리, 프로젝트 설정 변경 등 사용자가 수행할 수 있는 작업을 결정합니다. 인가는 보호된 애플리케이션 프로시저와 API 작업에 대해 서버 측에서 강제됩니다.

프로젝트 역할에는 `OWNER`, `ADMIN`, `MEMBER`, `VIEWER`가 있으며, 각 역할에는 특정 리소스와 작업에 대한 권한이 정의되어 있습니다. 따라서 사용자가 한 프로젝트에 접근 권한을 가지고 있다고 해서 동일한 조직 내 다른 프로젝트에 대한 접근 권한이 자동으로 부여되지는 않습니다.

플랫폼 관리자는 별도의 특권 역할이며, 운영 및 지원 목적으로 프로젝트에 접근할 수 있습니다. 고객은 일반적인 사용자 관리에는 프로젝트 역할과 최소 권한 접근 방식을 사용해야 합니다.

## API 및 SDK 접근

프로젝트 범위의 API 및 SDK 키는 특정 프로젝트에 바인딩됩니다. 이러한 키로 인증된 요청은 요청 시점에 임의의 프로젝트를 선택하는 것이 아니라, 해당 프로젝트의 범위 내에서만 동작합니다. 이 키 쌍은 프로젝트의 일반 API 범위에 대해 Basic 인증을 사용하며, Bearer 인증은 점수 제출에만 제한적으로 사용됩니다.

엔터프라이즈 조직은 [조직 범위 API 키](/docs/administration/scim-and-org-api)를 생성할 수도 있습니다. 이 키는 Basic 인증을 사용하며, 프로젝트, 사용자, 프로젝트 또는 조직 멤버십을 관리할 수 있도록 의도적으로 더 넓은 접근 권한을 가집니다. 이 키는 특권 자격 증명으로 취급하고, 조직 관리 워크플로에만 사용을 제한해야 합니다.

API 키는 비공개로 유지하고, 노출되었을 가능성이 있는 경우 즉시 교체(rotate)하세요. 더 세밀한 제어가 필요하다면 프로덕션, 스테이징, 개발 환경이나 서로 다른 팀별로 별도의 프로젝트와 키를 사용하세요.

## 백그라운드 처리 및 지원 서비스

Langfuse는 Postgres, ClickHouse, Redis/Valkey, 오브젝트 스토리지와 같은 지원 서비스를 사용합니다. 이러한 서비스는 Langfuse의 백엔드에서 접근하며 고객에게 직접 노출되지 않습니다.

이러한 서비스 전반에서도 애플리케이션이 테넌트 경계 역할을 유지합니다.

- 데이터베이스 레코드와 분석 쿼리는 해당되는 경우 프로젝트 범위를 사용합니다.
- 큐 및 백그라운드 작업(job) 페이로드는 프로젝트 컨텍스트를 포함하여, 워커가 의도된 프로젝트에 대한 작업만 처리하도록 합니다.
- 내보내기(export) 객체는 저장 경로에 프로젝트 컨텍스트를 포함하며, 시간 제한이 있는 인가된 다운로드 링크를 통해 전달됩니다.
- 서비스 자격 증명은 개별 고객이나 프로젝트가 아닌 Langfuse의 백엔드에서 관리됩니다.

이러한 지원 서비스는 별개의 독립적인 테넌트 인가 계층으로 설명되어서는 안 됩니다. 이들에 대한 접근은 백엔드 서비스, 자격 증명, 프로젝트를 인식하는 애플리케이션 로직, 네트워크 제어, 암호화, 모니터링, 그리고 Langfuse의 전반적인 보안 프로그램을 통해 통제됩니다.

## 공개 데이터 공유

Langfuse는 선택한 트레이스나 세션을 명시적으로 공개(public)로 표시하는 기능을 지원합니다. 공개 접근은 일반적인 멤버십 확인에 대한 의도적인 예외이며, 공유해도 안전한 데이터에 대해서만 활성화해야 합니다. 그 외의 모든 트레이스 및 세션 접근에는 적절한 프로젝트 인가가 필요합니다.

## 테스트 및 검증

Langfuse의 테넌트 격리 경계는 연례 제3자 침투 테스트(penetration testing)에 포함됩니다. Langfuse Cloud는 SOC 2 Type II 및 ISO 27001 감사도 받고 있습니다. 자세한 내용은 [보안 개요](/security), [침투 테스트 정보](/security/penetration-testing), [보안 FAQ](/security/security-faq)를 참고하세요.

## 셀프 호스팅을 선택해야 하는 경우

관리형 운영, 지역별 데이터 거주성(residency), 애플리케이션 수준의 논리적 격리를 원한다면 Langfuse Cloud가 적합한 선택입니다. 요구 사항상 고객이 직접 제어하는 VPC, 팀에서 전적으로 관리하는 프라이빗 연결, 전용 인프라 경계, 또는 고객이 운영하는 데이터베이스 접근 제어가 필요하다면 [셀프 호스팅 Langfuse](/self-hosting)를 고려하세요.

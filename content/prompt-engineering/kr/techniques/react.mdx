# ReAct: 추론과 행동

ReAct(reasoning + acting, 추론 + 행동)는 모델이 다음에 무엇을 해야 하는지 추론하는 것과 이미 알고 있지 않은 정보를 얻기 위해 행동(보통 도구나 함수 호출)을 취하는 것 사이를 번갈아 가며, 학습 과정에서 배운 내용만으로 답하려고 시도하지 않도록 하는 프롬프팅 패턴입니다.

## 일반적인 chain-of-thought만으로는 부족한 이유 [#why-cot-falls-short]

[Chain-of-thought](/prompt-engineering/kr/techniques/chain-of-thought)는 프롬프트에 이미 있는 정보나 모델의 학습 데이터에 대한 추론 능력을 향상시키지만, 현재 사실을 조회하거나 데이터베이스를 질의하거나 실제 세계에서 행동을 취할 수는 없습니다. "서울의 현재 날씨는 어떤가"에 대해 단계별로 추론하는 모델도, 곧바로 답으로 건너뛰는 모델과 마찬가지로 추측할 가능성이 큽니다 — 더 열심히 추론한다고 해서 실제로 조회하는 것을 대신할 수는 없습니다.

## 패턴 [#the-pattern]

각 단계에서 모델은 짧은 추론 진술(무엇을 알아내려 하는지), 행동(인자를 포함한 도구 호출)을 생성한 뒤, 계속 진행하기 전에 관찰 결과(도구의 결과)를 받습니다.

```text
Question: What's the population of the capital city of the country that won
the most recent World Cup?

Thought: I need to find which country most recently won the World Cup.
Action: search("most recent World Cup winner")
Observation: Argentina won the 2022 World Cup.
Thought: Now I need the capital of Argentina.
Action: search("capital of Argentina")
Observation: Buenos Aires.
Thought: Now I need the population of Buenos Aires.
Action: search("population of Buenos Aires")
Observation: approximately 3.1 million (city proper).
Thought: I have enough information to answer.
Answer: approximately 3.1 million.
```

이 루프는 모델이 답할 만큼 충분한 정보를 확보했다고 판단하거나 단계 제한에 도달할 때까지 계속됩니다.

## 도구 호출 대 직접 작성한 ReAct [#tool-calling-vs-hand-written]

대부분의 최신 모델 API는 네이티브 함수/도구 호출을 지원합니다. 이는 스키마로 사용 가능한 도구를 설명하면 모델이 특정한 "Thought/Action/Observation" 텍스트 형식이 아니라 구조화된 호출을 생성하는 방식입니다. 실제로는 프로덕션 환경에서 네이티브 도구 호출이 직접 작성한 ReAct 스타일 프롬프트를 대체로 대체했습니다 — 파싱하기에 더 신뢰할 수 있고, 모델이 정확한 텍스트 패턴을 재현하는 데 의존하지 않기 때문입니다. 근본적인 아이디어는 어느 쪽이든 동일합니다. 모델에게 추측 대신 정보를 찾아볼 이유를 제공하고, 하나의 끊기지 않는 생성이 아니라 도구 결과와 엮인 추론으로 교환 과정을 구성하는 것입니다.

## 효과적인 도구 설명의 조건 [#tool-descriptions]

네이티브 도구 호출을 사용하든 텍스트 기반 패턴을 사용하든, 모델이 도구를 _언제_, _어떻게_ 호출할지에 대한 판단은 도구가 어떻게 설명되어 있는지에 크게 좌우됩니다.

- 도구가 무엇을 하는지, 그리고 그만큼 중요하게 무엇을 하지 않는지를 명확히 명시하세요 — 내부 문서만을 다루는 검색 도구라면 그 사실을 명시해야 합니다. 그렇지 않으면 모델이 일반적인 웹 검색 결과를 기대하며 호출할 수 있습니다.
- 매개변수를 정확하게 설명하세요. 예상되는 형식을 포함해서요(날짜 매개변수라면 단순히 "date"라고 이름 붙이는 대신 예상 형식을 명시해야 합니다).
- 도구가 실패하거나 결과를 반환하지 않을 수 있다면 설명에 그 사실을 언급하세요. 그러면 모델은 빈 결과를 오류로 취급하는 대신 이를 처리할 명확한 방법을 갖게 됩니다.

## 추가할 가치가 있는 안전장치 [#guardrails]

- **단계 제한.** 추론/행동 주기의 횟수를 제한해, 같은 것을 반복해서 검색하는 등 루프에 갇힌 모델이 무한정 실행되는 대신 예측 가능하게 실패하도록 합니다.
- **행동 허용 목록.** 해당 작업에 실제로 필요한 도구만 노출하세요 — 도구 집합이 지나치게 넓으면 잘못되거나 불필요한 도구 호출의 가능성이 커집니다.
- **트레이스에 대한 관찰 가능성(observability).** ReAct는 단일 요청/응답이 아니라 여러 단계로 이루어진 트레이스를 만들어내기 때문에, 모든 thought/action/observation을 트레이싱하는 것이 모델이 _왜_ 잘못된 결론에 도달했는지를 디버깅할 수 있게 해주는 방법입니다 — 에이전트형 프롬프트를 구체적으로 테스트하는 것과 이것이 어떻게 연결되는지는 [프롬프트에 평가가 필요한 이유](/prompt-engineering/kr/evaluation/why-evaluate-prompts)를 참고하세요.

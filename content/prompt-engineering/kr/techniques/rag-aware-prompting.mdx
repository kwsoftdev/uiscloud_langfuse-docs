# RAG-aware 프롬프팅

검색 증강 생성(Retrieval-augmented generation, RAG)은 모델이 학습 과정에서 배운 것 이상의 정보를 활용해 답할 수 있도록, 검색된 문서를 프롬프트에 삽입합니다. 이는 프롬프트가 해야 할 일을 바꿉니다. 더 이상 모델에게 지시만 내리는 것이 아니라, 모델이 직접 작성하지 않았고 신뢰할 본질적인 이유가 없는 텍스트 블록을 어떻게 다룰지도 알려줘야 합니다.

## 검색된 컨텍스트가 무엇인지 모델에게 알려라 [#label-the-context]

검색된 구절은 명확하게 구분되어 참고 자료로 라벨링되어야 하며, 지시문과 뒤섞여서는 안 됩니다.

````text
Answer the question using only the information in the context below. If the
context doesn't contain the answer, say "I don't have enough information to
answer that" rather than guessing.

Context:
​```
{{retrieved_passages}}
​```

Question: {{user_question}}
````

제공된 컨텍스트를 고수하라는 명시적인 지시가 없으면, 검색 결과가 부족하거나 관련이 없을 때 모델은 종종 학습 데이터로 되돌아갑니다 — 이는 애초에 검색을 수행하는 목적을 무너뜨리며, 실제로는 여러분의 문서에 근거하지 않았음에도 자신 있어 보이는 답변을 만들어낼 수 있습니다.

## 관련 없거나 비어 있는 검색 결과를 명시적으로 처리하라 [#handle-empty-retrieval]

검색은 탐색 문제이며, 탐색이 항상 유용한 결과를 반환하는 것은 아닙니다. RAG 프롬프트에는 검색된 컨텍스트가 실제로 질문에 답하지 못할 때 무엇을 해야 하는지에 대한 명시적인 지시가 필요합니다 — 그렇지 않으면 모델은 즉흥적으로 답을 만들어내며, 실제 출처와 나란히 제시되는 즉흥적인 답변은 특히 RAG 애플리케이션에서 더 치명적인 오류 유형 중 하나입니다. 출처가 표시된 잘 정돈된 답변을 읽는 사용자는 그것이 실제로 근거가 있는지 없는지 쉽게 구분할 방법이 없기 때문입니다.

## 인용을 요청하고, 검증 가능하게 만들어라 [#citations]

모델이 어떤 구절이 어떤 주장을 뒷받침하는지 인용하기를 원한다면, 개방형으로 남겨두지 말고 그 방식을 구체적으로 지정하세요.

```text
When you make a claim based on the context, cite the source using [1], [2],
etc. matching the numbered sources below.

[1] {{source_1}}
[2] {{source_2}}
```

번호나 ID 기반의 인용은 모델에게 출처 이름을 인용하거나 바꿔 말하도록 요청하는 것보다 실제 검색 결과 집합에 대해 프로그램적으로 검증하기 더 쉽습니다. 후자는 근본적인 답이 맞더라도 모델이 틀릴 수 있습니다.

## 검색된 구절의 순서와 개수가 중요하다 [#order-and-quantity]

- 긴 컨텍스트에서 앞이나 뒤에 배치된 구절은 중간에 묻힌 구절보다 더 많은 주의를 받는 경향이 있습니다 — 검색 결과를 관련성에 따라 순위를 매긴다면, 위치가 중립적이라고 가정하지 말고 가장 관련성 높은 구절을 처음, 마지막, 혹은 둘 다에 배치해야 할지 고려해볼 가치가 있습니다.
- 검색된 컨텍스트가 많다고 해서 자동으로 더 나은 것은 아닙니다. 관련 없는 구절이 관련 있는 구절과 섞여 있으면 모델의 주의를 분산시키고 혼합되거나 근거 없는 답변이 나올 가능성을 높일 수 있으므로, 실제 데이터에서 더 작고 정밀도가 높은 검색 결과 집합이 더 크고 노이즈가 많은 집합보다 나은 성능을 내는지 테스트해볼 가치가 있습니다.

## RAG의 실패는 흔히 검색의 실패이지, 프롬프팅의 실패가 아니다 [#retrieval-vs-prompting]

RAG 답변이 틀렸을 때는 프롬프트를 다시 작성하기 전에, 매우 다른 두 가지 원인을 구분할 가치가 있습니다. 검색 단계가 애초에 올바른 구절을 표면화하는 데 실패한 것인지, 아니면 모델이 올바른 구절을 가지고 있었음에도 여전히 잘못 답한 것인지입니다. [전체 파이프라인을 트레이싱](/docs/observability/overview)하는 것 — 무엇이 검색되었는지, 어떤 순서로, 그리고 모델이 그것으로 무엇을 했는지 — 이 어느 쪽을 고쳐야 할지 짐작하는 대신 이 구분을 가능하게 만들어줍니다.

# 인증 및 SSO (셀프 호스팅)

Langfuse는 셀프 호스팅 인스턴스에서 이메일/비밀번호 인증과 SSO 인증을 모두 지원합니다.

<Callout>
  이메일/비밀번호 이외의 인증 방식을 사용하려면 `NEXTAUTH_URL` 환경 변수가 [올바르게
  구성](/self-hosting/configuration)되어 있는지 확인하세요.
</Callout>

<Callout type="info">
  Langfuse Cloud를 사용 중이라면 [Cloud 인증 및 SSO](/docs/administration/authentication-and-sso) 문서를 참고하세요.
</Callout>

## 이메일/비밀번호 [#auth-email-password]

이메일/비밀번호 인증은 기본적으로 활성화되어 있습니다. 사용자는 이메일과 비밀번호로 가입하고 로그인할 수 있습니다.

### 비밀번호 재설정

인스턴스에 **[트랜잭션 이메일](/self-hosting/configuration/transactional-emails)이 구성되어 있다면**, 사용자는 로그인 페이지의 "Forgot password" 링크를 통해 비밀번호를 재설정할 수 있습니다.

**트랜잭션 이메일이 설정되어 있지 않다면**, 다음 단계에 따라 비밀번호를 재설정할 수 있습니다.

    1.  데이터베이스에서 해당 사용자 계정에 연결된 이메일을 (예: 접두사를 추가하는 방식으로) 변경합니다.
    2.  이후 새 비밀번호로 다시 가입할 수 있습니다.
    3.  데이터베이스의 `organization_memberships` 테이블을 통해 기존에 속해 있던 조직을 다시 연결합니다.
    4.  마지막으로 데이터베이스의 `users` 테이블에서 기존 사용자 계정을 삭제합니다.

### SSO를 사용하기 위해 이메일/비밀번호 인증 비활성화

이메일/비밀번호 인증을 비활성화하려면 `AUTH_DISABLE_USERNAME_PASSWORD=true`로 설정하세요. 이 경우 대신 [SSO](#sso)를 설정해야 합니다.

운영 중인 인스턴스에서 이메일/비밀번호 방식에서 SSO로 전환하기로 결정했다면, 해당 SSO 제공업체에 `AUTH_<PROVIDER>_ALLOW_ACCOUNT_LINKING=true`를 활성화할 수 있습니다. 이렇게 하면 동일한 이메일 주소를 가진 계정이 자동으로 병합됩니다.

### 가입 시 이메일 인증 [#email-verification-on-signup]

`AUTH_EMAIL_VERIFICATION_REQUIRED=true`로 설정하면, 새 사용자가 이메일/비밀번호 가입 과정에서 비밀번호를 설정하기 전에 일회용 비밀번호(OTP)를 통해 이메일 주소를 인증하도록 요구할 수 있습니다. 이 기능을 사용하려면 [트랜잭션 이메일](/self-hosting/configuration/transactional-emails)이 구성되어 있어야 합니다.

이 설정을 활성화하면 가입 흐름은 다음과 같이 진행됩니다.

1.  사용자가 가입 페이지에서 이메일과 이름을 입력합니다.
2.  사용자가 이메일로 OTP를 받아 이를 입력하여 주소를 인증합니다.
3.  사용자가 `/auth/setup-password`에서 비밀번호를 설정하고 로그인됩니다.

참고 사항:

- SSO 로그인은 이 설정의 영향을 받지 않습니다.
- 이 플래그가 켜져 있으면 `POST /api/auth/signup`을 직접 호출할 경우 `403`을 반환합니다. 가입은 반드시 OTP 흐름을 거쳐야 합니다.
- 인증 완료 후 사용자가 흐름을 중단하면, 다시 가입을 시도할 때 OTP가 재전송되며 비밀번호 재설정 페이지를 통해서도 흐름을 완료할 수 있습니다.

### 기본 사용자 생성

기본 사용자를 프로그래밍 방식으로 생성하려면 [헤드리스 초기화](/self-hosting/administration/headless-initialization) 문서를 참고하세요. 이는 UI를 사용하지 않고 인스턴스를 초기화하고자 할 때, 예를 들어 CI/CD 파이프라인에서 Langfuse를 실행하거나 여러 환경에 프로그래밍 방식으로 Langfuse를 배포할 때 유용합니다.

## SSO

Langfuse에서 OAuth/SSO 제공업체 로그인을 활성화하려면 해당 제공업체에 필요한 환경 변수를 구성하세요.

동일한 이메일 주소를 가진 계정을 병합하려면 `AUTH_<PROVIDER>_ALLOW_ACCOUNT_LINKING`을 사용하세요. 이는 사용자가 서로 다른 제공업체나 이메일/비밀번호로 로그인하지만 동일한 이메일 주소를 사용하는 경우에 유용합니다. 이메일이 검증되지 않은 경우 보안 문제가 발생할 수 있으므로 이 설정을 사용할 때는 주의해야 합니다.

다른 제공업체가 필요하신가요? Langfuse는 Auth.js를 사용하며, 이는 [다양한 제공업체](https://next-auth.js.org/providers/)와 통합됩니다. 특정 제공업체 지원을 원하시면 [GitHub에 기능 요청](/ideas)을 등록해 주세요.

### Auth0

[NextAuth Auth0 제공업체 문서](https://next-auth.js.org/providers/auth0)

| 구성               | 값                                                                            |
| ------------------ | ----------------------------------------------------------------------------- |
| 필수 변수          | `AUTH_AUTH0_CLIENT_ID`<br/>`AUTH_AUTH0_CLIENT_SECRET`<br/>`AUTH_AUTH0_ISSUER` |
| 선택 변수          | 아래 [추가 구성](#additional-configuration) 섹션을 참고하세요.                |
| OAuth 리디렉션 URL | `/api/auth/callback/auth0`                                                    |

제공업체: `AUTH0`

### Authentik

[NextAuth Authentik 제공업체 문서](https://next-auth.js.org/providers/authentik)

| 구성               | 값                                                                                                         |
| ------------------ | ---------------------------------------------------------------------------------------------------------- |
| 필수 변수          | `AUTH_AUTHENTIK_CLIENT_ID`<br/>`AUTH_AUTHENTIK_CLIENT_SECRET`<br/>`AUTH_AUTHENTIK_ISSUER`                  |
| 선택 변수          | `AUTH_AUTHENTIK_AUTHORIZATION_URL`<br/><br/>아래 [추가 구성](#additional-configuration) 섹션을 참고하세요. |
| OAuth 리디렉션 URL | `/api/auth/callback/authentik`                                                                             |

제공업체: `AUTHENTIK`

참고 사항:

- `AUTH_AUTHENTIK_ISSUER`는 `https://<domain>/application/o/<slug>` 형식이어야 하며, 끝에 슬래시를 추가하는 것은 선택 사항입니다.
- **인가 URL 재정의:** Authentik 인스턴스가 내부용 URL과 외부용 URL을 다르게 사용하는 경우(예: 스플릿 DNS 또는 리버스 프록시 구성) 브라우저에서 접근 가능한 Authentik URL을 `AUTH_AUTHENTIK_AUTHORIZATION_URL`에 설정하세요. 이슈어 URL은 서버 측 토큰, userinfo, JWKS 검증에 계속 사용되며, 인가 URL 재정의는 브라우저 리디렉션에만 사용됩니다.

## SSO용 HTTP 프록시

Langfuse는 SSO 제공업체를 위한 HTTP 프록시 설정을 지원합니다.
SSO 제공업체 요청에 프록시를 사용하려면 `AUTH_HTTPS_PROXY` 또는 `AUTH_HTTP_PROXY`를 구성하세요.

## IdP 주도 SSO [#idp-initiated-sso]

Langfuse는 **IdP 주도 SSO**(Identity Provider-initiated Single Sign-On)를 지원합니다. 이를 통해 사용자는 Langfuse가 아닌 자신의 아이덴티티 제공업체(예: Okta, Azure AD, Keycloak, JumpCloud 등)에서 SSO 흐름을 직접 시작할 수 있습니다.

IdP 주도 SSO를 활성화하려면 아이덴티티 제공업체가 사용자를 다음 URL로 리디렉션하도록 구성하세요.

```
https://<YOUR_LANGFUSE_URL>/auth/sso-initiate?provider=<PROVIDER>
```

- `<YOUR_LANGFUSE_URL>`을 여러분의 Langfuse 인스턴스 URL로, `<PROVIDER>`를 구성한 제공업체 이름(예: `GOOGLE`, `GITHUB`, `AZURE_AD` 등, 위의 다른 제공업체 변수 참고)으로 바꾸세요.
- 아이덴티티 제공업체 설정에서 `Redirect to app to initiate login (OIDC Compliant)` 옵션을 사용하세요.
- IdP 주도 SSO는 Langfuse 버전 `>=v3.126.0`부터 사용할 수 있습니다.

## 추가 구성

다음은 추가 구성 변수입니다. `<PROVIDER>`를 제공업체 이름(예: `GOOGLE`, `GITHUB`, `AZURE_AD` 등, 위의 다른 제공업체 변수 참고)으로 바꾸세요.

| 변수                                           | 설명                                                                                                                                                                                                                                                                                        |
| ---------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `AUTH_DOMAINS_WITH_SSO_ENFORCEMENT`            | SSO로만 로그인이 허용되는 도메인의 쉼표로 구분된 목록입니다. 이 도메인들은 이메일/비밀번호 로그인이 비활성화됩니다. 예: `domain1.com,domain2.com`                                                                                                                                           |
| `AUTH_DISABLE_SIGNUP`                          | `true`로 설정하면 신규 사용자 가입이 비활성화됩니다. 기존 사용자만 로그인할 수 있습니다. 이는 아직 계정이 없는 상태에서 프로젝트 초대를 받은 사용자를 포함해, 가입을 시도하는 모든 신규 사용자에게 영향을 줍니다.                                                                           |
| `AUTH_SESSION_MAX_AGE`                         | 세션(JWT)의 최대 유효 기간을 분 단위로 설정합니다. 기본값은 30일(`43200`)입니다. 프론트엔드 애플리케이션이 5분마다 세션을 갱신하므로 이 값은 5분보다 커야 합니다.                                                                                                                           |
| `AUTH_IGNORE_ACCOUNT_FIELDS`                   | 계정을 생성할 때 SSO IDP 계정에서 무시할 필드의 쉼표로 구분된 목록입니다. 커스텀 IDP 제공업체의 오류를 수정할 때 사용하세요.                                                                                                                                                                |
| `AUTH_<PROVIDER>_ALLOW_ACCOUNT_LINKING`        | `true`로 설정하면 동일한 이메일 주소를 가진 계정을 병합할 수 있습니다. 이는 사용자가 서로 다른 제공업체나 이메일/비밀번호로 로그인하지만 동일한 이메일 주소를 사용하는 경우에 유용합니다. 이메일이 검증되지 않은 경우 보안 문제가 발생할 수 있으므로 이 설정을 사용할 때는 주의해야 합니다. |
| `AUTH_<PROVIDER>_CLIENT_AUTH_METHOD`           | 토큰 엔드포인트 인증 방식을 구성합니다. 지원되는 값: `client_secret_basic`(기본값), `client_secret_post`, `client_secret_jwt`, `private_key_jwt`, `tls_client_auth`, `self_signed_tls_client_auth`, `none`. PKCE 흐름에는 `none`을 사용하세요.                                              |
| `AUTH_<PROVIDER>_CHECKS`                       | 인증 검사 방식을 구성합니다. 지원되는 값: `nonce`, `none`, `pkce`, `state`. 여러 값을 쉼표로 구분하여 지정할 수 있습니다.                                                                                                                                                                   |
| `AUTH_<PROVIDER>_ID_TOKEN_SIGNED_RESPONSE_ALG` | 제공업체가 ID 토큰에 서명할 때 사용하는 알고리즘을 구성합니다. 지원되는 값: `RS256`, `RS384`, `RS512`, `ES256`, `ES384`, `ES512`, `PS256`, `PS384`, `PS512`, `HS256`, `HS384`, `HS512`. 이 설정은 GitHub, WorkOS 제공업체에서는 무시됩니다.                                                 |

## 문제 해결

- 이메일/비밀번호 이외의 인증 방식을 사용하려면 `NEXTAUTH_URL` 환경 변수가 [올바르게 구성](/self-hosting/configuration)되어 있는지 확인하세요.
- 오류: _"Please sign in with the same provider that you used to create this account"._ 이 오류는 계정을 생성할 때 사용한 제공업체와 다른 제공업체로 로그인을 시도할 때 발생합니다. 이를 해결하려면 계정을 생성할 때 사용한 제공업체로 로그인하거나, `AUTH_<PROVIDER>_ALLOW_ACCOUNT_LINKING=true`를 설정하여 계정 연결/전환을 허용해야 합니다(환경 변수명은 제공업체에 따라 다릅니다. 위 내용 참고).
- 비밀번호 재설정에 대해서는 [비밀번호 재설정](#password-reset) 섹션을 참고하세요.
- Langfuse 인증은 이메일 주소를 기반으로 합니다. SSO를 사용할 때는 사용자의 IDP 프로필에 이메일 주소가 포함되어 있는지 확인하세요.
- 제공업체와의 인증이 실패하면, 위에서 설명한 `CLIENT_AUTH_METHOD`와 `CHECKS`의 다양한 조합을 시도해 보세요.

## 관련 자료

- SSO를 구성한 후 사용자 프로비저닝, 역할 할당, 프로젝트 설정을 자동화하려면 [SCIM & Organization API](/docs/administration/scim-and-org-api)를 참고하세요.

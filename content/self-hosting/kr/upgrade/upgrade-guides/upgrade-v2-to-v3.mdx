# Langfuse v2에서 v3로 마이그레이션

<Callout type="info">

이번 업그레이드는 규모가 크며, 저희는 이를 최대한 원활하게 만들기 위해 노력했습니다. v3로 업그레이드하는 과정에서 궁금한 점이 있다면 [GitHub Issue](/issues)를 생성하거나 [지원팀](/support)에 문의해 주세요.

</Callout>

Langfuse v3(2024년 12월 6일 출시)는 새로운 백엔드 아키텍처를 도입하여 다양한 신규 기능과 성능 개선을 제공합니다.

이 가이드를 따라 다음을 수행하세요:

1. 아키텍처 변경 사항과 그 배경 이유를 이해합니다.
2. 그 밖의 호환성이 깨지는 변경 사항(breaking changes)에 대해 알아봅니다.
3. 업그레이드 단계를 따라 Langfuse v3로 성공적으로 마이그레이션합니다.

Langfuse v3의 아키텍처에 대해 더 알아보려면 [아키텍처 개요](/self-hosting#architecture-overview)로 이동하세요.

## 그 밖의 호환성이 깨지는 변경 사항

<Callout type="info">

버전 2.0.0(2023년 12월 출시) 이상의 Langfuse SDK를 사용하고 있다면 이러한 변경 사항의 영향을 받지 않습니다. Langfuse 팀은 소수의 팀(전체 사용자의 1% 미만)이 Langfuse SDK를 업그레이드하도록 도운 후, 아무런 문제 없이 이미 Langfuse Cloud를 v3로 업그레이드했습니다.

</Callout>

### SDK 요구 사항

**SDK v1.x.x는 더 이상 지원되지 않습니다**. 저희는 SDK와 API의 하위 호환성을 최대한 유지하려고 노력하지만, Langfuse Server v3로 업데이트하면서 호환성이 깨지는 변경 사항을 도입할 수밖에 없었습니다. 버전 2.0.0 미만의 SDK에 있는 특정 API는 새로운 백엔드 아키텍처와 호환되지 않습니다.

#### SDK v2 출시일

- Langfuse Python SDK v2는 2023년 12월 17일에 [출시](https://github.com/langfuse/langfuse-python/releases/tag/v2.0.1)되었습니다.
- Langfuse JavaScript SDK v2는 2023년 12월 18일에 [출시](https://github.com/langfuse/langfuse-js/releases/tag/v2.0.0)되었습니다.

#### SDK 버전 1.x.x를 사용 중인 경우의 업그레이드 옵션

- 기본 SDK 업그레이드: 1.x.x에서 2.x.x로의 업그레이드 경로([Python](/docs/sdk/python/low-level-sdk#upgrading-from-v1xx-to-v2xx), [JavaScript](/docs/sdk/typescript/guide#upgrade1to2))를 따르세요. JavaScript SDK의 경우 [2.x.x에서 3.x.x로의 업그레이드](/docs/sdk/typescript/guide#upgrade2to3)도 함께 고려하세요. 업그레이드 과정은 간단하며 많은 시간이 걸리지 않습니다.
- 선택적으로 [새로운 통합](/docs/get-started)으로 전환하기: 첫 번째 메이저 버전 이후, 저희는 Python용 [데코레이터(Decorators)](/docs/sdk/python/decorators)와 같이 코드를 Langfuse와 통합하는 다양한 새로운 방법을 구축했습니다. 더 편리한 통합 방법이 있는지 확인하려면 [퀵스타트](/docs/get-started)를 확인해 보시길 권장합니다.

#### 이번 변경의 배경

Langfuse v3는 이벤트 기반 백엔드 아키텍처에 의존합니다.
이는 SDK로부터 온 HTTP 요청을 승인(acknowledge)한 뒤, HTTP 바디를 백엔드에서 큐에 넣고 비동기적으로 처리한다는 의미입니다.
이를 통해 데이터베이스에 과부하를 주지 않으면서도 백엔드를 더 쉽게 확장하고 더 많은 요청을 처리할 수 있습니다.
2.0.0 미만의 SDK는 서버로 이벤트를 전송하고, 해당 이벤트의 데이터베이스 표현을 담은 동기식 응답을 기대합니다.
만약 이 데이터에 의존하여 코드에서 접근하고 있다면, 클라우드 버전의 경우 2024년 11월 11일부터, 셀프 호스팅의 경우 Langfuse v3로 업그레이드한 이후부터 해당 SDK는 정상적으로 동작하지 않게 됩니다.

### API 변경 사항

#### POST /api/public/ingestion

`/api/public/ingestion` 엔드포인트는 이제 비동기적으로 동작합니다.
들어오는 모든 이벤트를 수락하여 처리를 위해 큐에 넣은 뒤 207 상태 코드를 반환합니다.
즉, 이벤트는 백엔드에 수락된 직후 곧바로 사용할 수 있는 것이 _아니라_ 이후의 읽기 요청에서 순차적으로 나타나게 됩니다.

데이터 저장소를 Postgres에서 Clickhouse로 전환하면서, 일부 필드에 대한 업데이트 동작을 제거해야 했습니다. 아래에 나열된 모든 필드는 더 이상 업데이트할 수 없으며, 동일한 `id`에 대한 어떤 이벤트에서든 해당 필드가 설정된 경우에만 새로운 레코드가 데이터베이스에 기록됩니다.

- `observation.startTime`
- `observation.type`
- `observation.traceId`
- `trace.timestamp`
- `score.timestamp`
- `score.traceId`

개별 이벤트는 바디 내에 `string | string[] | Record<string, unknown>` 타입의 `metadata` 속성을 허용했습니다.
저희 UI와 엔드포인트에서 쿼리 및 이벤트 필터링을 수행할 때는 `Record` 타입만 지원됩니다.
따라서 앞으로는 `metadata`에 대해 객체(object) 타입을 강제합니다.
`{ metadata: string | string[] }` 형태로 들어오는 모든 이벤트는 `metadata`라는 키를 가진 객체로 매핑됩니다.
예를 들어 `{ event: { body: { metadata: "test" } } }`는 `{ event: { body: { metadata: { metadata: "test" } } } }`로 변환됩니다.

#### POST /api/public/scores

`/api/public/scores` 엔드포인트는 이제 비동기적으로 동작합니다.
`/api/public/ingestion` 엔드포인트와 정확히 동일하게 동작하지만, `{ id: string }` 타입의 바디와 함께 200 상태 코드를 반환합니다.
이전에는 이 엔드포인트가 생성된 점수 객체를 직접 반환했습니다.
이 변경 사항은 [API 레퍼런스](https://api.reference.langfuse.com/#tag/score/post/api/public/scores)와 일치하므로 호환성을 깨는 변경으로 간주되지 않습니다.

#### 지원 중단(Deprecated)된 엔드포인트

아래 엔드포인트는 v2 릴리스 이후 지원이 중단되었으며, 이에 따라 2024년 2월 이후로는 Langfuse SDK에서 사용되지 않고 있습니다.
Langfuse v3는 이 엔드포인트들로의 요청을 계속 수락합니다.
이들의 API 동작은 비동기 방식으로 변경되며, 생성된 전체 레코드 대신 생성된 객체의 id만 반환합니다.
이 엔드포인트들은 향후 릴리스에서 제거될 예정이니 유의하시기 바랍니다.

- POST /api/public/events
- POST /api/public/generations
- PATCH /api/public/generations
- POST /api/public/spans
- PATCH /api/public/spans
- POST /api/public/traces

### UI 동작 변경 사항

#### 트레이스 삭제

Langfuse UI 내에서 트레이스를 삭제하는 작업은 동기식 작업이었으며 트레이스가 즉시 제거되었습니다.
이제부터는 모든 트레이스가 삭제 예약(scheduled)되며, 짧은 기간 동안은 UI에 계속 표시될 수 있습니다.

#### 프로젝트 삭제

Langfuse 내에서 프로젝트를 삭제하는 작업은 동기식 작업이었으며 프로젝트가 즉시 제거되었습니다.
Langfuse v3에서는 프로젝트가 삭제된 것으로 표시되며 일반적인 UI 탐색 방법으로는 접근할 수 없게 됩니다.
프로젝트의 액세스 키는 즉시 폐기되지만, 나머지 데이터는 백그라운드에서 제거됩니다.

[S3/Blob Store](/self-hosting/deployment/infrastructure/blobstorage)에 저장된 정보는 삭제되지 않습니다.
이 작업은 관리자가 수동으로 수행해야 합니다.
이 프로세스는 향후 릴리스에서 자동화될 예정입니다.

## 마이그레이션 단계

문제가 발생하면 [지원팀](/support)에 문의하거나 [GitHub에 이슈](/issues)를 열어 주세요.

### 업그레이드를 시작하기 전에

업그레이드를 시작하기 전에, 해당 [v3 배포 가이드](/self-hosting)의 내용을 숙지했는지 확인하세요.
또한 업그레이드를 시작하기 전에 Postgres 데이터베이스를 백업할 것을 권장합니다.
그리고 Langfuse의 최신 버전, 가급적 v2.92.0 이후 버전을 사용하고 있는지 확인하세요.

다운타임 없는 업그레이드를 위해서는, Langfuse web 및 worker 컨테이너의 새 인스턴스를 프로비저닝하고, 새 인스턴스가 예상대로 동작하는지 검증한 후 트래픽을 이전할 것을 권장합니다.

<Callout type="warning">
  
다운타임 없는 업그레이드를 진행하는 경우, 트래픽을 새 인스턴스로 전환할 때까지 백그라운드 마이그레이션을 비활성화할 것을 권장합니다. 그렇지 않으면 새 인스턴스가 시작된 이후에 수집된 이벤트를 마이그레이션이 놓칠 수 있습니다. 트래픽이 전환될 때까지 새 Langfuse web 및 worker 컨테이너의 환경 변수에서 `LANGFUSE_ENABLE_BACKGROUND_MIGRATIONS=false`를 설정하세요. 이후에는 이 오버라이드를 제거하거나 `true`로 설정하세요.

또한 이 업그레이드는 v2.92.0과 v3.29.0 사이에서 잘 동작하는 것으로 알려져 있습니다.
v3의 최신 버전들은 v2가 여전히 의존하는 데이터베이스 엔티티를 제거합니다.
따라서 병렬 운영 시에는 v3.29.0을 사용하고, 마이그레이션이 완료된 이후에 최신 v3로 업그레이드할 것을 권장합니다.

</Callout>

### 업그레이드 단계

<Steps>

#### 1. 새 인프라 프로비저닝

필요한 모든 스토리지 구성 요소([Clickhouse](/self-hosting/deployment/infrastructure/clickhouse), [Redis](/self-hosting/deployment/infrastructure/cache), [S3/Blob Store](/self-hosting/deployment/infrastructure/blobstorage))를 배포하고 연결 정보를 준비해 두었는지 확인하세요.
새 배포에는 기존 Postgres 인스턴스를 재사용할 수 있습니다.
Postgres 연결 정보도 준비되어 있는지 확인하세요.

새로운 Langfuse web 및 worker 컨테이너에는 다음의 새로운 환경 변수가 필요합니다.
이를 제공하지 않으면 배포가 실패합니다.

- `CLICKHOUSE_URL`
- `CLICKHOUSE_USER`
- `CLICKHOUSE_PASSWORD`
- `CLICKHOUSE_MIGRATION_URL`
- `REDIS_CONNECTION_STRING` (또는 그 대체 변수)
- `LANGFUSE_S3_EVENT_UPLOAD_BUCKET`

#### 2. 새 Langfuse 컨테이너 시작

셀프 호스팅 가이드의 설정에 따라 Langfuse web 및 worker 컨테이너를 배포하세요.
새로운 스토리지 구성 요소에 대한 환경 변수와 Postgres 연결 정보를 설정했는지 확인하세요.

이 시점에서 새 Langfuse 인스턴스를 테스트해 볼 수 있습니다.
UI는 정상적으로 로드되지만, 트레이스, observation, 또는 점수는 아직 표시되지 않아야 합니다.
이는 정상적인 현상으로, 해당 요소들이 여전히 Postgres에 남아 있는 동안 데이터는 Clickhouse에서 읽어오기 때문입니다.

#### 3. v2에서 v3로 트래픽 전환

DNS 레코드나 로드밸런서 설정을 업데이트하여 트래픽을 새 Langfuse 인스턴스로 전환하세요.
모든 새 이벤트는 Clickhouse에 저장되며, 수집된 지 몇 초 이내에 UI에 표시되어야 합니다.

#### 4. 과거 데이터 마이그레이션 완료 대기

v3로의 마이그레이션 과정에서 [백그라운드 마이그레이션](/self-hosting/upgrade/background-migrations) 기능을 도입했습니다.
이를 통해 Langfuse는 서비스 가용성에 영향을 주지 않으면서 오래 걸리는 마이그레이션을 예약할 수 있습니다.
v3 릴리스의 일환으로, 새 스택을 배포하면 실행되는 네 가지 마이그레이션을 도입했습니다.

1. **비용 백필(Cost backfill)**: 모든 이벤트에 대한 비용을 계산하여 Postgres 데이터베이스에 저장합니다. 이전에는 이를 읽기 시점에 계산했기 때문에 읽기 성능에 부정적인 영향을 미쳤습니다.
2. **트레이스 마이그레이션**: 모든 트레이스를 배치 단위로 Postgres에서 Clickhouse로 마이그레이션합니다. 가장 최근의 트레이스부터 시작하므로, 마이그레이션 시작 직후 대시보드에서 확인할 수 있습니다.
3. **Observation 마이그레이션**: 모든 observation을 배치 단위로 Postgres에서 Clickhouse로 마이그레이션합니다. 가장 최근의 observation부터 시작하므로, 마이그레이션 시작 직후 대시보드에서 확인할 수 있습니다.
4. **점수 마이그레이션**: 모든 점수를 배치 단위로 Postgres에서 Clickhouse로 마이그레이션합니다. 가장 최근의 점수부터 시작하므로, 마이그레이션 시작 직후 대시보드에서 확인할 수 있습니다.

각 마이그레이션은 다음 마이그레이션이 시작되기 전에 완료되어야 합니다.
이벤트 테이블의 크기에 따라 이 과정은 여러 시간이 걸릴 수 있습니다.

문제가 발생하면 [백그라운드 마이그레이션 가이드](/self-hosting/upgrade/background-migrations#troubleshooting)의 문제 해결 섹션을 확인해 주세요.

#### 5. 기존 Langfuse 컨테이너 중지

새 이벤트가 Clickhouse에 저장되고 UI에 표시되는 것을 확인한 후, 기존 Langfuse 컨테이너를 중지할 수 있습니다.

</Steps>

## 배포 유형별 가이드

이 섹션에서는 업그레이드 과정에 도움이 되는 배포 유형별 가이드를 모아 두었습니다.
문제가 발생하거나 가이드가 부족하다고 느껴지면 자유롭게 가이드를 기여하거나 GitHub 이슈를 생성해 주세요.

### Docker Compose

Docker Compose를 통한 업그레이드에서는 서비스가 재시작되는 동안 짧은 다운타임이 발생해도 괜찮다고 가정합니다.
다운타임 없는 업그레이드를 원한다면, 일반 업그레이드 가이드에 설명된 단계를 따르면 됩니다.
외부 Postgres 인스턴스를 사용하는 경우에도 일반 가이드를 따르고, 해당 Postgres 인스턴스를 가리키는 새 v3 배포를 시작하기만 하면 됩니다.

<Steps>

#### 1. Postgres 인스턴스의 볼륨 이름 확인

[Langfuse 저장소](https://github.com/langfuse/langfuse/blob/v2/docker-compose.yml)의 docker-compose.yml을 사용해 Langfuse v2를 배포했다고 가정합니다.
데이터베이스의 볼륨 설정(아래 예시의 `database_data` 등)을 기록해 두세요.

```yaml
volumes:
  database_data:
    driver: local
```

#### 2. v3의 docker-compose.yml 복사본 생성

로컬 머신에 Langfuse v3의 [docker-compose 파일](https://github.com/langfuse/langfuse/blob/main/docker-compose.yml) 복사본을 생성하세요.
`langfuse_postgres_data` 볼륨 이름을 1단계에서 기록해 둔 이름으로 교체하세요.
`volumes` 섹션과 `postgres` 섹션을 모두 업데이트했는지 확인하세요.

#### 3. Langfuse v2 배포 중지 (다운타임 시작)

`docker compose down`을 실행하여 Langfuse v2 배포를 중지하세요.
모든 볼륨을 유지해야 하므로 `-v`는 _추가하지 않도록_ 주의하세요.

#### 4. Langfuse v3 배포 시작 (다운타임 종료)

`docker compose -f docker-compose.v3.yml up -d`를 실행하여 Langfuse v3 배포를 시작하세요.
`docker-compose.v3.yml`은 2단계에서 생성한 파일 이름으로 교체해야 합니다.
새 컨테이너가 실행되는 즉시 트레이스 수집과 다른 기능을 사용할 수 있습니다.
배포는 백그라운드에서 v2 배포로부터 데이터를 마이그레이션하므로, 한동안 UI에서 일부 데이터가 누락된 것처럼 보일 수 있습니다.

</Steps>

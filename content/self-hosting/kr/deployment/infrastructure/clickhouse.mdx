# ClickHouse

<Callout type="info">

이 문서는 ClickHouse 구성에 대해 자세히 다룹니다. 시작하려면 [배포 가이드](/self-hosting#deployment-options) 중 하나를 따라 진행하세요.

</Callout>

[ClickHouse](https://github.com/ClickHouse/ClickHouse)는 Langfuse에서 Trace, Observation, Score 엔티티를 위한 주요 OLAP 스토리지 솔루션입니다.
높은 쓰기 처리량과 빠른 분석 쿼리에 최적화되어 있습니다.
이 가이드는 Langfuse 내에서 ClickHouse를 구성하는 방법과 (선택적으로) 자체 ClickHouse를 사용할 때 유의해야 할 사항을 다룹니다.

Langfuse는 ClickHouse 버전 24.3 이상을 지원합니다.

## 구성

Langfuse는 ClickHouse 사용을 세밀하게 조정할 수 있도록 다음 환경 변수를 지원합니다.
이 변수들은 Langfuse Web 및 Langfuse Worker 컨테이너에 제공되어야 합니다.

| 변수                                          | 필수 여부 / 기본값 | 설명                                                                                                                                                                                                                                                                                                                                                                                                                 |
| --------------------------------------------- | ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `CLICKHOUSE_MIGRATION_URL`                    | 필수               | ClickHouse 인스턴스에 대한 마이그레이션 URL (TCP 프로토콜)입니다. 패턴: `clickhouse://<hostname>:(9000/9440)`                                                                                                                                                                                                                                                                                                        |
| `CLICKHOUSE_MIGRATION_SSL`                    | `false`            | 데이터베이스 마이그레이션 시 ClickHouse와 SSL 연결을 맺으려면 true로 설정합니다.                                                                                                                                                                                                                                                                                                                                     |
| `CLICKHOUSE_URL`                              | 필수               | ClickHouse 인스턴스의 호스트명입니다. 패턴: `http(s)://<hostname>:(8123/8443)`                                                                                                                                                                                                                                                                                                                                       |
| `CLICKHOUSE_USER`                             | 필수               | ClickHouse 데이터베이스의 사용자 이름입니다. SELECT, ALTER, INSERT, CREATE, DELETE 권한이 필요합니다.                                                                                                                                                                                                                                                                                                                |
| `CLICKHOUSE_PASSWORD`                         | 필수               | ClickHouse 사용자의 비밀번호입니다.                                                                                                                                                                                                                                                                                                                                                                                  |
| `CLICKHOUSE_DB`                               | `default`          | 사용할 ClickHouse 데이터베이스 이름입니다.                                                                                                                                                                                                                                                                                                                                                                           |
| `CLICKHOUSE_CLUSTER_ENABLED`                  | `true`             | ClickHouse 명령을 `ON CLUSTER`로 실행할지 여부입니다. 단일 컨테이너 구성에서는 `false`로 설정합니다.                                                                                                                                                                                                                                                                                                                 |
| `LANGFUSE_AUTO_CLICKHOUSE_MIGRATION_DISABLED` | `false`            | 자동 ClickHouse 마이그레이션을 비활성화할지 여부입니다.                                                                                                                                                                                                                                                                                                                                                              |
| `CLICKHOUSE_READ_ONLY_URL`                    |                    | 퍼블릭 API 읽기 및 일부 UI/필터 읽기 쿼리에 사용되는 선택적 읽기 전용 엔드포인트입니다. 설정하지 않으면 `CLICKHOUSE_URL`을 사용합니다. `CLICKHOUSE_USER`, `CLICKHOUSE_PASSWORD`, `CLICKHOUSE_DB`를 재사용합니다. [compute-compute가 분리된](https://clickhouse.com/docs/cloud/reference/warehouses) 클러스터에서만 유용합니다. [Scaling](/self-hosting/configuration/scaling#clickhouse-read-only-url)을 참고하세요. |

CLICKHOUSE_CLUSTER_ENABLED가 `true`로 설정되어 있으면 Langfuse는 클러스터 이름으로 `default`를 사용합니다.
`CLICKHOUSE_CLUSTER_NAME`을 다른 값으로 설정하여 이를 덮어쓸 수 있습니다.
이 경우 데이터베이스 마이그레이션이 서로 다른 클러스터에 대해 동적으로 실행될 수 없으므로 올바르게 적용되지 않습니다.
`LANGFUSE_AUTO_CLICKHOUSE_MIGRATION_DISABLED = true`로 설정하고 ClickHouse 마이그레이션을 수동으로 실행해야 합니다.
Langfuse 저장소를 클론하고, `./packages/shared/clickhouse/migrations/clustered/*.sql`에서 클러스터 이름을 조정한 뒤 `cd ./packages/shared && sh ./clickhouse/scripts/up.sh`를 실행하여
마이그레이션을 수동으로 적용하세요.

### 타임존 [#timezones]

<Callout type="warning">

**ClickHouse는 타임존을 UTC(기본값)로 설정한 상태로 실행해야 합니다.** Langfuse는 UTC가 아닌 ClickHouse 타임존을 지원하지 않습니다. [`timezone`](https://clickhouse.com/docs/en/operations/server-configuration-parameters/settings#timezone) 서버 설정 등을 통해 ClickHouse를 다른 타임존으로 실행하면 쿼리가 잘못되거나 비어 있는 결과를 반환하게 됩니다.

</Callout>

다음을 실행하여 ClickHouse 타임존을 확인할 수 있습니다:

```sql
SELECT timezone()
```

이 쿼리는 `UTC`를 반환해야 합니다. 그렇지 않다면 ClickHouse 서버 구성에서 사용자 지정 타임존 설정을 제거하고 서버를 재시작하세요.

동일한 요구 사항이 [Postgres](/self-hosting/deployment/infrastructure/postgres)에도 적용됩니다 — 모든 인프라 구성 요소는 UTC를 사용해야 합니다. 다른 타임존 지원을 고려해주길 원하신다면 이 [GitHub Discussion](https://github.com/orgs/langfuse/discussions/5046)에 투표해 주세요.

## 사용자 권한 [#user-permissions]

`CLICKHOUSE_USER`에 지정된 ClickHouse 사용자는 Langfuse가 올바르게 동작하도록 다음 권한을 가지고 있어야 합니다:

```sql
GRANT INSERT ON default.* TO 'user';
GRANT SELECT ON default.* TO 'user';
GRANT ALTER UPDATE, ALTER DELETE ON default.* TO 'user';
GRANT ALTER DROP INDEX ON default.* TO 'user';
GRANT CREATE ON default.* TO 'user';
GRANT DROP TABLE ON default.* TO 'user';
```

`'user'`를 실제 ClickHouse 사용자 이름으로 바꾸고, `default`가 아닌 다른 데이터베이스를 사용하는 경우 데이터베이스 이름도 조정하세요.

## 커스텀 도구를 위한 ClickHouse 직접 접근 [#direct-clickhouse-access]

셀프 호스팅 배포에서는 내부 ClickHouse 데이터베이스를 직접 제어할 수 있습니다.
내부 대시보드, 감사, 마이그레이션, 일회성 디버깅 등을 위해 직접 쿼리할 수 있습니다.

프로덕션 통합 및 커스텀 애플리케이션의 경우 [Public API](/docs/api-and-data-platform/features/public-api), [SDK query helpers](/docs/api-and-data-platform/features/query-via-sdk), [MCP 서버](/docs/api-and-data-platform/features/mcp-server), [Blob Storage Export](/docs/api-and-data-platform/features/export-to-blob-storage)를 우선적으로 사용하는 것이 좋습니다.
이러한 인터페이스는 Langfuse 릴리스 전반에서 호환성이 유지되는 대상입니다.

<Callout type="warning">

ClickHouse 스키마는 안정적인 API 계약이 아닙니다.
[Simplify Langfuse for Scale](/changelog/2026-03-10-simplify-for-scale)와 같은 주요 Langfuse 업그레이드, 백그라운드 마이그레이션, 성능 작업으로 인해 테이블, 컬럼, 중복 제거 동작, 조인 패턴이 언제든지 변경될 수 있습니다.
`traces`, `observations`, `scores` 또는 내부 구체화된 뷰를 읽는 커스텀 쿼리는 Langfuse 업그레이드 시마다 검증되어야 합니다.

</Callout>

ClickHouse를 직접 쿼리하는 경우:

- 가능하면 전용 읽기 전용 사용자 또는 전용 읽기 전용 컴퓨트 그룹을 사용하세요.
- 직접적인 분석 트래픽을 기본 수집 경로와 분리하세요. ClickHouse Cloud 또는 BYOC에서는 별도의 컴퓨트 그룹을 사용하고, 지원되는 Langfuse 읽기 경로(특히 Public API 및 필터/헬퍼 읽기)에 대해 `CLICKHOUSE_READ_ONLY_URL`을 구성하세요.
- 가능하면 항상 프로젝트 및 시간 필터를 포함하세요. Langfuse 트레이싱 데이터는 월별 파티셔닝을 포함해 프로젝트 및 시간 접근 패턴에 맞게 최적화되어 있습니다.
- Langfuse 테이블에 직접 쓰지 마세요. Langfuse 데이터를 생성하거나 업데이트할 때는 Public API 또는 SDK를 사용하세요.
- 누락된 API 필터나 필드 때문에 부득이하게 ClickHouse를 직접 쿼리해야 하는 경우, 사용 사례를 담아 [GitHub issue](https://github.com/langfuse/langfuse/issues)를 열어 주세요. API가 장기적으로 선호되는 확장 지점입니다.

## 커스텀 스키마 변경 [#custom-schema-changes]

Langfuse는 수정되지 않은 ClickHouse 스키마로 실행할 것을 권장합니다.

<Callout type="warning">

Langfuse의 ClickHouse 스키마와 마이그레이션은 각 릴리스와 하나의 단위로 설계, 테스트, 버전 관리됩니다.
`observations`, `events` 또는 기타 테이블 정의를 커스터마이징하면 해당 스키마는 저희가 테스트하는 범위를 벗어나게 되며, 향후 업그레이드와의 호환성을 유지하는 것은 사용자의 책임이 됩니다.
커스텀 스키마 변경으로 인해 발생하는 문제에 대해서는 최선의 노력으로 지원해 드립니다.

</Callout>

누락된 컬럼, 인덱스, 테이블 설정 때문에 커스텀 스키마 변경을 고려하고 있다면, 먼저 사용 사례를 담아 [GitHub issue](https://github.com/langfuse/langfuse/issues)를 열어 주세요. 표준 스키마에 대한 개선은 모든 배포에 도움이 되며 저희의 정규 테스트 및 업그레이드 프로세스에 포함됩니다.

## 배포 옵션

이 섹션에서는 다양한 배포 옵션과 예시 환경 변수를 다룹니다.

### Cloud/BYOC (권장) [#cloudbyoc]

ClickHouse Cloud는 확장 가능하며 완전 관리형인 ClickHouse 배포 옵션입니다.
[ClickHouse](https://clickhouse.cloud/)에서 직접 프로비저닝하거나 클라우드 제공업체 마켓플레이스를 통해 프로비저닝할 수 있습니다:

- [AWS Marketplace](https://aws.amazon.com/marketplace/pp/prodview-jettukeanwrfc)
- [Google Cloud Marketplace](https://console.cloud.google.com/marketplace/product/clickhouse-public/clickhouse-cloud)
- [Azure Marketplace](https://azuremarketplace.microsoft.com/en-us/marketplace/apps/clickhouse.clickhouse_cloud?tab=Overview)

ClickHouse Cloud 클러스터는 사용자의 클라우드 환경 및 VPC 외부에 프로비저닝되지만, ClickHouse는 AWS, GCP, Azure를 위한 [private link](https://clickhouse.com/docs/en/cloud/security/private-link-overview)를 제공합니다.

ClickHouse Cloud의 운영 모델을 유지하면서도 ClickHouse 데이터 플레인을 자체 클라우드 계정에 두고 싶다면 [ClickHouse BYOC](https://clickhouse.com/cloud/bring-your-own-cloud)를 고려하세요. BYOC는 사용자의 클라우드 계정 내 인프라에서 실행되는 완전 관리형 ClickHouse Cloud 배포이며, 엄격한 데이터 거주 요구 사항, 컴플라이언스 또는 VPC 경계 요구 사항이 있는 대규모 배포를 위해 설계되었습니다.

대규모 Langfuse 배포의 경우 ClickHouse Cloud 또는 BYOC를 권장합니다. Langfuse가 사용하는 셀프 매니지드 OSS ClickHouse 설정에서는 제공되지 않는 클라우드 네이티브 확장 프리미티브를 제공하기 때문입니다. ClickHouse Cloud와 BYOC는 [SharedMergeTree](https://clickhouse.com/docs/cloud/reference/shared-merge-tree)를 통해 스토리지와 컴퓨트를 분리함으로써 저장된 데이터와 독립적으로 컴퓨트를 확장하고, 레플리카 스토리지 오버헤드를 줄이며, 성장에 따른 수동 샤드 계획을 피할 수 있게 해줍니다. 또한 [warehouses](https://clickhouse.com/docs/cloud/reference/warehouses)를 통해 compute-compute 분리를 지원하므로, 동일한 데이터를 공유하면서도 CPU와 메모리를 두고 경쟁하지 않는 별도의 컴퓨트 그룹에서 수집 쓰기, 지원되는 Langfuse 읽기, 분석 쿼리, 임시 워크로드를 분리할 수 있습니다. Langfuse는 이 패턴을 퍼블릭 API 읽기 및 일부 UI/필터 읽기 트래픽에 대해 `CLICKHOUSE_READ_ONLY_URL`을 통해 활용할 수 있습니다.

도움이 필요하거나 ClickHouse 팀과 이야기하고 싶다면 [여기](https://clickhouse.com/company/contact)로 문의할 수 있습니다.

#### 예시 구성

ClickHouse 인스턴스에 연결하려면 다음 환경 변수를 설정하세요:

```yaml
CLICKHOUSE_URL=https://<identifier>.<region>.aws.clickhouse.cloud:8443
CLICKHOUSE_MIGRATION_URL=clickhouse://<identifier>.<region>.aws.clickhouse.cloud:9440
CLICKHOUSE_USER=default
CLICKHOUSE_PASSWORD=changeme
CLICKHOUSE_MIGRATION_SSL=true
```

#### 문제 해결

- **마이그레이션 중 'error: driver: bad connection in line 0' 오류**: 웹 컨테이너 시작 중 이 오류 메시지가 표시되면, `CLICKHOUSE_MIGRATION_SSL` 플래그가 설정되어 있는지, 그리고 Langfuse Web이 ClickHouse 환경에 접근할 수 있는지 확인하세요. 해당되는 경우 IP 화이트리스팅과 인스턴스가 Private Link에 접근할 수 있는지 확인하세요.
- **Code: 80. DB::Exception: It's not initial query. ON CLUSTER is not allowed for Replicated database. (INCORRECT_QUERY)** (ClickHouse Cloud Azure에서 발생): ClickHouse Cloud Azure는 ON CLUSTER와 Replicated 설정을 제대로 처리하지 못하는 것으로 보입니다. 당분간 `CLICKHOUSE_CLUSTER_ENABLED=false`로 설정하는 것을 권장합니다. 이는 성능이나 고가용성에 영향을 주지 않습니다.

### ClickHouse Kubernetes Operator [#clickhouse-kubernetes-operator]

공식 [ClickHouse Kubernetes Operator](https://clickhouse.com/docs/clickhouse-operator/overview) ([GitHub](https://github.com/ClickHouse/clickhouse-operator))는 업그레이드, 스케일링, 고가용성을 포함하여 Kubernetes에서 셀프 매니지드 ClickHouse 및 ClickHouse Keeper 클러스터의 배포, 구성, 관리를 자동화합니다.
이는 Kubernetes에서 셀프 매니지드 ClickHouse를 실행하기 위한 공식 지원 옵션입니다.

[operator 문서](https://clickhouse.com/docs/clickhouse-operator/overview)를 따라 클러스터를 프로비저닝한 후, 위에서 설명한 환경 변수를 사용해 Langfuse를 연결하세요.
동일한 [사용자 권한](#user-permissions) 및 [타임존](#timezones) 요구 사항이 적용됩니다.

### Kubernetes 상의 ClickHouse (Helm) [#clickhouse-on-kubernetes-helm]

[Bitnami ClickHouse Helm Chart](https://github.com/bitnami/charts/tree/main/bitnami/clickhouse)는 주어진 Kubernetes 클러스터를 사용해 프로덕션 준비가 된 ClickHouse 배포를 제공합니다.
저희는 이를 [Langfuse K8s](https://github.com/langfuse/langfuse-k8s)의 의존성으로 사용합니다.
Kubernetes에서 Langfuse를 배포하는 방법에 대한 자세한 내용은 [Langfuse on Kubernetes (Helm)](/self-hosting/deployment/kubernetes-helm)을 참고하세요.
이 배포 옵션은 커뮤니티에서 지원됩니다. Kubernetes에서 공식적으로 지원되는 셀프 매니지드 설정을 원한다면 [ClickHouse Kubernetes Operator](#clickhouse-kubernetes-operator)를 고려하세요.

#### 예시 구성

최소 프로덕션 설정의 경우, ClickHouse Helm 차트를 배포할 때 다음 values.yaml 오버라이드를 사용하는 것을 권장합니다:

```yaml
clickhouse:
  deploy: true
  shards: 1 # 고정값: Langfuse는 다중 샤드 클러스터를 지원하지 않습니다
  replicaCount: 3
  resourcesPreset: large # 또는 그 이상
  persistence:
    size: 100Gi # 초기 리사이징을 피하기 위해 처음부터 큰 볼륨을 사용하세요. 또는 블록 스토리지 기반 디스크를 고려하세요.
  auth:
    username: default
    password: changeme
```

- **shards**: 샤드는 ClickHouse를 수평으로 확장하는 데 사용됩니다. 단일 ClickHouse 샤드는 수 테라바이트의 데이터를 처리할 수 있습니다. 현재 Langfuse는 다중 샤드 클러스터를 지원하지 않으므로, 이 값은 반드시 1로 설정해야 합니다. 단일 샤드 클러스터의 확장 한계에 도달했다면 저희에게 문의하거나 ClickHouse Cloud/BYOC를 고려하세요.
- **replicaCount**: 각 샤드의 레플리카 수입니다. ClickHouse는 모든 인스턴스를 레플리카 수에 포함하므로, 레플리카 수가 1이면 중복성이 전혀 없다는 뜻입니다. 프로덕션 설정에서는 최소 3개의 레플리카를 권장합니다. 레플리카 수는 수동 개입이나 다운타임 없이 런타임에 늘릴 수 없습니다.
- **resourcesPreset**: ClickHouse는 분석 및 높은 동시성 요청에서 CPU와 메모리를 많이 사용합니다. 최소한 `large` resourcesPreset을 사용하고, 더 큰 배포에서는 그 이상을 권장합니다.
- **auth**: ClickHouse 데이터베이스의 사용자 이름과 비밀번호입니다. 원하는 값으로 덮어쓰거나 시크릿에서 마운트하세요.
- **disk**: ClickHouse Helm 차트는 각 레플리카의 볼륨을 생성할 때 기본 스토리지 클래스를 사용합니다. 관측성 워크로드는 디스크를 많이 사용하는 경향이 있으므로, 스토리지 클래스에 `allowVolumeExpansion = true`가 설정되어 있는지 확인하세요. AWS, GCP, Azure 같은 클라우드 제공업체에서는 기본값으로 제공됩니다.

Langfuse는 ClickHouse 구성에 특정 매개변수가 설정되어 있다고 가정합니다.
데이터베이스 마이그레이션을 수행하려면 다음 값이 제공되어야 합니다:

```xml
<!--
    복제된 테이블의 매개변수를 위한 치환값입니다.
     선택 사항입니다. 복제된 테이블을 사용하지 않는다면 생략할 수 있습니다.
     https://clickhouse.com/docs/en/engines/table-engines/mergetree-family/replication/#creating-replicated-tables 참고
-->
<!--
    <macros>
        <shard>01</shard>
        <replica>example01-01-1</replica>
    </macros>
-->
<!--
    <default_replica_path>/clickhouse/tables/{database}/{table}</default_replica_path>
    <default_replica_name>{replica}</default_replica_name>
-->
```

`macros` 및 `default_replica_*` 구성은 추가 설정 없이 Helm 차트에서 처리됩니다.

Langfuse가 동일한 클러스터 및 네임스페이스 내에서 실행된다고 가정하고, ClickHouse 인스턴스에 연결하기 위해 다음 환경 변수를 설정하세요:

```yaml
CLICKHOUSE_URL=http://<chart-name>-clickhouse:8123
CLICKHOUSE_MIGRATION_URL=clickhouse://<chart-name>-clickhouse:9000
CLICKHOUSE_USER=default
CLICKHOUSE_PASSWORD=changeme
```

#### 문제 해결

- **NOT_ENOUGH_SPACE 오류**: 이 오류는 ClickHouse의 디스크 공간이 부족할 때 발생합니다. Kubernetes 환경에서는 일반적으로 퍼시스턴트 볼륨 클레임(PVC)을 확장해야 함을 의미합니다. 다음 단계로 해결할 수 있습니다:

  **1. 현재 디스크 사용량 확인:**

  ```bash
  # PVC 상태 확인
  kubectl get pvc -l app.kubernetes.io/name=clickhouse

  # ClickHouse 파드 내부의 디스크 사용량 확인
  kubectl exec -it <clickhouse-pod-name> -- df -h /var/lib/clickhouse
  ```

  **2. PVC 확장 (allowVolumeExpansion: true가 설정된 스토리지 클래스 필요):**

  ```bash
  # PVC 직접 편집
  kubectl edit pvc data-<chart-name>-clickhouse-0

  # 또는 모든 ClickHouse PVC를 한 번에 패치
  kubectl patch pvc data-<chart-name>-clickhouse-0 -p '{"spec":{"resources":{"requests":{"storage":"200Gi"}}}}'
  kubectl patch pvc data-<chart-name>-clickhouse-1 -p '{"spec":{"resources":{"requests":{"storage":"200Gi"}}}}'
  kubectl patch pvc data-<chart-name>-clickhouse-2 -p '{"spec":{"resources":{"requests":{"storage":"200Gi"}}}}'
  ```

  **3. 확장 진행 상황 모니터링:**

  ```bash
  # PVC 상태 관찰
  kubectl get pvc -w

  # 파드가 새 공간을 인식하는지 확인
  kubectl exec -it <clickhouse-pod-name> -- df -h /var/lib/clickhouse
  ```

  **4. StatefulSet 재시작:**

  ```bash
  # 더 큰 볼륨을 사용하도록 모든 파드를 개별적으로 재시작
  kubectl rollout restart statefulset <chart-name>-clickhouse
  ```

  **예방 팁:**
  - 디스크 사용량에 대한 모니터링 알림을 설정하세요 (80% 용량에서 알림을 권장합니다)
  - `allowVolumeExpansion: true`가 설정된 스토리지 클래스를 사용하세요 (대부분의 클라우드 제공업체에서 기본값)
  - [volume-autoscaler](https://github.com/DevOps-Nirvana/Kubernetes-Volume-Autoscaler)와 같은 도구를 사용해 자동 PVC 확장 구현을 고려하세요
  - 성장 속도가 빠른 환경의 경우, 자동 확장을 위해 [디스크로서의 블록 스토리지](#blob-storage-as-disk)를 사용하는 것을 고려하세요

### Docker [#docker]

개발 목적으로 단일 [Docker](https://hub.docker.com/r/clickhouse/clickhouse-server) 컨테이너에서 ClickHouse를 실행할 수 있습니다.
중복성이 없으므로 **프로덕션 워크로드에는 권장되지 않습니다**.

#### 예시 구성

다음 명령으로 컨테이너를 시작하세요:

```bash
docker run --name clickhouse-server \
  -e CLICKHOUSE_DB=default \
  -e CLICKHOUSE_USER=clickhouse \
  -e CLICKHOUSE_PASSWORD=clickhouse \
  -d --ulimit nofile=262144:262144 \
  -p 8123:8123 \
  -p 9000:9000 \
  -p 9009:9009 \
  clickhouse/clickhouse-server
```

ClickHouse 인스턴스에 연결하려면 다음 환경 변수를 설정하세요:

```yaml
CLICKHOUSE_URL=http://localhost:8123
CLICKHOUSE_MIGRATION_URL=clickhouse://localhost:9000
CLICKHOUSE_USER=clickhouse
CLICKHOUSE_PASSWORD=clickhouse
CLICKHOUSE_CLUSTER_ENABLED=false
```

## 새 인스턴스로 마이그레이션

제공업체를 변경하거나 단일 노드에서 클러스터 구성으로 이전하는 등의 경우, Langfuse를 다른 ClickHouse 인스턴스로 이전하고 싶을 수 있습니다.
다음 접근 방식을 권장합니다:

1. Langfuse가 새 ClickHouse 인스턴스를 가리키도록 설정하여 들어오는 데이터가 그곳에 저장되도록 합니다.
2. 기존 인스턴스의 백업을 블롭 스토리지에 생성한 뒤 새 인스턴스로 복원합니다.

두 인스턴스에 동시에 데이터를 쓰는 무중단 전환도 기술적으로는 가능하지만, 저희가 테스트하지 않았으며 ClickHouse에서도 권장하지 않습니다.
이 옵션을 검토하고 싶다면 Langfuse Support에 문의해 주세요.

백업 및 복원 방법은 ClickHouse 마이그레이션 가이드를 참고하세요:

- [remoteSecure를 통한 ClickHouse to ClickHouse](https://clickhouse.com/docs/cloud/migration/clickhouse-to-cloud)
- [백업 및 복원을 통한 ClickHouse to ClickHouse](https://clickhouse.com/docs/cloud/migration/oss-to-cloud-backup-restore)

## 암호화

ClickHouse는 저장 데이터(data at rest)에 대한 디스크 암호화를 지원하여 민감한 데이터에 대한 추가 보안 계층을 제공합니다.

### 블록 스토리지를 이용한 자동 암호화

[디스크로서의 블록 스토리지](#blob-storage-as-disk) (AWS S3, Azure Blob Storage, Google Cloud Storage)를 사용하는 경우, 데이터는 클라우드 제공업체의 기본 암호화를 통해 자동으로 저장 시 암호화됩니다:

- **AWS S3**: 기본적으로 AES-256 암호화를 사용합니다
- **Azure Blob Storage**: 기본적으로 AES-256 암호화를 사용합니다
- **Google Cloud Storage**: 기본적으로 AES-256 암호화를 사용합니다

### 수동 디스크 암호화

로컬 디스크 스토리지나 추가적인 암호화 계층이 필요한 경우, ClickHouse는 AES_128_CTR 알고리즘을 사용한 구성 가능한 디스크 암호화를 지원합니다.

#### Kubernetes 구성

Bitnami ClickHouse Helm 차트를 사용하는 Kubernetes 배포의 경우, 커스텀 구성 파일을 생성하여 디스크 암호화를 구성할 수 있습니다:

```yaml
# values.yaml
clickhouse:
  extraConfigmaps:
    - name: encryption-config
      mountPath: /etc/clickhouse-server/config.d/encrypted_storage.xml
      data:
        encryption.xml: |
          <clickhouse>
            <storage_configuration>
              <disks>
                <encrypted_disk>
                  <type>encrypted</type>
                  <disk>default</disk>
                  <path>encrypted/</path>
                  <algorithm>AES_128_CTR</algorithm>
                  <key_hex id="0" from_env="CLICKHOUSE_ENCRYPTION_KEY"></key_hex>
                </encrypted_disk>
              </disks>
              <policies>
                <encrypted_policy>
                  <volumes>
                    <main>
                      <disk>encrypted_disk</disk>
                    </main>
                  </volumes>
                </encrypted_policy>
              </policies>
            </storage_configuration>
            <merge_tree>
              <storage_policy>encrypted_policy</storage_policy>
            </merge_tree>
          </clickhouse>
```

암호화 키를 환경 변수로 설정하세요:

```yaml
# values.yaml
clickhouse:
  extraEnvVars:
    - name: CLICKHOUSE_ENCRYPTION_KEY
      valueFrom:
        secretKeyRef:
          name: clickhouse-encryption-key
          key: key
```

암호화 키 시크릿을 생성하세요:

```bash
kubectl create secret generic clickhouse-encryption-key \
  --from-literal=key="00112233445566778899aabbccddeeff"
```

## 디스크로서의 블록 스토리지

ClickHouse는 블록 스토리지(AWS S3, Azure Blob Storage, Google Cloud Storage)를 디스크로 사용하는 것을 지원합니다.
이는 컨테이너 오케스트레이터 외부에 존재하는 자동 확장 스토리지에 유용하며, 데이터의 가용성과 내구성을 높여줍니다.
이 기능에 대한 전체 개요는 [ClickHouse External Disks 문서](https://clickhouse.com/docs/en/operations/storing-data)를 참고하세요.

아래에서는 Docker Compose를 사용하는 ClickHouse Docker 컨테이너를 위해 S3와 Azure Blob Storage를 디스크로 사용하는 config.xml 예시를 제공합니다.
메타데이터는 여전히 로컬 디스크에 저장되므로, ClickHouse 컨테이너에 퍼시스턴트 볼륨을 사용해야 테이블 접근 권한을 잃지 않는다는 점에 유의하세요.

<Callout type="warning">

ClickHouse 배포에서 블록 스토리지를 디스크로 사용할 때 다음 설정을 권장합니다:

- **버킷 버전 관리를 활성화하지 마세요**: ClickHouse는 병합 처리 중에 많은 파일을 쓰고 업데이트합니다. 버킷에 버전 관리를 활성화하면 전체 이력이 보존되어 스토리지 사용량이 빠르게 증가합니다.
- **삭제를 위한 라이프사이클 정책을 활성화하지 마세요**: ClickHouse의 내부 일관성 모델이 깨질 수 있으므로 삭제 라이프사이클 정책은 피하세요. 대신 Langfuse 애플리케이션이나 ClickHouse TTL을 통해 데이터를 삭제하세요.
- **중단된 멀티파트 업로드에 대한 라이프사이클 정책을 활성화하세요**: ClickHouse가 업로드를 시도했다가 완료 전에 중단하면 원치 않는 잔재가 남을 수 있습니다.

이 내용은 다음 [ClickHouse issue](https://github.com/ClickHouse/clickhouse-docs/issues/1385)에서 도출되었습니다.

</Callout>

### S3 예시

로컬 작업 디렉터리에 다음 내용의 config.xml 파일을 생성하세요:

```xml
<clickhouse>
    <merge_tree>
        <storage_policy>s3</storage_policy>
    </merge_tree>
    <storage_configuration>
        <disks>
            <s3>
                <type>object_storage</type>
                <object_storage_type>s3</object_storage_type>
                <metadata_type>local</metadata_type>
                <endpoint>https://s3.eu-central-1.amazonaws.com/example-bucket-name/data/</endpoint>
                <access_key_id>ACCESS_KEY</access_key_id>
                <secret_access_key>ACCESS_KEY_SECRET</secret_access_key>
            </s3>
        </disks>
        <policies>
            <s3>
                <volumes>
                    <main>
                        <disk>s3</disk>
                    </main>
                </volumes>
            </s3>
        </policies>
    </storage_configuration>
</clickhouse>
```

Access Key Id와 Secret Access key를 알맞은 AWS 자격 증명으로 교체하고, `endpoint` 요소 안의 버킷 이름을 변경하세요.
또는 자격 증명을 `<use_environment_credentials>1</use_environment_credentials>`로 교체하여 환경 변수에서 AWS 자격 증명을 자동으로 가져오게 할 수 있습니다.

이제 다음 Docker Compose 파일로 ClickHouse를 시작할 수 있습니다:

```yaml
services:
  clickhouse:
    image: clickhouse/clickhouse-server
    user: "101:101"
    container_name: clickhouse
    hostname: clickhouse
    environment:
      CLICKHOUSE_DB: default
      CLICKHOUSE_USER: clickhouse
      CLICKHOUSE_PASSWORD: clickhouse
    volumes:
      - ./config.xml:/etc/clickhouse-server/config.d/s3disk.xml:ro
      - langfuse_clickhouse_data:/var/lib/clickhouse
      - langfuse_clickhouse_logs:/var/log/clickhouse-server
    ports:
      - "8123:8123"
      - "9000:9000"

volumes:
  langfuse_clickhouse_data:
    driver: local
  langfuse_clickhouse_logs:
    driver: local
```

### Azure Blob Storage 예시

로컬 작업 디렉터리에 다음 내용의 config.xml 파일을 생성하세요.
아래 자격 증명은 기본 [Azurite](https://github.com/Azure/Azurite) 자격 증명이며 공개된 것으로 간주됩니다.

```xml
<clickhouse>
    <merge_tree>
        <storage_policy>blob_storage_disk</storage_policy>
    </merge_tree>
    <storage_configuration>
        <disks>
            <blob_storage_disk>
                <type>object_storage</type>
                <object_storage_type>azure_blob_storage</object_storage_type>
                <metadata_type>local</metadata_type>
                <storage_account_url>http://azurite:10000/devstoreaccount1</storage_account_url>
                <container_name>langfuse</container_name>
                <account_name>devstoreaccount1</account_name>
                <account_key>Eby8vdM02xNOcqFlqUwJPLlmEtlCDXJ1OUzFT50uSRZ6IFsuFq2UVErCz4I6tq/K1SZFPTOtr/KBHBeksoGMGw==</account_key>
            </blob_storage_disk>
        </disks>
        <policies>
            <blob_storage_disk>
                <volumes>
                    <main>
                        <disk>blob_storage_disk</disk>
                    </main>
                </volumes>
            </blob_storage_disk>
        </policies>
    </storage_configuration>
</clickhouse>
```

다음 Docker Compose 파일을 사용해 Azurite 서비스와 함께 ClickHouse를 시작할 수 있습니다:

```yaml
services:
  clickhouse:
    image: clickhouse/clickhouse-server
    user: "101:101"
    container_name: clickhouse
    hostname: clickhouse
    environment:
      CLICKHOUSE_DB: default
      CLICKHOUSE_USER: clickhouse
      CLICKHOUSE_PASSWORD: clickhouse
    volumes:
      - ./config.xml:/etc/clickhouse-server/config.d/azuredisk.xml:ro
      - langfuse_clickhouse_data:/var/lib/clickhouse
      - langfuse_clickhouse_logs:/var/log/clickhouse-server
    ports:
      - "8123:8123"
      - "9000:9000"
    depends_on:
      - azurite

  azurite:
    image: mcr.microsoft.com/azure-storage/azurite
    container_name: azurite
    command: azurite-blob --blobHost 0.0.0.0
    ports:
      - "10000:10000"
    volumes:
      - langfuse_azurite_data:/data

volumes:
  langfuse_clickhouse_data:
    driver: local
  langfuse_clickhouse_logs:
    driver: local
  langfuse_azurite_data:
    driver: local
```

이렇게 하면 ClickHouse 데이터가 Azurite 버킷 내에 저장됩니다.

## FAQ

### Langfuse를 셀프 호스팅하려면 ClickHouse가 필요한가요?

네, ClickHouse는 현재 Langfuse를 셀프 호스팅하기 위한 필수 구성 요소입니다. 현재 지원되는 대체 OLAP 데이터베이스는 없습니다. Langfuse는 트레이스, observation, 점수를 위한 주요 스토리지 솔루션으로 ClickHouse를 사용하지 않고는 셀프 호스팅할 수 없습니다.

모든 셀프 호스팅 배포에는 ClickHouse 인스턴스가 포함되어야 합니다.

### ClickHouse 스토리지 증가를 어떻게 관리하나요?

Langfuse에는 내장된 [Data Retention](/docs/administration/data-retention) 기능이 있어, 설정된 보관 기간(일수)이 지난 트레이스, observation, 점수, 미디어 자산을 자동으로 삭제합니다. 이 작업은 매일 밤 실행되며 ClickHouse와 블록 스토리지 양쪽의 정리를 처리합니다. ClickHouse TTL을 수동으로 설정할 필요는 없습니다.

보관 기능을 구성했음에도 ClickHouse 디스크 사용량이 계속 증가한다면, TTL 없이 제공되는 시스템 로그 테이블(`trace_log`, `text_log`, `opentelemetry_span_log` 등)이 원인인 경우가 많습니다. 이를 비활성화하거나 제한하는 방법은 [Scaling › ClickHouse system log tables](/self-hosting/configuration/scaling#clickhouse-system-log-tables)를 참고하세요.

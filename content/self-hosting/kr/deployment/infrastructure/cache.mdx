# 캐시(Redis/Valkey)

<Callout type="info">

이 문서는 Redis/Valkey 설정에 대한 심화 가이드입니다. 시작하려면 [배포 가이드](/self-hosting#deployment-options) 중 하나를 따라주세요.

</Callout>

Langfuse는 캐싱 레이어 및 큐로 Redis/Valkey를 사용합니다.
API를 통해 새 이벤트를 빠르게 수락하고 처리 및 삽입을 지연시키는 데 사용됩니다.
이를 통해 Langfuse는 요청 피크를 원활하게 처리할 수 있습니다.

AWS, Azure, GCP의 관리형 서비스를 사용하거나 직접 호스팅할 수 있습니다.
최소 버전 7 이상이 필요하며, 인스턴스에 `maxmemory-policy=noeviction`이 설정되어 있어야 합니다.

이 가이드는 Langfuse 내에서 Redis를 설정하는 방법과 자체 Redis를 사용할 때 유의해야 할 사항을 다룹니다.

## 공식 및 커뮤니티 지원 [#support]

Langfuse는 Redis 및 Redis 호환 스토어와 함께 작동합니다.
공식 지원 옵션과 커뮤니티 지원 옵션을 구분합니다.

| 옵션                                                       | 지원 수준 |
| ---------------------------------------------------------- | --------- |
| [Redis](https://redis.io/) (>= 7, 셀프 호스팅 또는 관리형) | 공식      |
| [Valkey](#valkey-vs-redis) (>= 8, 셀프 호스팅 또는 관리형) | 공식      |
| 기타 Redis 호환 스토어 (예: Dragonfly, KeyDB, Garnet)      | 커뮤니티  |

[Amazon ElastiCache](https://aws.amazon.com/elasticache/redis/), [Azure Cache for Redis](https://azure.microsoft.com/en-us/products/cache/), [GCP Memorystore](https://cloud.google.com/memorystore)와 같이 Redis 또는 Valkey를 실행하는 관리형 서비스는 공식 지원 수준에 포함됩니다.

## 설정

Langfuse는 Redis 사용을 세밀하게 조정할 수 있도록 다음 환경 변수를 지원합니다.
이 변수들은 Langfuse Web과 Langfuse Worker 컨테이너 모두에 제공되어야 합니다.

| 변수                      | 필수 / 기본값 | 설명                                                                                    |
| ------------------------- | ------------- | --------------------------------------------------------------------------------------- |
| `REDIS_CONNECTION_STRING` | 필수          | `redis[s]://[[username][:password]@][host][:port][/db-number]` 형식의 Redis 연결 문자열 |

### 캐시 설정 옵션

Langfuse는 성능 향상을 위해 API 키와 프롬프트에 대한 캐싱을 지원합니다. 다음 환경 변수는 캐싱 동작을 제어합니다.

| 변수                                 | 필수 / 기본값 | 설명                                                                                                                                                                 |
| ------------------------------------ | ------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `LANGFUSE_CACHE_API_KEY_ENABLED`     | `true`        | API 키 캐싱을 활성화 또는 비활성화합니다. API 키 캐싱을 비활성화하려면 `false`로 설정하세요. 평문 키는 Redis에 저장되지 않으며, 해시되거나 암호화된 키만 저장됩니다. |

### 타임아웃 설정

Langfuse는 자신이 여는 모든 Redis 연결에 소켓 수준의 워치독을 적용합니다. 연결이 데이터를 기다리는 중 30초 동안 아무것도 수신하지 못하면 클라이언트는 소켓을 파괴하고 다시 연결합니다. 이를 통해 중단된(hung) 연결이 BullMQ 워커 동시성 슬롯을 무한정 차단하는 것을 방지합니다. 또한 연결은 미들박스가 유휴 연결을 종료하지 않도록 10초 고정 TCP keep-alive를 사용합니다.

| 변수                      | 필수 / 기본값 | 설명                                                                                                                                                                                                                                                                                                                            |
| ------------------------- | ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `REDIS_SOCKET_TIMEOUT_MS` | `30000`       | 모든 Redis 연결(웹 및 워커, BullMQ 큐 연결 포함)에 대한 소켓 타임아웃(밀리초)입니다. 워치독을 비활성화하려면 `0`으로 설정하세요. 양수 값은 최소 `10000` 이상이어야 합니다. BullMQ의 블로킹 명령(`BZPOPMIN`)은 패킷 사이에 정상적으로 약 5초간 유휴 상태가 되므로, 값이 더 낮으면 정상적인 유휴 워커가 재연결을 반복하게 됩니다. |

#### 문제 해결: `Socket timeout. Expecting data, but didn't receive any in 30000ms.`

이 오류는 위에서 설명한 클라이언트 측 워치독에 의해 발생하는 것이며, Redis 서버에서 발생하는 것이 아닙니다. 즉, Redis 인스턴스가 정상이더라도 이 오류가 발생할 수 있습니다. 일반적인 원인은 다음과 같습니다.

- **Langfuse 컨테이너의 CPU 부족(가장 흔한 원인):** 워커 컨테이너가 CPU 제한(throttle)을 받는 등의 이유로 Node.js 이벤트 루프가 정체되면, 들어오는 Redis 데이터가 제때 처리되지 못해 Redis가 즉시 응답했음에도 워치독이 발동합니다. 평균 CPU 사용률이 낮아 보이더라도 영향을 받는 컨테이너의 CPU 할당량을 늘리면 여러 셀프 호스팅 배포 환경에서 이 오류가 해결된 사례가 있습니다.
- **네트워크 미들박스:** Langfuse와 Redis 사이의 연결을 조용히 끊는 NAT 게이트웨이, 프록시, 방화벽입니다.

`REDIS_SOCKET_TIMEOUT_MS`를 늘리는 것은 워치독의 대기 시간만 늘릴 뿐, CPU 부족 문제 자체를 해결하지는 못하며 실제로 중단된 연결로부터의 복구를 지연시킵니다.

## 배포 옵션

이 섹션에서는 다양한 배포 옵션과 예시 환경 변수를 다룹니다.

### 클라우드 제공업체의 관리형 Redis/Valkey

[Amazon ElastiCache](https://aws.amazon.com/de/elasticache/redis/), [Azure Cache for Redis](https://azure.microsoft.com/de-de/products/cache/), [GCP Memorystore](https://cloud.google.com/memorystore/?hl=en)는 완전 관리형 Redis 서비스입니다.
Langfuse는 읽기 복제본(read-replica) 간 페일오버를 처리하며, 수평 확장을 위한 Redis 클러스터 모드를 지원합니다.

**단일(standalone) Redis 인스턴스**의 경우 표준 설정을 사용하세요.

```bash
REDIS_CONNECTION_STRING=redis://username:password@your-redis-endpoint:6379
```

**Redis 클러스터 모드**의 경우 클러스터 지원을 활성화하세요.

```bash
REDIS_CLUSTER_ENABLED=true
REDIS_CLUSTER_NODES=your-cluster-endpoint:6379
REDIS_AUTH=your-redis-password  # if authentication is enabled
```

Langfuse 컨테이너가 VPC 내에서 Redis 인스턴스에 접근할 수 있는지 확인하세요.

<Callout type="warning">

큐 작업이 캐시에서 제거(eviction)되지 않도록 `maxmemory-policy` 파라미터를 반드시 `noeviction`으로 설정해야 합니다.

</Callout>

### Kubernetes(Helm)에서의 Redis

Bitnami는 [Redis](https://github.com/bitnami/charts/tree/main/bitnami/redis)와 [Valkey](https://github.com/bitnami/charts/tree/main/bitnami/valkey)용 Helm 차트를 제공합니다.
우리는 [Langfuse K8s](https://github.com/langfuse/langfuse-k8s)의 의존성으로 Valkey 차트를 사용합니다.
Kubernetes에 Langfuse를 배포하는 자세한 방법은 [Langfuse on Kubernetes (Helm)](/self-hosting/deployment/kubernetes-helm)를 참고하세요.

#### 예시 설정

최소한의 프로덕션 설정을 위해 Valkey Helm 차트를 배포할 때 다음의 values.yaml 오버라이드를 사용하는 것을 권장합니다.

```yaml
valkey:
  deploy: true
  architecture: standalone
  primary:
    extraFlags:
      - "--maxmemory-policy noeviction" # Necessary to handle queue jobs correctly
  auth:
    password: changeme
```

Redis 인스턴스에 연결하기 위해 다음 환경 변수를 설정하세요.

```yaml
REDIS_CONNECTION_STRING=redis://default:changeme@<chart-name>-valkey-master:6379/0
```

### Docker

단일 [Docker](https://hub.docker.com/_/redis/) 컨테이너에서 Redis를 실행할 수 있습니다.
이중화(redundancy)가 없으므로 **프로덕션 워크로드에는 권장하지 않습니다**.

#### 예시 설정

다음 명령으로 컨테이너를 시작합니다.

```bash
docker run --name redis \
  -p 6379:6379 \
  redis --requirepass myredissecret --maxmemory-policy noeviction
```

Redis 인스턴스에 연결하기 위해 다음 환경 변수를 설정하세요.

```yaml
REDIS_HOST=localhost
REDIS_PORT=6379
REDIS_AUTH=myredissecret
```

## Redis 클러스터 모드

Redis 클러스터 모드는 여러 Redis 노드에 데이터를 분산시켜 수평 확장을 가능하게 합니다.
Langfuse는 고가용성 및 처리량 증가 시나리오를 위해 Redis 클러스터 모드를 지원합니다.

### 설정

Redis 클러스터 모드를 활성화하려면 다음 환경 변수를 설정하세요.

```bash
REDIS_CLUSTER_ENABLED=true
REDIS_CLUSTER_NODES=redis-node1:6379,redis-node2:6379,redis-node3:6379,redis-node4:6379,redis-node5:6379,redis-node6:6379
REDIS_AUTH=your-redis-password  # if authentication is enabled
```

<Callout type="info">

프로덕션 배포의 경우 고가용성을 위해 마스터 노드 3개와 각각의 레플리카 1개(총 6개 노드)를 사용하는 것을 권장합니다.

</Callout>

### 예시 설정

#### AWS ElastiCache Redis 클러스터

```bash
REDIS_CLUSTER_ENABLED=true
REDIS_CLUSTER_NODES=clustercfg.my-redis-cluster.abc123.cache.amazonaws.com:6379
REDIS_AUTH=your-auth-token
REDIS_TLS_ENABLED=true
REDIS_TLS_SERVERNAME=clustercfg.my-redis-cluster.abc123.cache.amazonaws.com
```

#### 셀프 호스팅 Redis 클러스터

```bash
REDIS_CLUSTER_ENABLED=true
REDIS_CLUSTER_NODES=10.0.1.10:6379,10.0.1.11:6379,10.0.1.12:6379,10.0.1.13:6379,10.0.1.14:6379,10.0.1.15:6379
REDIS_AUTH=your-cluster-password
```

#### 사용자 지정 CA 인증서를 사용하는 엔터프라이즈 Redis

사용자 지정 인증 기관과 특정 TLS 요구 사항이 있는 엔터프라이즈 환경의 경우:

```bash
REDIS_CONNECTION_STRING=redis://username:password@redis.example.com:6379
REDIS_TLS_ENABLED=true
REDIS_TLS_CA_PATH=/app/ca/cacertbundle.pem
REDIS_TLS_CERT_PATH=/app/redis-certs/tls.crt
REDIS_TLS_KEY_PATH=/app/redis-certs/tls.key
REDIS_TLS_SERVERNAME=redis.example.com
REDIS_TLS_REJECT_UNAUTHORIZED=true
```

이 설정을 통해 다음이 가능해집니다.

- 사용자 지정 CA에 대한 올바른 인증서 검증
- 프록시된 연결을 위한 `REDIS_TLS_SERVERNAME`을 통한 SNI 지원
- 전역 우회 설정 없이도 안전한 인증서 검증

암호화된 개인 키의 경우 다음을 추가하세요.

```bash
REDIS_TLS_KEY_PASSPHRASE=your-key-passphrase
```

<Callout type="warning">

Redis 클러스터 모드를 사용할 때는 다음을 확인하세요.

- 모든 클러스터 노드가 Langfuse 컨테이너에서 접근 가능한지 확인
- 모든 클러스터 노드에서 동일한 인증 자격 증명 사용
- 클러스터 상태를 모니터링하고 노드 장애를 적절히 처리
- 큐 작업 제거를 방지하기 위해 모든 클러스터 노드에서 `maxmemory-policy=noeviction` 설정
- 클러스터 노드 전체에 걸쳐 수집(ingestion) 큐의 샤딩을 활성화하려면 `LANGFUSE_INGESTION_QUEUE_SHARD_COUNT`를 양의 정수로 설정. Redis 샤드 수의 2~3배를 권장
- 클러스터 노드 전체에 걸쳐 트레이스 업서트 큐의 샤딩을 활성화하려면 `LANGFUSE_TRACE_UPSERT_QUEUE_SHARD_COUNT`를 양의 정수로 설정. Redis 샤드 수의 2~3배를 권장

</Callout>

## Redis Sentinel 모드

Redis Sentinel은 전체 클러스터 모드의 복잡성 없이 Redis 배포에 고가용성을 제공합니다.
Redis 마스터 및 레플리카 인스턴스를 자동으로 모니터링하고, 페일오버를 처리하며, 서비스 디스커버리를 제공합니다.
따라서 여러 샤드에 걸친 수평 확장은 필요하지 않지만 자동 페일오버가 필요한 배포에 적합합니다.

### 설정

Redis Sentinel 모드를 활성화하려면 다음 환경 변수를 설정하세요.

```bash
REDIS_SENTINEL_ENABLED=true
REDIS_SENTINEL_NODES=sentinel1:26379,sentinel2:26379,sentinel3:26379
REDIS_SENTINEL_MASTER_NAME=mymaster
REDIS_AUTH=your-redis-password  # if authentication is enabled on Redis
REDIS_SENTINEL_PASSWORD=your-sentinel-password  # if authentication is enabled on Sentinels (optional)
```

### 예시 설정

#### Sentinel을 사용하는 셀프 호스팅 Redis

```bash
REDIS_SENTINEL_ENABLED=true
REDIS_SENTINEL_NODES=10.0.1.10:26379,10.0.1.11:26379,10.0.1.12:26379
REDIS_SENTINEL_MASTER_NAME=langfuse-master
REDIS_AUTH=your-redis-password
```

<Callout type="warning">

Redis Sentinel 모드를 사용할 때는 다음을 확인하세요.

- 모든 Sentinel 노드가 Langfuse 컨테이너에서 접근 가능한지 확인
- `REDIS_SENTINEL_MASTER_NAME`은 Sentinel 설정에 구성된 마스터 이름과 일치해야 함
- 큐 작업 제거를 방지하기 위해 Redis 마스터 및 레플리카에서 `maxmemory-policy=noeviction` 설정
- 기본적으로 `REDIS_TLS_ENABLED`는 Redis 마스터 및 레플리카 인스턴스에 대한 연결에만 TLS를 적용합니다. Sentinel 노드에 대한 연결도 암호화하려면 `REDIS_SENTINEL_TLS_ENABLED=true`를 설정하세요(아래 참고).

</Callout>

### Sentinel에서의 TLS

`REDIS_TLS_ENABLED=true`인 경우, Langfuse는 Redis 마스터 및 레플리카 인스턴스에 대한 연결에는 TLS를 적용하지만 Sentinel 노드에 대한 연결에는 적용하지 않습니다. Sentinel 노드에도 암호화된 연결이 필요한 경우 `REDIS_SENTINEL_TLS_ENABLED=true`를 설정하세요. 이 경우 Redis 연결을 설정하는 것과 동일한 TLS 옵션(`REDIS_TLS_CA_PATH`, `REDIS_TLS_CERT_PATH`, `REDIS_TLS_KEY_PATH`, `REDIS_TLS_SERVERNAME` 및 그 외 `REDIS_TLS_*` 변수)이 Sentinel 연결에도 재사용됩니다.

```bash
REDIS_SENTINEL_ENABLED=true
REDIS_SENTINEL_NODES=10.0.1.10:26379,10.0.1.11:26379,10.0.1.12:26379
REDIS_SENTINEL_MASTER_NAME=langfuse-master
REDIS_AUTH=your-redis-password
REDIS_TLS_ENABLED=true # required if REDIS_SENTINEL_TLS_ENABLED=true
REDIS_SENTINEL_TLS_ENABLED=true
```

## 크기 산정 권장 사항

Langfuse는 주로 워커가 처리해야 하는 이벤트 메타데이터를 큐잉하는 데 Redis를 사용합니다.
대부분의 경우 워커는 이벤트가 쌓이지 않도록 큐를 빠르게 처리할 수 있습니다.
분당 약 10만 건의 이벤트마다 Redis 인스턴스에 약 1GB의 메모리를 권장합니다.

## Redis 권한

Redis는 사용자가 접근할 수 있는 키와 명령을 제한할 수 있도록 지원합니다. [Redis ACL 문서](https://redis.io/docs/latest/operate/oss_and_stack/management/security/acl/)를 참고하세요.
Langfuse는 제공된 사용자가 지정된 데이터베이스 내의 모든 키와 명령에 접근할 수 있어야 합니다. 즉, 접근 제어는 `on ~* +@all`로 정의되어야 합니다.

## Valkey와 Redis 비교 [#valkey-vs-redis]

[Valkey](https://github.com/valkey-io/valkey)는 Redis의 오픈 소스(BSD) 대안으로 만들어졌습니다.
이는 Redis를 대체할 수 있는 드롭인(drop-in) 대체재이며 Redis 프로토콜과 호환됩니다.
관리자들에 따르면, 메이저 버전 8.0.0은 대부분의 경우 Redis v7과의 호환성을 유지합니다.
두 옵션 모두 [공식적으로 지원](#support)되며, Valkey는 [Kubernetes(Helm)](/self-hosting/deployment/kubernetes-helm) 배포에서 기본값입니다.
Redis와 Valkey 간의 동작 차이를 발견하면 [GitHub 이슈](/issues)를 생성해 주세요.

# 스케일링

이 가이드는 Langfuse 배포를 대규모로 운영하는 방법을 다루며, 최상의 성능을 얻기 위한 모범 사례와 튜닝 방법을 포함합니다.

| ------------------------------------------------------------------------------ | --------------------------------------------------------------- |

## 수집 처리량

Langfuse는 대량의 데이터를 수집하도록 설계되었습니다.
매우 높은 부하가 발생하는 경우, 처리량에 영향을 주는 추가 설정을 적용해야 할 수 있습니다.

### Worker 컨테이너 스케일링

대부분의 환경에서는 CPU 부하를 기준으로 worker 컨테이너를 스케일링하는 것을 권장합니다. 이는 측정하기 쉬운 직관적인 메트릭이기 때문입니다.
2 CPU 컨테이너의 부하가 50%를 초과하면 인스턴스가 포화 상태임을 나타내며, 컨테이너를 추가하여 처리량을 늘려야 합니다.

또한 Langfuse worker는 statsd를 통해 큐 길이 메트릭도 게시하며, 이를 worker 컨테이너 스케일링에 활용할 수 있습니다.
`langfuse.queue.ingestion.length`는 스케일링 결정을 내릴 때 주로 사용하는 핵심 메트릭입니다.
`ENABLE_AWS_CLOUDWATCH_METRIC_PUBLISHING=true`를 설정하면 큐 메트릭을 AWS CloudWatch에도 게시하여 AWS 메트릭 기반 오토스케일러를 구성할 수 있습니다.

### 수집 처리 과정에서 ClickHouse 읽기 줄이기

기본적으로 Langfuse worker는 ClickHouse에서 기존 이벤트를 읽어 들어오는 데이터와 병합합니다.
이로 인해 ClickHouse의 부하가 증가하고 전체 처리량이 제한될 수 있습니다.
이전 버전의 Langfuse에서 마이그레이션되지 않은 프로젝트의 경우, 전체 이벤트 히스토리가 S3에 존재하므로 이 과정은 선택 사항입니다.
`LANGFUSE_SKIP_INGESTION_CLICKHOUSE_READ_MIN_PROJECT_CREATE_DATE`를 첫 프로젝트가 생성되기 이전의 과거 날짜(예: `2025-01-01`)로 설정할 수 있습니다.
S3/blob storage의 삭제 수명 주기 규칙과 이벤트의 지연 업데이트가 결합되면 이벤트 히스토리에 중복이 발생할 수 있으니 유의하세요.
Langfuse SDK 또는 OpenTelemetry의 기본 통합 방식을 사용하는 경우에는 이 문제의 영향을 받지 않습니다.

### 수집과 사용자 인터페이스 분리

수집 부하가 높을 때 Langfuse 웹 인터페이스와 API 호출이 느려지거나 응답하지 않을 수 있습니다.
이런 경우 langfuse-web 배포를 수집 처리용과 사용자 인터페이스 처리용으로 분리하면 사용자 인터페이스의 응답성을 유지하는 데 도움이 됩니다.
langfuse-web 배포와 동일한 새 복제본을 생성하고, `/api/public/ingestion*`, `/api/public/media*`, `/api/public/otel*`로 향하는 모든 트래픽을 새 배포로 라우팅할 수 있습니다.

### S3(Blobstorage) 쓰기 동시성 늘리기

blob storage 백엔드는 원본 이벤트, 멀티모달 입력, 배치 내보내기 및 기타 파일을 저장하는 데 사용됩니다.
매우 높은 처리량 시나리오에서는 S3 클라이언트 라이브러리가 허용하는 소켓 수가 소진되어 요청이 스로틀링될 수 있습니다.
이런 현상이 발생하면 일반적으로 수집 및 미디어 워크로드를 처리하는 web 컨테이너의 메모리 사용량이 증가하는 것을 관찰할 수 있습니다.
이때 표시되는 로그 메시지는 다음과 같습니다: `@smithy/node-http-handler:WARN - socket usage at capacity=150 and 387 additional requests are enqueued.`.

이러한 상황에서는 `LANGFUSE_S3_CONCURRENT_WRITES`를 기본값인 50보다 큰 값으로 설정하여 동시 쓰기 수를 늘리는 것을 권장합니다.
추가되는 쓰기 소켓마다 약간의 메모리 오버헤드가 발생하므로, 값을 점진적으로 늘리며 서비스의 동작을 관찰하는 것을 권장합니다.

## 느린 UI 쿼리 또는 API 호출

UI에서 로딩 화면이 길어지거나 API 호출이 느려지는 경우, 대개 ClickHouse 데이터베이스의 리소스 부족 또는 시간 필터 누락과 관련이 있습니다.
트레이싱 데이터는 projectId와 시간을 기준으로 인덱싱되므로, 이 조건에 대한 필터를 추가하면 성능이 크게 향상됩니다.

모든 필터가 이미 적용되어 있다면, 더 큰 규모의 ClickHouse 인스턴스가 성능을 향상시킬 수 있습니다.
ClickHouse는 수직 확장을 위해 설계되었으므로, 인스턴스에 메모리를 추가하면 응답 시간이 더 빨라집니다.
워크로드에 적합한 메모리 크기를 선택하려면 [ClickHouse 문서](https://clickhouse.com/docs/operations/tips#using-less-than-16gb-of-ram)를 참고하세요.
일반적으로 대규모 배포에는 최소 16GiB의 메모리를 권장합니다.

### 읽기를 별도의 ClickHouse 컴퓨트 그룹으로 라우팅하기 [#clickhouse-read-only-url]

[컴퓨트-컴퓨트 분리](https://clickhouse.com/docs/cloud/reference/warehouses)를 지원하는 배포 환경 — 주로 [ClickHouse Cloud](https://clickhouse.com/cloud) 및 [BYOC](https://clickhouse.com/cloud/bring-your-own-cloud) — 에서는 Langfuse가 전용 읽기 전용 컴퓨트 그룹을 가리키도록 설정하여 대규모 분석 읽기 작업을 쓰기 경로로부터 격리할 수 있습니다.
이를 통해 대시보드 및 공개 API 읽기 트래픽이 프라이머리 컴퓨트에서 수집 삽입 및 백그라운드 병합 작업과 경합하지 않도록 할 수 있습니다.

`CLICKHOUSE_READ_ONLY_URL`을 설정하면 Langfuse는 UI 및 공개 API 읽기 쿼리를 지정된 엔드포인트로 라우팅하며, 쓰기, 마이그레이션, 수집은 계속해서 `CLICKHOUSE_URL`을 사용합니다:

자격 증명(`CLICKHOUSE_USER`, `CLICKHOUSE_PASSWORD`)과 데이터베이스 이름은 프라이머리 설정에서 재사용되므로, 읽기 전용 컴퓨트 그룹은 동일한 사용자를 허용해야 합니다.
두 변수는 모두 선택 사항이며 기본적으로 설정되지 않습니다. 단일 노드 ClickHouse나 컴퓨트가 분리되지 않은 클러스터에서는 엔드포인트가 프라이머리와 동일하므로 이점이 없습니다.

### OTel 프로젝트에서 FINAL 수식어 건너뛰기 [#skip-final-otel-projects]

Langfuse는 관측을 ClickHouse의 `ReplacingMergeTree`에 저장하며, 기본적으로 공개 API의 관측 쿼리에 `FINAL` 수식어를 추가하여 읽기 시점에 각 행의 최신 버전이 반영되도록 합니다.
이는 수집 과정에서 동일한 관측에 대해 여러 버전의 이벤트가 생성되는 경우(예: 레거시 SDK의 별도 start, end, update 이벤트) 필요하지만, 쿼리 시점의 병합 작업을 추가하여 읽기 속도를 저하시킵니다.

**OpenTelemetry만을 통해**(OTel SDK 또는 `/api/public/otel` 엔드포인트) 수집하는 프로젝트는 각 관측을 단일 불변 행으로 기록하므로, 이런 프로젝트에는 `FINAL`이 불필요합니다.
`LANGFUSE_SKIP_FINAL_FOR_OTEL_PROJECTS=true`를 설정하면 프로젝트별 추적 기능이 활성화됩니다. 프로젝트가 OTel 엔드포인트를 통해 수집하면 Langfuse는 이를 Redis에 24시간 TTL로 표시하고, 이후 해당 프로젝트의 관측 읽기(`GET /api/public/observations` 및 `GET /api/public/traces`의 관측 CTE)는 `FINAL` 수식어를 건너뜁니다.
TTL은 OTel 수집 호출마다 갱신되며, 프로젝트가 OTel 트래픽 전송을 중단하면 마커가 자동으로 만료되므로, 프로젝트가 레거시 수집 방식으로 전환하더라도 동작이 안전하게 원상 복구됩니다.

이는 일부 프로젝트만 OTel을 사용하는 혼합 배포 환경에서 이 최적화의 이점을 얻는 데 권장되는 방법입니다.

<Callout type="info">

인스턴스의 **모든** 프로젝트가 OTel만을 통해 수집한다는 것이 확실하다면, `LANGFUSE_API_CLICKHOUSE_DISABLE_OBSERVATIONS_FINAL=true`를 대신 설정하여 Redis 조회 없이 `FINAL` 수식어를 전역적으로 제거할 수 있습니다. 이 플래그는 프로젝트별 로직보다 우선 적용됩니다. 아직 레거시 수집 경로를 사용하는 프로젝트가 하나라도 있다면 이 플래그를 활성화하지 마세요. 그렇지 않으면 관측 읽기가 오래되었거나 중복된 행을 반환할 수 있습니다.

</Callout>

## 디스크 사용량 증가

LLM 트레이싱 데이터는 입력 및 출력이 추적되기 때문에 대용량 페이로드를 포함할 수 있습니다.
또한 ClickHouse는 시스템 테이블 내에 관측성 데이터를 저장합니다.
S3/Blob Storage 또는 ClickHouse에서 디스크 공간이 크게 증가하는 것을 발견했다면 다음을 권장합니다.

일반적으로 디스크 공간을 확보하는 가장 효과적인 방법은 [데이터 보존](/docs/data-retention) 정책을 설정하는 것입니다.
이 기능이 플랜에서 제공되지 않는다면 아래 옵션을 고려하세요.

<Callout type="info">

ClickHouse 디스크 증가가 운영상의 병목이 되는 대규모 배포 환경에서는 [ClickHouse Cloud 또는 BYOC](/self-hosting/deployment/infrastructure/clickhouse#cloudbyoc)를 고려하세요. 이러한 배포는 스토리지와 컴퓨트를 분리하므로, 스토리지는 Langfuse 쿼리를 처리하는 컴퓨트와 독립적으로 확장될 수 있습니다. 보존된 트레이싱 데이터는 여전히 스토리지를 소비하므로 이것이 보존 정책을 대체하지는 않지만, 셀프 관리형 OSS ClickHouse 환경에서 필요한 수동 디스크 계획, 볼륨 확장, 단일 샤드 용량 관리의 상당 부분을 줄여줍니다.

</Callout>

### S3 / Blob Storage 디스크 사용량

수명 주기 규칙을 구현하여 blob storage에서 오래된 파일을 자동으로 제거할 수 있습니다.
UI에서 접근하거나 업데이트하고자 하는 기간만큼 이벤트를 보관하는 것을 권장합니다.
대부분의 고객에게는 기본값인 30일이 적절한 선택입니다.

다만 이는 업로드된 미디어 파일을 저장하는 미디어 버킷에는 적용되지 않습니다. 이 버킷에 보존 정책을 설정하는 것은 다음과 같은 이유로 권장하지 않습니다.

1. 트레이스에서 참조하는 미디어 파일이 깨질 수 있습니다
2. 파일 업로드 상태가 Postgres에서 해시로 추적되므로 동일한 파일의 향후 업로드가 실패할 수 있습니다

대신 [Langfuse 데이터 보존 기능](/docs/data-retention)을 사용하여 미디어 파일을 적절히 관리하고 제품 전반에서 깨진 참조를 방지하는 것을 권장합니다.

### ClickHouse 디스크 사용량

ClickHouse 내의 데이터를 자동으로 제거하려면 [TTL](https://clickhouse.com/docs/guides/developer/ttl) 기능을 사용할 수 있습니다.
구성 방법에 대한 자세한 내용은 ClickHouse 문서를 참고하세요.
이는 ClickHouse 내의 `traces`, `observations`, `scores`, `event_log` 테이블에 적용됩니다.

다음 쿼리는 ClickHouse에서 가장 큰 테이블을 식별하는 데 도움이 됩니다:

```sql
SELECT table, formatReadableSize(size) as size, rows FROM (
    SELECT
        table,
        database,
        sum(bytes) AS size,
        sum(rows) AS rows
    FROM system.parts
    WHERE active
    GROUP BY table, database
    ORDER BY size DESC
)
```

#### 삭제가 느리거나 타임아웃되는 경우 [#clickhouse-deletion-timeouts]

데이터 보존 작업이나 프로젝트/트레이스 삭제가 타임아웃 오류로 실패한다면, 삭제 작업에 대한 클라이언트 측 ClickHouse HTTP 타임아웃이 소진되었을 가능성이 높습니다.
장기 실행 삭제 작업에 여유를 더 주려면 `LANGFUSE_CLICKHOUSE_DELETION_TIMEOUT_MS`(기본값 `600000`, 즉 10분)를 늘리세요.
이는 예약된 보존 작업과 트레이스, 관측, 점수, 데이터셋 실행 항목, 이벤트에 걸친 사용자 트리거 삭제 모두에 적용됩니다.

ClickHouse 25.7 이상에서는 경량 삭제 및 업데이트를 사용하여 뮤테이션 부하를 추가로 줄일 수 있습니다.
`CLICKHOUSE_LIGHTWEIGHT_DELETE_MODE`를 기본값인 `alter_update`에서 `lightweight_update`(또는 `lightweight_update_force`)로 설정하면 `DELETE` 문이 `ALTER TABLE ... DELETE` 뮤테이션 대신 경량 삭제를 통해 처리됩니다.
또한 `CLICKHOUSE_USE_LIGHTWEIGHT_UPDATE=true`를 설정하면 트레이싱 테이블 업데이트가 `ALTER TABLE ... UPDATE` 뮤테이션 대신 네이티브 `UPDATE` 문을 통해 처리됩니다.
두 설정 모두 삭제가 많은 워크로드에서 ClickHouse가 수행하는 백그라운드 병합 작업량을 줄여줍니다.

#### ClickHouse 시스템 로그 테이블 [#clickhouse-system-log-tables]

기본 ClickHouse 설정에서는 시스템 로그 테이블(`trace_log`, `text_log`, `opentelemetry_span_log`, `asynchronous_metric_log`, `metric_log`, `latency_log`)이 디스크 사용량을 지배할 수 있습니다.
이 테이블들은 기본적으로 TTL이 없으며, 쿼리 프로파일러는 `system.trace_log`에 지속적으로 기록합니다.
Langfuse는 이 테이블들을 읽지 않으므로 안전하게 줄일 수 있습니다. 두 가지 옵션이 있습니다.

**옵션 1 — 사용하지 않는 시스템 로그 테이블 비활성화.** `/etc/clickhouse-server/config.d/`에 파일을 마운트하여 Langfuse가 절대 읽지 않는 테이블을 제외하세요. 이는 [Langfuse Terraform AWS 모듈](https://github.com/langfuse/langfuse-terraform-aws/pull/26)에서 기본적으로 채택하고 있는 방식입니다:

```xml
<clickhouse>
    <trace_log remove="1"/>
    <text_log remove="1"/>
    <opentelemetry_span_log remove="1"/>
    <asynchronous_metric_log remove="1"/>
    <metric_log remove="1"/>
    <latency_log remove="1"/>
</clickhouse>
```

Bitnami ClickHouse Helm 차트에서는 `clickhouse.extraOverrides`를 통해 동일한 블록을 전달할 수 있습니다. `query_log`, `part_log`, `error_log`는 디버깅에 유용하고 크기도 작으므로 계속 활성화 상태로 두세요.

**옵션 2 — 공격적인 TTL 적용.** 디버깅을 위해 테이블을 계속 유지하고 싶다면, 짧은 TTL을 적용하고 대신 쿼리 프로파일러를 비활성화하세요:

```xml
<clickhouse>
    <profiles>
        <default>
            <query_profiler_real_time_period_ns>0</query_profiler_real_time_period_ns>
            <query_profiler_cpu_time_period_ns>0</query_profiler_cpu_time_period_ns>
        </default>
    </profiles>
    <trace_log>
        <engine>ENGINE = MergeTree PARTITION BY toYYYYMM(event_date) ORDER BY (event_date, event_time) TTL event_date + INTERVAL 7 DAY</engine>
    </trace_log>
    <opentelemetry_span_log>
        <engine>ENGINE = MergeTree PARTITION BY toYYYYMM(finish_date) ORDER BY (finish_date, finish_time_us) TTL finish_date + INTERVAL 7 DAY</engine>
    </opentelemetry_span_log>
    <query_log>
        <engine>ENGINE = MergeTree PARTITION BY toYYYYMM(event_date) ORDER BY (event_date, event_time) TTL event_date + INTERVAL 30 DAY</engine>
    </query_log>
</clickhouse>
```

<Callout type="info">

TTL 지시자는 시스템 테이블이 처음 생성될 때만 적용됩니다. 기존 설치 환경에서는 설정을 배포한 후 테이블에도 이를 소급 적용해야 합니다 — 예를 들어 `system.trace_log`의 경우:

```sql
SET max_table_size_to_drop = 0;
TRUNCATE TABLE system.trace_log;
ALTER TABLE system.trace_log MODIFY TTL event_date + INTERVAL 7 DAY;
```

원하는 테이블마다 이 과정을 반복하세요. 관련 논의는 업스트림 [langfuse/langfuse#13123](https://github.com/langfuse/langfuse/issues/13123)에서 확인할 수 있습니다.

</Callout>

## 높은 Redis CPU 부하

Redis 엔진 CPU 사용률이 높게(90% 이상) 관찰된다면 다음 사항을 확인하는 것을 권장합니다.

- 최소 4개의 CPU를 가진 인스턴스를 사용하세요. 이렇게 하면 Redis가 네트워킹과 백그라운드 작업을 별도의 CPU에 스케줄링할 수 있습니다.
- [Redis 클러스터 모드](/self-hosting/deployment/infrastructure/cache#redis-cluster-mode)가 활성화되어 있는지 확인하세요.

높은 CPU 사용률이 지속된다면 Langfuse가 사용하는 큐를 여러 노드에 걸쳐 샤딩할 수 있습니다.
샤딩을 활성화하려면 `LANGFUSE_INGESTION_QUEUE_SHARD_COUNT`와 `LANGFUSE_TRACE_UPSERT_QUEUE_SHARD_COUNT`를 1보다 큰 값으로 설정하세요.
각 큐-샤드는 Redis 내의 임의의 슬롯에 할당되므로, 노드 간 균등한 분산을 보장하려면 Redis 클러스터가 보유한 샤드 수의 약 2~3배에 해당하는 값을 권장합니다
(자세한 내용은 [Redis Cluster](https://redis.io/docs/latest/operate/oss_and_stack/management/scaling/) 문서를 참고하세요).

<Callout type="warning">

큐 샤딩은 고급 기능이며, 높은 Redis CPU 부하가 있고 위의 권장 사항을 이미 따른 경우에만 사용해야 합니다.
큐를 샤딩한 후에는 샤드 수를 _줄이지_ 마세요.
`LANGFUSE_INGESTION_QUEUE_PROCESSING_CONCURRENCY`와 `LANGFUSE_TRACE_UPSERT_WORKER_CONCURRENCY`는 _샤드당_ 값으로 집계되므로 이에 맞게 함께 조정해야 합니다.
기본적으로 worker당 동시성 목표는 20입니다. 즉, 큐-샤드가 10개라면 이 값을 2로 설정하세요.

</Callout>

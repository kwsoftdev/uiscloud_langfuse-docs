# 커스텀 베이스 경로

기본적으로 Langfuse는 도메인의 루트 경로(예 `https://langfuse.yourdomain.com`, `https://yourdomain.com`)에서 서비스됩니다.

기존 인프라에 Langfuse를 통합할 때처럼, 경우에 따라 Langfuse를 커스텀 베이스 경로(예 `https://yourdomain.com/langfuse`)에 배포하고 싶을 수 있습니다.

## 설정

<Callout type="warning">
  베이스 경로는 정적 자산(static assets)에 인라인으로 포함되므로, **web
  컨테이너**(langfuse/langfuse)에는 미리 빌드된 Docker 이미지를 사용할 수
  없습니다. 빌드 시점에 `NEXT_PUBLIC_BASE_PATH` 환경 변수를 설정하여 **소스에서
  이미지를 직접 빌드해야 합니다.** worker 컨테이너(langfuse/langfuse-worker)는
  미리 빌드된 이미지를 그대로 사용할 수 있습니다.
</Callout>

<Steps>

### 환경 변수 업데이트

커스텀 베이스 경로를 사용할 때는 `NEXTAUTH_URL`을 베이스 경로와 `/api/auth`를 포함한 전체 URL로 설정해야 합니다. 예를 들어 Langfuse를 `https://yourdomain.com/langfuse-base-path`에 배포하는 경우 다음과 같이 설정해야 합니다.

```bash filename=".env"
NEXT_PUBLIC_BASE_PATH="/langfuse-base-path"
NEXTAUTH_URL="https://yourdomain.com/langfuse-base-path/api/auth"
```

### 소스에서 Langfuse Web 이미지 빌드

`NEXT_PUBLIC_BASE_PATH`를 빌드 인자로 사용하여 Langfuse Web 컨테이너(`langfuse/langfuse`) 이미지를 소스에서 빌드합니다.

```bash /NEXT_PUBLIC_BASE_PATH/
# 저장소 클론
git clone https://github.com/langfuse/langfuse.git
cd langfuse

# 프로덕션 브랜치로 체크아웃
# main 브랜치에는 아직 릴리스되지 않아 불안정할 수 있는 변경 사항이 포함되어 있습니다
git checkout production

# NEXT_PUBLIC_BASE_PATH로 이미지 빌드
docker build -t langfuse/langfuse --build-arg NEXT_PUBLIC_BASE_PATH=/langfuse-base-path -f ./web/Dockerfile .
```

### Langfuse 실행

배포 가이드 중 하나에 따라 Langfuse를 배포할 때, web 컨테이너(`langfuse/langfuse`)의 미리 빌드된 이미지를 소스에서 빌드한 이미지로 교체합니다.

<Callout type="warning">
  **Kubernetes/Helm 배포:** Kubernetes/Helm과 함께 커스텀 베이스 경로를 사용할
  때는 liveness 및 readiness probe 경로를 커스텀 베이스 경로를 포함하도록
  업데이트해야 합니다. Helm `values.yaml` 파일의
  `langfuse.web.livenessProbe.path` 및 `langfuse.web.readinessProbe.path`
  항목을 업데이트하세요.
</Callout>

### Langfuse에 연결

Langfuse 인스턴스가 실행되면, 설정한 커스텀 베이스 경로를 통해 API와 콘솔 모두에 접근할 수 있습니다. SDK로 연결할 때는 호스트명에 커스텀 베이스 경로를 포함해야 합니다.

</Steps>
